PostgreSQL with Python

SQL lives inside applications most of the time. This module wires PostgreSQL into Python — connecting, reading, writing, and the safety rails (parameters, transactions) that separate scripts that work from scripts that survive contact with real data.
Estimated time: 75–100 minutes · Prerequisites: Modules 1–8 complete. Comfort with SELECT, INSERT, UPDATE. Basic Python (functions, dicts, lists, with-statements).

What this module covers

SQL is the query language; Python is the orchestration language. Most professional work uses both — a Python script reads from the database, transforms the data, and writes results back. Web apps, dashboards, ETL jobs, machine-learning pipelines, scheduled reports — every one of them sits a Python program in front of PostgreSQL.

  • Connecting — opening a session from Python to PostgreSQL with psycopg, the canonical driver.
  • Reading — running SELECT queries from Python, parameterizing safely, pulling rows into Python data structures (and into pandas when you want a DataFrame).
  • Writing — INSERTs, UPDATEs, bulk loads, and the transaction discipline that keeps half-finished writes from corrupting your data.

The SQL funnel returns

Module 8 was outside the SELECT funnel — file I/O is administrative. Module 9 is back inside it. Every SELECT you run from Python is still the same funnel; the difference is who types the query (a Python string instead of a psql prompt) and where the rows go (Python objects instead of a terminal).

The Python wrapper is incidental. The SQL underneath obeys the same rules as the last eight modules.

Lesson 9.1 — Connecting Python to PostgreSQL

The handshake. Five lines of Python that open a session, run a query, and close cleanly.

Why this matters

Every Python-to-database story starts with a connection. Get the connection pattern right once and the rest of the module is small variations. Get it wrong — leak connections, forget to close, hard-code passwords — and your script works on day one and breaks by week three.

The shape of a session

Five steps, in order:

  1. Import the driver — import psycopg
  2. Open a connection — psycopg.connect(...) with a connection string
  3. Get a cursor — the object that actually runs queries
  4. Execute — pass SQL (and parameters) to the cursor
  5. Close — release the connection back to the OS

Modern Python expresses steps 2–5 with with statements so closing is automatic, even if an exception is thrown halfway.

The canonical driver: psycopg

PostgreSQL's official Python driver is psycopg. Two versions are in common use:

  • psycopg2 — the long-time standard. Still widely deployed; stable, ubiquitous.
  • psycopg3 — the modern rewrite. Better async support, cleaner API, drop-in for most cases. Import as import psycopg (note: no 2 or 3 in the name).

This module uses psycopg3 syntax. If you're on psycopg2, replace import psycopg with import psycopg2 and the rest works almost identically — the few real differences are called out where they matter.

Install:

pip install "psycopg[binary]"

The [binary] extra avoids needing PostgreSQL development headers on your machine.

Your first connection

import psycopg

with psycopg.connect(
    "host=localhost dbname=dvdrental user=postgres password=secret"
) as conn:
    with conn.cursor() as cur:
        cur.execute("SELECT COUNT(*) FROM film;")
        (count,) = cur.fetchone()
        print(f"film table has {count} rows")

Reading top to bottom:

  • psycopg.connect(...) takes a connection string (libpq-style key=value pairs). It returns a connection object.
  • The outer with ensures the connection is committed-and-closed (or rolled-back-and-closed on exception) when the block ends.
  • conn.cursor() gives a cursor. The inner with closes the cursor when the block ends.
  • cur.execute(sql) sends the query. The SQL string is exactly what you'd type into psql.
  • cur.fetchone() returns the next row as a tuple — here, a single-element tuple holding the count. (count,) = ... unpacks it.

Connection strings vs keyword arguments

Both work. Pick the one your team uses.

# Style A — libpq connection string
psycopg.connect("host=localhost dbname=dvdrental user=postgres password=secret")

# Style B — keyword arguments
psycopg.connect(
    host="localhost",
    dbname="dvdrental",
    user="postgres",
    password="secret",
)

# Style C — URI (libpq supports this too)
psycopg.connect("postgresql://postgres:secret@localhost/dvdrental")

Never hard-code credentials

The examples above show a password in plain text for clarity. Never do that in real code. Read credentials from environment variables, a secrets manager, or a .env file (with python-dotenv):

import os
import psycopg

with psycopg.connect(
    host=os.environ["PGHOST"],
    dbname=os.environ["PGDATABASE"],
    user=os.environ["PGUSER"],
    password=os.environ["PGPASSWORD"],
) as conn:
    ...

Set the environment variables in your shell, your CI system, or your container orchestrator — never in the source file you commit.

Connection pools

Opening a connection is expensive: a TCP handshake, authentication, server-side process setup. A web server that opens a fresh connection per request will buckle under load.

The fix is a connection pool — a small set of connections kept open and handed out as needed:

from psycopg_pool import ConnectionPool

pool = ConnectionPool(
    "host=localhost dbname=dvdrental user=postgres password=secret",
    min_size=2,
    max_size=10,
)

with pool.connection() as conn:
    with conn.cursor() as cur:
        cur.execute("SELECT COUNT(*) FROM film;")
        print(cur.fetchone())

For scripts that run a few queries and exit, a single connection is fine. For long-running services, use a pool.

Common mistakes

Forgetting to close

conn = psycopg.connect(...)
cur = conn.cursor()
cur.execute("SELECT * FROM film")
# script ends; connection never closed

The connection lingers on the server until the OS notices the dropped TCP socket. A handful is harmless; a thousand and PostgreSQL refuses new sessions. Always use with blocks or try/finally to guarantee cleanup.

Hard-coded passwords in source

A password in source code ends up in git, in CI logs, in screen-shares. Treat it like a key — read it from the environment.

Reusing one cursor everywhere

Cursors are cheap. Open a new one per query (or per logical operation), close it when done. A cursor left open holds a server-side result set; many idle cursors are a memory leak.

Confusing psycopg2 and psycopg3

The imports differ (import psycopg2 vs import psycopg). The cursor() API is mostly identical, but a few behaviors changed — psycopg2.connect(...) does not commit on with exit (it just closes), while psycopg3's with conn: does commit. Read the docs for the version you're on.

The loop opens 10,000 connections and never closes them. PostgreSQL exhausts its max_connections (default 100) almost immediately, and after that every connect() call hangs or errors. Inside the loop, open a fresh cursor per iteration if you like; the connection stays put.

Lesson 9.2 — Reading data into Python

Running SELECTs from Python, parameterizing them safely, and pulling rows into Python data structures.

Why this matters

Most Python-DB code is read-heavy: a script pulls rows, processes them, prints a report or feeds a dashboard. Two skills make this category of work clean:

  1. Parameterized queries — the only safe way to inject user input into SQL. Skip this and you have a SQL injection vulnerability.
  2. The right fetch method — knowing when to use fetchone, fetchall, fetchmany, or pandas. Choosing badly is how a script that worked on test data crashes on production data.

Parameterized queries

You will be tempted to build SQL strings with f-strings or .format():

# WRONG. Never do this.
name = "Toy Story"
cur.execute(f"SELECT * FROM film WHERE title = '{name}';")

Two things break. First, special characters in name (apostrophes, semicolons) corrupt the SQL or open injection holes. Second, you've reinvented the quoting logic the driver already does for you. The right form passes parameters separately:

name = "Toy Story"
cur.execute("SELECT * FROM film WHERE title = %s;", (name,))

The %s is a placeholder (regardless of the parameter's actual type — it's not Python % formatting). The driver substitutes safely on the way to PostgreSQL.

Named parameters work too, which read better with several:

cur.execute(
    "SELECT * FROM film WHERE rating = %(rating)s AND length > %(min_length)s;",
    {"rating": "G", "min_length": 90},
)

This is the most important habit in this module. Memorize the pattern; never break it.

SQL injection in 30 seconds

If you build SQL with string concatenation and a user controls part of the string, they can append their own SQL. The classic:

# user_input came from a web form
user_input = "'; DROP TABLE film; --"
cur.execute(f"SELECT * FROM film WHERE title = '{user_input}';")
# Resulting SQL:
#   SELECT * FROM film WHERE title = ''; DROP TABLE film; --';

The semicolon ends your SELECT; DROP runs next; the -- comments out the trailing quote. With parameters, the same input is treated as a string — the entire thing is quoted and escaped, and the database looks for a film literally titled '; DROP TABLE film; -- (and finds none).

Fetching: one, many, or all

After cur.execute(...), the result set sits on the cursor. Pick a fetch method based on size:

cur.execute("SELECT film_id, title FROM film LIMIT 3;")

cur.fetchone()       # one row as a tuple, or None if exhausted
cur.fetchmany(50)    # next 50 rows as a list of tuples
cur.fetchall()       # all remaining rows as a list of tuples

Cursors are iterable, which is often the most Pythonic option:

cur.execute("SELECT film_id, title FROM film WHERE rating = %s;", ("G",))
for film_id, title in cur:
    print(film_id, title)

Iterating streams rows one at a time; you never hold the whole result set in Python memory.

Rule of thumb on fetch size

  • fetchone() — when you expect exactly one row (a COUNT, a SUM, a lookup by primary key).
  • iterate (for row in cur:) — when the result set is large or you don't know its size. Memory-safe by default.
  • fetchall() — when the result set is known to be small (under a few thousand rows) and you need it in a list to do something non-streaming with it (e.g., pass to pandas, JSON-encode).
  • fetchmany(n) — batched processing, useful when you want to chunk work but not stream one-by-one.

Calling fetchall() on a million-row table is how scripts crash with out-of-memory errors. Default to iterating; reach for fetchall consciously.

Row factories — getting dicts instead of tuples

Tuples are fast but positional — row[2] is fragile if the column order changes. psycopg3 has row factories that return dicts or named tuples:

import psycopg
from psycopg.rows import dict_row

with psycopg.connect("...") as conn:
    with conn.cursor(row_factory=dict_row) as cur:
        cur.execute("SELECT film_id, title, length FROM film LIMIT 3;")
        for row in cur:
            print(row["title"], row["length"])

For ad-hoc scripts, tuples are fine. For anything that lives more than a week, dicts read better and survive column reordering.

pandas: read_sql for the analytics path

When the next step is a DataFrame, skip the cursor and use pandas:

import pandas as pd
import psycopg

with psycopg.connect("...") as conn:
    df = pd.read_sql(
        "SELECT film_id, title, length, rating FROM film WHERE rating = %(r)s;",
        conn,
        params={"r": "G"},
    )

print(df.head())
print(df["length"].describe())

Common mistakes

String-formatting SQL

The single most common — and most dangerous — mistake in Python-DB code. Always use parameters. No exceptions for "trusted" input; trust changes over time, parameters don't.

fetchall() on huge result sets

A query that returns 10 million rows pulls 10 million tuples into Python memory before your code sees the first one. Iterate the cursor instead, or add LIMIT and paginate.

Reusing a cursor mid-iteration

If you start iterating a cursor and then call cur.execute(...) with a new query, the unread rows of the first query are discarded. Finish reading (or close the cursor) before reusing it. Better: one cursor per query.

Treating result tuples as immutable schema

row[3] is the fourth column today. Someone reorders the SELECT list tomorrow and row[3] is suddenly the wrong column — silently. Either spell columns out (SELECT film_id, title, length, rating) and unpack into named variables (for fid, title, length, rating in cur:) or use a dict row factory.

Lesson 9.3 — Writing data from Python

INSERT, UPDATE, transactions, and the bulk-load patterns that keep big writes fast and recoverable.

Why this matters

Reads can fail and you re-run them; nothing is broken. Writes are different — a half-finished write leaves the database in an inconsistent state. The discipline that separates production-grade write code from a script is transactions: a write either fully succeeds or fully rolls back, never half-applied.

Single-row INSERT

Same parameter pattern as SELECT — never f-string the values in:

import psycopg

with psycopg.connect("host=localhost dbname=dvdrental user=postgres password=secret") as conn:
    with conn.cursor() as cur:
        cur.execute(
            "INSERT INTO category (name) VALUES (%s) RETURNING category_id;",
            ("Educational",),
        )
        new_id = cur.fetchone()[0]
        print(f"inserted with id {new_id}")

Two things to notice:

  • The %s placeholder works for any column type. The driver formats correctly.
  • RETURNING category_id is a PostgreSQL feature — INSERT can return columns of the new row. Useful for grabbing auto-generated IDs without a second query.

Transactions and commit

By default, psycopg3 opens an implicit transaction at the first statement. Nothing is persisted until you commit:

with psycopg.connect("...") as conn:
    with conn.cursor() as cur:
        cur.execute("INSERT INTO category (name) VALUES (%s);", ("Documentary B",))
        # nothing written yet
    # exiting the `with conn` block commits if no exception was raised

The outer with conn: is your commit boundary. Exit cleanly → commit. Exception inside → rollback. That's the whole pattern, and it's why the with style is worth learning.

If you need finer control:

with psycopg.connect("...") as conn:
    with conn.cursor() as cur:
        try:
            cur.execute("UPDATE rental SET return_date = NOW() WHERE rental_id = %s;", (42,))
            cur.execute("UPDATE film SET rental_rate = rental_rate + 1 WHERE film_id = %s;", (1,))
            conn.commit()
        except Exception:
            conn.rollback()
            raise

Either both UPDATEs succeed (commit), or neither does (rollback). PostgreSQL guarantees atomicity within a transaction — that's the A in ACID.

executemany — many rows, one round trip

A loop of single-row INSERTs hits the database with one round trip per row:

# Slow. One network round trip per row.
for row in rows:
    cur.execute("INSERT INTO category (name) VALUES (%s);", (row,))

executemany batches the parameter list into one call:

rows = [("Sci-Fi B",), ("Romance B",), ("Drama B",)]
cur.executemany("INSERT INTO category (name) VALUES (%s);", rows)

Faster than a loop, and the natural form when your data already lives in a Python list.

COPY for bulk loads

executemany is fine for hundreds or low thousands of rows. For tens of thousands or more, PostgreSQL's COPY is dramatically faster — it bypasses the per-row parsing and writes through a streaming protocol. psycopg3 exposes it directly:

import psycopg

rows = [
    (1, "Apple", 1.00, "fruit"),
    (2, "Banana", 0.50, "fruit"),
    (3, "Cherry", 3.00, "fruit"),
]

with psycopg.connect("...") as conn:
    with conn.cursor() as cur:
        with cur.copy("COPY product (id, name, price, category) FROM STDIN;") as copy:
            for row in rows:
                copy.write_row(row)

For a real CSV file, even simpler — stream the file into COPY:

with open("products.csv", "rb") as f:
    with conn.cursor() as cur:
        with cur.copy("COPY product FROM STDIN WITH (FORMAT csv, HEADER true);") as copy:
            while data := f.read(8192):
                copy.write(data)

Same COPY you learned in Module 8 — same gains in speed — driven from Python.

The end-to-end pipeline

Putting the module together: read a CSV with pandas, clean it, write the cleaned rows back into the database with executemany.

import pandas as pd
import psycopg

# 1. Read
df = pd.read_csv("incoming_products.csv")

# 2. Clean (the patterns from Module 8, but in pandas)
df["name"] = df["name"].str.strip()
df["category"] = df["category"].str.lower()
df["price"] = pd.to_numeric(df["price"], errors="coerce")
df = df.dropna(subset=["price"])
df = df.drop_duplicates(subset=["name"], keep="first")

# 3. Write
rows = list(df.itertuples(index=False, name=None))

with psycopg.connect("host=localhost dbname=dvdrental user=postgres password=secret") as conn:
    with conn.cursor() as cur:
        cur.executemany(
            "INSERT INTO product (id, name, price, category) VALUES (%s, %s, %s, %s);",
            rows,
        )
    # commit on clean exit

That's the shape of most Python-DB jobs. Read → transform → write, with parameters on the way in and a transaction wrapping the write.

Common mistakes

Forgetting to commit

In raw psycopg (without the outer with conn:), writes are not persisted until you call conn.commit(). A script that runs INSERTs and exits without commit leaves the database unchanged — and nothing in the logs warns you. Prefer the with conn: pattern and let the context manager commit on exit.

One huge transaction

Wrapping a million-row load in a single transaction holds locks the entire time and bloats PostgreSQL's WAL. Either chunk the load (commit every 10,000 rows) or use COPY, which is optimized for this case.

Rolling back too late

try/except around conn.commit() only catches errors during commit itself. If an INSERT inside the block raises and you don't rollback, the connection is left in an aborted state and every subsequent statement errors with "current transaction is aborted". Either rollback in the except, or use with conn: which does it for you.

Bulk INSERT via string concatenation

# WRONG
sql = "INSERT INTO product VALUES " + ",".join(
    f"({row[0]}, '{row[1]}', {row[2]})" for row in rows
)
cur.execute(sql)

You've reinvented executemany, badly, with a SQL injection hole and a string-length limit. Use executemany or COPY.

Mixing reads and writes in one transaction without thinking

A transaction that reads a row, computes something in Python, then writes back is vulnerable to races: another transaction may have updated the row between your read and write. For high-contention paths, learn SELECT ... FOR UPDATE or design the write to be atomic in SQL (UPDATE … SET x = x + 1 rather than read-modify-write).

try:
    with psycopg.connect("host=localhost dbname=dvdrental user=postgres password=secret") as conn:
        with conn.cursor() as cur:
            cur.execute("INSERT INTO category (category_id, name) VALUES (%s, %s);", (999, "Probe A"))
            cur.execute("INSERT INTO category (category_id, name) VALUES (%s, %s);", (999, "Probe B"))  # duplicate PK
except psycopg.errors.UniqueViolation:
    print("rolled back due to duplicate key")

# Verify neither row landed
with psycopg.connect("host=localhost dbname=dvdrental user=postgres password=secret") as conn:
    with conn.cursor() as cur:
        cur.execute("SELECT COUNT(*) FROM category WHERE category_id = 999;")
        print("rows with id 999:", cur.fetchone()[0])  # expect 0

The with conn: block raises through, which triggers rollback before the connection closes. Both INSERTs vanish — that's the atomicity guarantee in action.

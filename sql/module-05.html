Advanced SQL

The next layer of tools. Functions for the values inside cells, and the ability to nest queries inside queries. After this module you can write almost any real-world query.
Estimated time: 100–130 minutes · Prerequisites: Modules 1–4 complete. Comfort with JOINs and GROUP BY.

What this module covers

So far we have learned clauses — SELECT, WHERE, GROUP BY, JOIN. Each clause shaped which rows and columns we got back. This module is different: it teaches functions — tools that transform the values inside the cells, plus the structural trick of putting whole queries inside other queries.

  • Date and time functions — extract parts of a date, do date arithmetic, format for display
  • Math and string functions — round numbers, concatenate strings, manipulate text
  • Subqueries — use the result of one query as input to another
  • Self-joins — join a table to itself to compare rows within it

The SQL funnel — zooming in, not down

Everything in this module fits into clauses you already know. Instead of moving down the funnel, we are zooming into the cells the funnel produces:

A function in the SELECT clause transforms a column. A function in WHERE makes a smarter filter. A subquery in WHERE replaces a hardcoded list with a computed one. The funnel is the same — you just have richer tools at each layer.

Dialect note for this module

This module hits dialect differences harder than any other. PostgreSQL is the book's target dialect, but the in-page playground runs SQLite — and they diverge most sharply on date functions and a few string functions.

Where this matters, examples will be paired: the PostgreSQL syntax (the "real answer" for your career), and the SQLite version that runs in the playground. Both are useful; they just call the same idea by different names.

Lesson 5.1 — Dates and timestamps

Almost every business question has a time component. SQL has rich tools for both.

The four time-related types

Type Holds Example
DATE Calendar date, no time 2024-05-30
TIME Time of day, no date 14:32:00
TIMESTAMP Date + time 2024-05-30 14:32:00
INTERVAL A duration 7 days, 2 hours 30 minutes

Our payment table stores payment_date as a TIMESTAMP. Most date functions accept either a DATE or a TIMESTAMP.

Getting the current moment

PostgreSQL also accepts NOW() as a synonym for CURRENT_TIMESTAMP. Use whichever reads more naturally.

Extracting parts of a date — PostgreSQL way

PostgreSQL uses EXTRACT(field FROM source) to pull one part out of a date:

SELECT EXTRACT(YEAR  FROM payment_date) AS year,
       EXTRACT(MONTH FROM payment_date) AS month,
       EXTRACT(DOW   FROM payment_date) AS day_of_week
FROM payment;

Valid fields include YEAR, MONTH, DAY, HOUR, MINUTE, DOW (day of week, 0=Sunday), DOY (day of year), QUARTER, WEEK.

Extracting parts of a date — SQLite way (works in this playground)

SQLite uses strftime(format, source) — a single function with a format string borrowed from C:

strftime — SQLite's EXTRACT equivalent
SQLite · dvdrental schema

Common format codes: %Y (year), %m (month), %d (day), %H (hour), %M (minute), %w (day of week 0–6), %j (day of year).

Why two ways to do this?

PostgreSQL gets EXTRACT from the SQL standard. SQLite chose strftime to keep its implementation tiny (it has no real date type — dates are just text strings under the hood). Both produce the same answers. When writing for a specific database, use that database's idiom.

The most common reason to use date functions: filtering. "Show me all payments from May":

Date arithmetic — PostgreSQL way

PostgreSQL has INTERVAL as a first-class type, so you can add it to a date:

SELECT payment_date,
       payment_date + INTERVAL '7 days'  AS one_week_later,
       payment_date - INTERVAL '1 month' AS one_month_earlier
FROM payment;

Formatting dates for display — TO_CHAR

For a report, you often want dates in a friendly format like "May 30, 2024" rather than 2024-05-30. PostgreSQL has TO_CHAR:

SELECT TO_CHAR(payment_date, 'Month DD, YYYY') AS pretty_date
FROM payment;

Common mistakes

Comparing a TIMESTAMP to a DATE without thinking

payment_date = '2024-05-30' may not match a payment recorded at 2024-05-30 14:32:00 if the database treats the right side as 2024-05-30 00:00:00 and does exact equality. Use a range instead: payment_date >= '2024-05-30' AND payment_date < '2024-05-31'.

Forgetting time zones

The moment a TIMESTAMP column gets a value, the time zone matters. PostgreSQL has TIMESTAMP WITH TIME ZONE for this. SQLite has no time zone support at all — it treats every value as a local string. In production code, always store in UTC and convert at display time.

Using EXTRACT inside an index-bound WHERE clause

WHERE EXTRACT(YEAR FROM payment_date) = 2024 forces the database to compute EXTRACT for every row before filtering — it cannot use an index on payment_date. WHERE payment_date >= '2024-01-01' AND payment_date < '2025-01-01' is logically equivalent and uses the index.

Lesson 5.2 — Math and string functions

Two grab-bags of tools you will reach for constantly. Learn them in one sitting.

Math operators

The four basics work as you expect: +, -, *, /. Plus modulo:

Operators
SQLite · dvdrental schema

The integer division trap

In most SQL databases, integer / integer = integer. So 10 / 3 is 3, not 3.33.... To get a floating-point result, cast one side:

SELECT 10 * 1.0 / 3;        -- portable: forces float by multiplying by 1.0
SELECT CAST(10 AS REAL) / 3; -- explicit cast

This bites everyone the first time. When dividing for a ratio or percentage, always promote to float first.

FLOOR(x) Largest integer ≤ x

RANDOM() A random number (range varies by dialect)

Common mistakes

Integer division for percentages

Covered above. Cast or multiply by 1.0.

Off-by-one in SUBSTR

SUBSTR positions are 1-based in PostgreSQL and SQLite (but 0-based in some other languages). SUBSTR('hello', 1, 3) returns 'hel', not 'ell'. Read the docs of your dialect when in doubt.

NULL propagation in ||

first_name || ' ' || NULL is NULL. If any concatenated part is NULL, the whole result is NULL. Use CONCAT() (treats NULL as empty) or COALESCE(col, '') to guard each side.

Using LIKE when a function would be clearer

To find rows where the title starts with a specific prefix:

WHERE title LIKE 'Ali%'         -- portable, uses index
WHERE SUBSTR(title, 1, 3) = 'Ali' -- equivalent, but cannot use index

Prefer LIKE for prefix matching — it's idiomatic AND fast.

Lesson 5.3 — Subqueries

A query inside a query. The biggest conceptual leap since GROUP BY.

The idea

A subquery (or "inner query" or "nested query") is a SELECT that lives inside another SELECT. The inner one runs first; its result is used by the outer one.

You have actually been avoiding subqueries up to now. Often a JOIN can replace a subquery and vice versa. Both are valid. Sometimes one reads more naturally than the other; sometimes only one is possible.

The simplest case:

Films longer than the average length
SQLite · dvdrental schema

Read inside-out. First, SELECT AVG(length) FROM film runs — it returns one number (about 96). Then the outer query becomes WHERE length > 96. You did not have to hardcode 96; the database computed it.

This is impossible to write without a subquery. You cannot put an aggregate directly in WHERE (Module 3.3). A subquery is the workaround.

Three shapes of subquery

Shape 1 — Scalar (returns one value)

The subquery returns exactly one row, one column — a single value. Usable anywhere a value is expected: in SELECT, in WHERE, in HAVING.

Each film + the global avg, side by side
SQLite · dvdrental schema

The same average is added to every row. Useful for showing each value alongside a benchmark.

Shape 2 — Row-set (returns many rows, one column)

The subquery returns a list of values. Use it with IN, NOT IN, ANY, or ALL:

Customers who have made at least one payment
SQLite · dvdrental schema

The inner query returns the list of customer_ids that appear in payment. The outer query keeps only customers whose id is in that list.

The negation:

Customers who have never paid (subquery version)
SQLite · dvdrental schema

Same answer as the anti-join from Module 4.2. Both work. NOT IN can be slow on big tables (it has to dedupe the inner list) and is dangerous with NULLs (Module 2.1) — many people prefer the LEFT JOIN version. Know both.

Shape 3 — Derived table (subquery in FROM)

The subquery returns a whole table-shaped result. Wrap it in parens, alias it, and use it like a table:

Each rating + its film count, then filter
SQLite · dvdrental schema

This is the same as GROUP BY rating ... HAVING COUNT(*) > 2. Either works. Derived tables shine when you need to do more with the aggregated result — joining it to another table, aggregating again, ordering by a computed column.

EXISTS — the existence test

EXISTS (subquery) returns true if the subquery returns any rows. It is the cleanest way to express "is there at least one matching row?":

Customers who have at least one payment (EXISTS version)
SQLite · dvdrental schema

The SELECT 1 is conventional — what the subquery returns does not matter, only whether it returns anything. Read literally: "for each customer, is there a payment with this customer's id?".

NOT EXISTS is the natural inverse — cleaner than NOT IN because it handles NULLs correctly.

Correlated subqueries

In the EXISTS example above, the inner query references the outer query's c.customer_id. That makes it a correlated subquery — it cannot be evaluated once; it runs per outer row. This is powerful but can be slow on large tables (it is conceptually a loop). For small datasets it is fine and often the clearest expression of intent.

Subquery vs JOIN — which to use

Many subqueries can be rewritten as JOINs. There are no strict rules, but useful guidelines:

Use a subquery when… Use a JOIN when…
You want to filter against an aggregate You want to display columns from both tables
You only need to test existence (EXISTS) The relationship is one-to-many you care about
The intent reads "find rows where … from another table" The intent reads "combine these tables"

When in doubt, write whichever you find more readable.

Common mistakes

Scalar subquery returning more than one row

SELECT title, (SELECT length FROM film) AS l FROM film;
-- error: more than one row returned by a subquery used as an expression

If a scalar subquery returns multiple rows, the database errors (PostgreSQL) or returns garbage (SQLite). Make sure the subquery is genuinely scalar — usually by including an aggregate (AVG, MAX) or LIMIT 1.

NOT IN with NULL in the list

WHERE x NOT IN (SELECT y FROM other)  -- if any y is NULL, returns nothing

Already covered in Module 2.1. Prefer NOT EXISTS to avoid this entirely.

Forgetting to alias a derived table

SELECT * FROM (SELECT rating, COUNT(*) FROM film GROUP BY rating);
-- error: subquery in FROM must have an alias

PostgreSQL requires every derived table to have an alias. Add AS something.

Correlated subqueries on huge tables

SELECT *,
       (SELECT COUNT(*) FROM payment p WHERE p.customer_id = c.customer_id) AS payment_count
FROM customer c;

This works, but it runs the inner query once per customer. For 10 customers, fine. For 10 million, you want a JOIN with GROUP BY instead.

Solution 5.3.4

SELECT *
FROM (
  SELECT rating, COUNT(*) AS n
  FROM film
  GROUP BY rating
) AS rating_counts
WHERE n > 2;

This is equivalent to GROUP BY rating ... HAVING COUNT(*) > 2. The derived table form lets you do further things with the aggregated result — join it, aggregate it again, sort by computed columns — that HAVING cannot.

Solution 5.3.5

The subquery SELECT title FROM film returns 15 rows. A scalar subquery is only allowed to return one row. PostgreSQL errors; SQLite returns the first row's title arbitrarily, paired with every outer row — nonsense. Fix: add an aggregate or a LIMIT to make the subquery genuinely scalar.

Lesson 5.4 — Self-joins

Joining a table to itself. The trick is in the aliasing.

The idea

Sometimes the rows you want to compare live in the same table. A classic example: employees and managers — both are people in the same employees table, linked by a manager_id column. To show each employee alongside their manager's name, you join the table to itself.

A self-join is no different from any other JOIN — same INNER/LEFT semantics, same ON clause. The only twist: you must alias the table twice, because SQL needs to distinguish the two copies.

The shape

SELECT  a.col, b.col
FROM    table a
JOIN    table b  ON  a.something = b.something_else;

a and b are two aliases for the same table. Without them, SQL would not know which copy col refers to.

Example — pairs of films at the same rating

Our film table has 15 rows. To list every pair of films that share a rating:

SQLite · dvdrental schema

Two crucial details:

  1. Different aliases (a and b) so we can tell the copies apart.
  2. a.film_id < b.film_id so we get each pair once. Without it, we would also get the mirror pair (B, A) for every (A, B), plus rows where a film is paired with itself.

Hierarchical example — employees and managers

If we had an employees table with id, name, manager_id (where manager_id is another row's id):

SELECT  e.name  AS employee,
        m.name  AS manager
FROM    employees e
LEFT JOIN employees m ON e.manager_id = m.id;

LEFT JOIN matters here: the CEO has no manager (NULL manager_id), and you usually want them in the result. INNER JOIN would silently drop them.

This is the canonical self-join example — every database textbook uses it. We do not have an employees table in dvdrental, so the example is on paper rather than in the playground.

Common mistakes

Forgetting the aliases

SELECT * FROM film JOIN film ON film.rating = film.rating;   -- error

Both copies are named film. SQL has no way to refer to one over the other. Alias both: FROM film a JOIN film b ON a.rating = b.rating.

Getting every pair twice (and self-pairs)

JOIN film b ON a.rating = b.rating

Without the a.film_id < b.film_id guard, you get: - Film A paired with Film B (good) - Film B paired with Film A (the duplicate) - Film A paired with Film A (the trivial self-pair)

The < (strict less-than) guard solves both at once.

Self-LEFT JOIN with the wrong direction

For the employees/managers example, FROM employees m LEFT JOIN employees e ON e.manager_id = m.id would give you "every manager and any employee under them, including managers with no reports". Different question. Always picture: what's on the left side is what's kept; what's on the right is what's attached.

Solution 5.4.2

SELECT a.first_name, b.first_name, addr.address
FROM customer a
JOIN customer b   ON a.address_id = b.address_id
                 AND a.customer_id < b.customer_id
JOIN address addr ON a.address_id = addr.address_id;

Solution 5.4.3

Without the a.film_id < b.film_id guard, you get every (A, B) pair, plus every (B, A) pair, plus every (A, A) self-pair. Add the guard:

JOIN film b ON a.rating = b.rating AND a.film_id < b.film_id

That keeps each unordered pair exactly once and removes the self-pairs.

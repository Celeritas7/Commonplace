Conditional Expressions & Views

The polish. Tools that make queries clearer, safer with NULLs, and reusable. After this module your SQL stops being a sequence of queries and starts being a small library.
Estimated time: 75–100 minutes · Prerequisites: Modules 1–6 complete. Comfort with SELECT, aggregates, JOINs, and the idea of NULL.

What this module covers

Three small tools and one large pattern. Each fills a gap in the SQL you already know.

  • CASE — if/then/else for columns. The single most-used conditional construct in SQL.
  • COALESCE, NULLIF, CAST — three short functions that handle NULLs and type conversions cleanly.
  • Views — save a query under a name and re-use it like a table. Where SQL stops being one-off and starts being shared infrastructure.

The SQL funnel returns

Module 6 took a break from the funnel because DDL is outside the read-side world. This module brings it back — everything here lives inside the clauses you already know:

CASE, COALESCE, NULLIF, and CAST mostly live inside SELECT (you use them to produce better columns in the result). They also appear in WHERE, ORDER BY, and HAVING. Views are queried FROM, like a table — they sit at the FROM layer.

Lesson 7.1 — CASE

SQL's if/then/else. The single most-used conditional construct in SQL.

Why this matters

CASE lets you compute a column whose value depends on conditions. "If salary is below 50k, label it 'entry'; if below 100k, 'mid'; otherwise 'senior'." That kind of bucketing is everywhere in reports, dashboards, and analytics queries.

Once you know CASE, you also unlock conditional aggregates — counting and summing only rows that meet a condition, without writing a separate query. That pattern alone is worth the lesson.

Two forms of CASE

CASE has two syntactic flavors. They are equivalent for simple equality checks; the searched form is more general.

Searched CASE — for arbitrary conditions

CASE
  WHEN condition1 THEN result1
  WHEN condition2 THEN result2
  ...
  ELSE default_result
END

Each WHEN holds any boolean expression. The first one that evaluates true wins.

Simple CASE — for equality against one expression

CASE column
  WHEN value1 THEN result1
  WHEN value2 THEN result2
  ...
  ELSE default_result
END

Less typing when every branch is comparing one column to a literal:

Rating to audience
SQLite · dvdrental schema

The simple form is a sugar for the searched form. You can always rewrite a simple CASE as a searched one. You cannot always go the other way — searched CASE handles range checks, NULL checks, multi-column conditions, and anything else a boolean expression supports.

ELSE is optional but recommended

If no WHEN matches and there is no ELSE, the result is NULL:

Films with rating PG-13, R, or NC-17 get NULL for audience. Sometimes that is what you want; often it is a bug. Make ELSE a habit.

Conditional aggregates — the killer pattern

This is what makes CASE worth its weight in gold. Combine CASE with SUM or COUNT to count or sum only the rows that match a condition:

This rotates a vertical GROUP BY result into a horizontal "pivot" — one row, one column per category. Without CASE you would need a separate query per rating and stitch them together. With CASE it is a single scan.

The same idea works with COUNT(CASE WHEN ... THEN 1 END) — COUNT ignores NULLs, so leaving the ELSE off (which produces NULL) also works:

SQLite · dvdrental schema

CASE in WHERE — rare but legal

You almost never need CASE in WHERE — boolean operators (AND/OR/NOT) cover the same ground more clearly. But it is legal:

WHERE CASE WHEN role = 'admin' THEN 1 ELSE 0 END = 1
-- equivalent to:
WHERE role = 'admin'

The second form is always clearer. CASE belongs in SELECT, in ORDER BY for custom sort, or in aggregates. Reach for it sparingly in WHERE.

CASE in ORDER BY — custom sort order

Want to sort by something that is not alphabetical or numeric? CASE.

Sort by rating, but in audience order, not alphabetical
SQLite · dvdrental schema

Alphabetically the order would be G, NC-17, PG, PG-13, R — confusing. The CASE expression assigns each rating a numeric rank that ORDER BY uses.

Common mistakes

Forgetting END

SELECT CASE WHEN x = 1 THEN 'a' WHEN x = 2 THEN 'b' ELSE 'c' FROM ...;   -- error

Every CASE needs a matching END. Easy to forget after several WHENs.

Mixing the two forms

CASE rating WHEN > 'G' THEN 'higher' END   -- error: cannot use operator in simple CASE

Simple CASE only does equality. For any other comparison, switch to searched: CASE WHEN rating > 'G' THEN 'higher' END.

Result type mismatch

CASE WHEN x > 0 THEN 'positive' WHEN x < 0 THEN -1 ELSE 0 END

The first branch returns text, the others return integers. PostgreSQL refuses; SQLite coerces silently. All branches of a CASE must return the same type.

Inscrutable nested CASE

CASE WHEN x THEN CASE WHEN y THEN ... ELSE CASE WHEN z ... END END END

Legal, but unreadable. If you find yourself nesting CASE more than two levels deep, the logic probably belongs in a view or a derived table.

Lesson 7.2 — COALESCE, NULLIF, CAST

Three short functions. Each solves a specific NULL or type problem cleanly.

COALESCE — first non-NULL value

COALESCE(a, b, c, ...) returns the first argument that is not NULL. If they are all NULL, it returns NULL.

The most common use: "if this column is NULL, show me this default instead":

Show 'no email' for customers without one
SQLite · dvdrental schema

You can chain it — first try email, then a phone field, then a default:

COALESCE(email, phone, 'no contact info')

The second most common use: combining with SUM to turn NULL totals into 0 (as in Module 4.2.4):

Total per customer — NULL for non-payers becomes 0
SQLite · dvdrental schema

Without COALESCE, customers who never paid would show NULL instead of 0 — usually less useful.

COALESCE vs ISNULL vs IFNULL

PostgreSQL uses COALESCE. SQL Server has ISNULL. MySQL and SQLite have IFNULL (which works in this playground too). They all do similar things, but COALESCE is the SQL standard, supports more than two arguments, and works in every database. Default to it.

NULLIF — turn a sentinel value into NULL

NULLIF(a, b) returns NULL if a equals b; otherwise returns a. Sounds odd at first — why would you want a NULL? Two reasons:

To avoid division by zero

x / 0 errors (PostgreSQL) or returns NULL/garbage (SQLite). x / NULLIF(divisor, 0) returns NULL instead — usually preferable to an error:

Average rental rate per film length unit (safe division)
SQLite · dvdrental schema

If any film had length = 0 (none do in our seed), the division would safely return NULL instead of crashing.

To mask sentinel values

Some datasets use -1 or 'unknown' to mean "no value recorded". NULLIF(rating, 'unknown') turns those sentinels into real NULLs so the rest of your SQL can use IS NULL, COALESCE, and aggregates that ignore NULLs:

SELECT NULLIF(rating, 'unknown') FROM film_imported;

CAST — convert between types

CAST(value AS type) explicitly converts a value to a different type. The PostgreSQL shorthand value::type does the same thing but is not portable to other databases.

Force integer-to-float division
SQLite · dvdrental schema

10 / 3 is 3 (integer division — covered in Module 5.2). CAST(10 AS REAL) / 3 is 3.333.... CAST is the explicit fix for that whole class of bug.

Other common conversions:

CAST('123' AS INTEGER)         -- parse string to number
CAST(price AS TEXT)            -- number to string for concatenation
CAST('2024-05-30' AS DATE)     -- text to date
CAST(rating AS TEXT)           -- ensure text type (defensive)

PostgreSQL accepts the shorthand:

'123'::INTEGER
price::TEXT
'2024-05-30'::DATE

Use the shorthand only in PostgreSQL-only code. CAST(...) works everywhere.

Combining the three

These functions compose naturally. A realistic example — compute average payment per customer, treating non-payers as 0, and tag whether they are above or below the company average:

COALESCE + CASE + AVG together
SQLite · dvdrental schema

This uses a common table expression (the WITH per_customer AS ... part) — a named subquery you can reference multiple times. We have not formally introduced CTEs; they are essentially "derived tables you give a name to first", and they compose beautifully with everything in this module.

Common mistakes

COALESCE with mixed types

COALESCE(email, 0)   -- error: cannot mix text and integer

All arguments to COALESCE must be the same type (or implicitly compatible). If you need a number when text is missing, cast: COALESCE(CAST(email AS INTEGER), 0). Usually the right answer is "pick a sensible default of the same type" — for text columns, an empty string '' or a label like 'unknown'.

NULLIF where you meant COALESCE

NULLIF(a, b) returns NULL when a = b. COALESCE(a, b) returns b when a is NULL. They look similar but do opposite things. Read the name: NULL-IF, COA-LESCE.

CAST that loses data

CAST(3.7 AS INTEGER)   -- returns 3, not 4 (truncation, not rounding)
CAST('abc' AS INTEGER) -- PostgreSQL: error. SQLite: 0.

CAST truncates and refuses (or silently returns 0). For rounding, use ROUND first. For parsing, validate the input string.

Lesson 7.3 — Views

Save a query under a name. Use it like a table. Reuse it across many queries.

What a view is

A view is a named SELECT statement, saved in the database. You query it the same way you query a table — but every time you do, the underlying SELECT runs fresh.

CREATE VIEW view_name AS
SELECT ... ;

Once created, the view appears in your database alongside tables. You can SELECT * FROM view_name, JOIN it, filter it, group it — everything that works on a table works on a view.

Why views matter

Three big reasons.

1. Hide complexity behind a name. A query that JOINs five tables and computes three derived columns is hard to read and easy to get wrong. Wrap it in a view named customer_lifetime_value and the next query is just SELECT * FROM customer_lifetime_value.

2. Standardize calculations. If three reports all compute "active customer" differently, you have a maintenance bug waiting. Define active_customer as a view once, and every report uses the same definition.

3. Security boundary. Grant a colleague access to a view that hides PII columns, without giving them access to the underlying table.

DROP VIEW and CREATE OR REPLACE VIEW

To remove a view:

DROP VIEW view_name;

To redefine an existing view in one step (PostgreSQL):

CREATE OR REPLACE VIEW view_name AS
SELECT ... ;

This is the safe pattern for evolving a view — it changes the definition without dropping and recreating.

Materialized views — store the result

A regular view recomputes every time you query it. A materialized view runs the query once, stores the result like a real table, and serves subsequent queries from the cache:

-- PostgreSQL only — SQLite does not have materialized views
CREATE MATERIALIZED VIEW customer_spend_cached AS
SELECT ... ;

REFRESH MATERIALIZED VIEW customer_spend_cached;

Tradeoff: instant reads (fast) versus stale data until the next REFRESH. Use them for expensive queries that power dashboards and tolerate some staleness.

When to make a view

A good rule of thumb: if you have copy-pasted a SELECT three times, make it a view. The third use is when "I'll just retype it" turns into "wait, did I get the JOIN condition right this time?".

Some good candidates:

  • A query that always does the same JOINs to assemble a "logical entity" (e.g., customer with their address, city, country)
  • A query that always applies the same business filter (e.g., active_customers = those with payments in the last 30 days)
  • A query that computes a standard metric (e.g., monthly_revenue)

Common mistakes

Forgetting that views recompute

SELECT * FROM expensive_view WHERE customer_id = 5;

The view runs its full underlying query — possibly scanning millions of rows — and then filters down to one row. The database optimizer often pushes the filter into the view's query (saving work), but not always. If a view is slow to use, consider a materialized view or rewriting the consuming query.

Views that hide bugs

CREATE VIEW v AS SELECT * FROM customer WHERE country = 'India';
SELECT COUNT(*) FROM v;

Six months later, someone reads the count and thinks it is "all customers". The view's filter is invisible to anyone who has not read the definition. Name views accurately: indian_customers, not customer_v2.

Forgetting DROP VIEW order with dependencies

If view B references view A, you cannot drop A while B exists. DROP VIEW A CASCADE (PostgreSQL) drops both. Be careful with CASCADE — it is also how you accidentally drop ten things at once.

Updating data through a view

Simple views (single table, no aggregates, no DISTINCT) are sometimes updatable — you can INSERT/UPDATE/DELETE through them and the changes propagate to the underlying table. Most non-trivial views are not updatable. When in doubt, treat views as read-only.

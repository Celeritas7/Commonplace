Filtering & Sorting

Narrow the rows you retrieve, decide what order they come back in, and cap how many you see. The three knobs that turn "fetch some data" into "answer a question".
Estimated time: 75–100 minutes · Prerequisites: Module 1 complete (SELECT, DISTINCT, COUNT). dvdrental sample database loaded.

What this module covers

Module 1 taught you to retrieve data. This module teaches you to retrieve the right data.

  • WHERE — keep only the rows that match a condition. The most-used clause after SELECT itself.
  • ORDER BY — decide what order the rows come back in. Without it, you have no guarantees.
  • LIMIT — cap how many rows you get. Pairs with ORDER BY to answer "top N" questions.

The SQL funnel — moving down a layer

In Module 1 we lived entirely at the top of the funnel. Now we descend:

WHERE sits one layer below SELECT/FROM. ORDER BY and LIMIT are further down still — they shape the result after the rows have already been chosen. Notice how each clause does less and less work as we move down: the heavy lifting is at the top, the polish is at the bottom.

Clause order is fixed

You write SQL clauses in a specific order: SELECT, FROM, WHERE, GROUP BY, HAVING, ORDER BY, LIMIT. Swap any two and you get a syntax error. The funnel is not just a metaphor — it is the literal order you type things.

Lesson 2.1 — WHERE

The first filter. Keep only the rows that match a condition.

Why this matters

A real database has thousands or millions of rows. You almost never want all of them. WHERE is how you say "only the rows that look like this".

Every query you write past Module 1 will have a WHERE clause. Reports, dashboards, app queries, data exports — all of them filter. Master WHERE and you can answer almost any question about your data.

The shape of a filtered query

WHERE goes immediately after FROM. The condition is anything that evaluates to true or false for each row:

SELECT columns
FROM table
WHERE condition;

For each row in the table, SQL evaluates the condition. If true, the row stays. If false (or NULL), the row drops.

Strings use single quotes, not double

In PostgreSQL, single quotes wrap a string literal. Double quotes wrap an identifier (a column or table name). Write WHERE rating = 'G', not WHERE rating = "G" — the second form makes PostgreSQL look for a column named G and error out.

Combining conditions with AND, OR, NOT

A single WHERE clause can have multiple conditions glued together:

AND: both must be true

OR: either is enough

NOT: invert the test

AND binds tighter than OR

In a chain like A OR B AND C, SQL reads it as A OR (B AND C) — not (A OR B) AND C. When mixing them, always use parentheses:

-- ambiguous, easy to misread:
WHERE rating = 'G' OR rating = 'PG' AND length > 100

-- explicit, what you almost certainly meant:
WHERE (rating = 'G' OR rating = 'PG') AND length > 100

IS NULL — testing for missing data

Some columns can be NULL — meaning "no value recorded". You cannot test for NULL with =. Use IS NULL and IS NOT NULL:

Customers without an email

The NULL trap that bites everyone

email = NULL does NOT find rows where email is NULL. It returns zero rows, every time. The query runs without error — exactly the kind of bug that ships to production.

NULL means "unknown", and asking "is unknown equal to NULL?" is itself unknown. Use IS NULL or IS NOT NULL. Always.

BETWEEN — range shortcut

BETWEEN a AND b is a compact way to say >= a AND <= b. Both endpoints are inclusive.

Films priced between $1 and $3
SQLite · dvdrental schema

These two queries are identical:

WHERE rental_rate BETWEEN 1.00 AND 3.00
WHERE rental_rate >= 1.00 AND rental_rate <= 3.00

BETWEEN is clearer when the range is short and the operator chain would be repetitive. Use whichever reads better.

BETWEEN is inclusive on both ends

length BETWEEN 90 AND 120 includes films of exactly 90 minutes AND exactly 120 minutes. If you wanted "strictly more than 90 and strictly less than 120", you need length > 90 AND length < 120.

IN — set membership

IN (a, b, c) is a compact way to say = a OR = b OR = c. Useful when you have a known list of acceptable values.

You can negate it with NOT IN:

NOT IN and NULL — a sharp edge

column NOT IN (a, b, NULL) returns no rows. If any value in the list is NULL, the whole expression evaluates to NULL (which is not true, so the row drops).

This catches everyone the first time. If your NOT IN list comes from a subquery, filter out NULLs first.

LIKE — pattern matching

LIKE lets you match string patterns. Two wildcards:

  • % matches any sequence of characters (including none)
  • _ matches exactly one character

LIKE is case-sensitive in PostgreSQL. PostgreSQL also has ILIKE for case-insensitive matching:

SELECT title FROM film WHERE title ILIKE 'a%';
-- matches both "Academy Dinosaur" and "ace Goldfinger"

To get the same effect in SQLite or other dialects, lowercase both sides explicitly:

Portable case-insensitive match
SQLite · dvdrental schema

ILIKE is PostgreSQL-only

ILIKE is a PostgreSQL extension. SQLite (this playground) does not recognize it. The LOWER(...) LIKE '...' pattern above works everywhere — slightly more verbose but fully portable across PostgreSQL, MySQL, and SQLite.

Common mistakes

Quoting numbers

SELECT * FROM film WHERE length = '100';   -- works in SQLite, breaks elsewhere

Numbers do not take quotes. PostgreSQL will sometimes implicitly cast and sometimes throw a type error depending on the column type. Be explicit: WHERE length = 100.

Comparing to NULL with =

SELECT * FROM customer WHERE email = NULL;   -- always zero rows

Already covered above, but it bears repeating because nobody escapes this bug once.

Stacking ANDs when you mean OR

WHERE rating = 'G' AND rating = 'PG';   -- no row matches both

A column can only hold one value per row. rating = 'G' AND rating = 'PG' asks for rows where the rating is simultaneously G and PG. Use OR or IN ('G','PG').

Forgetting parentheses with mixed AND/OR

Covered above. AND binds tighter than OR. Add parens.

Lesson 2.2 — ORDER BY

Decide the order rows come back in. Without it, you have no guarantees.

Why this matters

Here is a fact that surprises everyone the first time: SQL does not guarantee any row order unless you ask for one. A query may return rows in insertion order today and a different order tomorrow after the table is rebuilt. If order matters — for a report, a leaderboard, a paginated UI — you must say so.

ORDER BY also pairs with LIMIT (next lesson) to answer "top N" questions: the highest-paid customer, the longest film, the most recent rental.

Basic syntax

SELECT columns
FROM table
WHERE condition           -- optional
ORDER BY column [ASC|DESC];

ORDER BY sits below WHERE in the funnel — it works on rows that have already been filtered.

Films ordered shortest to longest
SQLite · dvdrental schema

Default direction is ASC (ascending — smallest first, A first, oldest first). Add DESC to flip it.

Longest films first
SQLite · dvdrental schema

Sorting by multiple columns

When you list two or more columns, SQL sorts by the first, then breaks ties using the second, then the third, etc.

By rating, then by length within each rating
SQLite · dvdrental schema

ASC/DESC applies per column — here rating is ascending alphabetically, but within each rating group, length is descending.

Tiebreakers matter for pagination

If you are paginating results (page 1, page 2, page 3...), order by enough columns to make ties impossible. If two rows have the same value in your sort column and ties are not broken, those rows can appear on different pages between requests — or both, or neither.

A common pattern: ORDER BY created_at DESC, id DESC — most-recent first, with id as a deterministic tiebreaker.

Sorting by an expression

You are not limited to sorting by raw columns. Any expression works:

Sort by total payment value (per row)
SQLite · dvdrental schema
Sort by an alias defined in SELECT
SQLite · dvdrental schema

You can also sort by column position — ORDER BY 2 means "the second column in SELECT". It works, but avoid it: rearranging the SELECT list silently changes the sort.

NULL handling

NULLs sort to one end of the result. PostgreSQL puts them last by default in ascending order, first in descending. You can override this with NULLS FIRST or NULLS LAST:

SELECT first_name, email FROM customer
ORDER BY email NULLS LAST;

SQLite vs PostgreSQL on NULL ordering

SQLite (this playground) puts NULLs first in ascending order — opposite of PostgreSQL. If consistent NULL placement matters, always be explicit with NULLS FIRST / NULLS LAST on PostgreSQL.

Common mistakes

Assuming insertion order

SELECT * FROM film;   -- looks ordered by film_id, but no guarantee

The rows look ordered today. They may not be tomorrow. If the order matters, write ORDER BY film_id explicitly.

Sorting after limiting

ORDER BY runs before LIMIT (the funnel shows this — ORDER BY is above LIMIT). If you LIMIT first and then ORDER BY in your application, you get a random subset sorted, not the top subset.

DESC applies only to the column it follows

ORDER BY rating, length DESC;   -- rating ascending, length descending
ORDER BY rating DESC, length;   -- rating descending, length ascending

Each column has its own direction. If you want both descending, say so: ORDER BY rating DESC, length DESC.

Lesson 2.3 — LIMIT

Cap how many rows you get. Pairs with ORDER BY for "top N" questions.

Why this matters

LIMIT is the simplest clause in SQL and one of the most useful. Three reasons you reach for it constantly:

  • Exploration — peek at the first few rows of an unfamiliar table without dumping thousands of rows into the console.
  • Top N — get the 5 longest films, the 10 newest customers, the 3 biggest payments. Always ORDER BY + LIMIT.
  • Pagination — show 25 results per page. LIMIT + OFFSET is the building block of every "Page 1 of 47" UI you have ever seen.

Basic syntax

SELECT columns
FROM table
WHERE condition           -- optional
ORDER BY column           -- optional, but usually you want it
LIMIT n;
Peek at the first 3 films
SQLite · dvdrental schema

Without ORDER BY, "first 3" is whatever 3 SQLite happens to return — same caveat as in Lesson 2.2. Useful for exploration, dangerous for reports.

The "top N" pattern

LIMIT shines when paired with ORDER BY. This is the single most common SQL pattern in business reporting:

Top 5 longest films
SQLite · dvdrental schema

Read it left to right: filter (none here), sort longest-first, keep the top 5. You can answer "highest", "lowest", "newest", "oldest", "biggest", "smallest" — any superlative — with ORDER BY + LIMIT.

3 customers who paid the most
SQLite · dvdrental schema

Pagination with OFFSET

LIMIT can be paired with OFFSET to skip the first N rows. Page 1 starts at offset 0, page 2 at offset 25, page 3 at offset 50, etc.

Films 6–10 by length
SQLite · dvdrental schema

Pagination needs a stable sort

If your ORDER BY has ties, OFFSET-based pagination breaks: a row can appear on two pages, or no page. Always sort by enough columns that no two rows tie. A unique column like id as the final tiebreaker is usually enough.

Common mistakes

LIMIT without ORDER BY

SELECT * FROM film LIMIT 5;   -- five arbitrary rows

Fine for exploration. Wrong for anything a user will see. If you mean "the 5 newest" or "5 random", say so with ORDER BY.

LIMIT before ORDER BY

SELECT * FROM film LIMIT 5 ORDER BY length DESC;   -- syntax error

Clause order is fixed. ORDER BY comes first, LIMIT last.

Using LIMIT for aggregation

SELECT COUNT(*) FROM film LIMIT 1;   -- the LIMIT does nothing useful

COUNT already returns one row. Adding LIMIT 1 is harmless but reveals a misunderstanding of what COUNT does. (It is also a hint that you might want LIMIT 5 on a query before the COUNT — i.e., "count the top 5", which needs a subquery, not a bare LIMIT.)

Confusing LIMIT with TOP

LIMIT n is the PostgreSQL/MySQL/SQLite syntax. SELECT TOP n is SQL Server's. FETCH FIRST n ROWS ONLY is the SQL-standard syntax. They all do the same thing. PostgreSQL accepts both LIMIT and FETCH FIRST.

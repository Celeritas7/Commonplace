Creating & Modifying Tables

So far we have only read data. Now you become the one who builds the database — designing tables, putting data in, changing it, and evolving the schema as needs change.
Estimated time: 100–130 minutes · Prerequisites: Modules 1–5 complete. Comfort with SELECT, JOIN, and the idea of NULL.

What this module covers

Until now, you were a reader of databases — someone who consumed data that someone else designed and loaded. This module flips that. You design the tables, you load the data, you change it, and when requirements shift, you evolve the schema without breaking anything.

  • Designing tables — picking data types, primary and foreign keys, constraints that prevent bad data getting in
  • DML — Data Manipulation Language — INSERT new rows, UPDATE existing ones, DELETE the ones you no longer need
  • Schema evolution — ALTER an existing table to add/rename/drop columns; DROP a whole table when it has outlived its purpose

These commands are called DDL (Data Definition Language — CREATE, ALTER, DROP) and DML (Data Manipulation Language — INSERT, UPDATE, DELETE). Together they cover everything except reading, which is SELECT (sometimes called DQL).

A different kind of SQL

Everything before this module was idempotent — running the same SELECT twice gives the same answer; nothing in the database changes. Everything in this module is mutating — running an UPDATE or DELETE changes the database. If you run it twice, the second run may do nothing (good) or destroy more data (very bad).

This shift in mental model is what makes DDL/DML feel risky to newcomers. Treat it that way — until you have habits like "WHERE first, then DELETE" — and you will not have a horror story to tell.

The funnel takes a break this module

The SQL funnel (SELECT, WHERE, GROUP BY, ORDER BY, LIMIT) was a visual aid for reading data. DDL and DML are outside that world entirely — they have their own shapes. We'll return to the funnel in Module 7.

In its place, the recurring image for this module is the table lifecycle: a table is created, fills with rows, gets edited and reshaped, and is eventually dropped. Every DDL and DML command moves a table from one stage of its life to the next.

Lesson 6.1 — Designing tables

Types, keys, constraints. The decisions you make here will shape every query against the table forever.

Data types

Every column has a type, and the type controls what can go in. Pick wrongly and you either lose data (truncated strings) or waste space and time (numbers stored as text). The PostgreSQL types you will reach for 95% of the time:

Type Holds
INTEGER Whole numbers — ids, counts, ages
BIGINT Very large whole numbers (over 2 billion)
NUMERIC(p, s) / DECIMAL Exact decimals — money, percentages
REAL / DOUBLE PRECISION Floating-point — measurements, scientific
TEXT Variable-length string — preferred default
VARCHAR(n) String with maximum length n
BOOLEAN TRUE / FALSE
DATE Calendar date
TIMESTAMP Date + time
UUID Globally-unique identifier (PostgreSQL only)

Use TEXT for strings, NUMERIC for money

PostgreSQL gives TEXT and VARCHAR(n) the same performance. TEXT is more flexible — no arbitrary length limit. Reach for VARCHAR(n) only when the limit is a real business constraint (e.g., 2-letter country codes).

For money, never use FLOAT or DOUBLE — floating-point math gives wrong answers for cents. NUMERIC(10, 2) (10 digits total, 2 after the decimal) gives exact decimal arithmetic. SQLite is more forgiving here but the rule still applies.

Primary keys

A primary key is the column (or combination) that uniquely identifies each row. Almost every table should have one. Without a primary key:

  • You cannot tell which row to UPDATE or DELETE when two rows look identical.
  • Other tables cannot reference rows in this one.
  • Many tools assume one exists and behave strangely without it.

The conventional pattern is an integer id column that auto-increments:

A minimal table with a primary key

INTEGER PRIMARY KEY in SQLite auto-increments by default. In PostgreSQL, the modern equivalent is id INT GENERATED ALWAYS AS IDENTITY or the older id SERIAL.

You only get one primary key per table, but it can span multiple columns (a composite key):

PRIMARY KEY (student_id, course_id)

We use this in the dvdrental film_category bridge table — each (film, category) pair is unique.

Foreign keys

A foreign key is a column that references another table's primary key. It says "this value must be one that exists over there".

Add an enrollment table referencing student
SQLite · dvdrental schema

student_id must be the id of an existing student. If you try to insert an enrollment for student_id 999 when no such student exists, the database refuses.

Foreign keys catch the bugs your application would have missed

Without a foreign key constraint, your application code is responsible for never writing an enrollment for a nonexistent student. With one, the database guarantees it — even if a buggy script tries.

This is the single most useful guard you can put on your data. Add foreign keys for every column that should reference another table, not just the ones where you remember to.

Constraints — beyond the type

Types stop you from storing text in an INTEGER column. Constraints stop you from storing wrong data of the right type — like a negative age or a duplicate email.

Constraint Means
NOT NULL Column may not be NULL
UNIQUE No two rows may have the same value
PRIMARY KEY NOT NULL + UNIQUE, plus marks this as the identifier
REFERENCES x(y) Foreign key — value must exist in x.y
CHECK (expr) Custom rule that must evaluate true for every row
DEFAULT v Value used when INSERT omits this column

Put them all together — this is what a real production table looks like:

A well-constrained employee table
SQLite · dvdrental schema

Reading top to bottom: every employee has an id, a unique non-null email, a non-null name, a non-negative salary, a hire_date that defaults to today, and an optional manager who must be another employee.

The full CREATE TABLE shape

CREATE TABLE table_name (
  column_1   TYPE [constraints],
  column_2   TYPE [constraints],
  ...
  -- table-level constraints (optional):
  PRIMARY KEY (col_a, col_b),
  FOREIGN KEY (col) REFERENCES other_table(other_col),
  CHECK (some_condition)
);

Column constraints (like NOT NULL or single-column UNIQUE) go on the column. Multi-column constraints (composite PK, multi-column UNIQUE, table-level CHECK) go at the bottom.

Common mistakes

Using FLOAT for money

Already covered — use NUMERIC. The bugs from FLOAT cents are subtle and hard to track down.

Forgetting NOT NULL

If you intend a column to always have a value, say so. NOT NULL is the cheapest defense against bad data. Adding it after the fact requires a migration that handles existing NULLs.

Skipping the primary key

A table without a primary key is a hot mess waiting to happen. Add id INTEGER PRIMARY KEY as a default reflex. You will almost never regret it.

Foreign keys to nothing

student_id INTEGER REFERENCES student(id) only protects you if the constraint is enforced. In SQLite, foreign keys are off by default — you have to PRAGMA foreign_keys = ON per connection. PostgreSQL enforces them always.

Solutions

Solution 6.1.1

CREATE TABLE product (
  id        INTEGER PRIMARY KEY,
  name      TEXT NOT NULL,
  price     NUMERIC(10, 2) NOT NULL CHECK (price >= 0),
  category  TEXT DEFAULT 'uncategorized'
);

Solution 6.1.2

CREATE TABLE order_item (
  id         INTEGER PRIMARY KEY,
  product_id INTEGER REFERENCES product(id),
  quantity   INTEGER NOT NULL CHECK (quantity >= 1)
);

Solution 6.1.3

Three issues: 1. No primary key — there is no way to uniquely identify a row, so UPDATE/DELETE on a specific row is impossible. 2. FLOAT for price — floating-point math gives wrong answers for money. Use NUMERIC(10, 2). 3. No NOT NULL anywhere — every column can be NULL. Is a row with product_id = NULL, price = NULL, qty = NULL actually valid? Probably not. Most of these should be NOT NULL.

A safer version:

CREATE TABLE order_line (
  id         INTEGER PRIMARY KEY,
  product_id INTEGER NOT NULL REFERENCES product(id),
  price      NUMERIC(10, 2) NOT NULL CHECK (price >= 0),
  qty        INTEGER NOT NULL CHECK (qty > 0)
);

INSERT … SELECT — copy data from another table

A powerful variant: instead of literal values, the source can be a SELECT. The columns of the SELECT must match the columns of the INSERT in order and type.

INSERT INTO archive_film SELECT * FROM film WHERE release_year < 2000;

This copies every old film into an archive table in one query.

UPDATE — changing rows

UPDATE table_name
SET column1 = value1, column2 = value2
WHERE condition;

The WHERE is technically optional — but almost always required. UPDATE without WHERE changes every row in the table. That has destroyed more data than any other SQL mistake.

Safety habit: SELECT first, then UPDATE

Before you run any UPDATE, run the SELECT version of the WHERE first:

SELECT * FROM product WHERE category = 'stationery';   -- check the rows
UPDATE product SET price = price * 1.20 WHERE category = 'stationery';  -- now update them

This catches the most common mistake — UPDATE with a too-broad WHERE, or no WHERE at all. Five seconds of SELECT can save hours of restoring from backup.

DELETE — removing rows

DELETE FROM table_name WHERE condition;

Same warning as UPDATE: DELETE without WHERE removes every row in the table. The empty table remains; the rows are gone.

SQLite · dvdrental schema

The SELECT-first habit applies double for DELETE

SELECT * with the planned WHERE before every DELETE. There is no Ctrl-Z. Even with backups, restoring is hours of pain compared to five seconds of typing.

What about RETURNING?

PostgreSQL has a useful extension: every INSERT, UPDATE, and DELETE can return the affected rows.

DELETE FROM product WHERE id = 6 RETURNING *;
-- gets back the row that was just deleted

SQLite supports this from version 3.35 onwards. Useful for confirming what changed without a separate SELECT.

Common mistakes

UPDATE or DELETE without WHERE

Covered above. The instinct should be "WHERE first" — write the WHERE before the UPDATE.

UPDATE that violates a constraint

UPDATE employee SET salary = -100 WHERE id = 5;
-- error: CHECK constraint failed: salary >= 0

Constraints from CREATE TABLE keep enforcing themselves through every UPDATE. This is the whole point of constraints — they refuse to let your data go bad.

Inserting a foreign key that does not exist

INSERT INTO order_item (id, product_id, quantity) VALUES (1, 999, 5);
-- error: FOREIGN KEY constraint failed

product_id 999 does not exist in the product table. Foreign key constraints stop the insert. Good — exactly what you want.

Forgetting that UPDATE = full row rewrite

In most databases (including PostgreSQL), UPDATE … SET x = 5 doesn't just write x; it rewrites the whole row internally. For huge rows or massive batch updates, that has performance implications. Not something to worry about until you have a real performance problem.

Lesson 6.3 — Schema evolution: ALTER and DROP

The only constant is change. ALTER and DROP let your tables follow requirements without breaking anything.

The two truths about schema change

  1. Real applications evolve. The columns you need next year are not the columns you needed last year.
  2. Existing data is precious. Schema changes that destroy data are usually wrong; schema changes that preserve and migrate it are usually right.

That tension is the whole story of ALTER. It is also why production database migrations are reviewed more carefully than almost any other change.

Changing types and constraints

Each database has its own quirks here. PostgreSQL supports rich ALTER TABLE syntax:

-- PostgreSQL
ALTER TABLE product ALTER COLUMN price TYPE NUMERIC(12, 4);
ALTER TABLE product ALTER COLUMN price SET NOT NULL;
ALTER TABLE product ALTER COLUMN price DROP NOT NULL;
ALTER TABLE product ADD CONSTRAINT product_price_positive CHECK (price > 0);
ALTER TABLE product DROP CONSTRAINT product_price_positive;

SQLite is more restrictive — older versions could only add columns, not modify them. Modern SQLite supports renaming and a handful of changes, but for serious type changes the SQLite idiom is: create a new table, copy rows over, drop the old, rename the new. PostgreSQL handles this in one statement.

Migrations in production

When a schema change runs against a large production database, two things matter:

  1. It should not lock the table for long — users can be reading and writing during the migration.
  2. It should be reversible — if something goes wrong, you can roll back.

PostgreSQL has tools like CREATE INDEX CONCURRENTLY and patterns like "add a new column, backfill, switch reads, drop old column" for safe online migrations. This is a deep topic; the key habit is to plan migrations as multi-step deployments, not single statements.

DROP TABLE table_name;

The table and all its data are gone. There is also DROP TABLE IF EXISTS table_name; which silently does nothing if the table is not there — useful in scripts.

DROP is irreversible

DROP TABLE has no undo, no WHERE clause. The entire table — schema, all rows, all indexes — disappears. The only "safety net" is backups.

Habit: never type DROP TABLE x without first typing SELECT COUNT(*) FROM x to remind yourself how much data you are about to vaporize.

TRUNCATE — empty a table fast

TRUNCATE TABLE table_name removes all rows but keeps the table. It is much faster than DELETE FROM table_name because it does not log every row deletion. PostgreSQL supports it; SQLite does not (use DELETE FROM table_name instead).

Common mistakes

ALTER that locks production

ALTER TABLE huge_table ADD COLUMN foo TEXT DEFAULT 'x' may rewrite every row depending on the database — locking the table for minutes or hours on a busy production system. Always test ALTER timing against a production-sized dataset before running it for real.

DROP TABLE in the wrong environment

The number-one production database incident: someone meant to DROP a table in dev and was connected to prod. Always confirm which connection you are on before any DROP or DELETE.

Forgetting that foreign keys block DROP

If order_item has a foreign key to product, you cannot drop product while orders reference it. Either drop order_item first, or use DROP TABLE product CASCADE (PostgreSQL) which drops dependent objects too — powerful and dangerous.

Importing & Exporting Data

Getting data into and out of a database. CSVs you receive from elsewhere, exports for analysts who don't speak SQL, the routine data hygiene that comes with both.
Estimated time: 75–100 minutes · Prerequisites: Modules 1–7 complete. Comfort with CREATE TABLE, INSERT, UPDATE, and CAST.

What this module covers

A surprising fraction of working SQL is getting data in and out of the database, not querying what is already there. CSVs from spreadsheets, exports for the marketing team, dumps for backup, JSON from APIs — every database has interfaces with the outside world. This module covers the routine ones.

  • Importing CSVs — loading external files into tables. The COPY family (PostgreSQL) and the .import mechanism (SQLite).
  • Exporting data — getting query results out as CSV, for downstream tools or non-SQL audiences.
  • Cleaning what arrives — real-world CSVs are messy. Type mismatches, NULL sentinels, encoding gotchas, duplicates. Standard patterns for fixing them.

A note on what you can run in this module

Most of Modules 1–7 ran every example against the in-page playground. This module is different: file-based import and export inherently need a real file system, and our in-browser SQLite has none.

So some examples are display-only (sql code blocks) — the canonical PostgreSQL syntax you would type into pgAdmin or psql. Other examples are playground-runnable — they simulate the import by using INSERT, then exercise the same cleaning and exporting patterns you would use on real loaded data. Each block is marked.

Like Module 6 (DDL), this module is mostly outside the SELECT funnel. Importing and exporting are administrative operations — they touch the boundary between the database and the outside world. The funnel returns in Module 9.

Lesson 8.1 — Importing data

Getting a CSV into a table. The first 10 minutes of most analytics projects.

The shape of an import

Every import has the same three steps, regardless of database:

  1. Create the destination table — define the columns and types
  2. Load the file — point the database at the CSV; it fills the table
  3. Validate — run a few sanity SELECTs to make sure types parsed and row counts match

Step 2 is where the dialects differ.

PostgreSQL — the COPY command

The standard PostgreSQL import is the COPY command:

COPY product (id, name, price, category)
FROM '/absolute/path/to/products.csv'
WITH (FORMAT csv, HEADER true, DELIMITER ',');

Reading left to right: - COPY product (cols...) — which table and columns we are filling - FROM '/path/to/file.csv' — the file path, as seen by the PostgreSQL server (not your laptop). This is the source of most COPY confusion. - WITH (...) — options. HEADER true skips the first line, DELIMITER is comma by default, NULL 'NULL' sets which string means NULL.

If the file is on your laptop and PostgreSQL is on a remote server, COPY cannot reach it. The fix is \copy — see below.

PostgreSQL — \copy in psql

\copy is a meta-command inside the psql client. Same syntax minus a backslash, but the file path is resolved on your machine:

\copy product (id, name, price, category) FROM 'products.csv' CSV HEADER

This is what you'll actually use 90% of the time. COPY (no backslash) is for files already on the database server — typically only true for self-hosted databases or specific data-pipeline setups.

pgAdmin — the GUI route

For one-off imports, the pgAdmin GUI is friendly: right-click a table, choose Import/Export Data, point at a CSV, tick "Header", click OK. Under the hood it generates a \copy and runs it. Useful when you forget the exact syntax.

SQLite — .import

SQLite's command-line tool has its own meta-command:

.mode csv
.import 'products.csv' product

Different syntax, same idea: tell SQLite to read a CSV and route it into the product table.

Common import pitfalls

Wrong file path

COPY ... FROM '/data/products.csv' is the server's view of the file. If psql is on your laptop and PostgreSQL is on a remote host, the server has no file called /data/products.csv and the command fails with "could not open file". Use \copy instead.

NULL representation

CSVs encode NULL in many ways: empty string, NULL, \N, NA, None. PostgreSQL defaults to treating empty strings as the NULL marker, but if your file uses NULL (literal text), you need WITH (NULL 'NULL').

Encoding

A file with non-ASCII characters (accented names, emoji, Japanese, etc.) saved as Windows-1252 will load with garbled characters into a UTF-8 database. Always save CSVs as UTF-8 and tell COPY so: WITH (FORMAT csv, ENCODING 'UTF8').

Header column count vs declared column count

COPY product (id, name, price) expects three columns per row. A CSV header with four columns will cause every row to error. Check the column list matches the file.

Types loose enough to swallow garbage

If you create the table with all TEXT columns, the import will probably succeed — but you have just loaded a table of strings, including "4.99" instead of 4.99. Cleaning becomes Lesson 8.3.

Lesson 8.2 — Exporting data

Getting the result of a query out as a file. The reverse of import, with mostly the same syntax.

Why export

Three common needs:

  • Hand off to non-SQL audiences — your boss wants the data in Excel; your marketing colleague wants a CSV they can drop into a dashboard tool.
  • Pipeline output — your job is the SQL stage of a larger workflow; downstream tools (Python, R, Tableau) consume your CSV.
  • Backups and snapshots — periodically dump a table for archival.

PostgreSQL — COPY ... TO

The reverse of COPY ... FROM. Pick a query, pick a destination file:

COPY (
  SELECT id, name, price FROM product WHERE price > 5
) TO '/path/to/expensive_products.csv'
WITH (FORMAT csv, HEADER true);

Same caveats as import: the path is the server's view of the file system. For files on your laptop, use \copy in psql:

\copy (SELECT id, name, price FROM product WHERE price > 5) TO 'expensive.csv' CSV HEADER

Quick output to your own screen

Sometimes you just want to see the query result in a way you can copy/paste. In psql:

\pset format csv
SELECT id, name, price FROM product WHERE price > 5;

Outputs CSV directly to the terminal. Paste it into Excel or send it. No file involved.

pg_dump — exporting whole tables or databases

For backups and migrations, pg_dump is the workhorse. Run it from the shell, not from SQL:

pg_dump -h localhost -U postgres -d dvdrental -t film > film.sql
pg_dump -h localhost -U postgres -d dvdrental > whole_database.sql

The output is a series of CREATE TABLE + INSERT statements — a complete reproducible snapshot. To restore: pipe the file back into psql.

SQLite — .output and .mode

.headers on
.mode csv
.output products.csv
SELECT * FROM product;
.output stdout

Switch the output destination to a file, run a query, switch back. Same idea, different syntax.

Exporting to JSON

For programmatic consumers, JSON is often nicer than CSV. PostgreSQL has rich JSON support:

SELECT json_agg(row_to_json(t))
FROM (SELECT id, title, rating FROM film WHERE rating = 'G') t;

That gives you a single JSON array, one object per row. Pipe it to a file via \copy and you have JSON output ready for any modern consumer.

SQLite's JSON is more limited but workable for small exports.

Common export pitfalls

Permission denied

COPY ... TO '/etc/passwd' will fail (good) and so will COPY ... TO '/users/me/file.csv' if PostgreSQL runs as a different user. Use \copy for client-side paths.

CSV that breaks on the receiving end

If your data contains commas, quotes, or newlines, naive string concatenation produces invalid CSV. Use COPY/\copy which handles quoting correctly. For ad-hoc concatenation, prefer pipe-delimited or tab-delimited.

Stale exports

A CSV is a snapshot. If anyone consumes it and your underlying data changes, they have stale numbers. For repeatable analytics, prefer giving them query access to the database (or a view) over emailing CSVs.

Lesson 8.3 — Cleaning what arrives

Real CSVs are messy. The routine pattern for turning a freshly-imported table into something queryable.

The mess you typically inherit

Open any CSV someone hands you, and you will likely find some combination of:

  • Leading or trailing whitespace — ' Notebook ' instead of 'Notebook'.
  • Mixed case — 'Stationery', 'stationery', 'STATIONERY' all meaning the same category.
  • Sentinel strings for NULL — 'NULL', 'NA', 'N/A', '-', ''. Some columns have one convention, some another.
  • Numbers in text columns — prices stored as '4.99' instead of as a NUMERIC.
  • Dates as text in mixed formats — '2024-05-30', '5/30/2024', ''.
  • Duplicates — the same row appearing twice because someone exported the same file twice.

The cleaning script as a recipe

After enough imports you will find yourself running the same five UPDATEs every time. Wrap them in a stored procedure or a .sql script you run after every load. The investment pays back fast — bug-free, repeatable cleanup is what separates "the analyst" from "the analyst who can be trusted with new data".

Common cleaning mistakes

Updating before backing up

If a cleaning UPDATE turns out to be wrong, you cannot rewind without a backup. For one-time analyses, this is acceptable. For production tables, always back up first — or work in a staging table (load into product_import_staging, clean it, then INSERT … SELECT into product).

Inconsistent NULL handling

COALESCE(price, 0) treats NULL as zero. WHERE price > 0 excludes NULLs entirely. Both are valid, depending on intent — decide deliberately whether "no price recorded" should count as zero, exclude, or error.

Premature deduplication

Two rows look identical but might represent two different real-world events — two payments of the same amount on the same day from the same customer, for instance. Before DELETE-ing as duplicates, confirm with the source: "are these really duplicates, or two separate events?".

Cleaning that hides upstream bugs

If every batch needs the same five fixes, the source system is producing dirty data. Push fixes upstream — into the export step, the data entry form, or the ETL job — rather than silently cleaning forever. Cleanup is a sign of an upstream problem, not the end state.

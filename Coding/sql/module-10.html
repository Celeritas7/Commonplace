Capstone — Tiffinly

One extended exercise that ties everything together. Design a schema for a small tiffin delivery startup, load it with data, answer business questions across modules 1–8, wrap it with views and a Python operationalization layer. No new syntax — just integration.
Estimated time: 120–150 minutes · Prerequisites: Modules 1–9 complete.

What this module covers

Every previous module taught one tool. This one asks you to pick the right tool for each step of a real problem and string them into something useful. No new syntax. The skill being tested is recognition — looking at a question and knowing it's a GROUP BY question, or a LEFT JOIN question, or a CASE question.

The capstone is built around a single fictional company:

Tiffinly — a small tiffin (lunchbox) delivery startup in Pune. Customers subscribe to a weekly meal plan and receive home delivery on weekdays. Tiffinly wants to know who its customers are, what they're paying, who's eating reliably, and where service is breaking down.

You will:

  • Build the schema (Lesson 10.1) — five tables, the relationships between them, and a populated dataset to query against. Module 6 territory.
  • Analyze the business (Lesson 10.2) — answer six concrete questions Tiffinly's founder would actually ask. Modules 1–5, plus Module 7's CASE for the conditional ones.
  • Operationalize (Lesson 10.3) — bundle the recurring queries into views, sketch the export step, and write the Python script that produces a daily report. Modules 7, 8, 9.

Lesson 10.1 — Build the schema

The Tiffinly data model. Five tables, the relationships between them, and seed data to query against.

Why this matters

Before you can analyze a business, you need a data model that fits it. The exercise here is small but real — pick the right primary keys, the right foreign keys, and types that won't betray you later. Most schema problems are decided in the CREATE TABLE statements; everything downstream pays the cost or reaps the benefit.

The Tiffinly data model

Five tables, all prefixed tf_ so they don't collide with the dvdrental schema in the playground.

tf_customer     ──┐
                  │ owns
tf_subscription ──┤
   │              │
   │ chosen plan  │
   ▼              │
tf_plan           │ generates
                  │
                  ▼
            tf_delivery ──→ tf_rating (optional)

Reading the diagram top to bottom: a customer owns one or more subscriptions. Each subscription chose one meal plan. Each subscription generates many deliveries over its lifetime. Each delivery may (or may not) receive a customer rating.

Notice the constraints:

  • NOT NULL on every column that's required for a row to make sense (a delivery without a date is meaningless).
  • UNIQUE on tf_customer.email (one account per email) and tf_rating.delivery_id (a delivery can only be rated once).
  • REFERENCES to declare every foreign key — so the database will (in PostgreSQL) reject orphan rows.
  • CHECK (stars BETWEEN 1 AND 5) so a five-star rating system actually stays one-to-five. Cheap to add now, painful to enforce in the application later.

Solution 10.1.3

The CHECK constraint CHECK (stars BETWEEN 1 AND 5) catches it — PostgreSQL rejects the row with a check-constraint-violation error. (SQLite enforces it too when constraints are honored.) This is exactly the value of declaring constraints in the schema: every code path that tries to insert garbage is caught at the database, not just the one path the application developer remembered to validate.

Lesson 10.2 — Analyze the business

Six questions Tiffinly's founder would actually ask. Each answered with one query, naming the funnel clauses that do the work.

Why this matters

A schema is only useful insofar as it can answer questions. This lesson is six representative questions, with the funnel called out for each. By the end you should be able to read a business question and immediately see which clauses you need.

Q1 — Monthly recurring revenue (MRR)

How much subscription revenue is currently coming in per month?

Sum the monthly_price of every plan attached to an active subscription. That's a JOIN (subscriptions to plans) with a WHERE (active only) and an aggregate (SUM).

Funnel: FROM joins → WHERE filters → SELECT aggregates.

SQLite · dvdrental schema

Q2 — Revenue by area

Which neighborhoods are paying us the most?

Now bring tf_customer in (for area) and GROUP BY area.

Funnel: FROM joins three tables → WHERE filters active → GROUP BY area → SELECT sums → ORDER BY revenue desc.

Q2 — MRR broken down by area
SQLite · dvdrental schema

Q3 — Late-or-missed rate by area

Where is service breaking down?

Bucket every delivery as "on time" or "not on time", then take the average of "not on time" as a rate. CASE inside an AVG is the cleanest form.

Funnel: FROM joins → SELECT computes a CASE → GROUP BY area → ORDER BY worst area first.

Q3 — Fraction of deliveries late or missed, by area
SQLite · dvdrental schema

Q4 — Top customers by delivered meals

Who are our highest-volume eaters this month?

Count delivered rows per customer, sort, take the top.

Funnel: FROM joins → WHERE filters by status + date range → GROUP BY customer → ORDER BY count desc → LIMIT.

Q4 — Top 3 customers by delivered meals in May 2025
SQLite · dvdrental schema

Q5 — Customers who never rated anything

Who's silent? They're either thrilled, indifferent, or about to churn.

A LEFT JOIN from customer to rating, keeping only the customers where no rating exists.

Funnel: FROM left-joins customer to (subscription → delivery → rating) → WHERE filters to NULL on the rating side → SELECT names.

Q5 — Customers with zero ratings ever
SQLite · dvdrental schema

Notice: GROUP BY + HAVING COUNT = 0 expresses "never has any rating" cleanly. We saw this pattern in Module 4 (anti-joins) and Module 7 (HAVING on aggregates).

Q6 — Plan-level satisfaction

Which plan is leaving customers least happy?

Average stars per plan, with a column counting how many ratings the average is computed from (so the founder can spot tiny-sample averages).

Funnel: FROM joins delivery + subscription + plan + rating → GROUP BY plan → SELECT AVG, COUNT → ORDER BY avg asc.

Q6 — Average stars per plan
SQLite · dvdrental schema

The LEFT JOIN to tf_rating matters — without it, any plan with zero ratings would silently disappear from the report rather than showing up as n_ratings = 0. Inner joins hide; outer joins reveal.

Solutions

Solution 10.2.1

SELECT ROUND(AVG(p.monthly_price), 2) AS avg_price
FROM tf_plan p
WHERE EXISTS (
  SELECT 1 FROM tf_subscription s
  WHERE s.plan_id = p.plan_id AND s.status = 'active'
);

Solution 10.2.2

SELECT DISTINCT c.name, c.area
FROM tf_customer c
JOIN tf_subscription s ON s.customer_id = c.customer_id
JOIN tf_delivery d     ON d.subscription_id = s.subscription_id
WHERE d.status = 'missed'
ORDER BY c.name;

Solution 10.2.3

SELECT substr(signed_up_on, 1, 7) AS sign_up_month, COUNT(*) AS new_customers
FROM tf_customer
GROUP BY substr(signed_up_on, 1, 7)
ORDER BY sign_up_month;

In PostgreSQL the more idiomatic form is DATE_TRUNC('month', signed_up_on). The substr form is the cross-dialect fallback for SQLite.

Lesson 10.3 — Operationalize

Wrap the recurring queries into views, sketch the daily export, write the Python script that runs the report.

Why this matters

Lesson 10.2's queries are correct, but if the founder has to re-paste them every morning they'll fall out of use within a week. Operationalizing means making the work reproducible — give the queries names (views), give them a schedule (cron + Python), give them an output (CSV or email). What was a one-time analysis becomes a system.

Step 1 — Capture the recurring queries as views

Two queries from 10.2 deserve to be views — they will be queried over and over: the by-area MRR and the by-area service-quality rate. Once they're views, the founder (or a dashboard tool) can SELECT * FROM v_mrr_by_area and get fresh numbers without remembering joins.

Step 2 — Sketch the export (Module 8 territory)

The founder wants the daily report as a CSV in her inbox. In PostgreSQL (display-only — the playground has no file system):

\copy (SELECT * FROM v_mrr_by_area) TO 'mrr_by_area.csv' CSV HEADER
\copy (SELECT * FROM v_delivery_quality) TO 'delivery_quality.csv' CSV HEADER

Two CSVs, one \copy each, run from psql. That's the file-based path from Module 8.

The Python path (next step) skips the intermediate CSV entirely and emails the result directly. Either works; pick based on who consumes the output.

Step 3 — The Python daily report (Module 9 territory)

A short script that queries both views, prints a formatted summary, and is the kind of thing you'd put behind cron to run at 7 a.m. every day. Display-only — runs in a real Python environment, not the playground.

"""
Tiffinly daily report.
Runs the two operational views and prints a summary.
Schedule with cron:  0 7 * * *  /usr/bin/python3 /opt/tiffinly/daily_report.py
"""

import os
import datetime as dt
import psycopg
from psycopg.rows import dict_row

DSN = (
    f"host={os.environ['PGHOST']} "
    f"dbname={os.environ['PGDATABASE']} "
    f"user={os.environ['PGUSER']} "
    f"password={os.environ['PGPASSWORD']}"
)

def main():
    today = dt.date.today().isoformat()
    print(f"Tiffinly daily report — {today}")
    print("=" * 50)

    with psycopg.connect(DSN) as conn:
        with conn.cursor(row_factory=dict_row) as cur:

            cur.execute("SELECT * FROM v_mrr_by_area ORDER BY mrr_inr DESC;")
            print("\nMRR by area")
            print("-" * 50)
            total_mrr = 0
            for row in cur:
                print(f"  {row['area']:18s}  ₹{row['mrr_inr']:>8,.0f}  ({row['active_subscriptions']} subs)")
                total_mrr += float(row['mrr_inr'])
            print(f"  {'TOTAL':18s}  ₹{total_mrr:>8,.0f}")

            cur.execute("SELECT * FROM v_delivery_quality ORDER BY bad_rate DESC;")
            print("\nDelivery quality by area")
            print("-" * 50)
            for row in cur:
                flag = "  ⚠" if row['bad_rate'] > 0.25 else ""
                print(f"  {row['area']:18s}  bad_rate={row['bad_rate']:.1%}  "
                      f"(of {row['total_deliveries']} deliveries){flag}")

if __name__ == "__main__":
    main()

Reading the script in terms of the last nine modules:

  • The psycopg.connect(...) + with block is Module 9 (Lesson 9.1).
  • The dict_row factory and cursor iteration is Module 9 (Lesson 9.2).
  • The queries are calls to views from this lesson — which wrap the queries from 10.2 — which combine Modules 1–5.
  • The ⚠ flag uses Python rather than SQL CASE, but the same conditional logic could move into the view as a CASE WHEN bad_rate > 0.25 THEN 'alert' ELSE 'ok' END column. Either is reasonable; the SQL form keeps the rule in the database and survives a UI rewrite.

Tiffinly's founder now has a one-command daily report. That's the level of integration the book has been building toward.

Challenge 10.3.3 — Concept check

The founder asks: "Can we just keep adding columns to v_mrr_by_area instead of building more views?" Give one reason for and one reason against. Which would you pick, and why?

Solutions

Solution 10.3.1

DROP VIEW IF EXISTS v_customer_summary;
CREATE VIEW v_customer_summary AS
SELECT
  c.customer_id,
  c.name,
  c.area,
  COUNT(s.subscription_id) AS total_subscriptions,
  CASE WHEN SUM(CASE WHEN s.status = 'active' THEN 1 ELSE 0 END) > 0
       THEN 1 ELSE 0 END   AS is_currently_active
FROM tf_customer c
LEFT JOIN tf_subscription s ON s.customer_id = c.customer_id
GROUP BY c.customer_id, c.name, c.area;

SELECT * FROM v_customer_summary ORDER BY customer_id;

Solution 10.3.2

SELECT customer_id, name, area, total_subscriptions
FROM v_customer_summary
WHERE is_currently_active = 0
ORDER BY customer_id;

Solution 10.3.3

For wide views: one query covers more questions; consumers don't need to learn six view names. Against wide views: every consumer pays the cost of every column even if they only want one, and a single view that tries to answer everything becomes hard to evolve without breaking someone. The right pick is usually small, focused views that each answer one question — composed by the consumer if they need a wider picture. That mirrors how good functions are written: one job, well, with a clear name.

Where to go from here

The book ends here, but SQL doesn't. A short, opinionated list of what to learn next, in roughly the order it tends to come up:

  • Window functions — ROW_NUMBER, RANK, LAG, LEAD, SUM() OVER (...). Module 5 touched subqueries; window functions extend that to running totals, ranks within groups, and time-series comparisons. Once you have these, a whole class of "I'd need a script for this" problems becomes one query.
  • Common table expressions (CTEs) — WITH x AS (SELECT ...) SELECT ... FROM x. Briefly seen in Module 5; treating CTEs as first-class scaffolding for big queries makes them readable instead of nested-soup.
  • Indexes and EXPLAIN — what makes one query 50ms and the same query 50 seconds. Read execution plans, learn when to add an index (and when not to). The first time you make a slow query ten thousand times faster, you stop being afraid of large tables.
  • Transactions in depth — isolation levels, locking, and the patterns (SELECT ... FOR UPDATE, optimistic concurrency) that make concurrent writes correct.
  • Database design beyond five tables — normalization (3NF, BCNF), denormalization for analytics, dimensional modeling for warehouses. Module 6 taught the mechanics; the next step is the theory.
  • ORMs and migrations — SQLAlchemy, Django ORM, Prisma, Drizzle. ORMs let you express schema and queries from your application's language. Migrations (Alembic, Django migrations) version-control the schema itself. Most production codebases have both.
  • Analytical tooling — dbt for SQL-as-code in analytics, BigQuery / Snowflake / DuckDB for warehouse-scale work, Metabase / Superset for self-serve dashboards.
  • Query specialization — PostGIS for spatial queries, full-text search, JSONB for semi-structured data. Each is a deep specialization, but worth knowing exists so you reach for the right tool when the question arrives.

You don't need any of these to be useful. You need exactly what's in this book to be useful. The list above is the long road from "I know SQL" to "I'm the person at the company who knows SQL." Most of that road is walked one query at a time, against a real database, solving real problems.

The end. Open a psql prompt and pick a question.

JOINs

Pull data from two or more tables in one query. JOINs are where SQL stops being a query language and starts feeling like working with a network of related ideas.
Estimated time: 90–120 minutes · Prerequisites: Modules 1–3 complete. Comfort with WHERE, aliasing, and the idea that columns have types.

What this module covers

Until now, every query touched one table. Real databases hold data in many tables — and the interesting questions almost always need information from more than one of them.

  • INNER JOIN — keep only rows that match in both tables. The default and most common join.
  • OUTER JOINs (LEFT, RIGHT, FULL) — keep unmatched rows too, filling the gaps with NULL.
  • UNION — stack rows from two queries on top of each other (different from JOIN, which puts them side-by-side).

JOINs are the single biggest "click" moment in SQL. After this module, you can answer questions like "which customers have made payments and what films did they rent?" in one query.

Why JOINs exist — normalization in 60 seconds

Look at our payment table:

Peek at payments — note: no customer name
SQLite · dvdrental schema

Every payment has a customer_id but no customer name. The name lives in the customer table, and customer_id is the bridge between them. This split is deliberate — it is called normalization:

  • Each fact lives in exactly one place. A customer's name is in customer, full stop. Never duplicated across every payment they made.
  • Updates are cheap and safe. If Aniket changes his name, you update one row in customer — every payment automatically reflects the new name.
  • Tables stay small and focused. payment has 14 rows of payments; it does not bloat into a denormalized log of every customer detail.

The cost: to see a customer's name next to their payment, you must JOIN customer ON customer.customer_id = payment.customer_id. That join is what this module teaches.

The SQL funnel — JOINs live in FROM

JOINs are part of the FROM clause. The funnel does not move for this module — we are still at the top, broadening what SELECT pulls from:

The trick is that FROM stops being a single table and becomes "the result of combining tables".

The shape of every JOIN

Every JOIN you will ever write follows this skeleton:

SELECT  columns
FROM    left_table
JOIN    right_table  ON  left_table.id = right_table.id
WHERE   ...                 -- optional

The ON clause is the matching rule. Almost always it equates two columns — usually a primary key on one side and a foreign key on the other. Memorize this shape.

Lesson 4.1 — INNER JOIN

Keep only rows that have a match in both tables. The default join.

Visual: what INNER JOIN keeps

The dark purple area — rows that appear in both tables — is what comes back. Everything else is dropped.

The shape

SELECT  columns
FROM    left_table
INNER JOIN  right_table  ON  left_table.key = right_table.key;

The word INNER is optional. Plain JOIN means INNER JOIN.

Every payment, with the customer's name
SQLite · dvdrental schema

You get 14 rows — the number of payments. Each payment row has been paired with its matching customer row.

What about customers with no payments?

INNER JOIN drops them. We have 10 customers and 14 payments, but 3 customers (Chen, Lucas, Anna) have never paid. They do not appear in the result because there is no row in payment to match against. LEFT JOIN solves that — Lesson 4.2.

Aliasing tables

Typing customer.first_name and payment.amount over and over gets old. Use table aliases — pick a short letter or two, declare it right after the table name:

Same query, aliased
SQLite · dvdrental schema

Conventional aliases: c for customer, p for payment. Whatever you pick, be consistent inside a query.

Always qualify shared column names

If both tables have a column named customer_id, you cannot write SELECT customer_id — SQL does not know which one you mean and throws an "ambiguous column" error. Always qualify: c.customer_id or p.customer_id.

For columns that exist in only one table, qualification is optional — but doing it consistently makes queries readable.

Common mistakes

Forgetting the ON clause

SELECT * FROM customer JOIN payment;   -- error or cross product

Without ON, you get either an error (PostgreSQL) or a Cartesian product — every customer paired with every payment, all 10 × 14 = 140 rows of meaningless data. Always include ON.

Joining on the wrong column

SELECT * FROM customer c JOIN payment p ON c.customer_id = p.payment_id;

This runs. It might even return some rows (if a customer_id happens to equal a payment_id). Every row will be wrong. The bug is silent. Always check that ON references the right column on each side — usually a primary key matched to a foreign key.

Ambiguous columns

SELECT customer_id FROM customer c JOIN payment p ON c.customer_id = p.customer_id;
-- error: column reference "customer_id" is ambiguous

Qualify the column: SELECT c.customer_id (or p.customer_id — they are equal, so either works after a join on equality).

Challenge 4.1.4 — Bug spotting

What is wrong with this query?

SELECT first_name, amount
FROM customer
JOIN payment ON customer.customer_id = customer.customer_id;

Solutions

Solution 4.1.1

SELECT c.first_name, p.amount
FROM customer c
JOIN payment p ON c.customer_id = p.customer_id
ORDER BY p.amount DESC;

Solution 4.1.2

SELECT r.rental_date, f.title, c.first_name
FROM rental r
JOIN film f     ON r.film_id     = f.film_id
JOIN customer c ON r.customer_id = c.customer_id;

Notice the alignment — lining up JOIN ... ON ... = ... makes multi-table joins much more readable.

Solution 4.1.3

SELECT c.first_name, COUNT(*) AS payment_count
FROM customer c
JOIN payment p ON c.customer_id = p.customer_id
GROUP BY c.customer_id, c.first_name
ORDER BY payment_count DESC;

We group by customer_id too (not just first_name) in case two customers share a first name — same idea as Module 3.2 Challenge 1.2.3.

Solution 4.1.4

The ON clause references the same table on both sides: customer.customer_id = customer.customer_id. That is always true, so every customer is matched with every payment — a 140-row Cartesian product, not the 14-row join you wanted. Fix:

SELECT first_name, amount
FROM customer
JOIN payment ON customer.customer_id = payment.customer_id;

Lesson 4.2 — OUTER JOINs (LEFT, RIGHT, FULL)

Keep unmatched rows too. The most-asked-about JOIN in interviews.

Visual: the three outer joins

all of A + matches in B
all of B + matches in A
all of A + all of B

The dark purple area is what comes back. Wherever there is no match on one side, the columns from that side come back as NULL.

LEFT JOIN — the workhorse

This is the join you will reach for second most after INNER. "Give me every X, plus their Y if they have one":

Every customer + their payments (if any)
SQLite · dvdrental schema

You get 17 rows: the 14 customer-payment matches, plus 3 rows for Chen, Lucas, and Anna (who have never paid). Their amount column is NULL.

The word OUTER is optional. LEFT JOIN and LEFT OUTER JOIN are identical.

Finding rows that have no match

A LEFT JOIN with a WHERE x IS NULL filter is the standard "find the orphans" pattern:

You get only Chen, Lucas, and Anna — the 3 customers without payments. This is one of the most useful queries in any analytics toolkit. Variations: "users who never logged in", "products never ordered", "categories with no films".

Films that have never been rented
SQLite · dvdrental schema

RIGHT JOIN — same shape, mirrored

A RIGHT JOIN B is equivalent to B LEFT JOIN A. Same rows, same NULLs, same logic — just written with the tables in the opposite order.

-- These two return the same rows:
SELECT ... FROM payment p RIGHT JOIN customer c ON p.customer_id = c.customer_id;
SELECT ... FROM customer c LEFT  JOIN payment p ON c.customer_id = p.customer_id;

Most SQL writers never use RIGHT JOIN

Conventionally, you put the "main" table first and LEFT JOIN the secondary tables onto it. This reads naturally — "customers, plus their payments" — and is easier to extend with more joins.

If you find yourself wanting RIGHT JOIN, swap the table order and use LEFT JOIN instead. Same result, cleaner read.

FULL JOIN — everything from both sides

FULL OUTER JOIN (often just FULL JOIN) keeps every row from both tables. Where there is no match, the missing side is NULL.

Every customer and every payment, matched where possible
SQLite · dvdrental schema

Less common in practice — you usually have a "main" table and a "lookup" table, and LEFT JOIN expresses that better. FULL JOIN shines when both sides are equal citizens: comparing two snapshots of the same data ("which IDs are in this morning's file but not last night's?", and vice versa, in one query).

The "anti-join" recap

These are so useful they deserve their own name. Use a LEFT/RIGHT/FULL JOIN with WHERE other.key IS NULL to find unmatched rows.

-- A's that have no matching B
SELECT * FROM A LEFT JOIN B ON A.key = B.key WHERE B.key IS NULL;

-- B's that have no matching A
SELECT * FROM A RIGHT JOIN B ON A.key = B.key WHERE A.key IS NULL;
-- or equivalently:
SELECT * FROM B LEFT JOIN A ON B.key = A.key WHERE A.key IS NULL;

Common mistakes

Confusing INNER JOIN result with LEFT JOIN result

You ran a LEFT JOIN and the result has the same row count as INNER. This means every left row found a match — not a bug, just the data. To verify there are unmatched rows, count them: WHERE right.key IS NULL.

NULL columns from the unmatched side

After a LEFT JOIN, the right-side columns can be NULL. Filtering on them later with = will silently drop those rows:

SELECT c.first_name, p.amount
FROM customer c
LEFT JOIN payment p ON c.customer_id = p.customer_id
WHERE p.amount > 0;
-- Wait — Chen, Lucas, Anna are missing again!

The WHERE p.amount > 0 evaluates to NULL for unmatched rows (NULL > 0 is NULL, not true), so they drop. You may have meant WHERE p.amount > 0 OR p.amount IS NULL. Or just use INNER JOIN if you do not want the unmatched rows.

LEFT JOIN with the smaller table on the left

-- We want "every film + how many times it was rented"
SELECT r.rental_id, f.title
FROM rental r LEFT JOIN film f ON r.film_id = f.film_id;   -- wrong direction

This drops films that were never rented (films 14 and 15) because rental is on the left. The "main" table — the one whose rows you want all of — must be on the left. Fix: put film on the left.

Challenge 4.2.5 — Code review

A teammate wrote this and says it returns the same as INNER JOIN. Why?

SELECT c.first_name, p.amount
FROM customer c
LEFT JOIN payment p ON c.customer_id = p.customer_id
WHERE p.amount IS NOT NULL;

Solutions

Solution 4.2.1

SELECT c.first_name, p.amount
FROM customer c
LEFT JOIN payment p ON c.customer_id = p.customer_id
ORDER BY c.first_name;

Solution 4.2.2

SELECT f.title
FROM film f
LEFT JOIN rental r ON f.film_id = r.film_id
WHERE r.rental_id IS NULL;

Solution 4.2.3

SELECT c.name
FROM category c
LEFT JOIN film_category fc ON c.category_id = fc.category_id
WHERE fc.film_id IS NULL;

Solution 4.2.4

SELECT c.first_name, COALESCE(SUM(p.amount), 0) AS total
FROM customer c
LEFT JOIN payment p ON c.customer_id = p.customer_id
GROUP BY c.customer_id, c.first_name
ORDER BY total DESC;

COALESCE(a, b) returns the first non-NULL argument. You will meet it more formally in Module 7 (Conditional Expressions), but it earns its keep here: turning NULL totals into 0 for customers who never paid.

Solution 4.2.5

WHERE p.amount IS NOT NULL drops every row where the right side was NULL — which is precisely the rows LEFT JOIN added. The query effectively becomes an INNER JOIN. A filter on the right-side columns in WHERE is the most common way to accidentally turn a LEFT JOIN back into an INNER JOIN.

If you want a true LEFT JOIN with a filter on the right side, put the filter in the ON clause instead:

LEFT JOIN payment p ON c.customer_id = p.customer_id AND p.amount > 5

This filters which payments match — but still keeps every customer.

Lesson 4.3 — UNION

Stack rows from two queries on top of each other. Not a JOIN — a separate idea.

JOIN vs UNION — different shapes

This is the single most important distinction:

Operation Combines tables Result columns Result rows
JOIN side-by-side A + B columns rows where keys match (or all, depending on type)
UNION one on top of other columns of A (must match B) all rows from A + all rows from B

JOINs answer "what do these rows look like combined?". UNIONs answer "what does it look like if I treat both row sets as one bigger set?".

Visual

The shape

SELECT col1, col2 FROM A
UNION
SELECT col1, col2 FROM B;

Rules: 1. The two SELECTs must return the same number of columns 2. The columns must have compatible types in the same positions 3. The column names of the first SELECT become the result's column names

UNION vs UNION ALL

  • UNION removes duplicate rows (collapses rows that match across all columns).
  • UNION ALL keeps every row, duplicates included.

UNION ALL is faster (no deduplication step). Use it whenever you know there are no duplicates, or you want to keep them.

Every customer_id that appears in customer OR payment
SQLite · dvdrental schema
Same data, but keep every occurrence
SQLite · dvdrental schema

UNION returns 10 rows (the 10 distinct customer_ids). UNION ALL returns 24 (10 from customer + 14 from payment, no deduplication).

A realistic use case

Imagine your sales come from two channels — online and in-store — and they are recorded in separate tables. UNION ALL lets you analyze them as one combined stream without copying data around:

SELECT order_id, amount, 'online'   AS channel FROM online_orders
UNION ALL
SELECT order_id, amount, 'in_store' AS channel FROM store_orders;

The hardcoded 'online'/'in_store' literal becomes a column in the result, tagging where each row came from. This pattern — adding a constant column to disambiguate sources — is essential when UNIONing similar tables.

Common mistakes

Column count mismatch

SELECT first_name FROM customer
UNION
SELECT first_name, last_name FROM customer;   -- error: each UNION query must have the same number of columns

Fix: pick a column count and stick with it across both SELECTs.

Column type mismatch

SELECT customer_id FROM customer     -- integer
UNION
SELECT first_name  FROM customer;    -- text

Some databases error. Some implicitly cast everything to text. Either way, the result is junk. Make sure the i-th column of each SELECT has the same meaning and type.

Using UNION when you meant JOIN

If you find yourself wanting to "combine" customer and payment, you probably want JOIN, not UNION. UNION stacks them; JOIN matches them up.

Using UNION when you meant UNION ALL

If you know there are no duplicates (e.g. UNIONing rows from two different tables that share no IDs), UNION ALL is faster and clearer. Reach for UNION only when you actually want deduplication.

Solution 4.3.2

SELECT first_name, 'customer' AS source FROM customer;

Solution 4.3.3

UNION is right — you want two values from the same row stacked into one column:

SELECT first_name AS name FROM customer
UNION ALL
SELECT last_name  AS name FROM customer;

A JOIN would not make sense here — there is only one table involved, and you want the values in one column, not side-by-side.

Solution 4.3.4

Column count mismatch: 2 columns in the first SELECT, 1 in the second. Either drop a column from the first or add a placeholder to the second:

SELECT customer_id, first_name FROM customer
UNION
SELECT NULL,        amount     FROM payment;   -- compiles, but the result is weird; ask whether you really wanted UNION

The fact that fixing the column count produces nonsense is itself a hint: this probably should have been a JOIN, not a UNION.

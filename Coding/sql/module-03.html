Module 3 of 10

Aggregating Data

Stop thinking row-by-row. Start asking questions that collapse many rows into one — totals, averages, counts per group, filtered by what those totals look like.
Estimated time: 75–100 minutes · Prerequisites: Modules 1 and 2 complete. Comfort with SELECT, WHERE, and the COUNT function.

What this module covers

So far, every query returned one result row per input row. This module teaches the second great pattern of SQL: collapsing many rows into one — or many groups into many summary rows.

  • Aggregation functions — SUM, AVG, MIN, MAX, and revisiting COUNT. They take many rows and return one number.
  • GROUP BY — apply an aggregation per category instead of across the whole table.
  • HAVING — filter on aggregated values (which WHERE cannot do).

This is where SQL stops feeling like "fetch some rows" and starts feeling like "answer a business question".

The SQL funnel — two more layers

We are now descending into the middle of the funnel:

GROUP BY and HAVING are stacked: HAVING only makes sense in the presence of GROUP BY. Together they let you say "for each category, compute something — but only show me the categories where that something looks a certain way".

The mental shift this module asks of you

So far, one input row produced one output row. From this module on, that stops being true. A GROUP BY query can take 10 million rows and return 5. You are no longer reading data — you are summarizing it.

This is the single biggest conceptual jump in SQL. Once you have it, everything past this point (window functions, CTEs, analytics) builds on it.

Lesson 3.1 — Aggregation functions

Functions that take many rows of input and return one row of output.

Why this matters

In Module 1 you learned COUNT(*) — your first aggregate. Aggregates feel different from regular SELECT because they violate the "one input row, one output row" rule. Instead, they fold a whole column down to a single value.

There are five you will use constantly. Learn them in one sitting:

Function Returns
COUNT(...) Number of rows (revisited from Module 1)
SUM(col) Sum of all values in a numeric column
AVG(col) Mean of all values in a numeric column
MIN(col) Smallest value (works on numbers, dates, and text)
MAX(col) Largest value (works on numbers, dates, and text)

The "you can't have both" rule

What does this query mean?

SELECT title, AVG(length) FROM film;

It is asking for every film title (15 rows) AND one average across all films (1 row). Those two answers cannot fit in the same result table.

PostgreSQL refuses to run it. SQLite (this playground) will run it but return nonsense — picking a single title arbitrarily and pairing it with the average. Either way: you cannot mix aggregate and non-aggregate columns in a SELECT unless you tell SQL how to group them. That is what GROUP BY is for, and it is the entire content of Lesson 3.2.

Why this rule exists

Aggregates produce one row. Regular columns produce one row per input row. SQL has no way to combine those shapes without instructions. GROUP BY is the instruction: "compute the aggregate per category, and pair each result with that category". After GROUP BY, the shapes match — one row per category.

Common mistakes

SUMming something that is not a number

SELECT SUM(title) FROM film;   -- error or weird result

title is text. SUM expects numbers. Most databases will reject this; SQLite will return 0 (it casts unparseable text to 0 silently). Always check the column's data type.

Mixing aggregate and non-aggregate columns

Covered above. This is the most common GROUP BY mistake by far.

Forgetting that AVG ignores NULLs

AVG(salary) averages the rows where salary is not NULL. If 200 of 1000 employees have NULL salary, the average is over 800 rows, not 1000. Sometimes that is what you want — sometimes you need COALESCE(salary, 0) first to count NULLs as zero. Decide deliberately.

Lesson 3.2 — GROUP BY

Apply an aggregation per category instead of across the whole table.

Why this matters

SELECT SUM(amount) FROM payment gives you total revenue. That is useful, but the moment someone says "OK, but per customer?", you need GROUP BY.

GROUP BY answers questions of the form "for each X, what is the [count / total / average / min / max] of Y?". That covers a huge fraction of every business question ever asked.

The shape of a grouped query

SELECT  grouping_column, AGG(other_column)
FROM    table
WHERE   condition          -- optional, filters BEFORE grouping
GROUP BY grouping_column
ORDER BY ...               -- optional
LIMIT n;                   -- optional

The grouping column appears twice: once in SELECT (so you can see which group is which), once in GROUP BY (the instruction).

Count of films per rating
SQLite · dvdrental schema

One row per distinct rating value. Five rows back (since there are five ratings).

The golden rule

Every non-aggregate column in SELECT must also appear in GROUP BY.

If you SELECT a column that is not aggregated and not grouped, what would each row pair it with? There is no single answer. PostgreSQL errors. SQLite returns garbage. Either way, you have to obey:

-- ILLEGAL: title is neither aggregated nor grouped
SELECT title, rating, COUNT(*) FROM film GROUP BY rating;

-- LEGAL: every non-aggregate is in GROUP BY
SELECT rating, COUNT(*) FROM film GROUP BY rating;

-- ALSO LEGAL: group by both, get one row per (title, rating)
SELECT title, rating, COUNT(*) FROM film GROUP BY title, rating;

The rule, restated

If a column appears in SELECT but is not wrapped in an aggregate function, it must appear in GROUP BY. No exceptions.

This is the single most common error new SQL writers hit. When you get a "column must appear in GROUP BY" error, do not panic — just add the column.

Grouping by multiple columns

GROUP BY can take a comma-separated list. The result is one row per combination:

WHERE filters the raw rows before grouping. This is what you want when the filter is on a per-row attribute:

This is the entire intuition for Lesson 3.3. Hold onto it.

Common mistakes

Forgetting the column in GROUP BY

SELECT rating, AVG(length) FROM film;
-- PostgreSQL: error. Add: GROUP BY rating

The fix is mechanical — copy the non-aggregate column into a GROUP BY clause.

Putting aggregates in GROUP BY

SELECT rating, COUNT(*) FROM film GROUP BY rating, COUNT(*);   -- nonsense

You group by raw columns. Aggregates are the result of grouping, not part of the instruction.

Trying to filter aggregates in WHERE

SELECT rating, COUNT(*) FROM film WHERE COUNT(*) > 3 GROUP BY rating;
-- error: cannot use aggregate in WHERE

WHERE runs before aggregation. The count does not exist yet. You need HAVING — see Lesson 3.3.

Lesson 3.3 — HAVING

Filter on aggregated values. WHERE cannot do this.

Why HAVING exists

You learned in Module 2 that WHERE filters rows. So why is there a second filter clause?

Because WHERE runs before aggregation. By the time WHERE looks at the data, no aggregates exist yet — there are no counts, no totals. You cannot ask "give me customers whose total spend exceeds $10" with WHERE, because the total has not been computed.

HAVING runs after GROUP BY and aggregation. By the time HAVING looks at the data, each group is a single row with its aggregates already computed. Now you can filter on them.

Customers who paid more than $5 in total
SQLite · dvdrental schema

Read that bottom to top: group payments by customer, sum each customer's amount, keep only the customers whose sum exceeds 5.

HAVING vs WHERE — the one-line rule

Clause Filters Runs Can reference aggregates?
WHERE Individual rows Before GROUP BY No
HAVING Whole groups After GROUP BY Yes

They can both appear in the same query. WHERE drops rows before grouping; HAVING drops groups after.

Both filters in one query
SQLite · dvdrental schema

What HAVING can reference

HAVING can reference: - The grouping columns (HAVING rating = 'PG') - Aggregate expressions (HAVING COUNT(*) > 5, HAVING SUM(amount) BETWEEN 10 AND 100) - Combinations with AND/OR

You can use any aggregate function in HAVING, not just the one in SELECT:

Ratings whose average film length exceeds 100
SQLite · dvdrental schema

Notice — AVG(length) is in HAVING but not SELECT. That is fine. The database computes both, uses one for filtering, returns the other.

Style: where to put the filter

If a filter can go in WHERE, put it in WHERE. Only use HAVING for things WHERE cannot do (filtering aggregates).

WHERE rating = 'G' GROUP BY ... is faster than GROUP BY rating HAVING rating = 'G' because the WHERE version drops rows before doing the (expensive) grouping work.

Common mistakes

Putting aggregates in WHERE

WHERE SUM(amount) > 5   -- error: aggregate not allowed in WHERE

Use HAVING.

Putting per-row conditions in HAVING

GROUP BY customer_id HAVING amount > 5

This errors or behaves strangely. amount is per-row, not per-group. Put it in WHERE.

Using HAVING without GROUP BY

PostgreSQL technically allows this — it treats the whole table as one group. Rarely useful and confusing to read. If you find yourself writing HAVING without a GROUP BY, you probably wanted WHERE.

Referencing a SELECT alias in HAVING

SELECT customer_id, SUM(amount) AS total
FROM payment
GROUP BY customer_id
HAVING total > 5;   -- PostgreSQL: error. SQLite: works.

Some databases allow the alias in HAVING, some require the full expression HAVING SUM(amount) > 5. PostgreSQL requires the full expression. Write it the long way for portability.

Solution 3.3.2

FROM payment
GROUP BY customer_id
HAVING SUM(amount) >= 10;

Solution 3.3.3

SELECT rating, COUNT(*) AS n, AVG(length) AS avg_len
FROM film
WHERE length > 60
GROUP BY rating
HAVING AVG(length) > 100;

This uses every clause you have learned so far: SELECT (with aggregates), FROM, WHERE, GROUP BY, HAVING. Read it top to bottom — that is literally the order SQL evaluates.

Solution 3.3.4

You cannot put aggregates in WHERE — WHERE runs before aggregation. Use HAVING:

SELECT customer_id, SUM(amount) FROM payment
GROUP BY customer_id
HAVING SUM(amount) > 5;

Solution 3.3.5

SELECT customer_id, SUM(amount) AS total
FROM payment
GROUP BY customer_id
ORDER BY total DESC
LIMIT 2;

This is the "top N by aggregate" pattern, possibly the single most common analytics query you will ever write. Memorize the shape.

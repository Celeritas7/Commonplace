Retrieving Data

Three foundational ways to pull data out of a SQL database. Every query you will ever write builds on these three skills.
Estimated time: 60–90 minutes · Prerequisites: pgAdmin installed, dvdrental sample database loaded — or use the in-page playground

What this module covers

Three foundational ways to pull data out of a SQL database. Every query you will ever write builds on these three skills.

  • SELECT — the universal command for retrieving any column from any table.
  • SELECT DISTINCT — retrieve only the unique values, ignoring duplicates.
  • COUNT — count rows, with subtle but important behavior around NULLs and duplicates.

The SQL funnel

Every SQL query you write follows the same syntactic stack. You write the clauses in a fixed order from top to bottom, and each layer narrows the result further than the one above it. Memorize this funnel and you have memorized the structure of SQL.

Module 1 lives entirely at the top of the funnel — the SELECT layer. We will not touch WHERE, GROUP BY, or anything below it. That is intentional: master the act of retrieving data first, then learn how to filter it.

Lesson 1.1 — SELECT

The single most-used statement in SQL. Pull any columns from any table.

Why this matters

SELECT is the statement you will type more than any other in SQL. Reading data is what databases are built for, and almost every analysis, dashboard, report, and API call ultimately resolves into a SELECT query somewhere.

Once SELECT clicks, the rest of SQL — WHERE, GROUP BY, JOIN, subqueries — is just additions to this single core skeleton.

The shape of a query

Every basic SELECT query answers two questions:

  • Which columns do I want back?
  • From which table do they come?

That is the entire idea. The syntax mirrors it directly:

SELECT column_name
FROM table_name;

How PostgreSQL actually reads this

You write SELECT first, but the engine resolves FROM first. It must know which table it is dealing with before it can look up your columns. This is why a typo in the table name surfaces before a typo in the column name.

Pulling many columns

Separate column names with commas. They come back in the order you list them — not the order they sit in the table:

Custom column order
SQLite · dvdrental schema

Order is a feature, not a constraint

When building reports or feeding data into another tool, list columns in the order you want them displayed. Do not rely on the table's internal column order — it can change as the schema evolves.

Selecting every column with *

The asterisk is shorthand for "every column in this table, in the order the table defines them."

Star (every column)
SQLite · dvdrental schema

It is convenient when you are exploring an unfamiliar table and want a quick look at its shape. It is the wrong choice almost everywhere else.

Avoid SELECT * in real queries

  • It returns every column, including ones you do not need — wasting bandwidth between the database and your application.
  • It hides intent. A reader cannot tell which columns your code actually depends on.
  • It breaks silently when the schema changes. A new column added upstream can suddenly appear in your output and downstream parsers may misalign.

Prefer naming columns explicitly, even when there are many of them.

Style and readability

SQL is lenient. Capitalization, spacing, and semicolons are mostly optional — but consistent style is what makes a query readable six months later.

Convention Required? Why bother
UPPERCASE keywords No SELECT, FROM, WHERE stand out from your column and table names at a glance.
Trailing semicolon No Marks the end of the statement clearly — essential when you chain queries.
One clause per line No Makes diffs in version control readable and edits less error-prone.

Common beginner mistakes

Missing comma between columns

SELECT first_name last_name FROM actor;   -- silent rename!

Without the comma, SQL interprets last_name as an alias for first_name and returns only one column. The query runs without error — the worst kind of bug.

Trailing comma

SELECT first_name, last_name, FROM actor;   -- syntax error

A comma after the last column is a syntax error. Watch for it especially when editing column lists.

Forgetting FROM

SELECT first_name;   -- error: missing FROM-clause entry

Every column reference needs a table. There are exceptions for expressions and built-in functions, but for real data you always need FROM.

Lesson 1.2 — SELECT DISTINCT

Strip duplicate values from your result. Still at the SELECT layer of the funnel.

The problem DISTINCT solves

Real tables almost always contain duplicate values in some columns. The film table has thousands of rows, but only five distinct rating values (G, PG, PG-13, R, NC-17). If you only want to know which ratings exist, plain SELECT is wasteful:

The wasteful way
SQLite · dvdrental schema

You wanted five rows. You got fifteen. DISTINCT fixes this in one keyword.

Syntax

SELECT DISTINCT column_name
FROM table_name;

DISTINCT goes immediately after SELECT and before the column list. It applies to whatever follows it.

DISTINCT on one column
SQLite · dvdrental schema

See it work on dvdrental

SELECT DISTINCT rating FROM film;       -- returns 5 rows
SELECT DISTINCT name FROM category;     -- returns 16 distinct categories
SELECT DISTINCT country FROM city;      -- one row per country

DISTINCT across multiple columns

You can apply DISTINCT to a combination of columns. The result is one row per unique combination — not per unique value in each column separately:

DISTINCT across two columns
SQLite · dvdrental schema

This returns rows like (G, 3), (G, 4), (PG, 3), (PG, 5), etc. — every unique pair, not every unique rating or every unique duration.

DISTINCT applies to the whole row, not just one column

A common misconception is that DISTINCT only deduplicates the first column. It does not. It treats the entire selected tuple as the unit of uniqueness.

Common use cases

DISTINCT is most useful for three kinds of questions:

  • What categories exist? — list every unique rating, genre, country, status, etc.
  • How varied is this column? — is the column nearly all the same value, or genuinely diverse?
  • Are there unexpected values? — a quick sanity check before deeper analysis.

Common mistakes

Expecting DISTINCT to apply to one column when several are listed

SELECT DISTINCT rating, title FROM film;
-- This returns ~15 rows, not 5.
-- Why? Every title is unique, so every pair (rating, title) is also unique.

If you wanted just the five distinct ratings, drop the title column.

Using DISTINCT to fix a bad join

If a JOIN produces duplicate rows, slapping DISTINCT on the front silences the symptom but hides the real problem — a missing join condition or a one-to-many relationship you did not account for. Investigate the join, do not paper over it.

DISTINCT vs GROUP BY confusion

GROUP BY also deduplicates and is more powerful (it lets you aggregate). For simple "give me unique values" questions, DISTINCT is shorter and clearer. You will meet GROUP BY in Module 3.

Solution 1.2.2

One row for every unique (customer_id, amount) pair. If customer 5 paid $4.99 ten times, that becomes one row. If customer 5 paid $4.99 once and $9.99 once, that is two rows. The count depends on the data, not on the row count of the source table.

Solution 1.2.3

The query is fine. The issue is that two different customers can share the same first_name + last_name. DISTINCT only collapses exact duplicates. If you want truly unique customers, select customer_id too — or accept that names are not unique identifiers.

Lesson 1.3 — COUNT

Your first aggregate function. Lives inside SELECT, takes a column or *.

Why we count

Before any analysis, you almost always want to know how big the dataset is. How many customers? How many orders this month? How many distinct products? COUNT answers all of these.

COUNT is your first encounter with an aggregate function — a function that takes many rows of input and returns one row of output. Aggregates feel different from regular SELECT, and learning the mental shift now makes Module 3 (GROUP BY) much smoother.

Three flavors of COUNT

COUNT comes in three forms that look almost identical and behave very differently:

Form Counts
COUNT(*) Every row, including rows where every column is NULL.
COUNT(column) Rows where that specific column is NOT NULL.
COUNT(DISTINCT column) Unique non-NULL values in that column.

COUNT(*) — total rows

The most common form. Asks "how many rows are in this table?"

COUNT(*) — total rows
SQLite · dvdrental schema

COUNT(*) is fast — PostgreSQL has efficient ways to count rows without inspecting any column values. Use it whenever you just want a row total.

COUNT(column) — non-NULL rows

Looks similar but means something different. Counts only rows where the named column has a value (is not NULL).

COUNT(column) — non-NULL only
SQLite · dvdrental schema

In our customer table, two customers have NULL email and eight have one — so COUNT(email) returns 8, while COUNT(*) returns 10.

The NULL trap

COUNT(column) silently ignores NULL rows. If you are checking data completeness, that is exactly what you want — the difference between COUNT(*) and COUNT(email) tells you how many emails are missing.

But if you just want a row total, never write COUNT(column_that_can_be_null). It will under-count without warning. Use COUNT(*) instead.

COUNT(DISTINCT column) — unique values

Combines the lessons of Module 1.2 and 1.3. Counts how many unique non-NULL values exist in a column.

COUNT(DISTINCT) — uniques
SQLite · dvdrental schema

This returns 5 — the count of distinct ratings. The single most useful combination in exploratory data analysis. "How many distinct X are there?" is a question you will ask constantly.

Naming the result

By default the result column is named count — which is unhelpful in a report. Use AS to give it a meaningful name:

Aliasing the count
SQLite · dvdrental schema

AS is your first taste of column aliasing

AS lets you rename a column in the result without changing the underlying data. You will use it constantly with aggregates and JOINs. For now: AS new_name goes right after the expression you want to rename.

Common mistakes

Confusing COUNT(*) with COUNT(column)

Reviewing someone's code and seeing COUNT(specific_column) is a flag to check whether they meant "total rows" or "rows with a value". 90% of the time they meant the first and should have written COUNT(*).

Counting in the WHERE clause

SELECT * FROM film WHERE COUNT(*) > 10;   -- error

You cannot use COUNT inside WHERE. WHERE filters rows before aggregation, so the count does not exist yet. To filter on a count, you need HAVING — which works only with GROUP BY (Module 3).

Forgetting AS in reports

A column named "count" in a CSV export gets confusing fast. Always alias aggregates in any output that someone else will read.

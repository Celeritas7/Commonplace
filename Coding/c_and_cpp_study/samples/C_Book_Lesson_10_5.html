Undefined Behaviour

C's speed comes from trusting you to follow the rules. Break one — read past an array, use freed memory, overflow a signed int — and you've entered undefined behaviour: the standard imposes no requirements on what happens. It might crash, print garbage, or seem to work perfectly until the day it doesn't. Knowing UB, and how to catch it, is what separates fragile C from solid C.

1 Why this matters

The standard's shrug 🎲

C trades safety for speed: it doesn't check array bounds, doesn't track whether memory is still valid, doesn't catch integer overflow — that would cost performance. The deal is that you promise to play by the rules. When you break one, the program has undefined behaviour: the language standard says, in effect, "we make no guarantees about what happens." Not a crash you can rely on, not a defined error — anything. The truly dangerous part is that UB often appears to work — until a different compiler, optimization level, input, or machine makes it fail, sometimes as a security hole. This is the most important safety topic in the whole C book.

The goal isn't fear — it's recognising the handful of rules that, if broken, void all guarantees.

2 What UB actually means

No requirements, none 🤷

It helps to distinguish UB from its milder cousins. Only undefined behaviour means "literally anything goes":

KindWhat the standard guarantees
implementation-definedsome defined choice, documented (e.g. size of int)
unspecifiedone of a few allowed options, not documented which
undefinednothing — any result, including none
With UB the compiler is even allowed to assume it never happens and optimize on that basis — which is why UB can delete your bounds check, reorder code, or make a loop run forever. There's no required diagnostic: the program may compile cleanly with no warning and still be broken. So "it compiled and ran" tells you nothing about whether you've triggered UB. The only safe move is to not write it — and to use tools that catch it (section 5).
3 Classic #1 — out of bounds

Reading past the edge 📏

Accessing an array outside its valid indices — even just reading — is undefined. C does no bounds checking:

int a[3] = {10, 20, 30};   // valid indices: 0, 1, 2
int x = a[5];              // 🔴 out of bounds — UB
a[3] = 99;                  // 🔴 writing past the end — UB (and dangerous)
Reading a[5] might return whatever bytes happen to be there (garbage), crash, or — with optimizations — do something stranger. Writing out of bounds (a "buffer overflow") is worse: it corrupts whatever memory sits next to the array, a leading cause of crashes and security exploits. The fix is discipline: never index past length - 1, and carry the size alongside the array so loops can check it (for (i = 0; i < n; i++)).
4 Classic #2 — dead memory

Using what's gone 🗑️

Two relatives here: using memory after you've freed it (dangling pointer), and reading a variable you never initialised:

int *p = malloc(sizeof *p);
free(p);
*p = 5;          // 🔴 use-after-free — p no longer owns that memory

int y;            // uninitialized
printf("%d", y);   // 🔴 reading an indeterminate value — UB
After free(p), the pointer still holds an address, but that memory may be reused for anything — so reading or writing through it is UB. The same goes for a variable that was never given a value: it doesn't quietly default to 0; its bits are indeterminate, and merely using the value is undefined. Defences: set pointers to NULL after free (so a stray use crashes loudly instead of silently corrupting), and always initialise variables when you declare them.
5 Classic #3 — signed overflow & how to avoid UB

Overflow, and your toolkit 🛡️

Overflowing a signed integer is UB — it is not guaranteed to wrap around (unsigned arithmetic does wrap, by definition):

int x = INT_MAX;
int y = x + 1;        // 🔴 signed overflow — UB (NOT guaranteed INT_MIN)

unsigned u = UINT_MAX;
u = u + 1;            // ✅ defined: wraps to 0 (unsigned wraps by the rules)

🚨 "It worked when I ran it" proves nothing

The deadliest myth about UB is that a program which runs and gives the right answer is therefore correct. UB can lie low for months: it might happen to work at -O0 but break at -O2, work on your machine but fail on a user's, or work today and corrupt data after one more code change — because the compiler is permitted to assume UB never occurs and optimize accordingly. You can't test UB away; you avoid it. Your toolkit: compile with -Wall -Wextra (catches many cases), run under a sanitizer like -fsanitize=address,undefined (AddressSanitizer/UBSan flag out-of-bounds, use-after-free, and overflow at runtime), use valgrind for memory errors, initialise everything, and NULL out freed pointers. These turn silent UB into a loud, locatable error.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

✅
Trusting "it worked when I tested it" UB can pass every test and still be a bug — it may break at a higher optimization level, on another machine, or after an unrelated change. You avoid UB; you can't test it away.
🧯
Not turning on warnings or sanitizers Build with -Wall -Wextra and run under -fsanitize=address,undefined (or valgrind). They turn silent UB into a loud, located error.
0️⃣
Assuming uninitialized = 0, or signed overflow wraps Local variables aren't auto-zeroed, and signed overflow is UB (only unsigned wraps). Initialise everything; use unsigned or wider types when wrap/overflow is a real risk.

Practice exercises 💪

Easy Catch it with a sanitizer. Write a tiny program that reads a[5] from a 3-element array. Compile normally and run (it may "work"); then compile with -fsanitize=address and run again to see it reported.
Medium Dangle on purpose. malloc an int, free it, then read through the pointer. Run under AddressSanitizer to see "heap-use-after-free." Then fix it by setting the pointer to NULL after free and checking before use.
Stretch Overflow, defined vs not. Add 1 to INT_MAX and to UINT_MAX and print both. Build with -fsanitize=undefined and observe that the signed case is flagged while the unsigned one is defined to wrap. Note how you'd detect overflow safely (check before adding).

🎓 What to remember

  • Undefined behaviour = no guarantees — crash, garbage, or "works," with the compiler free to assume it never happens.
  • It's worse than a crash because it can silently appear to work, then fail on another build, machine, or day.
  • Classic causes: out-of-bounds access, use-after-free / uninitialized reads, and signed integer overflow.
  • You avoid UB, you don't test it away — bounds-check, initialise, NULL freed pointers.
  • Tools catch it: -Wall -Wextra, AddressSanitizer/UBSan (-fsanitize=...), and valgrind.

🧠 PART VII · MEMORY MANAGEMENT

malloc and free

The heap is yours to use — and these two functions are the door. malloc requests a block of memory at runtime and hands you a pointer to it; free gives it back. Master the pattern (request → check → use → free → null) and you can build data that outlives any function.

1 Why this matters

Asking the heap for memory 🧠

Last lesson you learned why the heap exists — for memory that outlives a function or whose size you only learn at runtime. malloc is how you actually get it: you ask for a number of bytes, and it carves out a block and returns a pointer to it. When you're done, free hands the block back. This pair is the heart of manual memory management in C.

The mechanics are small, but the discipline matters: what you take, you must return. We'll build the standard, safe pattern step by step.

2 malloc — request bytes

How many bytes? Ask sizeof 📦

malloc(n) asks for n bytes and returns a pointer to the start of the block (or NULL if it couldn't). You almost never count bytes by hand — you let sizeof do it:

#include <stdlib.h>   // malloc, free live here

int *p = malloc(sizeof(int));   // enough room for one int
sizeof(int) gives the right byte count for an int on this machine, so the request is always correct. The returned block is uninitialized — it holds leftover garbage until you put something there. (In C you don't need to cast malloc's result; it returns a void * that converts to any pointer type.)
3 Check, then use

Did it succeed? ⚠️

malloc can fail (return NULL) if memory runs out. Before you use the pointer, check it — then read and write through it like any pointer:

int *p = malloc(sizeof(int));
if (p == NULL) return 1;   // out of memory — bail out safely

*p = 42;                 // now the block holds 42
printf("%d\n", *p);       // 42
The NULL check matters because dereferencing NULL crashes (you saw this in 6.8). For tiny requests it almost never fails, but checking is the habit to build — especially for big allocations. Once checked, the heap block behaves exactly like any other memory you point at.
4 free — give it back

Every malloc needs a free 🧹

Heap memory stays yours until you release it. When you're finished, call free on the pointer — and then set the pointer to NULL, because after freeing it dangles (6.8):

free(p);     // return the block to the heap
p = NULL;    // p was dangling — make it safe

🚨 Forget to free = a memory leak

If you never call free, the heap never reclaims the block — your program holds memory it no longer uses. That's a memory leak (the subject of 7.4). The rule is simple and strict: every malloc must have a matching free. And never use a pointer after freeing it (use-after-free) — setting it to NULL right after free protects you.

5 A runtime-sized array

malloc(n * sizeof(int)) 🗄️

Here's the payoff from 7.1: an array whose size is decided at runtime. Ask for room for n elements, and index the result exactly like a normal array:

int n = 5;                          // could come from user input
int *a = malloc(n * sizeof(int));   // room for n ints
if (a == NULL) return 1;

for (int i = 0; i < n; i++)
    a[i] = i * i;                   // use it like any array

free(a);                            // one free for the whole block
The block is one contiguous chunk, so a[i] works through the same pointer arithmetic as a stack array (6.4) — but its size came from a variable, which a stack array can't do. One malloc gets the whole array; one free releases it all. This is your first dynamic array, and 7.5 builds a complete one.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🧹
Forgetting to free Every malloc needs a matching free, or the block leaks — the heap won't reclaim it for you. One allocation, one release.
💥
Use-after-free (or reading uninitialized memory) After free the pointer dangles — don't dereference it. And a fresh malloc block is garbage until you set it. Initialize before reading; set the pointer to NULL after freeing.
⚠️
Not checking malloc's return malloc returns NULL if it fails. Dereferencing NULL crashes — check if (p == NULL) before using the block, especially for large requests.

Practice exercises 💪

Easy One int. malloc space for a single int, store a value, print it, then free the block and set the pointer to NULL. Walk through each line's purpose.
Medium A dynamic array. Read n from a variable, malloc(n * sizeof(int)), fill it in a loop, print it, and free it. Notice the size came from a variable — a stack array can't do that.
Stretch The safe pattern. Write the full sequence with comments: malloc → check for NULL → use → free → set NULL. Explain in one line each why every step is there.

🎓 What to remember

  • malloc(size) requests size bytes from the heap and returns a pointer (or NULL). Use sizeof for the size.
  • The block is uninitialized — set values before reading them.
  • Check the return for NULL before using it.
  • Every malloc needs a matching free; after freeing, set the pointer to NULL to avoid dangling.
  • Allocate an array with malloc(n * sizeof(type)) and index it normally — a runtime-sized array on the heap.

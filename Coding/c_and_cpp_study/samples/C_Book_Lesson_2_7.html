Type Conversion — Silent and Explicit

When you write 3.0 + 5, C quietly turns the 5 into a double first. Most conversions happen automatically — but a few throw away data without a peep. This lesson is C's conversion rules and the (type) cast that puts you back in control.

1 Why this matters

Different types, one expression 🧠

C is fussy about types in memory — an int and a double are stored completely differently. Yet you mix them all the time: area = 3.14159 * r * r; where r is an int. For that to work, C has to convert one type into another. Most of the time it does this for you, silently, following fixed rules.

That silence is usually a convenience — but it's also where a whole family of beginner bugs hides. A conversion can quietly drop the decimals off a number, wrap a value past the edge of its type, or flip a comparison upside down. Knowing when C converts and what it costs is the difference between code that works and code that's subtly wrong. And when the automatic behavior isn't what you want, the (type) cast lets you take the wheel.

2 Implicit conversions

The conversions C does for you 🪄

An implicit conversion happens with no special syntax. There are two situations where C inserts one automatically:

WhenExampleWhat happens
Assignmentdouble d = 5;The int 5 becomes 5.0
Assignmentint n = 3.9;The 3.9 is truncated to 3
Mixed arithmetic3.0 + 5The 5 is converted to 5.0 first
Function argumentssqrt(9)The int 9 becomes 9.0

When two different types meet in an arithmetic expression, C follows the usual arithmetic conversions — a ranking that picks one common type for both operands. The simple version you'll use every day:

RankTypesRule of thumb
HighestdoubleIf either side is double, both become double
↓floatElse if either is float, both become float
↓longElse the smaller integer is widened toward the larger
Lowestintchar and short are always promoted to int first
Integer promotion is the one beginners miss: anything smaller than int (a char or short) is promoted to int before arithmetic even starts. That's why 'A' + 1 is the int 66, not a char. The result only becomes a char again if you store it back into one.
3 Where silence bites

The two conversions that cause real bugs 💥

🚨 Integer division before the conversion

double avg = sum / count; with two ints computes sum / count as integer division first — remainder discarded — and only then widens the result to double. 7 / 2 becomes 3.0, not 3.5. The conversion happens too late to help. Cast one operand before dividing: (double)sum / count.

🚨 Signed meets unsigned

When a signed and an unsigned value meet, C converts the signed one to unsigned — and a negative number wraps into a huge positive. So -1 < 1u is false: -1 becomes the largest unsigned value, which is not less than 1. This bites hard in loops that count down with an unsigned index.

Both traps share a root cause: the conversion is invisible, so you don't notice it changed your math. The fixes are equally simple — convert deliberately with a cast, and keep signed and unsigned values from mixing in comparisons. The Mixed-Type Calculator below lets you reproduce both live.

4 The explicit cast

(type)value — taking control 🔧

When you want a conversion that C wouldn't do on its own — or you want it to happen now, not later — you write an explicit cast. The syntax is the target type in parentheses, in front of the value:

CastEffect
(double)sum / countForces floating-point division — fixes the average bug
(int)3.9Truncates to 3 (toward zero), on purpose and visibly
(char)65Turns the code 65 into the character 'A'
(unsigned)xReinterprets x using unsigned rules
A cast produces a new value — it never changes the original variable. (double)sum hands the expression a double copy of sum; sum itself stays an int. And a cast binds tightly (it's a unary operator), so (double)sum / count casts only sum, then divides. That's exactly what you want.

One honest caveat: C's cast is a blunt instrument. It will happily do dangerous conversions — like turning a pointer into an integer — without complaint. Use it for the everyday numeric conversions above, and be suspicious of any cast that's doing something more exotic. (C++ later splits this one cast into four safer, named ones — but in C, this is the tool.)

5 Truncation, overflow, safety

What survives a conversion ⚖️

Whether a conversion is safe comes down to whether the destination type can hold the value:

  • Widening is safe. int → long, int → double, char → int — the bigger type holds everything. No data lost.
  • Float → integer truncates. The fractional part is chopped off toward zero. (int)3.9 is 3; (int)-3.9 is -3. It does not round.
  • Narrowing integers can overflow. (char)300 doesn't fit in a signed 8-bit char, so it wraps around to 44. The value is mangled, usually silently.
  • Any nonzero → a true condition. In an if, C treats any nonzero value as true and 0 as false.
The mental checklist: is the destination at least as roomy as the source? If yes, the conversion is lossless and you can relax. If no, something might be thrown away — make it a deliberate cast so the next reader knows you meant it, and double-check the value actually fits.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

➗
Dividing two ints and expecting a fraction double r = a / b; does integer division first, so 7 / 2 is 3.0. Cast one operand before the division: (double)a / b gives 3.5.
🔀
Comparing signed and unsigned for (unsigned i = n; i >= 0; i--) never ends — an unsigned is always >= 0, and it wraps to a huge number instead of going negative. Keep counters signed, or rethink the loop.
📦
Stuffing a big value into a small type char c = 300; doesn't fit and wraps silently. If you must narrow, cast on purpose — (char)x — and make sure the value is in range first.

🎓 What to remember

  • C converts types automatically in assignments, mixed arithmetic, and function calls — usually invisibly.
  • Usual arithmetic conversions: if either operand is double, both become double; char/short are always promoted to int first.
  • Two silent killers: integer division before widening, and signed-vs-unsigned comparisons that wrap.
  • The (type) cast makes a conversion deliberate and visible — and binds tightly, so (double)a / b casts only a.
  • Widening is safe; narrowing and float→int lose data. Ask: is the destination roomy enough for the value?

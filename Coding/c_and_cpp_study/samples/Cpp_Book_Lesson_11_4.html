📂 PART XI · I/O AND FILES

Binary Files & Random Access

Text I/O is readable but bulky and lossy for raw data. Binary I/O writes an object's exact bytes with .write() and reads them back with .read() — fast and exact. And with seekg/seekp you can jump straight to any position, turning a file into a random-access store of fixed-size records.

1 Why this matters

Bytes, not text 💾

Everything so far has been text I/O: << turns a number into its decimal digits, >> parses them back. That's readable, but for raw data it's wasteful (the int 1000000 takes 7 characters as text, 4 bytes as binary) and can lose precision when round-tripping floating-point. Binary I/O skips the translation: it copies an object's exact bytes to disk and back. The tools are the stream member functions .write() and .read(), used on a stream opened with std::ios::binary. Pair that with random access — seekg/seekp to move the read/write position anywhere in the file — and a file becomes an array of fixed-size records you can index into directly.

This closes Part XI: you'll have the full I/O toolkit — console, files, string streams, formatting, and now raw bytes with positioning.

2 Writing bytes

.write() with ios::binary ✍️

ostream::write takes a const char* and a byte count. To write any object, hand it the object's address (cast to const char*) and its sizeof:

#include <fstream>

struct Rec { int id; double score; };   // plain data — safe to copy as bytes
Rec r{ 7, 9.5 };

std::ofstream out("data.bin", std::ios::binary);     // binary mode!
out.write(reinterpret_cast<const char*>(&r), sizeof r);
Three pieces matter. std::ios::binary turns off any text translation (on Windows, text mode would rewrite newline bytes — disastrous for binary data). reinterpret_cast<const char*>(&r) reinterprets the object as a raw byte buffer — write deals in bytes, so it needs a char*. And sizeof r is exactly how many bytes that object occupies. The result is a chunk of sizeof(Rec) bytes appended to the file — an exact image of the object in memory. Writing an array or a std::vector's buffer works the same way: point at the data, pass the total byte size.
3 Reading bytes

.read() back into an object 📖

istream::read is the mirror image: give it the destination address (as char*) and the byte count, and it fills the object:

std::ifstream in("data.bin", std::ios::binary);

Rec r;
in.read(reinterpret_cast<char*>(&r), sizeof r);   // fill r from the bytes

if (in)                       // did we get all sizeof(r) bytes?
    use(r.id, r.score);
// in.gcount() tells how many bytes were actually read
Because the bytes are an exact copy, read reconstructs the object perfectly — r.id == 7, r.score == 9.5, no parsing, no rounding. Always check the stream afterward: if the file ended early, read sets the fail state (and in.gcount() reports how many bytes it actually managed to read). The cast here drops the const — you're writing into r — but it's the same idea: treat the object as a byte buffer just long enough to copy into it. Read in the same struct layout you wrote, and you get your data back intact.
4 Random access

seekg, seekp, tellg/tellp 🎯

Streams keep a position — where the next read or write happens. You can move it directly, so you don't have to read a file front-to-back:

CallMeaning
seekg(pos)move the get (read) position to byte pos
seekp(pos)move the put (write) position to byte pos
seekg(off, dir)relative: dir = ios::beg / ios::cur / ios::end
tellg() / tellp()report the current read / write position
 
// jump straight to record number n (0-based) and read it:
in.seekg(n * sizeof(Rec));            // byte offset of record n
Rec r;
in.read(reinterpret_cast<char*>(&r), sizeof r);
This is what makes a binary file of fixed-size records a random-access store: record n always lives at byte offset n * sizeof(Rec), so you can read or overwrite any record in one jump — no scanning. seekg moves the read head, seekp the write head; the two-argument form seeks relative to the beginning (ios::beg), the current spot (ios::cur), or the end (ios::end) — e.g. seekg(-sizeof(Rec), std::ios::end) lands on the last record. tellg()/tellp() hand back the current position (as a byte count from the start), handy for measuring sizes or remembering a spot.
5 What's safe to dump

Only plain data — and mind portability ⚠️

Byte-dumping works only for types whose bytes are the data. That's trivially-copyable types: numbers, fixed arrays, and structs built from those:

🚨 Never byte-dump a std::string or std::vector

A std::string or std::vector object is small — it holds a pointer to heap memory plus a size, not the characters/elements themselves. So write(reinterpret_cast<char*>(&myString), sizeof myString) saves the pointer and bookkeeping, not the text — and when you read it back into another run, that pointer is meaningless (it points into the old, gone process), giving garbage or a crash. To persist such data, write the contents: e.g. write the length, then the characters/elements. Second caution: a raw byte dump is not portable. Endianness (byte order), struct padding, and type sizes can differ between machines and compilers, so a .bin written on one system may not read correctly on another. For data you'll share or keep long-term, prefer a defined format (text, or an explicit binary layout you control field-by-field).

For self-contained, same-program scratch data (a cache, a save file you also load), byte-dumping POD records is simple and fast. For interchange, reach for a real format.

8 Mistakes, exercises, recap

Wrap up — and Part XI complete! 🏆

Three mistakes to avoid 🚨

🧵
Byte-dumping a std::string/std::vector Their sizeof bytes hold a pointer + size, not the contents. Writing them saves a pointer that's garbage on reload. Persist the length then the actual characters/elements.
📄
Forgetting std::ios::binary In text mode, some platforms translate newline bytes (0x0A ↔ 0x0D 0x0A), corrupting binary data. Always open binary files with std::ios::binary.
🌍
Assuming a .bin is portable Endianness, struct padding, and type sizes vary across machines/compilers. A raw dump is fine for the same program, but use a defined format for data you'll share or keep.

Practice exercises 💪

Easy Round-trip an int. Write an int to a file with out.write(...) in binary mode, then read it back with in.read(...) into a new variable and confirm they match.
Medium A record file. Define struct Rec { int id; double v; };, write five of them to data.bin, then use seekg(n * sizeof(Rec)) to read back record #3 without reading the others.
Stretch Update in place. Open the record file with an fstream (in | out | binary), seekp to record #2, overwrite just that record with write, then seekg back and read it to confirm only that record changed.

🎓 What to remember

  • Binary I/O copies exact bytes: out.write(reinterpret_cast<const char*>(&obj), sizeof obj) and in.read(reinterpret_cast<char*>(&obj), sizeof obj).
  • Open with std::ios::binary to avoid newline translation; check the stream / gcount() after reading.
  • Random access: seekg/seekp move the read/write position; record n sits at n * sizeof(Rec); tellg/tellp report position.
  • Only byte-dump trivially-copyable types — never std::string/std::vector (they store pointers, not data).
  • Raw dumps aren't portable (endianness, padding, sizes) — use a defined format for interchange. Part XI complete! 🏆

Composition vs Inheritance

Real programs build classes out of other classes — and there are two ways to do it. Composition (a class has-a another) and inheritance (a class is-a another). Choosing the right one is a core design skill — and the guideline is famous: prefer composition.

1 Why this matters

Building with other classes 🧠

So far you've designed single classes. But a real type is usually made from other types — a Car involves an Engine and Wheels; a SavingsAccount is a kind of Account. C++ gives you two ways to reuse other classes: composition — putting objects inside your class as members — and inheritance — building your class on top of a base class. Picking the right one shapes whether your code stays flexible or grows rigid.

This lesson closes Part VI by naming the choice and giving you a reliable test for it. (The full mechanics of inheritance are Part VII — here we focus on when to use which.)

2 Composition — has-a

A class contains other objects 🧩

Composition means your class has other objects as members. A Car doesn't become an engine — it contains one and uses it:

struct Engine {
    int start() { return 1; }
};

struct Car {
    Engine engine;              // Car HAS-an Engine (composition)
    int start() { return engine.start(); }   // delegate to it
};
The Car reuses Engine by holding one and delegating work to it (engine.start()). This is the everyday way to combine classes — you've been doing it whenever a class had a std::string or std::vector member. It's flexible: you can swap the engine, change it, or add more parts without disturbing Car's identity.
3 Inheritance — is-a

A class extends a base 🌳

Inheritance means your class is a kind of another class, and automatically gets its members. A Dog is-an Animal:

struct Animal {
    void breathe() { /* ... */ }
};

struct Dog : Animal {       // Dog IS-an Animal (inheritance)
    void bark() { /* ... */ }
};
// a Dog can breathe() (inherited) AND bark() (its own)
With Dog : Animal, every Dog automatically has whatever Animal provides, plus its own additions. This expresses a genuine "is-a" relationship and is the gateway to polymorphism — treating different derived types through a common base. The full story (how the syntax works, virtual functions, and more) is Part VII; for now, just recognise it as the "is-a" tool.
4 The test

"is-a" or "has-a"? 🎯

To choose, ask one question about the two types. If "X is a Y" sounds right, that's inheritance. If "X has a Y" fits better, that's composition:

RelationshipSounds right?Use
a Dog and an Animal"a Dog is an Animal" ✓inheritance
a Car and an Engine"a Car has an Engine" ✓composition
a Square and a Shape"a Square is a Shape" ✓inheritance
a Person and a Name"a Person has a Name" ✓composition
The test usually makes the answer obvious. "A car is an engine" sounds wrong — a car has one — so Car should contain an Engine, not inherit from it. Trust the plain-English phrasing; it captures the real relationship.
5 Prefer composition

The famous guideline ⚖️

When both could technically work, there's a well-known rule of thumb: prefer composition over inheritance. Composition keeps classes loosely coupled and easy to change; inheritance ties a class tightly to its base:

  • Composition is flexible — swap or modify the contained parts without touching the rest.
  • Inheritance is powerful but rigid — the derived class depends on the base's internals and is harder to change later.

🚨 Don't force an "is-a" that isn't there

A classic beginner mistake is reaching for inheritance just to reuse code — e.g. making Car inherit from Engine because it "needs engine stuff." But a car isn't an engine, so that relationship is a lie that will tangle your design. Reuse by composition (contain an Engine) unless there's a genuine "is-a." Save inheritance for true is-a relationships where you also want polymorphism — exactly what Part VII is about.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔧
Using inheritance when it's really has-a A Car isn't a kind of Engine, so it shouldn't inherit from one — it should contain one (composition). Apply the is-a test before reaching for inheritance.
🏗️
Overusing inheritance Deep inheritance hierarchies are rigid and tightly coupled. Prefer composition for reuse; reserve inheritance for genuine "is-a" relationships (and when you want polymorphism — Part VII).
💪
Thinking composition is "weaker" It's usually the more flexible choice — you can change or swap the contained parts freely. Most code reuse in good designs is composition, not inheritance.

Practice exercises 💪

Easy A car has an engine. Write a struct Engine { int hp = 120; } and a struct Car { Engine engine; }. Make a Car and print c.engine.hp — composition in action.
Medium Delegate. Give Engine a start() method, and have Car delegate: int start() { return engine.start(); }. Call c.start() — the car reuses the engine by composition.
Stretch Decide. For each pair, say is-a or has-a and which you'd use: (Bird, Animal), (Book, Page), (Manager, Employee), (Library, Book). Write a one-line justification for each.

🎓 What to remember

  • Two ways to build classes from others: composition (has-a) and inheritance (is-a).
  • Composition: a class contains other objects as members, reusing them by holding and delegating.
  • Inheritance: a class derives from a base, inheriting its members (full treatment in Part VII).
  • The test: "is X a Y?" → inheritance; "does X have a Y?" → composition.
  • Prefer composition over inheritance — it's more flexible; reserve inheritance for true is-a (and polymorphism).

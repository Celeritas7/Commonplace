🧯 PART X · ERRORS AND EXCEPTIONS

Error Codes & std::expected

Exceptions are for the exceptional. When a failure is routine and expected — a missing key, a parse that fails, a file that isn't there — it's often cleaner to return the error than to throw it. This lesson covers the classic approaches and the modern one: std::expected<T, E> (C++23), which holds either a value or a reason it failed.

1 Why this matters

When not to throw 📬

Exceptions (10.1–10.5) shine for rare, exceptional failures that the immediate caller usually can't fix and shouldn't have to check for at every step. But many failures aren't exceptional at all: a lookup that misses, a user who types "abc" where a number was expected, a config file that may or may not exist. For these, throwing is heavy-handed — it's costly when it happens, and it hides the failure from the function's signature. The alternative is to make failure a return value: the caller can see, right in the type, that this operation might not produce a result, and is nudged to handle it. C++ offers a progression of tools for this — from old C-style error codes to std::optional and, newest, std::expected.

Think of it as choosing the right channel: throw for "this shouldn't happen," return for "this might not work, and that's normal."

2 The old way

Return codes & out-parameters 🪧

The classic C approach (still everywhere): return a status code, and pass the real result back through an out-parameter:

bool parse_int(const std::string& s, int& out) {   // true = success
    // ... fill `out`, or return false on failure
}

int n;
if (parse_int(text, n)) {   // must check the return!
    use(n);
} else {
    // handle the failure
}
This works, but it has real downsides. The return code is easy to ignore — nothing forces you to check it, so an unchecked failure silently uses a garbage out. The value and the status are split apart (one returned, one through a reference), which reads awkwardly. And a bare bool or int doesn't say why it failed. The standard std::error_code improves on raw ints (it carries a category and a message), but the core awkwardness — value here, error there, easy to skip — remains. The modern types fix exactly this.
3 Value or nothing

std::optional<T> 🎁

When a function either yields a value or nothing (and you don't need to say why), std::optional<T> (from <optional>) packs both into the return type:

#include <optional>

std::optional<int> parse_int(const std::string& s) {
    if (ok) return value;        // success: holds an int
    return std::nullopt;          // failure: holds nothing
}

auto r = parse_int(text);
if (r) std::cout << *r;          // check, then dereference
int safe = r.value_or(0);        // or supply a fallback
An optional is either engaged (holds a T) or empty. You test it with if (r) or r.has_value(), read the value with *r or r.value(), and r.value_or(fallback) gives a default when it's empty. The win over an out-parameter: the "might be absent" possibility is baked into the return type, so callers can't accidentally use a result that isn't there — at least not without dereferencing, which is the one thing to be careful about (see the gotcha). What optional can't do is tell you why it's empty.
4 Value or reason

std::expected<T, E> (C++23) ✅

When you want the value or the reason it failed, reach for std::expected<T, E> (from <expected>, C++23). It holds either a T (success) or an E (the error), in one object:

#include <expected>

std::expected<int, std::string> parse_int(const std::string& s) {
    if (ok) return value;                       // success: just return the T
    return std::unexpected("not a number");    // failure: wrap the E
}

auto r = parse_int(text);
if (r) std::cout << *r;              // has value → read it with * or .value()
else   std::cout << r.error();       // no value → read the reason
expected is like optional with a payload on the failure side. Test it the same way (if (r) / r.has_value()); on success read *r / r.value(); on failure read r.error() for the E you returned. You construct a success by just returning the T, and a failure with std::unexpected(e). This gives you the clarity of return-based errors and the "why" that exceptions carry — without the cost of throwing — which is why it's the recommended modern choice for routine, recoverable failures. (It needs a C++23 compiler: -std=c++23.)
5 Choosing

Throw or return? 🧭

A rough guide — match the tool to how exceptional the failure really is:

SituationReach for
value or absent, no reason neededstd::optional<T>
value or a reason, caller handles itstd::expected<T,E>
rare / unrecoverable here, deep call stackthrow (exception)
legacy / C API boundaryerror code

🚨 Reading the value when there isn't one — and ignoring the result

Two related traps. First, dereferencing without checking: *r (or r->m) on an empty optional/expected is undefined behaviour, just like dereferencing a dangling pointer — always test if (r) first, or use value_or. (Note r.value() on an empty one is different: it throws std::bad_optional_access / std::bad_expected_access rather than UB.) Second, ignoring the result entirely: unlike a thrown exception, a returned error sits quietly in the return value — if you don't inspect it, the failure is lost. The whole point of these types is that you look at them. Treat an unchecked expected the way you'd treat an unchecked parse result: a bug waiting to happen.

8 Mistakes, exercises, recap

Wrap up — and Part X complete! 🏆

Three mistakes to avoid 🚨

💥
Dereferencing without checking *r on an empty optional/expected is undefined behaviour. Test if (r) first, or use value_or(fallback). (r.value() instead throws on empty — different, but still not free.)
🙈
Ignoring a returned error Unlike a thrown exception, a returned error won't interrupt anything — if you don't inspect the expected, the failure vanishes. Always check the result you got back.
⚖️
Using the wrong channel Throwing for routine, expected failures is costly and noisy; returning expected for a truly unrecoverable error deep in the stack forces every layer to forward it by hand. Match the tool to the failure.

Practice exercises 💪

Easy Optional fallback. Write std::optional<int> first_even(const std::vector<int>& v) returning the first even number or std::nullopt. Call it on a vector with no evens and print result.value_or(-1).
Medium Expected with a reason. Write std::expected<double, std::string> safe_divide(double a, double b) that returns a / b, or std::unexpected("divide by zero") when b == 0. Call it both ways and print either *r or r.error().
Stretch Same function, two styles. Write a lookup(key) that fails sometimes — once returning std::expected, once throwing an exception. Write a caller for each. Reflect on which reads better here and why; decide which failures in your own code are "expected" vs "exceptional".

🎓 What to remember

  • Throw for the exceptional; return for the expected. Routine, recoverable failures are often better reported as a value.
  • std::optional<T> = value or nothing; check with if (r), read *r, default with value_or.
  • std::expected<T, E> (C++23) = value or a reason; success returns the T, failure returns std::unexpected(e), read the reason with r.error().
  • Don't dereference an empty one (UB) and don't ignore the result — these types only help if you check them.

🧠 PART VII · MEMORY MANAGEMENT

Memory Leaks

A memory leak is heap memory you allocated but never freed — often because you lost the only pointer to it. Each leak is invisible at first, but they pile up: a long-running program slowly eats memory until it crawls or crashes. Here's how they happen and how to stop them.

1 Why this matters

The silent bug 🧠

A leaked block doesn't crash your program or print a wrong answer — that's what makes leaks sneaky. The code seems to work. But every leak is heap memory the program can never reclaim, so memory use only climbs. In a quick script that exits in a second, nobody notices. In a server or a game that runs for hours, leaks add up until the system runs out of memory and everything grinds to a halt. Learning to avoid them is what separates "it works on my machine for 5 seconds" from "it runs reliably for days."

The good news: the rule is the one you already know — every malloc needs a matching free. This lesson shows the ways that rule gets broken, and how to keep it.

2 What a leak is

Lost memory you can't get back 💧

A leak happens when allocated heap memory is never freed — and often when you've lost the only pointer to it, so you couldn't free it even if you wanted to:

int *p = malloc(100);   // 100 bytes reserved on the heap
// ... we use p ...
// function ends, p disappears — but the 100 bytes are STILL reserved
// 💧 leaked: nobody can ever free that block now
The block stays marked "in use" for the rest of the program's life, even though no one is using it. The heap doesn't track your intentions — it only frees what you explicitly hand back. Lose the pointer (or just forget to free), and that memory is gone for good.
3 Cause #1 — overwriting the pointer

Losing the only handle ⚠️

A very common leak: you point p at a new block before freeing the old one. The old block is now unreachable:

int *p = malloc(100);   // block A
p = malloc(200);        // 💧 block A leaked — p no longer points to it

// the same happens in a loop:
for (int i = 0; i < n; i++)
    p = malloc(50);     // 💧 every block but the last is leaked
Once you reassign p, the address of block A is gone — there's no way left to free it. The fix is to free the old block before reassigning, or to keep a separate pointer for each allocation. In a loop, that usually means freeing inside the loop body.
4 Cause #2 — an exit before free

Free on every path 🚪

The other classic: a return (often on an error) that skips the free. The happy path frees the block; the error path leaks it:

int *p = malloc(100);
if (something_failed)
    return 1;            // 💧 leaves without freeing p

// ... more work ...
free(p);                // only reached on the success path

🚨 Every code path must free what it allocated

It's not enough to free once at the bottom — if any return, break, or error branch can leave the function first, that path leaks. Make sure every way out frees the block: free before each early return, restructure so there's a single exit, or use the goto cleanup; pattern common in C. The mental check: "for each malloc, is there a free on every path that follows?"

5 Spotting & preventing

Habits and tools 🔍

  • Match every malloc with a free — pair them in your head as you write the allocation.
  • Free where you allocated when you can — the same function that takes the memory gives it back, so ownership is clear.
  • Free before reassigning or returning — don't overwrite or abandon a pointer that still owns a block.
  • Set the pointer to NULL after freeing, so a stray reuse is caught rather than silent.
Even careful programmers miss some, so we lean on tools: Valgrind and AddressSanitizer run your program and report exactly which allocations were never freed, with line numbers. They turn invisible leaks into a clear list to fix. We'll meet them properly in 7.6 — for now, know that you don't have to find every leak by eye.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

⚠️
Overwriting the only pointer to a block p = malloc(); p = malloc(); loses the first block forever. Free (or save) the old block before pointing p somewhere new.
🚪
Returning or erroring out before free An early return after a malloc leaks the block on that path. Free on every exit — before each early return, or with a single cleanup point.
🔁
Allocating in a loop but freeing once A free after the loop releases only the last block. Free inside the loop, or store every pointer and free them all.

Practice exercises 💪

Easy Clean function. Write a function that mallocs a block, uses it, and frees it — no leak. Then delete the free line and describe, in one sentence, what now leaks.
Medium Loop, two ways. Write a loop that mallocs each iteration and frees inside the loop (no leak). Then move the free outside the loop and explain why all but the last block leak.
Stretch Free on every path. Take a function that does malloc then return 1; on an error before the free. Rewrite it so the block is freed on every path (free before the early return, or a single cleanup label).

🎓 What to remember

  • A leak is heap memory you allocated but never freed — often because you lost the only pointer to it.
  • It's silent: the program runs, but memory use grows; long-running programs eventually crash or slow.
  • Common causes: overwriting the only pointer, returning before free, and allocating in a loop but freeing once.
  • Prevent it: match every malloc with a free on every path; free where you allocated; null the pointer.
  • Tools find leaks for you — Valgrind and AddressSanitizer report exactly what wasn't freed (7.6).

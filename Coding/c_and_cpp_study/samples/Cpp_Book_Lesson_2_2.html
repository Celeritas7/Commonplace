Basic Types, Sizes & Ranges

In 2.1 we met the types. Now we look inside: how many bytes each takes, what range of values it can hold, what overflow does, and how modern C++ gives you guaranteed sizes with int32_t and friends.

1 Why this matters

Size and range — the two facts that bite 🦷

Every C++ type takes a specific number of bytes in memory. That number determines the range of values the type can hold. Pick too small and your value silently wraps; pick too big and you waste memory. C++ adds tools to make this safer (std::numeric_limits, fixed-width types) — but you still need to know the basics.

Two questions you'll meet constantly: "What size should my counter be?" and "What happens at INT_MAX + 1?" (Spoiler for the second in C++: undefined behavior — the worst kind.) This lesson hands you the numbers, the intuition, and the modern C++ way to ask the compiler for guaranteed sizes.

The sizeof operator gives you the exact size on your machine: std::cout << sizeof(int); prints 4 almost everywhere.

3 Fixed-width types: the C++ way

When you need a guaranteed size 🎯

<cstdint> — the fixed-width types

When portability matters — when your code must behave identically on a 32-bit microcontroller, a Windows desktop, and a 64-bit Linux server — use fixed-width types from the <cstdint> header:

int8_t, int16_t, int32_t, int64_t  ←  signed, exactly that many bits
uint8_t, uint16_t, uint32_t, uint64_t  ←  unsigned versions

Each name promises exactly the bit width it says, on every conforming platform. uint32_t is always 32 bits, anywhere.

#include <cstdint> #include <iostream> int main() { int32_t a = 1000000; // exactly 4 bytes, signed uint8_t red = 255; // exactly 1 byte, unsigned — like an image pixel channel int64_t big = 9000000000LL; // exactly 8 bytes std::cout << sizeof(a); // always prints 4 }

Use these when: working with binary file formats, network protocols, image data, or any code that must be portable across machines. For general application code, plain int is fine.

5 Floating-point: float vs double

Why double is almost always the right choice 🎯

float and double both store decimal numbers, but with different precision. float has about 7 decimal digits; double has about 15. In C++, unsuffixed decimal literals are always double — 3.14 is a double; 3.14f is a float.
float f = 0.1f + 0.2f; // f ≈ 0.30000001192 double d = 0.1 + 0.2; // d ≈ 0.30000000000000004

Both are slightly wrong (binary can't represent decimal exactly), but double is wrong further out. For almost every program — finance, science, games — double is the right pick.

Use float when: giant data buffers where memory matters (3D meshes, ML weights, image arrays), or specific platforms where double-precision is slower. Otherwise: double.

7 Common mistakes

Three traps with C++ types 🚧

🙅
"Signed integer overflow wraps around in C++." It does not. Signed overflow is undefined behavior — the compiler is free to assume it never happens and optimize accordingly. Some real bugs only appear when you turn on optimization. If you need wrapping math, use an unsigned type (or uint32_t) where wrap-around is guaranteed.
🙅
"auto x = 3.14; gives me a float." It gives you a double. C++ decimal literals default to double. To get a float, you'd write auto x = 3.14f; — but you usually shouldn't bother. double is the right default in C++.
🙅
"size_t is just an unsigned int." It's an unsigned type, yes — but it's the type used by sizeof, .size() on containers, etc. On 64-bit systems it's usually a 64-bit unsigned, not a 32-bit one. Comparing a signed int i with vector.size() triggers the sign-compare warning we mentioned in 2.1 — keep the types consistent. In modern C++ a common fix is for (size_t i = 0; i < v.size(); ++i) or, better, the range-for loop.
8 Your turn

Get a feel for sizes 💪

Easy Measure your machine. Write a small program that prints sizeof each type. Use std::cout << sizeof(int) << '\n'; and similar. Compare to what the size table claims. Especially check long — on Linux/Mac it's usually 8 bytes; on Windows it's still 4.
Medium Meet std::numeric_limits. Include <limits>. Print these values:
std::cout << std::numeric_limits<int>::max() << '\n';
std::cout << std::numeric_limits<int>::min() << '\n';
std::cout << std::numeric_limits<double>::max() << '\n';
std::cout << std::numeric_limits<double>::epsilon() << '\n';
Run it. The last one (epsilon) is the smallest difference a double can distinguish near 1.0 — it's the source of all "0.1 + 0.2 isn't 0.3" weirdness.
Stretch Catch a portability bug. On godbolt.org, write a function that does int32_t a = 100000; int16_t b = a;. Compile with -Wconversion -Wall. Read the warning. Now repeat without the flags — note that the conversion is silent. This is exactly why brace-init (int16_t b{a};) refuses to compile narrowing conversions — to make them explicit.

🎓 Quick recap

  • Sizes you'll see on every modern machine: char = 1, short = 2, int = 4, long long = 8, float = 4, double = 8, bool = 1.
  • Default to int for whole numbers, double for decimals.
  • Fixed-width types (int32_t, uint8_t, etc.) from <cstdint> guarantee exact sizes — use for binary data, network protocols, portable code.
  • Signed overflow is undefined behavior in C++. Unsigned overflow wraps cleanly to 0. Use the right type for the behavior you want.
  • Use std::numeric_limits<T>::max() / ::min() from <limits> for safe access to the exact range of any type.
  • sizeof(type) still works in C++ for "how many bytes does this take on this machine."

Bitwise Operators — A Quick Tour

Most operators work on whole numbers. These six work on the individual bits inside them: &, |, ^, ~, <<, >>. You won't reach for them daily, but when you need to pack flags, read hardware, or squeeze space, nothing else will do.

1 Why this matters

Working one bit at a time 🧠

Every integer is stored as a row of bits — 0s and 1s. The number 5 is really 00000101 in an 8-bit char. Most operators treat that as the value "five." Bitwise operators ignore the value and work on each bit position directly, like flipping switches in a row.

Why would you want that? Three big reasons: flags (cram eight on/off settings into one byte instead of eight variables), hardware (device registers are read and written bit by bit), and performance/space (shifts multiply and divide by powers of two, and bit tricks pack data tight). This is a quick, practical tour — enough to recognize bit manipulation in the wild and use the everyday patterns. The deep dive lives in Appendix B.

2 The six operators

What each one does 🔧

OperatorNamePer-bit rule
&AND1 only if both bits are 1
|OR1 if either bit is 1
^XOR1 if the bits differ
~NOTflips every bit (unary)
<<left shiftslide bits left, fill with 0s
>>right shiftslide bits right

Take 5 & 3. Line the two numbers up bit by bit, then apply the AND rule to each column:

5 = 0000 0101
3 = 0000 0011
─────────────
5&3 = 0000 0001 = 1
Don't confuse these with the logical operators from lesson 2.5. & is bitwise AND; && is logical AND. 5 & 3 is 1 (bit math), but 5 && 3 is 1 meaning "both are truthy." Different jobs — same-looking symbols. Same for | vs ||.
3 XOR and NOT in action

The flip operators 🔀

XOR (^) sets a bit to 1 only when the two bits differ. That makes it the "toggle" operator — XOR-ing a bit with 1 flips it; XOR-ing with 0 leaves it alone. A fun property: x ^ x is always 0, and x ^ 0 is always x.

6 = 0000 0110
3 = 0000 0011
─────────────
6^3 = 0000 0101 = 5

NOT (~) is unary — it flips every single bit. In an 8-bit byte, ~00000101 becomes 11111010. Because it flips the high bits too, the result on a signed type is usually negative (~5 is -6). You'll mostly use ~ to build a "clear this bit" mask, as you'll see in a moment.

4 Shifts

<< and >> — sliding bits 🚀

A left shift slides every bit one position to the left and drops a 0 in on the right. That doubles the value. A right shift slides bits right, which halves it (integer division):

ExpressionBinaryValueSame as
1 << 00000000112⁰
1 << 30000100082³
5 << 100001010105 × 2
20 >> 1000010101020 ÷ 2
20 >> 200000101520 ÷ 4
The pattern 1 << n is everywhere — it builds a value with exactly one bit set, at position n. That's how you name individual bits: bit 0 is 1 << 0, bit 3 is 1 << 3, and so on. Those single-bit values are the building blocks for flags.
5 The killer app: flags & masks

Eight switches in one byte 🎛️

Here's where bitwise earns its place. Instead of eight separate bool variables, you store eight on/off settings in the bits of a single integer, and use bit operations to flip them. Each pattern is a one-liner you'll memorize:

GoalPatternHow it works
Set a bitflags |= MASK;OR forces the bit to 1
Clear a bitflags &= ~MASK;AND with the inverse forces it to 0
Toggle a bitflags ^= MASK;XOR flips it
Test a bitif (flags & MASK)nonzero means the bit is set

where MASK is a single-bit value like 1 << 2. These four lines — set, clear, toggle, test — are the entire vocabulary of flag handling, and you'll see them in every codebase that touches hardware, graphics, or file permissions. Play with them in the Flag Manipulator below.

🚨 Bitwise binds looser than ==

From lesson 2.6: if (flags & MASK == 0) is read as flags & (MASK == 0) because == binds tighter than &. Almost never what you want. Always parenthesize the test: if ((flags & MASK) == 0).

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

⚡
Using &&/|| where you meant &/| The logical versions give 1 or 0 based on truthiness, not bit math. 6 & 3 is 2; 6 && 3 is 1. For bit work, use the single-character forms.
🔢
Forgetting the precedence parentheses flags & MASK == 0 parses as flags & (MASK == 0) — == binds tighter. Always write (flags & MASK) == 0.
➖
Shifting or ~-ing signed values carelessly Right-shifting a negative number and shifting by too many bits are implementation-defined or undefined. For bit manipulation, prefer unsigned types (unsigned int, uint8_t) so behavior is well-defined.

Practice exercises 💪

Easy Even or odd. Write a one-line test using & that's true when an int n is odd. (Hint: what's the lowest bit of an odd number?) Verify in the Bit Visualizer with n & 1.
Medium Permission bits. Define three masks READ = 1<<0, WRITE = 1<<1, EXEC = 1<<2. Start from 0, grant READ and WRITE, then revoke WRITE, and print whether EXEC is set. Walk it through the Flag Manipulator.
Stretch Swap without a temp. The famous XOR swap: given a and b, exchange their values using only a ^= b; b ^= a; a ^= b; — no temporary variable. Trace why it works on paper with two small numbers, using x ^ x == 0.

🎓 What to remember

  • Six operators on bits: & AND, | OR, ^ XOR, ~ NOT, << left shift, >> right shift.
  • << doubles, >> halves; 1 << n builds a value with one bit set at position n.
  • Flag patterns: set |=, clear &= ~, toggle ^=, test &. The whole vocabulary in four lines.
  • Don't confuse &/| with &&/||, and parenthesize (flags & MASK) == 0.
  • Use unsigned types for bit work so shifts and ~ behave predictably.

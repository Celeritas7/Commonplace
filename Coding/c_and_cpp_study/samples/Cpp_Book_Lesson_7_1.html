🧬 PART VII · INHERITANCE & POLYMORPHISM

Single Inheritance

In 6.11 you learned when to use inheritance — for a real "is-a" relationship. Now the mechanics: class Dog : public Animal builds Dog on top of Animal, so every Dog automatically gets Animal's members and can add its own.

1 Why this matters

Build on what exists 🧠

When two types share an "is-a" relationship — a Dog is an Animal, a SavingsAccount is an Account — inheritance lets the more specific type reuse everything the general type already provides. You write the shared behaviour once in the base class, then each derived class gets it for free and adds its own specifics. No copy-paste, no duplication.

This opens Part VII, the heart of object-oriented C++. We start with the simplest case — one class deriving from one base — and the rules that govern it. (The big payoff, polymorphism, comes a couple of lessons on.)

2 The syntax

class Dog : public Animal 🌳

You derive a class by naming its base after a colon. Dog below inherits everything Animal offers, and adds bark():

class Animal {
public:
    void breathe() { std::cout << "breathing\n"; }
};

class Dog : public Animal {   // Dog IS-an Animal
public:
    void bark() { std::cout << "woof\n"; }
};
The word public before Animal is the kind of inheritance — public inheritance, which models "is-a" and keeps the base's public interface public. (There are protected and private variants, but public is by far the common one; we'll mention the others next lesson.) Animal is the base class; Dog is the derived class.
3 Inherited members in action

A Dog can do both 🔗

Because Dog inherited from Animal, a Dog object can call Animal's methods directly — as if they were its own — alongside its own additions:

Dog d;
d.breathe();   // "breathing" — inherited from Animal
d.bark();      // "woof"      — Dog's own method
There's no special syntax to reach the inherited members — d.breathe() just works, because a Dog is an Animal and therefore has everything an Animal has. The derived class is the base, plus more. That's the whole reuse mechanism in one sentence.
4 Construction order

Base first, then derived 🏗️

A derived object is built in layers: the base part is constructed first, then the derived part. With print statements in each constructor you can watch it:

class Animal { public: Animal() { std::cout << "A"; } };
class Dog : public Animal { public: Dog() { std::cout << "D"; } };

Dog d;   // prints "AD" — Animal() runs before Dog()
It makes sense: the derived class is built on top of the base, so the foundation (the Animal part) must be ready first. Destruction runs in the reverse order — derived destructor first, then base — mirroring the construct→destruct rule from 6.5. The base is always set up before the derived class's constructor body runs.
5 Access through inheritance

What the derived class can touch 🔑

Inheritance interacts with the access specifiers from 6.3. The derived class can use the base's public and protected members — but not its private ones:

Base member is…Usable in the derived class?
publicyes (and from outside too)
protectedyes (but not from outside)
privateno — hidden even from derived classes

🚨 class defaults to private inheritance

If you write class Dog : Animal (without public), you get private inheritance — the base's public interface becomes private in Dog, so outside code can't use the inherited members, and it no longer reads as "is-a." For the normal is-a relationship, always write : public Animal. (A struct defaults to public inheritance, mirroring the member-access default from 6.3 — but spell out public anyway for clarity.) Also remember: private base members stay private to the base; if a derived class needs them, the base should make them protected.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔑
Forgetting public (with class) class Dog : Animal is private inheritance by default, which hides the base's interface. For an is-a relationship write : public Animal.
🏗️
Expecting the derived constructor to run first The base part is constructed first, then the derived. (Destruction is the reverse.) Don't rely on derived members being set up before the base.
🚫
Reaching for the base's private members Private base members are hidden even from derived classes. If a derived class needs to use one, the base should declare it protected (or expose it through a method).

Practice exercises 💪

Easy Dog is an Animal. Write struct Animal { void breathe(); } and struct Dog : Animal { void bark(); }. Make a Dog and call both breathe() (inherited) and bark() (its own).
Medium Watch the order. Give Animal and Dog constructors that each print a letter. Create a Dog and confirm the output shows the base constructor running before the derived one.
Stretch Access levels. Add a protected member to Animal and use it from Dog (works). Then add a private one and try to use it from Dog — watch the compiler reject it. The access rules in action.

🎓 What to remember

  • Single inheritance: class Derived : public Base builds a class on top of another.
  • The derived class inherits the base's public and protected members and adds its own.
  • public inheritance models an "is-a" relationship (from 6.11).
  • Construction runs base-first, then derived (destruction is the reverse).
  • A class defaults to private inheritance, a struct to public — write : public Base to be explicit.

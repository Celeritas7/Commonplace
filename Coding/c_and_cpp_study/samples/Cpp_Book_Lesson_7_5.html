Virtual Destructors

Polymorphism means holding derived objects through a base pointer — and eventually delete-ing them through it. But if the base destructor isn't virtual, only the base part is destroyed: the derived part's cleanup never runs, leaking memory. One keyword fixes it, and it's the rule for every polymorphic base.

1 Why this matters

Cleanup that goes missing 🧠

You've learned to wield objects polymorphically through a base pointer: Base* p = new Derived();. The natural counterpart is delete p; when you're done. But here's the catch — p is a Base*, so the compiler, by default, only knows to run Base's destructor. If Derived allocated memory or opened a file, its destructor — the one that frees that resource — never runs. The object half-dies: base part cleaned, derived part leaked. This is one of C++'s most important "you must know this" rules, and the fix is a single keyword.

It's the loose end the last two lessons kept pointing at — let's tie it off.

2 The problem

A non-virtual destructor leaks 🚨

Suppose Derived owns a resource it frees in its destructor. Delete it through a Base* whose destructor isn't virtual, and only ~Base runs:

struct Base {
    ~Base() { /* not virtual */ }
};
struct Derived : Base {
    int* data = new int[100];
    ~Derived() { delete[] data; }   // frees the buffer
};

Base* p = new Derived();
delete p;   // only ~Base runs → ~Derived skipped → data leaks! (undefined behaviour)
Because the destructor isn't virtual, the call is resolved by the pointer's static type (Base*) — exactly like the non-virtual functions back in 7.3. So ~Base runs and the story ends there; ~Derived, which would have freed data, is never called. The 100-int buffer leaks. Formally, deleting a derived object through a base pointer without a virtual destructor is undefined behaviour — so you can't even rely on what happens.
3 The fix

Make the base destructor virtual ✅

Add virtual to the base destructor. Now delete p dispatches at runtime to the real object's destructor and chains all the way up:

struct Base {
    virtual ~Base() = default;   // virtual destructor — the fix
};
struct Derived : Base {
    int* data = new int[100];
    ~Derived() { delete[] data; }
};

Base* p = new Derived();
delete p;   // ~Derived() runs (frees data), THEN ~Base() → full cleanup ✅
With a virtual destructor, delete p behaves like any virtual call: it looks at the actual object (a Derived) and starts with ~Derived, then automatically runs ~Base after. The whole object is destroyed properly, in order. = default just asks the compiler to generate the (empty) destructor body for you — you only needed it to be virtual.
4 The rule

Polymorphic base ⇒ virtual destructor 📜

The guideline is simple and worth memorizing:

If a class…then…
has any virtual functiongive it a virtual destructor
is meant as a base for delete via base*give it a virtual destructor
is a standalone value (no inheritance)a normal destructor is fine
In short: any class you intend to use polymorphically should have a virtual destructor — almost always written virtual ~Base() = default;. It costs essentially nothing and prevents a whole category of silent leaks. If a class has even one virtual function, that's the signal it'll be used through base pointers, so its destructor should be virtual too. (Adding it does give the class a vtable, but if it already has virtuals, that cost is already paid.)
5 Order of destruction

Reverse of construction 🔁

When the full chain runs, destructors fire in the opposite order to constructors — most-derived first, base last:

// construct: Base() then Derived()
Base* p = new Derived();   // prints: b d

// destruct (virtual): Derived() then Base()
delete p;                  // prints: D B

🚨 Same idea applies to smart pointers

Soon you'll use std::unique_ptr<Base> and std::shared_ptr<Base> instead of raw new/delete. They call the destructor for you — but they obey the same rule: if you store a Derived in a unique_ptr<Base> and the base destructor isn't virtual, the derived part still won't be cleaned up. The virtual-destructor rule is fundamental, no matter how the delete is triggered.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🧹
A polymorphic base with a non-virtual destructor If a class has virtual functions but a non-virtual destructor, deleting derived objects through a base pointer leaks the derived part (undefined behaviour). Make the destructor virtual.
🤔
Adding virtual to everything A small value type with no inheritance doesn't need a virtual destructor — it adds a vtable for no reason. The rule is for classes used polymorphically, not every class.
🏗️
Trying to make a constructor virtual Constructors can't be virtual — the object's type isn't fully formed yet. Only destructors (and ordinary methods) can. The cleanup is what needs the dynamic dispatch.

Practice exercises 💪

Easy See the leak path. Give Base and Derived destructors that each std::cout a letter. With a non-virtual ~Base, run Base* p = new Derived; delete p; and note only "B" prints.
Medium Fix it. Add virtual to ~Base and run again — now you see "D" then "B". Confirm the full chain runs, derived first.
Stretch Catch the real leak. Make Derived hold an int* data = new int[100]; freed in ~Derived. Run the non-virtual version under a sanitizer or Valgrind (C book 7.6!) and watch it report the leak; then make the destructor virtual and confirm it's clean.

🎓 What to remember

  • Deleting a derived object through a base pointer only runs the base destructor — unless it's virtual.
  • A non-virtual base destructor skips the derived destructor → leak / undefined behaviour.
  • The fix: virtual ~Base() = default; — then delete runs the full chain.
  • The rule: any class with virtual functions (a polymorphic base) should have a virtual destructor.
  • Destruction is reverse of construction — derived first, then base.

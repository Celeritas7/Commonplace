Text Files: fopen & friends

So far your programs forgot everything the moment they ended. Files change that: they let a program save data and read it back later. C's standard library gives you a small, dependable toolkit — fopen to open, fprintf and fgets to write and read, fclose to finish. This opens Part IX.

1 Why this matters

Programs that remember 🧠

Until now, every variable vanished when main returned. To save a high score, load a config, process a log, or write a report, a program needs to talk to files — data that lives on disk between runs. C handles this through its standard I/O library (<stdio.h>, the same header that gives you printf). The model is simple and consistent: you open a file to get a handle, read or write through that handle, and close it when you're done. Master those four steps and files stop being mysterious. This opens Part IX — Files and the Outside World.

The same printf-style tools you know, now pointed at a file.

2 Opening & closing

fopen gives a FILE* 🔑

fopen takes a filename and a mode, and returns a FILE * handle — or NULL if it couldn't open. Always check, and always fclose when done:

FILE *f = fopen("data.txt", "r");   // open for reading
if (f == NULL) {                       // ALWAYS check!
    perror("could not open data.txt");
    return 1;
}
// ... use f ...
fclose(f);                             // release the file when finished
ModeMeaningIf the file…
"r"readmust exist, or fopen returns NULL
"w"writecreated, or truncated (emptied!)
"a"appendcreated, or written to at the end
The FILE * is your handle to the open file — you pass it to every read/write call. The mode decides what's allowed and what happens to existing contents: "r" needs the file to exist, "w" starts it fresh (wiping anything there), and "a" keeps the old contents and adds to the end. Checking for NULL is not optional — files fail to open all the time (missing, no permission), and using a NULL handle crashes.
3 Writing

fprintf — printf to a file 📝

Writing is just printf with a FILE * as the first argument. Everything you know about format strings carries over:

FILE *f = fopen("scores.txt", "w");
fprintf(f, "%s %d\n", "Ada", 95);   // writes "Ada 95\n" to the file
fprintf(f, "%s %d\n", "Bel", 88);
fputs("done\n", f);                  // fputs writes a plain string
fclose(f);
fprintf(f, …) is to a file what printf(…) is to the screen — same format specifiers, same behaviour, just directed at f. (In fact printf(...) is essentially fprintf(stdout, ...).) Use \n yourself when you want line breaks — nothing is added automatically. For writing a ready-made string with no formatting, fputs is the simple choice.
4 Reading

fgets — a line at a time 📄

The safe, everyday way to read text is fgets: it reads up to one line into a buffer you provide, and won't overflow it. It returns NULL at end of file:

char line[256];
while (fgets(line, sizeof line, f) != NULL) {   // read until EOF
    printf("got: %s", line);   // note: line still has its '\n' on the end
}
fgets(buf, size, f) reads at most size - 1 characters (leaving room for the terminating \0), stopping at a newline or end of file — so it can never write past your buffer. It keeps the trailing \n as part of the line, which surprises people; you often trim it. Looping until fgets returns NULL is the standard way to read a whole file line by line (the focus of the next lesson). There's also fscanf for parsing formatted fields, but fgets is the safer default.
5 The workflow

Open, check, use, close 🔁

Every file interaction follows the same four beats — the habit that keeps file code correct:

FILE *f = fopen("notes.txt", "a");   // 1. open (append)
if (!f) { perror("open"); return 1; }     // 2. check
fprintf(f, "new note\n");                  // 3. use
fclose(f);                                // 4. close

🚨 "w" erases the file the instant you open it

Opening with "w" truncates the file to empty immediately — before you write a single byte. If you meant to add to existing contents, that's data loss; use "a" (append) to keep what's there. And don't forget fclose: it flushes any buffered output to disk and releases the handle. Skip it and your last writes may never reach the file, and you leak a resource — exactly the kind of cleanup C++ will later automate with RAII.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

❓
Not checking fopen for NULL Files fail to open often (missing, no permission). Using a NULL FILE* crashes. Check every fopen before using the handle.
🗑️
Using "w" when you meant "a" "w" truncates the file to empty the moment it opens — instant data loss. To preserve and add, use "a".
↩️
Forgetting the newline — or fclose fgets keeps the trailing \n; trim it if needed. And fprintf adds no newline for you. Always fclose to flush buffered writes and release the handle.

Practice exercises 💪

Easy Write then read. Open "hello.txt" with "w", fprintf two lines, fclose. Reopen with "r" and fgets them back, printing each. Check both fopen calls.
Medium Append a log. Write a program that opens a file with "a" and adds a timestamped line each run. Run it a few times and confirm the file grows instead of resetting.
Stretch Copy a file. Open one file for reading and another for writing, then loop with fgets/fputs to copy line by line. Handle the case where the source can't be opened.

🎓 What to remember

  • fopen(name, mode) returns a FILE * — or NULL on failure. Always check.
  • Modes: "r" read (must exist), "w" write (truncates!), "a" append (keeps & adds).
  • fprintf(f, …) writes like printf; fputs writes a plain string.
  • fgets(buf, size, f) reads a line safely, keeps the \n, and returns NULL at EOF.
  • Open → check → use → fclose — close to flush writes and free the handle.

Lambda Expressions

Sometimes you need a tiny function for one spot — a comparison, a quick transform — and naming it feels like overkill. A lambda is an anonymous function you write inline, right where it's used. It's one of modern C++'s most-loved features, and the STL is built to take them.

1 Why this matters

A function, right where you need it 🧠

You've written named functions all chapter. But some functions are so small and so local — "double this number," "is this name longer than five letters?" — that giving them a name and putting them elsewhere actually makes the code harder to read. A lambda lets you write the function inline, at the exact spot you use it.

This matters most for the part of C++ coming up next: the STL algorithms. Functions like std::sort and std::count_if take a function that tells them how to compare or what to count — and a lambda is the natural way to hand them that little piece of behaviour without breaking your flow. Learn the shape now, and the STL chapter will feel easy.

2 The shape

Brackets, parens, braces 🔧

A lambda has three little pieces: a capture list [ ], a parameter list ( ), and a body { }. You can store it in a variable (its type is unspeakable, so use auto) and call it just like a function:

auto doubler = [](int x) { return x * 2; };
//            ^^      ^^^^^^    ^^^^^^^^^^^^^^^^
//          capture  params         body

std::cout << doubler(5);   // 10 — call it like any function
Read it as "a function that takes an int x and returns x * 2" — just with no name and no separate declaration. The return type is usually deduced for you (here, int). The [ ] looks odd at first, but it's the lambda's signature feature: it's how a lambda can grab variables from the surrounding code, which is the next piece.
3 Captures

Reaching into the surroundings 🎒

Unlike a normal function, a lambda can use variables from the scope where it's written — but only the ones you list in the [ ]. That's capturing:

int factor = 3;
auto scale = [factor](int x) { return x * factor; };  // captures factor
std::cout << scale(10);   // 30
Capture listMeaning
[ ]capture nothing (can only use its parameters)
[x]capture x by value — a copy, frozen at creation
[&x]capture x by reference — the live variable
[=]capture everything used, by value
[&]capture everything used, by reference
By value ([x]) takes a snapshot at the moment the lambda is created — later changes to x don't affect it. By reference ([&x]) ties the lambda to the live variable, so it sees the current value when called. Same distinction you met in 4.2 — now for the variables a lambda borrows. The Capture Lab below makes the difference visible.
4 Where you'll use them

Handing behaviour to algorithms 🚀

The payoff: functions that take other functions. The STL is full of them. You give std::sort a lambda that says how to compare, or std::count_if a lambda that says what to count:

// sort longest-name-first
std::sort(names.begin(), names.end(),
          [](const std::string& a, const std::string& b) {
              return a.size() > b.size();
          });

// count how many scores are 40 or above
int passed = std::count_if(scores.begin(), scores.end(),
                           [](int s) { return s >= 40; });
Don't worry about the exact STL syntax yet — that's its own chapter. The point is the pattern: the algorithm does the looping, and your lambda supplies the one decision that makes it yours. Written inline, the rule sits right next to the call, so the code reads as one thought instead of jumping off to a named helper elsewhere.
5 Watch out

Captured by reference? Mind the lifetime ⚠️

The one real hazard is capturing by reference something that won't live long enough. If a lambda captures a local by reference and then outlives that local, the reference dangles — undefined behaviour:

🚨 A reference capture that outlives its variable

If you return a lambda from a function with [&] capturing a local, that local is destroyed when the function returns — the lambda is left pointing at garbage. For lambdas you store and use later, prefer capturing by value ([=] or [x]) so they own a safe copy. Reference capture is great for short-lived lambdas used right away (like the STL calls above), where the variables clearly outlive the lambda.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🎒
Using an outside variable you didn't capture A lambda can only use surrounding variables that are in its [ ]. [](){ return factor; } won't compile if factor isn't captured — write [factor] (or [=]).
⏳
A dangling reference capture Capturing a local by reference ([&]) and using the lambda after that local is gone is undefined behaviour. For lambdas you keep around, capture by value.
🧊
Expecting a by-value capture to see later changes [x] snapshots x when the lambda is created. Changing x afterward doesn't change what the lambda sees — use [&x] if you want the live value.

Practice exercises 💪

Easy Your first lambda. Store [](int n){ return n * n; } in an auto variable called square, then print square(6).
Medium Capture a factor. Set int k = 5; and write a lambda that multiplies its argument by k (capture k by value). Call it on a few numbers. Then change k after creating the lambda and confirm the lambda still uses the old value.
Stretch Value vs reference. Make two lambdas that both read a variable n — one with [n], one with [&n]. Change n, call both, and explain why they print different things. (This is the whole capture lesson in one experiment.)

🎓 What to remember

  • A lambda is an anonymous function written inline: [capture](params){ body }.
  • Store it in auto and call it like any function; the return type is usually deduced.
  • The [ ] captures surrounding variables: [x] by value (a snapshot), [&x] by reference (live).
  • Their big use is the STL — handing a small rule to algorithms like std::sort and std::count_if.
  • Beware dangling reference captures; for lambdas you keep, capture by value.

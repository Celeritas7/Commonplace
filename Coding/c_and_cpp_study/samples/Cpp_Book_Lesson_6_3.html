Access Specifiers

A class can decide who's allowed to touch each member. public members are part of the interface anyone can use; private members are the hidden internals only the class itself may touch. This is encapsulation — and it's what makes objects safe to use.

1 Why this matters

Hide the internals 🧠

Think of a car: you use the steering wheel and pedals (the interface), but the engine internals are sealed away. A class works the same way — you expose a small, safe set of operations and hide the raw data behind them. That way, no outside code can put an object into a broken state, and you're free to change the internals later without breaking anyone. The tool for this is access specifiers.

You've actually relied on this already: a std::vector hides its internal array and length, and only lets you touch them through methods like push_back and size. Now you'll do the same for your own classes.

2 public vs private

Who's allowed in 🔐

Two labels do most of the work. public: marks members anyone can use from outside; private: marks members only the class's own methods can touch. Everything after a label has that access until the next label:

class BankAccount {
private:
    double balance = 0;       // hidden — only methods below can touch it
public:
    void deposit(double amt) { balance += amt; }
    double getBalance() { return balance; }
};
From outside, acc.deposit(50) and acc.getBalance() work — they're public. But acc.balance is a compile error — it's private. Inside the class, though, deposit and getBalance reach balance freely. Private doesn't mean "frozen"; it means "only reachable from within the class."
3 The standard pattern

Private data, public methods 🎛️

The everyday recipe for a well-behaved class: keep the data private, and offer public methods as the only way in. Read access is a getter; write access is a setter:

class Temperature {
private:
    double celsius = 0;
public:
    double get() { return celsius; }          // getter (read)
    void set(double c) { celsius = c; }          // setter (write)
};
This looks like extra typing for little gain at first — why not just make celsius public? The payoff is control: every read and write now flows through a method you own, so you can validate, log, or compute on the way. Outside code depends only on get()/set(), never on the raw field.
4 Why bother

Guarding the rules 🛡️

Because writes go through a setter, you can enforce rules (invariants) that must always hold. A bank balance should never go negative — a setter can refuse bad input, something a public field could never do:

void withdraw(double amt) {
    if (amt > 0 && amt <= balance)   // guard the invariant
        balance -= amt;
    // else: ignore / report an error — never go negative
}
With balance private, every change passes through your code, so the rule "balance ≥ 0" can't be broken from outside. If it were public, any line anywhere could write acc.balance = -9999; and corrupt the object. Encapsulation turns "please don't do that" into "the compiler won't let you."
5 protected, and the defaults

A third option & struct vs class 🧬

There's a third specifier, protected: — like private, but also reachable by derived classes (subclasses). You'll meet it properly with inheritance in Part VII; for now just know it exists. And recall the only difference between struct and class:

SpecifierWho can access the member
publicanyone — it's the interface
privateonly the class's own methods
protectedthe class + its derived classes (Part VII)

🚨 class defaults to private

Members before the first label are private in a class but public in a struct — that's the only built-in difference between the two keywords. So a class with no public: label exposes nothing, and outside code can't even construct-and-use it meaningfully. Add a public: section for the interface. Convention: use struct for plain public data bundles, class when you want encapsulation.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🚪
Touching private members from outside Only the class's own methods can reach a private member. From main, go through a public method (a getter or setter) — don't try acc.balance directly.
🌍
Making everything public If all data is public, anyone can break the object's rules. Keep data private and expose a small, intentional public interface — that's the whole point of a class.
🏷️
Forgetting class is private by default Without a public: label, a class exposes nothing and outside code can't use it. Add the label (or use struct for a plain public bundle).

Practice exercises 💪

Easy Private with a getter. Write a class Circle with a private double radius and a public double getRadius(). Construct one, set the radius through a setter, and print it via the getter from main.
Medium A guarded setter. Give Circle a void setRadius(double r) that ignores negative values (keeps the old radius). Try setting -5 and confirm the radius doesn't change — the invariant "radius ≥ 0" holds.
Stretch Seal a struct. Take a plain struct with public fields and convert it into a class with private data plus public methods. Notice main can no longer poke the fields directly — you've encapsulated it. Reflect on what that buys you.

🎓 What to remember

  • Access specifiers control who reaches a member: public (anyone), private (only the class's methods), protected (class + derived — Part VII).
  • The standard pattern: private data + public methods — a small, safe interface.
  • Private data protects invariants and lets you change internals without breaking callers.
  • A setter can validate before storing; a getter exposes controlled read access.
  • struct defaults to public, class to private — add labels to be explicit.

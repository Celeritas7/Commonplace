constexpr

Some values never change and could be known the moment you compile — an array size, a lookup table, a converted constant. constexpr tells the compiler "work this out now, at compile time," so the finished result is baked straight into your program. It's free at runtime, and it unlocks uses (like array sizes) that ordinary variables can't fill.

1 Why this matters

Compute it before it runs ⚡

Most code runs when the program runs. But some values are fixed and computable the instant you compile — the number of days in a week, a buffer size, a small lookup table. constexpr (C++11) lets you ask the compiler to evaluate an expression at compile time and store the finished answer in the program. Two payoffs: it's free at runtime (the work already happened), and a constexpr value is a true compile-time constant, so it can do things a normal variable can't — like sizing a built-in array or being a template argument. It fits Part IX because compile-time computation and templates are the two halves of "things the compiler does for you before main runs."

Think of it as moving work from runtime back to compile time, where it costs nothing later.

2 constexpr variables

A constant the compiler computes 🔢

A constexpr variable must be initialised with something computable at compile time. The compiler works it out and bakes the result in:

constexpr int daysPerWeek = 7;
constexpr int hoursPerWeek = daysPerWeek * 24;   // computed at compile time → 168

int buffer[hoursPerWeek];   // OK — array size needs a compile-time constant
Everything to the right of = must itself be known at compile time, so the compiler can finish the calculation before the program runs. The result is a constant baked into the binary — no multiplication happens at runtime. This is also why buffer[hoursPerWeek] works: a built-in array's size must be a compile-time constant, which a plain const int set from, say, user input could not guarantee, but a constexpr always can.
3 const vs constexpr

"Won't change" vs "known now" 🔍

They look similar but answer different questions. const means "I won't modify this"; constexpr means "this is computable at compile time" (and is implicitly const too):

DeclarationKnown at compile time?
constexpr int a = 5 * 4;yes — always a compile-time constant
const int b = 5 * 4;maybe — here yes, but…
const int c = readNumber();no — value comes at runtime
A const just promises you won't change the variable — its value might still only be known at runtime (like c above, set from input). A constexpr makes a stronger promise: the value is fully computable while compiling. So every constexpr is const, but not every const is constexpr. When you want a genuine compile-time constant (for an array size, a template argument, a fixed table), reach for constexpr.
4 constexpr functions

Functions that run at compile time 🛠️

Mark a function constexpr and the compiler can run it during compilation when given compile-time arguments — yet it still works normally at runtime when given runtime ones:

constexpr int square(int x) { return x * x; }

constexpr int s = square(5);   // computed at COMPILE time → 25
int arr[square(4)];           // 16 — used as a compile-time size

int n = readNumber();
int r = square(n);            // same function, now runs at RUNTIME
A constexpr function is dual-purpose: call it with constants in a constant context and it folds away to a literal at compile time; call it with runtime values and it behaves like any ordinary function. The compiler decides based on the arguments and where the result is used. This is what makes constexpr powerful — you write the logic once and the compiler runs it early whenever it can. (Early C++11 limited these functions to a single return; C++14 and later allow loops, locals, and branches, so they read like normal code.)
5 When it applies

Constants in, constant out 🎯

The rule of thumb: constexpr can compute at compile time only when everything it needs is a compile-time constant. Feed it a runtime value and it gracefully falls back to running at runtime:

🚨 constexpr is a "can", not a "must" — and the inputs must be constant too

constexpr on a function means it's eligible to run at compile time, not that it always does. If you call square(n) with a runtime n, it simply runs at runtime like a normal function — no error. The compile-time magic only happens when the arguments are themselves compile-time constants and the result is used in a constant context (initialising a constexpr variable, an array size, a template argument). The common mistake: writing constexpr int s = square(userInput); — that's a compile error, because you demanded a compile-time value but handed it a runtime one. Either make the input constexpr, or store the result in a plain int. (Related: consteval in C++20 forces compile-time evaluation always — a niche tool you can note and move on.)

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🏃
Feeding a runtime value to a constexpr variable constexpr int s = square(userInput); won't compile — you demanded a compile-time value but gave a runtime one. Use a plain int, or make the input constexpr.
🔀
Confusing const with constexpr const only means "won't change" — its value can still be runtime-only. For a genuine compile-time constant (array size, template arg), you need constexpr.
🤔
Expecting constexpr to force compile-time A constexpr function only runs at compile time when its inputs are constant and the result is used in a constant context; otherwise it runs at runtime. (Use C++20 consteval to force it.)

Practice exercises 💪

Easy Compile-time size. Declare constexpr int N = 4 * 5; and use it to size an array int grid[N];. Confirm it compiles, then try the same with a const int set from std::cin and see it fail as an array size.
Medium Dual-purpose function. Write constexpr int cube(int x). Use it once to initialise a constexpr variable (compile time) and once on a value read from input (runtime). Confirm both work.
Stretch Compile-time table. Write a constexpr function factorial(int n) (C++14 style, with a loop or recursion) and build a small constexpr array of the first few factorials. Verify the values are computed at compile time (e.g. use one as an array size).

🎓 What to remember

  • constexpr asks the compiler to compute a value at compile time and bake the result into the program — free at runtime.
  • A constexpr variable must be initialised from compile-time-known values; it's a true compile-time constant (array sizes, template args).
  • const ≠ constexpr: const means "won't change" (maybe runtime); constexpr means "known while compiling." Every constexpr is also const.
  • A constexpr function runs at compile time with constant inputs, and at runtime otherwise — same code, both worlds.
  • It's a "can," not a "must" — demanding a compile-time value from runtime input is a compile error.

Operators Part 1 — Arithmetic, Assignment, Comparison

Operators are the verbs that act on your variables. Today: the math operators (+, −, *, /, %), the assignment family (=, +=, *=, ...), and the comparison operators that return true or false. Plus the deadliest beginner trap: == vs =.

1 Why this matters

Operators turn data into decisions ⚙️

Variables hold values — but a program that just stores values does nothing useful. Operators are what let your program act on those values: add them, compare them, update them. Without operators, all you have is filing.

You'll use these constantly. Add two numbers? Operator. Check if something equals something else? Operator. Update a counter? Operator. By the end of this lesson, you'll have met about 15 of them — enough to write programs that compute real things.

2 Arithmetic operators

The math you already know — plus one new face 🧮

OperatorMeaningExampleResult
+Addition10 + 313
-Subtraction10 - 37
*Multiplication (asterisk, not ×)10 * 330
/Division10 / 33 (integer division!)
%Modulo (remainder)10 % 31

Four of those look familiar. The fifth — modulo (%) — gives you the remainder of a division. 10 % 3 is 1 because 10 ÷ 3 is 3 remainder 1. It's surprisingly useful: n % 2 == 0 tells you if n is even; seconds % 60 gives you the seconds part of a clock.

🚨 Integer division truncates!

In C, 10 / 3 is 3, not 3.333…. When both sides of / are integers, you get integer division — the fractional part is discarded.

To get a real division, at least one side must be a floating-point: 10.0 / 3 gives 3.333…, and so does 10 / 3.0. Once one side is a double, the other gets converted automatically.

This trips up nearly every beginner once. You write double avg = (a + b + c) / 3;, expecting a decimal, and get a whole number. The fix: write 3.0 instead of 3, or cast: (double)(a + b + c) / 3.

3 Assignment operators

= and its compound friends 📝

You've already used = — that's plain assignment: take what's on the right, put it into the variable on the left. C adds six compound versions that combine an arithmetic op with assignment.
OperatorShorthand forExample
=(plain assignment)x = 10
+=x = x + nx += 5
-=x = x - nx -= 5
*=x = x * nx *= 2
/=x = x / nx /= 2
%=x = x % nx %= 3

They're pure typing-savings: x += 5 is identical in behavior to x = x + 5. Use whichever reads more naturally for the situation — but the compound forms are very common in real code, so you'll see them everywhere.

4 Comparison operators (and the deadliest trap)

Comparison ops return 1 (true) or 0 (false) 🔍

OperatorMeaningExampleResult
==Equal to5 == 51 (true)
!=Not equal to5 != 31 (true)
<Less than3 < 51 (true)
>Greater than3 > 50 (false)
<=Less than or equal5 <= 51 (true)
>=Greater than or equal5 >= 60 (false)

In C, comparisons don't return a special bool value — they return 1 for true and 0 for false (as plain ints). That's why you can do int x = (a < b); and store the result.

🚨 The == vs = trap — every C programmer's first big bug

= is assignment. == is comparison. Mix them up and your code does something very different from what you meant:

if (x = 5) { ... } // BUG: assigns 5 to x, then tests "is 5 truthy?" → always runs!
if (x == 5) { ... } // correct: compares x to 5

The single = version compiles fine — C considers it a perfectly valid expression. Many compilers warn about it (-Wall on by default), but you should still develop the habit of double-checking.

One defensive trick some old-school C programmers use: write the constant on the left — if (5 == x). If you typo it to if (5 = x), it's a compile error (you can't assign to a literal). Not everyone likes the style — but it's a real, working defense.

7 Common mistakes

Three classic operator traps 🚧

🙅
"= and == are the same." They are not, and the bug they create is silent. if (x = 5) compiles, assigns 5 to x, then tests whether 5 is non-zero (it is — so the branch always runs). Compile with -Wall; GCC will warn. If you find yourself typing = in an if, stop and recheck.
🙅
"10 / 3 is 3.33." Not in C with integer operands. Integer division truncates: 10 / 3 == 3. To get a real division, make at least one operand a floating-point: 10.0 / 3 or (double)x / y. This is the classic average-of-integers bug.
🙅
"% works on doubles." Modulo is integer-only in C. 10.5 % 3.0 doesn't compile. For floating-point remainder, use fmod(10.5, 3.0) from <math.h>. (You'll see this if you try to use % with any double.)
8 Your turn

Real arithmetic 💪

Easy A tiny calculator. Write a program that declares int a = 17, b = 5; and prints all five arithmetic results on separate lines: a+b, a-b, a*b, a/b, a%b. Then change them to double and rerun — what changes for a/b? What about a%b?
Medium Even or odd? Write a program that declares int n = 17; and prints whether it's even or odd. Hint: use n % 2 — if it's 0, even; if 1, odd. Use an if with ==. (Yes, we're foreshadowing lesson 3.1 — but this exercise is small enough to figure out.)
Stretch The Yoda trick. Write a program with int x = 10; and an if. Try if (x = 0) on purpose. Compile with -Wall — what does GCC say? Now flip it: if (0 = x). Read that error message. Now write it the right way: if (x == 0). Sometimes the defensive Yoda style (if (0 == x)) really does prevent the bug — note the trade-off in readability vs safety.

🎓 Quick recap

  • Arithmetic: + - * / and modulo % (remainder).
  • Integer division truncates: 10 / 3 = 3. Make one operand a double to get real division.
  • Assignment: =, plus the compound forms += -= *= /= %= — pure typing savings.
  • Comparison: == != < > <= >= — return 1 for true, 0 for false.
  • The deadly trap: = assigns, == compares. Compile with -Wall to catch the bug at compile time.

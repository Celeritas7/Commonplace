Capstone: A To-Do App

Time to build something real. We'll make a command-line to-do manager that remembers your tasks between runs by saving them to a file. It ties together everything: a struct for the data, a std::vector to hold it, file streams to persist it, and string parsing to load it back.

1 What we're building

The graduation project 🏗️

A to-do manager you run from the terminal. You can add a task, list all tasks, mark one done, and remove one — and because it saves to a file, your list is still there next time you run it. That last part is what makes it feel like a real tool rather than a toy. Building it pulls together the threads of the whole book: a struct Task for the data (Part VIII), a std::vector<Task> to hold them (Part V), file streams to save and load (Part XI), getline + parsing to read the file back (11.2), and a command loop driving it all. No new syntax — just everything you know, assembled.

Try the finished program in the interactive below, then walk through how each piece is built. By the end you'll have a mental blueprint you could type out yourself.

2 The data model

A Task, and a list of them 🗂️

Start with the data. A task needs an id (to refer to it), a done flag, and its text. A small class owns the collection and the "next id" counter:

struct Task {
    int id;
    bool done;
    std::string text;
};

class TaskList {
    std::vector<Task> tasks_;
    int nextId_ = 1;
public:
    void add(const std::string& text);
    bool markDone(int id);
    bool remove(int id);
    void print() const;
    void save(const std::string& path) const;
    void load(const std::string& path);
};
Task is a plain data struct; TaskList encapsulates the std::vector<Task> and the nextId_ counter so the rest of the program never touches them directly — it just calls add, markDone, remove, and so on (the same separation you saw in Part VI). Keeping a nextId_ that only ever increases means ids stay unique even after deletions, so done 3 always means the same task. This little class is the program's brain; the rest is plumbing around it.
3 Saving & loading

Persistence: one line per task 💾

The whole point is that tasks survive between runs. We save each task as a text line — id|done|text — and load by parsing those lines back:

void TaskList::save(const std::string& path) const {
    std::ofstream out(path);                 // truncate + write (11.1)
    for (const Task& t : tasks_)
        out << t.id << '|' << t.done << '|' << t.text << '\n';
}

void TaskList::load(const std::string& path) {
    std::ifstream in(path);
    if (!in) return;                        // no file yet — start empty
    tasks_.clear();
    std::string line;
    while (std::getline(in, line)) {          // one task per line (11.2)
        std::istringstream ls(line);
        std::string idStr, doneStr, text;
        std::getline(ls, idStr,   '|');
        std::getline(ls, doneStr, '|');
        std::getline(ls, text);              // rest of line = the text
        Task t{ std::stoi(idStr), doneStr == "1", text };
        tasks_.push_back(t);
        nextId_ = std::max(nextId_, t.id + 1);
    }
}
save opens an ofstream (which truncates, so the file always reflects the current list) and writes one line per task. load does the reverse: getline pulls a whole line, then an istringstream with the three-argument getline(ls, field, '|') splits on the | delimiter — the final getline(ls, text) grabs everything left, so the task text may safely contain spaces. We rebuild nextId_ as we go so new ids won't collide with loaded ones. If the file doesn't exist yet, load just returns and we start with an empty list — a brand-new user's first run.
4 The commands

add, done, remove, list ⚙️

Each command is a small method on TaskList. They're short because the data model does the heavy lifting:

void TaskList::add(const std::string& text) {
    tasks_.push_back(Task{ nextId_++, false, text });   // unique id, not done
}

bool TaskList::markDone(int id) {
    for (Task& t : tasks_)
        if (t.id == id) { t.done = true; return true; }
    return false;                                       // no such id
}

bool TaskList::remove(int id) {
    auto it = std::remove_if(tasks_.begin(), tasks_.end(),
                  [&](const Task& t){ return t.id == id; });
    if (it == tasks_.end()) return false;
    tasks_.erase(it, tasks_.end());                     // erase-remove idiom
    return true;
}
add appends a new Task with the next id and done = false. markDone scans for the matching id and flips its flag (returning false if none matched, so the caller can report "no such task"). remove uses the erase-remove idiom from the STL: std::remove_if shifts the matching element to the end and returns where the "kept" part stops, then erase trims it off. Returning a bool from markDone/remove lets the command loop give friendly feedback instead of failing silently.
5 The main loop

Read a command, do it, save 🔁

Finally, main loads the file once, then loops: read a command line, dispatch to the right method, and save after anything that changed:

int main() {
    TaskList list;
    list.load("todos.txt");                       // restore previous session

    std::string line;
    std::cout << "> ";
    while (std::getline(std::cin, line)) {
        std::istringstream cmd(line);
        std::string action;  cmd >> action;          // first word = command

        if (action == "add") {
            std::string text;  std::getline(cmd, text);   // rest = task text
            list.add(text.substr(1));                // drop leading space
        } else if (action == "done") {
            int id;  cmd >> id;  list.markDone(id);
        } else if (action == "rm") {
            int id;  cmd >> id;  list.remove(id);
        } else if (action == "list") {
            list.print();
        } else if (action == "quit") break;

        list.save("todos.txt");                   // persist every change
        std::cout << "> ";
    }
}

🚨 Save after every change, and remember why it's text not binary

Two design notes. First, persistence timing: we call save after each command so a crash or a forgotten quit never loses data — for a tiny file that's cheap and safe (a bigger app might save less often or on exit). Second, why a text format and not the binary byte-dump from 11.4? Because Task contains a std::string, which holds a pointer to its characters — byte-dumping the struct would save the pointer, not the text, and reload as garbage. Writing the text out as id|done|text stores the actual characters, and as a bonus the file is human-readable and editable. That's the right call whenever your records contain strings or other heap-owning types.

    8 Mistakes, extensions, recap

    Wrap up 📚

    Three mistakes to avoid 🚨

    🆔
    Reusing ids after deletion If you set the next id from tasks_.size(), deleting then adding can reuse an id and confuse done/rm. Keep a separate, only-increasing nextId_ (and restore it on load).
    📑
    Splitting text on a character it may contain We split fields on |, so a task containing | would break parsing. Choose a delimiter the text won't have (or escape it), and read the last field with plain getline so spaces survive.
    💾
    Forgetting to save Changes live only in the in-memory vector until save writes them out. Call save after every mutating command (or at least before exit), or the file and the list drift apart.

    Extend it 💪

    Easy Pretty listing. Make print() show [x] for done and [ ] for not, with the id right-aligned in a width-3 column using std::setw (11.3).
    Medium A due date. Add a std::string due field to Task. Update save/load to round-trip it (mind the delimiter), and an add syntax that accepts it.
    Stretch Undo. Keep a stack of previous states (or of inverse commands) so a undo command can roll back the last change. Decide what "a state" is and how much history to keep.

    🎓 What you built

    • A real, persistent program — add/list/done/remove tasks that survive between runs by saving to a file.
    • Data model: a struct Task held in a std::vector<Task>, encapsulated by a TaskList class with a unique-id counter.
    • Persistence: save writes one id|done|text line per task; load uses getline + an istringstream to parse them back.
    • Commands are small methods (add, markDone, remove); main is a read-dispatch-save loop.
    • Text format, not a binary dump, because Task holds a std::string — and it's human-readable too.

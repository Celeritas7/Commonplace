switch — Clean Multi-Way Branching

When you're choosing between many fixed values — a menu option, a day number, a command key — a long if / else if chain gets ugly fast. switch is the tidy alternative, with one famous catch: the break.

1 Why this matters

One value, many destinations 🧠

Some decisions aren't "is this true?" but "which one of these is it?" A menu where 1 means new game, 2 means load, 3 means quit. A traffic light that's red, yellow, or green. When you're matching a single value against a list of fixed possibilities, switch reads cleaner than a tall stack of else ifs — and signals your intent at a glance.

You could write all of this with if / else if (you learned that last lesson). switch doesn't add new power — it adds clarity for one specific shape of decision: comparing one integer or character against several constant values. The price of that clarity is a quirk unique to C's switch — once a case matches, execution keeps going until it hits a break. Master that, and switch is your friend.

2 Anatomy of a switch

The parts 🧩

switch (choice) {
    case 1:
        printf("New game\n");
        break;
    case 2:
        printf("Load game\n");
        break;
    default:
        printf("Unknown option\n");
}
  • switch (choice) — C evaluates choice once, then jumps to the matching case.
  • case 1: — a label. If choice equals 1, execution jumps here. Case values must be constant integers or characters.
  • break; — exits the switch. Without it, execution "falls through" into the next case.
  • default: — the catch-all, run when no case matches. Optional, but usually a good idea.
Think of the case labels as entry doors, not separate boxes. C finds the matching door, walks in, and then keeps walking downward through whatever follows — across other case labels — until a break stops it or the switch ends. That walking-downward behavior is the whole story of switch.
3 Fall-through

The break you can't forget 💥

🚨 Forgetting break = fall-through

If you leave out a break, execution doesn't stop at the end of that case — it spills into the next one. Given choice = 1 with no breaks, this prints all three lines:

case 1: printf("one ");  // no break!
case 2: printf("two ");  // no break!
case 3: printf("three");
// output: one two three

This is the #1 switch bug. The fix is simple: end every case with break;.

Fall-through isn't always a bug, though — sometimes it's exactly what you want. To make several values run the same code, stack their case labels with nothing between them:

switch (key) {
    case 'a':
    case 'A':
        printf("Move left\n");
        break;        // 'a' and 'A' share this code
}
The rule of thumb: intentional fall-through is only for stacked, empty case labels (grouping values that do the same thing). Any case that has its own code should end with break. When you deliberately fall through after running code, leave a comment like /* fall through */ so the next reader knows it wasn't a mistake.
4 switch vs if

Which one when 🤔

switch isn't always the right call. It only works for one narrow shape of decision; outside that, if / else if is clearer or simply required.

Use switch when…Use if / else if when…
Matching one value against fixed constants (1, 2, 3 or 'a', 'b')Testing ranges (score >= 90)
Many discrete cases — a menu, a command, a stateCompound conditions (x > 0 && y < 10)
The value is an int or charComparing doubles, strings, or anything non-constant
The hard limits: switch only branches on integer-like values (int, char, enums), and every case must be a compile-time constant — you can't write case x: or case 0...9: or test a range. If your decision involves a range, a comparison, a string, or two variables, reach for if. If it's "pick the matching value from this fixed list," reach for switch.
5 A couple more rules

Small print worth knowing 📋

  • default can go anywhere, but putting it last reads best. If it's missing and nothing matches, the switch simply does nothing.
  • Case order doesn't matter for correctness (unlike an if chain) — C jumps straight to the match. Order them however reads clearest.
  • You can't compare ranges or non-constants. case 1: case 2: case 3: is fine; case 1...3: is not standard C.
  • Declaring a variable inside a case can cause trouble — wrap a case body in its own braces { } if you need a local variable there.
A handy mental model: a switch is really a computed jump. C looks at the value, leaps directly to the matching label, and runs forward from there. That's why it can be faster than a long if chain for many cases — and why fall-through exists at all. It's not testing each case; it's jumping to one and then flowing.

    case 1: printf("one ");
    case 2: printf("two ");
    case 3: printf("three ");
    default: printf("other");
}

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

💥
Forgetting break Without it, execution falls through into the next case and runs its code too. End every case that has its own body with break; unless you mean to fall through.
📏
Trying to switch on a range or a string case score >= 90: and case "yes": don't work — cases must be constant integers or chars. For ranges and strings, use if.
🧰
Declaring a variable in a case without braces case 1: int y = 5; can fail to compile because the variable's scope crosses other case labels. Wrap the case body in its own { } block when you need a local.

Practice exercises 💪

Easy Day namer. Read an int day (1–7) and print the weekday name with a switch. Add a default that prints "Invalid day". Don't forget the breaks!
Medium Vowel counter. Read a char and print "vowel" or "consonant". Use stacked case labels (case 'a': case 'e': ...) so all five vowels share one branch — intentional fall-through done right.
Stretch Tiny calculator. Read two numbers and an operator char ('+', '-', '*', '/'). Use a switch on the operator to compute and print the result. Handle division by zero, and use default for an unknown operator.

🎓 What to remember

  • switch (value) jumps to the matching case label, then runs forward from there.
  • break; exits the switch. Without it, execution falls through into the next case — the classic bug.
  • Intentional fall-through is for stacked empty case labels that share one block of code.
  • default catches everything unmatched; cases must be constant int/char values.
  • Use switch for discrete constants; use if for ranges, compound conditions, and non-integers.

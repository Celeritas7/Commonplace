Scope and Lifetime

Two questions about every variable: where can the code see it (its scope), and how long does it exist (its lifetime)? Functions made these matter — each one has its own private variables that appear and vanish as the function runs.

1 Why this matters

Variables aren't visible everywhere 🧠

Now that you write functions, a natural question appears: if main has a variable x and a helper also has an x, are they the same? (No.) Can the helper see main's variables? (No.) The answers come from two ideas: scope — the region of code where a name is usable — and lifetime — the span of time the variable actually exists in memory.

Getting these right is what lets functions stay independent: each one has its own private variables that nobody else can touch, created fresh when the function runs and cleaned up when it returns. That isolation is exactly what made functions testable back in 4.1. Let's see where variables live and how long they last.

2 Local scope

Declared inside, seen inside 🏠

A variable declared inside a function is local to it — only that function's code can use it. Two functions can each have an x and they're completely separate:

void helper(void) {
    int x = 99;          // helper's own x
    printf("%d\n", x);   // 99
}

int main(void) {
    int x = 1;           // main's own x — unrelated
    helper();
    printf("%d\n", x);   // still 1
}
The parameter names in a function header are local too — they're variables that live inside the function. This is the whole reason a function can't accidentally stomp on the caller's data: the names simply aren't visible across the boundary. (To reach out, you'd pass a pointer — that's a later chapter.)
3 Block scope

Every { } makes a room 🚪

Scope isn't only per-function — it's per block. Any pair of braces { } opens a new, smaller scope. A variable declared inside is invisible outside it:

int main(void) {
    for (int i = 0; i < 3; i++) {
        int sq = i * i;       // sq lives only in the loop body
        printf("%d ", sq);
    }
    // printf("%d", sq);  // ERROR: sq doesn't exist here
    // printf("%d", i);   // ERROR: i (C99 loop var) is gone too
}
Think of each { } as a room: things you declare inside are only on that room's table, and they're cleared away when you leave. The loop counter i declared in for (int i = …) belongs to the loop, so it doesn't exist after the loop ends. Declaring variables in the smallest block that needs them is good style — it keeps names from leaking and colliding.
4 Lifetime

Born on entry, gone on exit ⏳

Scope is about where; lifetime is about when. A local variable has automatic lifetime: it's created when control reaches its declaration and destroyed when its block ends. Each call to a function gets a brand-new set:

void tick(void) {
    int count = 0;   // fresh 0 every single call
    count++;
    printf("%d ", count);
}
// tick(); tick(); tick();  prints: 1 1 1  (not 1 2 3)
That's why count prints 1 every time, not 1 2 3 — it doesn't remember anything between calls, because it's destroyed when tick returns and recreated next time. A local variable's value does not survive past the end of its block. (If you genuinely need it to persist, that's what static is for — the very next lesson.)
5 Global scope & shadowing

The shared shelf, and name clashes 🌍

A variable declared outside all functions is global: visible to every function, and it lives for the entire program. Handy, but use it sparingly — anyone can change it, which makes bugs hard to trace.

int g = 10;            // global — lives the whole program

int main(void) {
    int g = 99;        // a LOCAL g — shadows the global inside main
    printf("%d\n", g);  // 99 — the nearest g wins
}
When an inner name matches an outer one, the inner one shadows the outer for the duration of its scope — the closest declaration wins. It's legal but confusing, so avoid reusing names across scopes when you can. Prefer passing values in as parameters over reaching for globals; your functions stay independent and easy to reason about.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🚪
Using a variable outside its block A variable declared inside { } (or a for header) doesn't exist after that block. Referring to it later is a "not declared in this scope" error.
🔁
Expecting a local to remember its value Locals are recreated every call. If you need a value to persist between calls, that's static (next lesson) — not an ordinary local.
🌍
Reaching for a global to "share" data Globals are visible everywhere, which sounds convenient but makes bugs hard to track — any function might have changed it. Prefer passing values as parameters and returning results.

Practice exercises 💪

Easy Two x's. Write a helper with a local x = 50 and a main with a local x = 7. Print both. Confirm that calling helper doesn't change main's x.
Medium Fresh every time. Write a function with a local int n = 0; n++; printf("%d ", n);. Call it three times and predict the output before running it. (Then peek ahead: what would static int n = 0; change?)
Stretch Shadow hunt. Make a global int total = 100;. Inside main, declare a local int total = 5; and print it. Then write a separate function that prints total with no local — and see which total each one uses.

🎓 What to remember

  • Scope = where a name is visible; lifetime = how long the variable exists.
  • Locals are visible only inside their block (function or any { }), including loop counters.
  • Locals are automatic: created on entry, destroyed when the block ends — fresh on every call.
  • Globals live the whole program and are visible everywhere — powerful, but use sparingly.
  • An inner name shadows an outer one; the nearest declaration wins. Avoid the clash when you can.

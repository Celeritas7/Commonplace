const Member Functions

Some methods only read an object; they shouldn't change it. Marking such a method const makes that a promise the compiler enforces — and it's what lets the method run on const objects. This habit, called const-correctness, is a hallmark of solid C++.

1 Why this matters

Some methods only look 🧠

A class usually has two flavours of method: ones that change the object (a setter, deposit, increment) and ones that merely read it (a getter, area, print). C++ lets you mark the read-only ones const, which does two valuable things: it documents your intent ("this won't touch the object"), and it's required for the method to be callable on a const object or through a const reference.

This matters more than it first appears, because passing objects by const& (cheap and safe) is everywhere in real C++ — and that only works if the methods you call are const. Get into the habit now.

2 What it is

const after the parentheses 🏷️

You make a method const by writing the keyword after the parameter list, before the body. It's a promise: "this function will not modify the object":

class Temperature {
    double celsius = 20;
public:
    double get() const { return celsius; }   // read-only → const
    void set(double c) { celsius = c; }       // modifies → NOT const
};
Read double get() const as "get returns a double and promises not to change the object." The const goes right after the (). A method that changes a member — like set — must not be const, because it breaks that promise.
3 const objects need const methods

The promise unlocks access 🔒

Here's the payoff. On a const object, you can only call const methods — the compiler refuses anything that might modify it:

const Temperature t;   // a const object — must not change

t.get();   // ✅ ok — get() is const, safe to call
t.set(5);  // ❌ error — set() could modify t, forbidden on a const object
It makes sense: if t is const, calling a method that might change it would break the guarantee, so the compiler won't allow it. Only the methods that promised not to modify (the const ones) are permitted. That's why a getter must be const to be usable on const objects.
4 The compiler enforces it

Break the promise, get an error 🛡️

A const method isn't just a label — inside it, the object is treated as read-only. Try to modify a member and the program won't compile:

double get() const {
    celsius = 0;     // ❌ compile error — can't modify in a const method
    return celsius;
}
This is the safety net: the const keyword turns "I promise not to change the object" into a guarantee the compiler checks for you. If a method genuinely needs to modify the object, the fix is simple — don't mark it const. The keyword and the behaviour must agree.
5 Where it shows up

const& and const-correctness 📦

The everyday reason this matters: passing objects by const reference (const T&) is the standard way to hand a big object to a function cheaply without copying it. Through such a reference, only const methods are callable:

void report(const Temperature& t) {
    std::cout << t.get();   // ✅ get() is const
    // t.set(0);            // ❌ would not compile
}
So if your getters aren't const, you can't even read the object through a const& — a constant source of beginner compile errors. The rule of thumb is simple and worth adopting forever: mark every method that doesn't modify the object const. Do that and your classes "just work" with const objects and const references.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔑
Forgetting const on getters A getter that isn't const can't be called on a const object or through a const&. Mark every read-only method const so the class works with const objects.
✏️
Modifying a member in a const method The compiler rejects it — that's the guarantee at work. If a method needs to change the object, it simply must not be const.
📍
Putting const in the wrong place For a const method it goes after the parameter list: int get() const;. const before the return type (const int get()) makes the return value const instead — a different thing.

Practice exercises 💪

Easy A const getter. Write class Circle with a private radius and a double getRadius() const. Create a const Circle and call the getter — confirm it compiles and prints.
Medium Feel the error. Add a non-const void setRadius(double). Try calling it on a const Circle and read the compiler error. Then call it on a normal Circle — fine.
Stretch Pass by const reference. Write void print(const Circle& c) that prints c.getRadius(). Notice it only compiles because getRadius is const — remove the const from the getter and watch print break.

🎓 What to remember

  • A const member function promises not to modify the object — write const after the parameter list: int get() const;.
  • const objects (and const references) can only call const methods.
  • The compiler enforces the promise — modifying a member inside a const method is an error.
  • Mark every read-only method const — that's "const-correctness."
  • It's what lets you pass objects by const& — cheap and safe — and still read them.

A Note on goto

C has a keyword that jumps execution straight to a labeled line, anywhere in the function: goto. You'll meet it in old code, so it's worth understanding — but modern C almost never uses it, and this short lesson explains why.

1 Why this matters

Know it, then move on 🧭

This is a short one. goto is a real C keyword, and you'll occasionally see it in old codebases, textbooks, or systems code. So you should recognize it and understand what it does. But it's the one piece of C flow control that the whole industry agreed, decades ago, to mostly stop using — and this lesson is about why, so you can confidently leave it alone.

Everything you've learned in Part III — if, switch, loops, break, continue — exists precisely so you don't need goto. They're the "structured" replacements that made code readable. Think of this lesson as the historical footnote that ties a bow on the chapter.

2 What goto is

A jump to a label 🏷️

goto transfers execution, unconditionally, to a label elsewhere in the same function. A label is just a name followed by a colon:

    i = 0;
start:                       // a label
    printf("%d ", i);
    i++;
    if (i < 3)
        goto start;          // jump back up to 'start:'
    printf("done");
// prints: 0 1 2 done
That's a loop built out of goto — and it works. But notice you have to mentally follow the jump to understand it: read down, hit the goto, scroll your eyes back up to the label, repeat. A for loop says the same thing in one tidy header. The Jump Tracer below lets you step through this exact program and watch execution leap around.
3 Why it's discouraged

The road to spaghetti 🍝

🚨 Unrestricted jumps make code unreadable

Because goto can jump anywhere in a function — forward, backward, into the middle of things — a program full of them becomes impossible to follow. You can't tell how you arrived at a given line or where you'll go next without tracing every jump by hand. This tangled style earned the nickname "spaghetti code," and in 1968 a famous letter titled "Go To Statement Considered Harmful" kicked off the move to structured programming — the loops and ifs you now use.

The fix the industry adopted: build all control flow out of a few well-behaved structures — sequence, selection (if/switch), and iteration (loops) — each with one obvious entry and exit. Those are easy to read because the structure tells you the flow. goto throws that guarantee away, so it's avoided by default.

4 The one place it survives

Error cleanup in C 🧹

✅ A single forward jump to a cleanup block

C has no automatic cleanup (no destructors like C++). So when a function grabs several resources and any step can fail, a tidy pattern is one goto that jumps forward to a shared cleanup section that releases everything:

f = fopen(...);
if (!f) goto cleanup;
buf = malloc(...);
if (!buf) goto cleanup;
... use them ...
cleanup:
  free(buf); fclose(f);

You'll see this a lot in the Linux kernel and C libraries. It's accepted because it's one direction (forward), one target, and replaces deeply nested ifs — the opposite of spaghetti.

This is the honest exception: a disciplined, forward-only goto for error handling can be cleaner than the alternatives in plain C. Outside that narrow pattern, reach for structured control flow.

5 What to use instead

You already have the tools 🧰

Instead of goto, to……use
repeat a blockfor / while loop
stop a loop earlybreak

Every row above is something you learned this chapter. That's the takeaway: structured control flow covers essentially every case where a beginner might reach for goto. Knowing goto exists is useful for reading other people's code; using it should be a deliberate, rare choice — basically just the cleanup pattern.

8 Recap & what's next

Wrap up 📚

Two things to avoid 🚨

🍝
Using goto for ordinary control flow Building loops or skipping code with goto instead of for/while/break/continue makes code hard to follow. Use the structured tools — that's what they're for.
↩️
Jumping backward or into blocks The only broadly-accepted goto is a single forward jump to a cleanup label. Backward jumps (fake loops) and jumps into the middle of other blocks are the spaghetti the world moved away from.

Practice exercise 💪

Medium De-goto it. Take the goto loop from this lesson (start: … goto start;) and rewrite it as a plain for loop that prints 0 1 2. Notice how the for version makes the loop's bounds obvious in one line — no jump to trace.

🎓 What to remember

  • goto label; jumps unconditionally to a labeled line in the same function.
  • It's discouraged because unrestricted jumps create unreadable "spaghetti code."
  • Structured flow replaced it: loops, if/switch, break, continue, return cover essentially every case.
  • The one accepted use: a single forward jump to an error-cleanup block in plain C.
  • Bottom line: recognize it when reading old code; reach for it almost never.

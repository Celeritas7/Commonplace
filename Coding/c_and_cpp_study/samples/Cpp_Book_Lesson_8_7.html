Why new & delete Are Rare

This is where Part VIII pays off. With RAII, containers, and smart pointers in hand, modern C++ has almost no reason to write raw new and delete — and every reason not to. This lesson is the decision guide: for any object, which tool do you reach for? Spoiler — it's almost never new.

1 Why this matters

The payoff of the whole Part 🧠

Coming from C, it's natural to assume "heap object = new, then delete later." But every lesson in Part VIII has been quietly removing the need for that. RAII (8.1) ties cleanup to scope; copy and move (8.2–8.3) make owning objects safe to pass around; the Rule of Zero (8.4) means you don't hand-write resource management; and the smart pointers (8.5–8.6) wrap heap ownership for you. Put together, they mean modern C++ code almost never contains a raw new or delete — and code that does is usually a bug waiting to happen. This finale ties it all into one simple decision.

The question to ask about any object is just: who owns it, and when does it die? The right tool answers that for you.

2 The ladder of choices

Reach for these, in order 🪜

For almost any object, walk down this list and stop at the first that fits. Raw new is at the bottom for a reason:

NeedReach for
a local object, normal lifetimea value on the stack — Point p;
a collection of thingsstd::vector / std::string / std::map
one heap object, single ownerstd::unique_ptr
one heap object, shared ownersstd::shared_ptr
raw new / deletealmost never (see §4)
Most objects don't need the heap at all — a plain value on the stack is the simplest and fastest, and it cleans itself up. Collections go in a container, which manages its own storage. Only when you genuinely need a heap object with a lifetime longer than one scope do you reach for a smart pointer — unique_ptr for a single owner, shared_ptr when ownership is truly shared. Each rung answers "who owns it, when does it die?" without you writing any cleanup.
3 Why avoid raw new/delete

Four ways it goes wrong ⚠️

Manual new/delete isn't just old-fashioned — it's a reliable source of the worst bugs in C++:

void f() {
    Widget* w = new Widget;   // manual heap object
    if (early) return;          // 🔴 leak — delete skipped
    risky();                    // 🔴 throws? leak — delete skipped
    delete w;                   // only runs on the happy path
    // 🔴 forget it → leak;  🔴 do it twice → crash
}
Four classic failures: leaks (you forget the delete, or a return/exception jumps over it), double frees (you delete the same pointer twice), dangling pointers (you use it after delete), and exception unsafety (a throw between new and delete leaks). A unique_ptr eliminates all four at once: its destructor runs on every exit path, it can't be accidentally copied into two owners, and there's no manual delete to forget, repeat, or skip.
4 When they still appear

Rare, but not extinct 👀

"Almost never" isn't "never." Raw new/delete still have a few legitimate homes — and raw pointers remain perfectly fine for one job:

  • Inside a resource wrapper: someone has to implement unique_ptr, a custom container, or a memory pool — that code uses new/delete so your code doesn't.
  • Interfacing with older or C APIs that hand back raw pointers you must manage (often wrapped in a smart pointer immediately).
  • Raw pointers as non-owning observers: a Widget* or Widget& that just looks at an object it doesn't own is normal and good — you simply never delete it.
The key distinction is ownership, not pointers themselves. A raw pointer that owns (you're responsible for deleteing it) is the thing to avoid. A raw pointer that merely refers to an object owned elsewhere — passed into a function that just reads it, say — is idiomatic. Own with values, containers, and smart pointers; observe with raw pointers and references.
5 The mental model

Who owns it, when does it die? 🧠

Everything in Part VIII reduces to making ownership visible in the type. You should be able to read a declaration and know exactly who's responsible for the object's lifetime:

Widget w;                          // I own it; dies at end of scope
std::vector<Widget> ws;             // the vector owns them all
std::unique_ptr<Widget> up;         // I'm the one owner; dies with me
std::shared_ptr<Widget> sp;         // shared; dies with the last owner
Widget* observer = up.get();       // I just look — I do NOT own or delete

🚨 The bug isn't the heap — it's untracked ownership

Beginners often think "heap = dangerous." It isn't; unclear ownership is. A raw owning new says nothing about who must free it or when, so two readers make opposite assumptions and you get a leak or a double free. Smart pointers fix this by encoding the answer in the type: unique_ptr means "exactly one owner," shared_ptr means "shared, freed at the last release." Once ownership is in the type, the compiler and the destructor enforce it for you — which is exactly why raw new/delete fade away. If you ever feel the urge to write new, ask "who will delete this, and can a smart pointer say so instead?"

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🆕
Reaching for new out of habit Coming from C, new feels default. In C++ start at the top of the ladder: a value, then a container, then a smart pointer. new is the last resort.
🧩
Using the heap when a value would do A local object with a normal lifetime should just be a value on the stack — simpler, faster, and self-cleaning. Don't heap-allocate it for no reason.
👻
Confusing observing with owning A raw pointer/reference that just looks at an object is fine — never delete it. The thing to avoid is a raw pointer that owns and must be manually freed.

Practice exercises 💪

Easy Heap to stack. Take a snippet that does Point* p = new Point{1,2}; ...; delete p; and rewrite it as a plain stack value. Confirm it's shorter and needs no delete.
Medium new → unique_ptr. Take a function that news a polymorphic object, uses it, and deletes it, and rewrite it with std::make_unique. Add an early return and confirm it no longer leaks (the old version would).
Stretch Own vs. observe. Write a function void use(const Widget&) (observes) and a function that owns a Widget via unique_ptr, passing up.get() or *up to use. Show the observer never deletes and the owner cleans up automatically.

🎓 What to remember

  • Modern C++ rarely writes new/delete — RAII, containers, and smart pointers handle ownership for you.
  • The ladder: value → container (vector/string) → unique_ptr → shared_ptr; raw new is the last resort.
  • Manual new/delete invites leaks, double frees, dangling pointers, and exception unsafety — smart pointers remove all four.
  • Raw pointers are fine for observing (non-owning); avoid raw pointers that own.
  • Make ownership visible in the type — answer "who owns it, when does it die?" and the cleanup takes care of itself.

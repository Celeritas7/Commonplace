🧬 PART VII · INHERITANCE & POLYMORPHISM

Inheritance Access

When you wrote : public Animal in 7.1, that public was the inheritance mode — and it can also be protected or private. The mode controls how the base's members are exposed through the derived class. public is the everyday choice; here's the short overview of all three.

1 Why this matters

One keyword, big effect 🧠

The word right before the base class — class Dog : public Animal — isn't decoration. It's the inheritance access mode, and it decides how everyone outside sees the members Dog inherited from Animal. With public (the normal case), Animal's public interface stays public on a Dog. Swap in protected or private and that inherited interface gets locked down. You'll almost always want public — but knowing what the alternatives do (and that the default is private for a class) saves you from baffling errors.

This is a short, "good to know" lesson. The headline: use public for "is-a," and understand why.

2 public inheritance

The everyday one 🌳

With public inheritance, the base's members keep their access level in the derived class — public stays public, protected stays protected. The inherited public interface is fully usable from outside:

class Animal { public: void breathe() {} };

class Dog : public Animal { };

Dog d;
d.breathe();   // ✅ breathe() is public on Dog too
This is the mode that models a true "is-a" relationship — a Dog is an Animal, so it should offer everything an Animal offers. It's what 7.1 used and what you'll reach for the vast majority of the time. The other two modes restrict the inherited interface instead.
3 protected & private inheritance

Locking down the interface 🔒

The other two modes shrink how the inherited members are exposed. They're uncommon — think of them as "implemented in terms of," not "is-a":

Inheritance modeBase's public members become…Usable from outside?
publicpublic in the derived classyes
protectedprotected in the derived classno (but further-derived classes can)
privateprivate in the derived classno (only inside the derived class)
With private inheritance, the derived class can still use the base internally, but no one outside can call the inherited methods through it — which is much closer to composition than to "is-a." With protected, only further-derived classes get access. Both are niche; you'll rarely write them, but you'll now recognise them.
4 What it doesn't change

The base's privates stay private 🎯

One thing the inheritance mode never affects: the base's own private members. Those are hidden from the derived class no matter which mode you use — exactly as in 7.1:

class Base {
    int secret;          // private to Base
public:
    int value;
};
class D : public Base {
    void f() {
        value = 1;       // ✅ public base member, fine
        // secret = 1;       // ❌ still hidden — mode can't unlock it
    }
};
So keep two ideas separate. The base's own access specifiers (6.3) decide what a derived class can touch at all — and private base members are always off-limits. The inheritance mode only decides how the members the derived class did inherit are re-exposed to the outside world. The mode can restrict, never expand.
5 The takeaway

Default to public ⚖️

The practical rule is short: use public inheritance for is-a relationships — which is nearly always what you want. Reach for protected/private inheritance only in special cases (and often composition is the clearer choice anyway, per 6.11):

🚨 class defaults to private inheritance

If you forget the keyword — class Dog : Animal — you get private inheritance, so the inherited public interface becomes private and d.breathe() won't compile from outside. This is a common, confusing beginner error. Always write the mode explicitly: class Dog : public Animal. (A struct defaults to public inheritance, mirroring its member-access default — but spell out public anyway so the intent is clear at a glance.)

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔑
Forgetting the keyword (class default = private) class D : Base is private inheritance, which hides the base's interface. For an is-a relationship, always write : public Base.
🔐
Thinking the mode unlocks the base's privates It never does. The base's private members stay hidden from the derived class regardless of mode. The mode only re-exposes the members that were inherited.
🤔
Reaching for private/protected inheritance casually They're niche ("implemented-in-terms-of"). For is-a use public; for reuse without is-a, composition (6.11) is usually clearer.

Practice exercises 💪

Easy Public works. Write struct Base { void hi(); } and struct D : public Base { }. Call d.hi() from main — it compiles, because the inherited interface stays public.
Medium Private blocks it. Change to class D : private Base { } and try calling the inherited method from main — read the compiler error. Then switch back to public and watch it compile.
Stretch Implemented-in-terms-of. With private inheritance, add a public method in D that internally calls the base's method. D uses the base but doesn't expose it — the "implemented-in-terms-of" idea private inheritance is for.

🎓 What to remember

  • The inheritance mode (the keyword before the base) controls how the base's members are exposed through the derived class.
  • public inheritance keeps the base's public/protected members public/protected — models "is-a" (the everyday choice).
  • protected/private inheritance make them protected/private in the derived class — rare, "implemented-in-terms-of," not is-a.
  • The mode never grants access to the base's private members (always hidden).
  • Default: class → private, struct → public — always write the keyword (usually public).

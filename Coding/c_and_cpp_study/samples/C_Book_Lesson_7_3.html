🧠 PART VII · MEMORY MANAGEMENT

calloc and realloc

malloc is the workhorse — but two cousins handle common needs more neatly. calloc gives you memory that's already zeroed; realloc resizes a block you already have (perfect for growing a dynamic array). Same heap, same free — just the right tool for the job.

1 Why this matters

Two specialists 🧠

malloc covers most needs, but two situations come up so often that C gives them dedicated functions. First: you want a block that starts at all zeros (counters, a clean buffer) — calloc does that for you. Second: you have a block and need it bigger or smaller — realloc resizes it in place where it can, copying your data along. Knowing these saves you boilerplate and bugs.

They're part of the same family as malloc/free (all in <stdlib.h>, all freed with free) — just tuned for these two jobs.

2 calloc — zeroed memory

Count, size, and all zeros 0️⃣

calloc takes two arguments — a count and the size of each element — and returns a block of count × size bytes that's already set to all zeros:

int *a = calloc(5, sizeof(int));   // 5 ints, every one is 0
if (a == NULL) return 1;

printf("%d\n", a[0]);   // 0 — guaranteed, no garbage
free(a);
The difference from malloc is the guarantee: malloc hands you uninitialized garbage, while calloc gives you a clean, zeroed block. That's exactly what you want for a counters array or any buffer you'll read before fully writing. (The two-argument form — count and element size — also makes the intent clear.)
3 realloc — resize a block

Grow your array 📏

realloc takes an existing pointer and a new size, and gives you back a block of that size — keeping your old data. It's how you grow a dynamic array when you need more room:

int *a = malloc(2 * sizeof(int));
a[0] = 1; a[1] = 2;

a = realloc(a, 4 * sizeof(int));   // now room for 4; a[0],a[1] preserved
a[2] = 3; a[3] = 4;
free(a);
Crucially, realloc may move the block to a new address (if it can't grow it in place), so it returns a pointer you must reassign — a = realloc(a, …). Any old copy of the pointer becomes dangling. Your existing data is copied across automatically, up to the smaller of the old and new sizes.
4 The realloc safety pattern

Use a temp pointer 🛟

There's a subtle trap: if realloc fails, it returns NULL but leaves your original block intact. If you assigned that NULL straight back to your pointer, you'd lose the original — a leak. So assign to a temporary first, and only update on success:

int *tmp = realloc(a, n * sizeof(int));
if (tmp == NULL) {
    // realloc failed; 'a' still points to the old, valid block
    free(a);            // handle the error; don't leak
    return 1;
}
a = tmp;                // success — now it's safe to update a

🚨 a = realloc(a, …) can leak on failure

Writing a = realloc(a, n); directly is convenient but risky: if realloc returns NULL, you've just overwritten a with NULL and lost the only pointer to the still-allocated original block — a memory leak. The temp-pointer pattern above avoids it. For learning examples the direct form is common, but remember the safe version for real code.

5 Which one when

The three, side by side 🧰

FunctionUse it when…Notes
malloc(n)you'll fill the block yourselffast; memory is uninitialized
calloc(c, s)you want it zeroedcount + element size; all zeros
realloc(p, n)you need to resize a blockmay move it — reassign the pointer
They share one cleanup: whatever you allocate — with malloc, calloc, or realloc — you release with a single free on the final pointer. Pick calloc when zeros matter, realloc when sizes change, and malloc for everything else. That's the whole toolkit for heap allocation.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🛟
Assigning realloc's result straight to the same pointer If realloc fails it returns NULL, and a = realloc(a, …) would overwrite your only pointer to the still-valid block — a leak. Use a temp: tmp = realloc(a, n); if (tmp) a = tmp;.
🚚
Forgetting realloc may move the block It can return a different address, so always use the returned pointer. Any old copy of the pointer is now dangling — don't keep using it.
0️⃣
Expecting malloc to zero memory malloc gives garbage, not zeros. If you need a clean, zero-filled block, use calloc — or initialize the malloc'd memory yourself before reading it.

Practice exercises 💪

Easy Zeroed array. calloc 5 ints and print them all without setting any — confirm they're all 0. Then free the block.
Medium Grow it. malloc a 3-int array, fill it, then realloc to 6 ints (reassign the pointer), fill the new slots, print all six, and free. Confirm the first three values survived.
Stretch Safe grow. Write the temp-pointer realloc pattern: store the result in tmp, check it for NULL, and only assign back on success. Explain in one line why assigning realloc directly to a is risky.

🎓 What to remember

  • calloc(count, size) allocates count × size bytes and zero-initializes them — two args, zeroed memory.
  • realloc(ptr, newsize) resizes a block; it may return a new address, so always reassign the pointer to the result.
  • realloc can fail (return NULL) while leaving the original valid — use a temp pointer so you don't leak it.
  • malloc gives garbage; use calloc for zeros, realloc to resize.
  • All three are released with the same free on the final pointer.

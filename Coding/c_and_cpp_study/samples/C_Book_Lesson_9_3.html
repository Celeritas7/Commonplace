Writing a File Safely

Reading a file can't really hurt you. Writing can — it's where data gets lost. Two surprises bite beginners: output is buffered, so it isn't on disk until you flush or close; and opening with "w" empties the file the instant it opens, so a failed write can wipe a good file and leave nothing. This lesson covers writing correctly, and the temp-file-and-rename trick the pros use to never lose data.

1 Why this matters

Where data goes to die 💾

You met fopen and fprintf back in 9.1. So why a whole lesson on writing? Because writing is where programs lose data, and the failures are quiet. You write a config, the program is killed, and the file is empty — because your output never left the buffer. Or you overwrite a file with "w", the write fails halfway, and now both the old data and the new data are gone. Reading from a file leaves it untouched; writing changes the world, so it deserves care. This lesson is the habits that make writes survive.

Two ideas do most of the work: flush before you trust it's saved, and write a temp file, then rename it into place.

2 Mode decides what survives

"w" empties, "a" keeps ✂️

The mode you pass to fopen decides the fate of whatever was already in the file — and "w" is ruthless. Always check the handle for NULL too:

FILE *f = fopen("out.txt", "w");   // file is EMPTIED right now, before any write
if (!f) { perror("open out.txt"); return 1; }   // writes fail too — always check

The key fact: "w" truncates the file at the moment of opening, not when you write. So the old contents are gone immediately — even if your program then crashes without writing a thing. Use "a" when you're adding to a log and must never lose the past. Use "w" when you genuinely mean "replace everything" — but read section 5 first, because replacing safely takes one more step.
3 The writing calls

fprintf, fputs, fputc ✍️

Three everyday ways to put text into a file. None of them adds a newline for you — that's always your job:

fprintf(f, "%s scored %d\n", name, score);  // formatted, like printf
fputs("a plain line\n", f);                  // a ready-made string
fputc('!', f);                                // a single character
fprintf is printf aimed at a file — all the same %d/%s/%f specifiers. fputs writes a string with no formatting (slightly faster, no format parsing). fputc writes one character. Whichever you use, you decide where the \ns go; the library never inserts them. (There's also fwrite for raw binary blocks — that's a brief topic later in this Part.)
4 Output is buffered

It isn't saved until you flush 🫧

When you call fprintf, the bytes usually go into an in-memory buffer, not straight to disk. They reach the disk when the buffer fills, when you call fflush(f), or when you fclose(f). If the program dies before that, the buffered bytes are simply gone:

fprintf(f, "important data\n");   // sits in a buffer — NOT on disk yet
fflush(f);                          // force it to disk now
// ...
fclose(f);                          // flushes any remaining buffer, then closes
This is why fclose matters so much: it flushes the buffer before releasing the handle. Forget it (or get killed first) and your last writes never land. fflush(f) forces a flush at any point — useful for a log you want on disk immediately, or right before risky work. And fclose itself can fail (a full disk reports the error only then), so for data that matters, check its return value: if (fclose(f) != 0) perror("fclose");.
5 Replacing a file safely

Write a temp, then rename 🔁

Here's the danger with "just open it with "w" and overwrite": the file is emptied at open, so if the write fails partway (disk full, crash, bug), you're left with a truncated or half-written file and the original is gone. The fix professionals use: write to a temporary file, and only when it's complete, rename it over the original:

FILE *f = fopen("out.txt.tmp", "w");   // write to a temp, not the real file
if (!f) { perror("open tmp"); return 1; }
// ... write everything, checking for errors ...
if (fclose(f) != 0) { perror("fclose"); return 1; }   // flush + verify
rename("out.txt.tmp", "out.txt");   // atomic swap: out.txt is old OR new, never half

🚨 "w" destroys the original before you write a byte

Open the real file directly with "w" and you've already lost the old contents — so a write that fails halfway leaves you with nothing complete. Writing to out.txt.tmp first means a failure just leaves a junk temp file you can delete; the real out.txt is untouched. rename then replaces the file in one atomic step: at every instant, out.txt is either the entire old version or the entire new one — never a torn mix. This is exactly how editors and databases avoid corrupting your files on a crash.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🫧
Forgetting fclose (or fflush) Buffered output isn't on disk until a flush. If the program ends without fclose, your last writes vanish. Always close; flush early for data you can't lose.
🗑️
Overwriting the real file with "w" "w" empties the file at open, so a failed write leaves nothing. Write to a temp file and rename it into place to keep the original until the new one is complete.
❓
Ignoring write return values fprintf, fputs, and even fclose can fail (full disk, bad handle). For data that matters, check them — a silent failure is worse than a loud one.

Practice exercises 💪

Easy Save and reopen. Open "scores.txt" with "w", write three name score lines with fprintf, then fclose. Reopen with "r" (from 9.2) and print them back to prove they're really on disk.
Medium Flush vs. no flush. Write a line, then loop or sleep before closing. Run once with an fflush after the write and once without, killing the program early each time, and observe when the line makes it to disk.
Stretch Safe replace. Write a function that rewrites a config file using the temp-file-and-rename pattern: open name.tmp with "w", write everything (checking fprintf and fclose), and only rename on success. On any failure, delete the temp and leave the original intact.

🎓 What to remember

  • "w" truncates at open — the old contents are gone immediately; "a" keeps them and appends.
  • Write with fprintf / fputs / fputc — and add your own \n; the library never does.
  • Output is buffered — it isn't on disk until fflush or fclose. Forgetting to close loses your last writes.
  • To replace a file safely, write a temp file then rename it over the original — an atomic, all-or-nothing swap.
  • Check return values of the write calls and fclose when the data truly matters.

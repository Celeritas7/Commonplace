Constants, Literals, and const

Sometimes a value should never change — π, the days in a week, the maximum length of a username. Today: the three ways C lets you say "this is constant," plus the const keyword that turns a variable into a promise.

1 Why this matters

Locking down what shouldn't change 🔒

An entity whose value doesn't change is called a constant — where a variable is an entity whose value can be changed.

Your notes had it exactly right. A variable is a container whose contents can change. A constant is a value that can't. Some things in your programs really shouldn't change — π is always π, there are always 12 months in a year, your buffer size shouldn't shift mid-loop. Marking those as constants tells the compiler (and any future reader) "this is fixed, don't touch."

And when you make a mistake and try to change one anyway? The compiler catches it. That's the whole win — preventable bugs prevented at compile time.

2 The three flavors of constants

Literals, #define, and const 📝

Types of constants — primarily, there are three types of constants.

Your notes hinted at the three. Modernized for today's C, they are:

1

Literals — values written directly in code

The simplest: numbers and characters you just write. 5, 3.14, 'A', "hello" are all literals. They're constants by nature — no name needed.

int age = 25;          // 25 is an int literal
double pi = 3.14159;     // 3.14159 is a double literal
char grade = 'A';        // 'A' is a char literal (its value is 65)
2

#define — the preprocessor macro (older style)

Names a constant value at the preprocessor level. Before the compiler ever sees your code, every PI is replaced with 3.14159 — pure text substitution.

#define PI 3.14159
#define MAX_USERS 100

double area = PI * r * r;   // PI becomes 3.14159 before compile
3

const — a variable that promises not to change (modern preference)

A normal variable, but with the const keyword stuck on. The compiler enforces the promise: try to assign to it and you get a compile error.

const double PI = 3.14159;
const int MAX_USERS = 100;

PI = 3.14;   // ERROR — can't assign to a const variable

Use const whenever possible. It's typed, scoped, and the compiler protects it. #define still has its place (header guards, conditional compilation) but for naming a constant value, const is the modern win.

3 Literal suffixes — making the type explicit

The little letters that change everything 🔤

By default, 42 is an int and 3.14 is a double. But you can force a different type by adding a one or two-letter suffix to the literal. This matters when the default type isn't what you want.

Also notable: integer literals in different bases:

  • 42 — decimal (base 10)
  • 052 — octal (base 8, starts with 0). Watch out — leading zero changes the meaning!
  • 0x2A — hexadecimal (base 16, starts with 0x). Useful for bit patterns and memory addresses.
  • 0b101010 — binary (C99 and later, starts with 0b). Direct bit specification.

All four of those equal 42 in value — same number, written different ways for different occasions.

4 The const keyword in detail

A promise the compiler enforces 🤝

The simplest use of const is what you've seen: const int MAX = 100;. But const goes deeper — it can also lock down function parameters and (when we meet pointers in Part VI) what a pointer can change.
// A const variable: must initialize at declaration const int DAYS_PER_WEEK = 7; const int x; // ERROR — const must be initialized DAYS_PER_WEEK = 8; // ERROR — can't assign to a const // A const function parameter — "I promise I won't modify this" void printPrice(const double price) { printf("$%.2f", price); price = 0; // ERROR — can't modify the const parameter }

Why bother marking parameters const if you weren't planning to modify them anyway? Documentation + protection. The function signature now says "I treat this as read-only", and the compiler catches you if you accidentally try to modify it. In bigger programs this is a real bug-preventer.

Modern convention: mark anything that doesn't need to change as const. It's free documentation and free bug prevention. The C++ community even has a term for the habit — "const correctness." (More on this when we get to C++ in Part VI of that book.)

7 Common mistakes

Three traps with constants 🚧

🙅
"Leading zero is harmless formatting." It's not — leading zero on an integer literal means octal (base 8). 012 is 10, not 12. Drop the leading zero, or pad with spaces. The only "safe" leading zero is 0 itself.
🙅
"#define and const are interchangeable." Mostly, yes. But #define MAX 100 has no type, no scope, and no respect for namespaces — it's just text substitution. const int MAX = 100; has all three. Modern C prefers const for values; #define survives for things like header guards (#ifndef MY_H) and platform conditionals (#ifdef _WIN32).
🙅
"3.14 is a float." In C, 3.14 is a double. Writing float f = 3.14; compiles but does a silent narrowing conversion from double to float. To get a real float literal, write 3.14f with the f suffix. (The compiler will usually warn if you have -Wall on.)

🎓 Quick recap

  • A constant is a value that can't change. Three flavors: literals (values in code), #define macros, const variables.
  • Use const whenever you can — it's typed, scoped, compiler-enforced.
  • Literal suffixes change the type: 42L is long, 42U is unsigned, 3.14f is float, 3.14 is double.
  • Integer bases: decimal (42), octal (052), hex (0x2A), binary (0b101010) — all equal 42. Beware leading-zero octal!
  • Mark function parameters const when you don't intend to modify them. Free documentation + protection.

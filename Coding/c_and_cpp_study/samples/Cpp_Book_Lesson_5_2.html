std::array

When you know the size up front and don't need it to grow, std::array is the fixed-size companion to vector. It lives on the stack like a raw array but, unlike one, it knows its own size, doesn't decay to a pointer, and can bounds-check.

1 Why this matters

Fixed size, modern safety 🧠

std::vector grows and shrinks, storing its data on the heap. But plenty of collections have a size you know at compile time and never changes — the 3 RGB channels of a colour, the 7 days of a week, a 9-cell tic-tac-toe board. For those, std::array gives you a fixed-size container that's as cheap as a raw array (it sits on the stack) but carries the modern conveniences vector has.

Think of it as "a raw C array, with the rough edges filed off." Same fixed size and stack storage, but now it knows its length, copies cleanly, plays nicely with the STL algorithms, and offers a checked .at(). When the size is fixed, it's the safe choice over both a raw array and a (slightly heavier) vector.

2 Creating one

Type and size, both fixed 🧱

Include <array>. The template takes two things — the element type and the size — both fixed at compile time:

#include <array>

std::array<int, 5> a = {10, 20, 30, 40, 50};
//          ^type ^size

std::cout << a[0];        // 10 — index like any array (zero-based)
std::cout << a.size();    // 5  — it knows its length
The size 5 is part of the type: a std::array<int, 5> and a std::array<int, 3> are different types, and the size can't change at runtime — no push_back here. You index with [ ] exactly like a raw array or a vector, and you initialise it with a braced list the same way.
3 What it fixes

Knows its size, doesn't decay, copies cleanly ✅

Everything that made a raw C array awkward, std::array smooths over:

  • It knows its size: a.size() always works — no sizeof trick, no passing a length.
  • It doesn't decay: pass it to a function and it stays a std::array, length intact (unlike a raw array, which becomes a pointer).
  • It has value semantics: auto b = a; makes a real, independent copy — changing b doesn't touch a (a raw array can't even be assigned like that).
  • It can bounds-check: a.at(i) throws on a bad index; a[i] stays fast and unchecked.
That second point is the direct cure for the previous C lesson's headache: because a std::array is a real object (not a bare pointer), its size travels with it across function calls. void print(const std::array<int,5>& a) still knows a.size() is 5 inside the function. 4 Choosing a container

array vs vector vs raw 🗂️

SizeLives onKnows size?
raw int a[5]fixedstackno (decays)
std::array<int,5>fixedstackyes
std::vector<int>grows/shrinksheapyes
A simple way to choose: need it to grow at runtime? → vector. Fixed size known at compile time? → std::array. You'll almost never reach for a raw array in modern C++ — std::array does the same job with none of the sharp edges, at the same cost.
5 Watch out

The size is part of the type 🧊

Because the size is baked into the type, you can't resize a std::array or assign one of a different size to it. And like every container, [ ] still doesn't bounds-check — only .at() does.

🚨 Out-of-range [ ], and size mismatches

a[5] on a std::array<int,5> is past the end — undefined behaviour, just like a raw array. Use a.at(5) to get a thrown std::out_of_range instead. Also, you can't do std::array<int,5> a = b; if b is a std::array<int,3> — different sizes are different types, so they don't assign.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🧊
Trying to resize it std::array has no push_back — its size is fixed at compile time. If you need to grow, you wanted std::vector.
🚧
Assuming [ ] bounds-checks It doesn't — a[5] on a size-5 array is undefined behaviour, same as a raw array. Use a.at(i) when you want a checked access that throws.
📐
Forgetting the size is part of the type A std::array<int,3> and a std::array<int,5> are different types. A function taking one size won't accept the other (templates handle "any size" — a later topic).

Practice exercises 💪

Easy RGB colour. Make a std::array<int, 3> holding red, green, blue values. Print each with a range-for, and print .size().
Medium Pass it, keep the size. Write int sum(const std::array<int,5>& a) that loops with a.size() (no length parameter!). Call it and confirm the size came along — the very thing a raw array couldn't do.
Stretch Copy semantics. Make a std::array<int,4> a, copy it with auto b = a;, change one element of b, and print both. Confirm a is untouched — then recall that you can't do this with a raw array at all.

🎓 What to remember

  • std::array<T, N> is a fixed-size array (type and size fixed at compile time); needs #include <array>.
  • It lives on the stack like a raw array — cheap, no heap allocation.
  • It fixes the raw-array pain: knows its size(), doesn't decay, copies by value, and offers checked .at().
  • No resizing — if you need to grow, use std::vector.
  • Choose: grows at runtime → vector; fixed compile-time size → std::array; raw arrays rarely.

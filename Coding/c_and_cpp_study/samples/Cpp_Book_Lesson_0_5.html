⭐ PART 0 · BEFORE YOU CODE

Read, Write, Run, Fix, Repeat

There are basically only five things you do as a programmer. Every working day, every project, every lesson coming up — these five are it. Master the rhythm and the rest follows.

1 Why this matters

The honest job description 📋

Movies and TV make programming look like a single act: someone types furiously, sweat on the brow, and bam — the program works. That's not how it goes. Real programming is a tight little loop of five activities, and the people who get good are the ones who don't fight the loop.

This lesson is short — but it might be the most important one in Part 0. If you internalize the loop now, you won't be surprised when writing C++ turns out to be 30% writing and 70% everything else. That surprise sinks a lot of beginners.

2 The five activities

This is the whole job 🎯

That's it. Five things. Every programmer, from a beginner to a 25-year veteran maintaining Chromium, spends their working day doing these five activities in some mix. Hover any card.
📖
Read
Existing code, docs, error messages, other people's work
✍️
Write
New code, modifications, comments, tests
▶️
Run
Compile, execute, see what actually happens
🔧

Debug when it doesn't behave (and it won't)
🔁

Tighten the loop. Go again. Until done.
3 The proportions will surprise you

How a programmer's hour actually breaks down ⏱️

Beginners imagine programming is "mostly writing." It isn't. A rough — but representative — breakdown of where the time goes for an experienced C++ developer:

Time spent in each activity

📖 Read
~40%
✍️ Write
~20%
▶️ Run
~10%
🔧 Fix
~25%
💭 Think
~5%

Reading dominates. You read existing code to understand it, read documentation to figure out how to use a library, read compiler errors to figure out what went wrong (C++ errors can be long — they're really worth learning to read), and read your own code from yesterday to remember what you were doing. Reading skill is at least as valuable as writing skill.

4 It's a loop, not a sequence

The tight loop that defines the work 🔁

The five activities don't happen once and then you're done. They cycle, fast, in a tight loop. Good C++ programmers can complete a full read-write-run-fix-repeat cycle in seconds — sometimes 30 cycles per hour. Beginners often spend hours stuck because their loop is too slow.

A good programmer doesn't try to write 200 lines and then run them once. They write 5 lines, run, see if it works, fix if not, write 5 more. Tiny laps around the loop, many times. That's the rhythm. C++ in particular rewards short cycles because the compiler is your best ally — short loops let it catch type errors and bad casts seconds after you make them.

7 Common mistakes

Beginner traps in the loop 🚧

🙅
"Errors mean I'm failing." Errors are feedback, not failure. A C++ error message is the compiler telling you exactly what it didn't understand. Senior programmers see compiler errors all day; what they've learned is to read them carefully instead of panicking. C++ errors are notoriously long — read from the top.
🙅
"I'll write the whole class, then compile it." This is the slow-loop trap. Big-bang releases have big-bang bugs. Write a tiny piece (one method, one function), compile, see it work, then build on it. Tight loops let the C++ compiler catch type errors seconds after you make them.
🙅
"Asking for help means I'm not a real programmer." Reading other people's code, documentation (cppreference.com is your friend), and Stack Overflow answers is part of the READ step. Real C++ programmers do it every day. Refusing to read = trying to invent the STL from scratch. Don't.
8 Your turn

Practice the rhythm 💪

Easy Time yourself. Next time you do any repetitive task (cooking, organizing photos, replying to emails), notice the loop you're in. What's your read step? Write step? Fix step? Repeat step? You already loop — programming just makes it explicit.
Medium Read someone else's C++. Find a small open-source C++ program on GitHub (search "small C++ example" or "C++ beginner project"). Pick one with under 200 lines. Spend 15 minutes just reading it. Don't try to understand everything. Notice what makes sense, what doesn't. Reading C++ is the most underrated skill — and the one you'll spend most time on.
Stretch Read a real C++ error message. Open any online C++ compiler (godbolt.org / wandbox.org). Type something deliberately wrong — like std::vector<int> v; v.push_back("hello"); — and read the error. C++ errors can be 30+ lines for one mistake. The skill is finding the actual problem line. This is half of the FIX step.

🎓 Quick recap

  • Programming is five activities: Read, Write, Run, Fix, Repeat.
  • Writing is the small part. Reading and fixing together usually take more time than typing new code.
  • It's a tight loop, not a sequence. Lap many times per hour. C++ compilers reward tight loops — they catch type errors instantly.
  • Errors and bugs are feedback, not failure. Senior programmers see them all day; what's different is how they read them.

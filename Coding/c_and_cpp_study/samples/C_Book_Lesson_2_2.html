Basic Types, Sizes & Ranges

In 2.1 we met the types. Now we look inside: how many bytes each one takes, what range of values it can hold, what happens when you exceed that range, and how to pick the right type for a value.

1 Why this matters

Size and range — the two facts that bite 🦷

Every C type takes a specific number of bytes in memory. That number determines the range of values the type can hold. Pick a type that's too small, and your value silently wraps to garbage. Pick one that's too big, and you waste memory in a tight loop.

You'll meet two related questions all the time: "How big should this counter be?" and "What happens if I add 1 to 2,147,483,647 in an int?" (Spoiler for the second: chaos.) This lesson hands you the numbers and the intuition.

2 The size table

How many bytes does each type take? 📏

C's rules are slightly fuzzy here: the standard guarantees minimum sizes but lets the compiler pick larger ones if the platform makes that sensible. On nearly every modern 64-bit machine, the actual sizes are stable — here they are:

char1 byte-128 to 1270 to 255
short2 bytes-32,768 to 32,7670 to 65,535
int4 bytes−2.1 billion to +2.1 billion0 to 4.3 billion
long4 or 8 bytesplatform-dependentplatform-dependent
long long8 bytes±9.2 × 10¹⁸0 to 1.8 × 10¹⁹
float4 bytes≈ ±3.4 × 10³⁸ (about 7 decimal digits of precision)
double8 bytes≈ ±1.8 × 10³⁰⁸ (about 15 decimal digits)
_Bool1 byte0 or 1 (just true/false)

Two practical takeaways:

  • For whole numbers, just use int 95% of the time. It's 4 bytes on every modern machine, gives you ±2 billion, and the CPU's most efficient size.
  • For decimals, just use double. float saves 4 bytes per number but loses precision; rarely worth the trade.

The sizeof operator gives you the exact size at runtime: printf("%zu", sizeof(int)); prints 4 on most machines.

3 Signed vs unsigned

Where the negative numbers come from 🔄

For integer types, the same number of bits can store either signed (positive and negative) or unsigned (only non-negative) values. signed char covers −128 to +127; unsigned char covers 0 to 255. Same 8 bits, different interpretation.

You make the choice at declaration time:

int a = -5; // signed by default signed int b = -5; // same as above, explicit unsigned int c = 5; // only non-negative values unsigned int d = -1; // compiles! but d becomes 4,294,967,295 — wrap

By default, int, short, long, etc. are signed. The exception is char — whether it's signed or unsigned by default depends on the compiler (annoying). When it matters, write signed char or unsigned char explicitly.

Rule of thumb: use signed (the default int) unless you have a specific reason for unsigned. Mixing signed and unsigned in comparisons causes subtle bugs.

5 Floating-point: float vs double

Why double is almost always the right choice 🎯

float and double both store decimal numbers, but with different precision. float has about 7 decimal digits of precision; double has about 15. That's a huge gap.

An example of why precision matters:

float f = 0.1f + 0.2f; // f ≈ 0.30000001192 double d = 0.1 + 0.2; // d ≈ 0.30000000000000004

Both are slightly wrong (a well-known fact of how decimals are stored in binary), but double is wrong much later in the decimal expansion. For almost every program you'll write, double is the right pick.

When to use float: giant graphics arrays where memory matters (think 3D vertex buffers — millions of floats), or tiny embedded systems without a hardware double-precision unit. Otherwise, double.

7 Common mistakes

Three traps with C types 🚧

🙅
"Bigger type = safer, so I'll use long long everywhere." long long takes 8 bytes vs int's 4. For a single variable, no big deal. For an array of a million values, that's 4 MB vs 8 MB. The CPU also moves int faster than long long on most operations. Default to int unless you actually need the bigger range.
🙅
"Adding 1 to INT_MAX gives INT_MIN." For unsigned types it wraps reliably. For signed types, signed overflow is undefined behavior in C — the compiler is allowed to do anything, including breaking your program in clever optimization passes. If you need wrapping math, use an unsigned type or a fixed-width type like int32_t.
🙅
"Comparing int and unsigned int is fine." It's the source of one of the most common C bugs. if (signed_x < unsigned_y) — the compiler converts signed_x to unsigned before comparing. If signed_x is negative, it becomes a huge positive number, and your comparison gives the wrong answer. Either keep types consistent or cast carefully. Most compilers warn about this with -Wsign-compare.
8 Your turn

Get a feel for sizes 💪

Easy Measure your machine. Write a small program that prints the size of each type:
printf("char: %zu\n", sizeof(char));
printf("short: %zu\n", sizeof(short));
printf("int: %zu\n", sizeof(int));
printf("long: %zu\n", sizeof(long));
printf("long long: %zu\n", sizeof(long long));
printf("float: %zu\n", sizeof(float));
printf("double: %zu\n", sizeof(double));
Run it. Note what your machine reports — especially what long shows.
Medium Trigger overflow on purpose. Declare unsigned char c = 255;. Run a loop that prints c and increments it 5 times. Watch it go 255 → 0 → 1 → 2 → 3 → 4. Now do the same with signed char c = 127; — see it become -128. Now signed int from INT_MAX (use <limits.h> for the constant): compile with -Wall — note the compiler's warning that this is undefined behavior.
Stretch Fixed-width types. Read the man page or online docs for <stdint.h> — it defines int8_t, int16_t, int32_t, int64_t and their unsigned counterparts. These are exactly the size their name says, on every platform. Rewrite the easy-exercise program to print sizes of these types too. Why might code that needs to work on many machines prefer these names?

🎓 Quick recap

  • Each C type takes a specific number of bytes: char = 1, short = 2, int = 4, long long = 8, float = 4, double = 8.
  • The size determines the value range. int gives you ±2 billion; unsigned int gives 0 to 4.3 billion.
  • Default to int for whole numbers and double for decimals. Pick others only with a specific reason.
  • Signed overflow is undefined behavior. Unsigned overflow wraps cleanly to 0.
  • sizeof(type) gives the exact size on your machine — use printf("%zu", sizeof(int)).
  • For guaranteed sizes across platforms, use int32_t, uint64_t, etc. from <stdint.h>.

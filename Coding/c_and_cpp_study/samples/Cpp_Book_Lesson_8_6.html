shared_ptr & weak_ptr

unique_ptr gives one object exactly one owner. But sometimes several parts of a program need to share an object, and it should live until the last of them is done. std::shared_ptr handles that with a reference count; std::weak_ptr is its non-owning companion that observes without keeping the object alive — and breaks the cycles shared ownership can create.

1 Why this matters

When one owner isn't enough 🧠

unique_ptr (8.5) is perfect when a single owner controls an object's lifetime. But some designs genuinely need shared ownership: several objects hold the same node of a graph, or a cache and a worker both reference the same resource, and it must stay alive as long as any of them needs it — then be freed exactly once when the last one lets go. std::shared_ptr does this by keeping a reference count: each owner adds one, each releases one, and the object is deleted when the count reaches zero. It's the last smart pointer you need to know.

Powerful — but shared ownership is heavier than unique, so it's a tool for when you truly need it, not a default.

2 shared_ptr & the count

Many owners, one count 🔢

Create one with std::make_shared<T>(args). Unlike unique_ptr, you can copy it — each copy is another owner, and use_count() tells you how many there are:

auto p = std::make_shared<Widget>();   // count = 1
auto q = p;                            // copy → both own it, count = 2
std::cout << p.use_count();             // prints 2
// use p->, *p just like any pointer
A shared_ptr holds two things: the pointer to your object, and a pointer to a small control block that stores the reference count. Copying the shared_ptr copies both and bumps the count; you still use it with -> and * like any pointer. Prefer make_shared — it allocates the object and the control block together in one step, which is faster and exception-safe.
3 How the count moves

Zero means goodbye ➕➖

Every new owner increments the count; every owner that's destroyed or reset decrements it. When it hits zero, the object is deleted — automatically, exactly once:

auto p = std::make_shared<Widget>();   // count = 1
{
    auto q = p;     // count = 2
}                  // q destroyed → count = 1 (Widget still alive)
// ... when p is destroyed too → count = 0 → Widget deleted
This is the key mental model: the object lives exactly as long as the count is above zero, and dies the instant the last owner goes away — wherever in the program that happens to be. You never call delete; the control block does it when the final shared_ptr is gone. That's what "the object lives until the last owner is done" means in practice.
4 weak_ptr

Look without owning 👀

A std::weak_ptr points at an object managed by a shared_ptr but doesn't own it — it doesn't change the count, so it never keeps the object alive. To use it, ask for a temporary shared_ptr with .lock(), which gives you one if the object is still alive, or nullptr if it's gone:

std::weak_ptr<Widget> w = p;   // observes — count is unchanged

if (auto sp = w.lock()) {       // got a shared_ptr → still alive
    sp->use();
} else {
    // expired — the object was already deleted
}
Think of weak_ptr as a safe "I'd like to use this if it still exists" reference. It's perfect for caches, observers, or back-references where you must not extend the object's lifetime. Because it doesn't own anything, you can't dereference it directly — you lock() it to atomically check "alive?" and get a real owning shared_ptr in the same step, avoiding a dangling-pointer race.
5 Cycles & cost

When sharing leaks 🔗

Shared ownership has one famous failure mode and one ongoing cost:

🚨 Two shared_ptrs pointing at each other never reach zero

If object A holds a shared_ptr to B and B holds a shared_ptr back to A, each keeps the other's count at 1 forever — even after every external owner is gone. The count never reaches zero, so neither is deleted: a reference cycle leak. The fix is to make one direction a weak_ptr (e.g. a parent owns its child with shared_ptr, the child refers back to the parent with weak_ptr) — the weak side doesn't count, so the cycle can unwind. Also note the cost: a shared_ptr carries a control block and updates the count atomically (thread-safe), which is real overhead. So prefer unique_ptr by default and reach for shared_ptr only when ownership is genuinely shared.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔗
Creating a reference cycle Two shared_ptrs pointing at each other keep both counts above zero forever — a leak. Make one direction a weak_ptr to break it.
👥
Reaching for shared_ptr by default Shared ownership has overhead (control block, atomic count) and muddies who owns what. Default to unique_ptr; use shared_ptr only when ownership is truly shared.
💀
Dereferencing a weak_ptr without locking A weak_ptr may point at a deleted object. Always .lock() and check the result before using it — don't assume it's still alive.

Practice exercises 💪

Easy Watch the count. Make a shared_ptr and print use_count(). Copy it inside a block, print the count again (2), let the block end, and print once more (back to 1).
Medium Last one out. Give a class a destructor that prints "freed". Hold it in two shared_ptrs, reset one (no "freed" yet), then reset the other and confirm "freed" prints exactly when the count hits zero.
Stretch Break a cycle. Build two nodes that point at each other with shared_ptr and observe (via destructor prints) that neither is freed — a leak. Then change one link to a weak_ptr and confirm both are now destroyed.

🎓 What to remember

  • std::shared_ptr allows multiple owners via a reference count; the object is deleted when the count hits zero.
  • Copy = +1 owner, destroy/reset = −1; use_count() reports how many own it. Make with make_shared.
  • std::weak_ptr observes without owning — it doesn't change the count; .lock() gives a shared_ptr if still alive.
  • Reference cycles leak — two shared_ptrs pointing at each other never reach zero; break one with a weak_ptr.
  • Prefer unique_ptr; shared_ptr has real overhead — use it only for genuine shared ownership.

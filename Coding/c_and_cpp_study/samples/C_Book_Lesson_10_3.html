A First Makefile

Once a project has more than a file or two, typing the full gcc command every time gets old fast. A Makefile records how to build your project, and the make tool runs it — rebuilding only the parts that actually changed. You don't need the whole arcane language; about ten lines covers everyday use, and that's what we'll learn.

1 Why this matters

Stop retyping the build 🛠️

In 10.1 you built a multi-file program with gcc main.c util.c -o prog. That's fine for two files, but real projects have dozens, with flags and link options you'd never remember to type the same way twice. A Makefile is a small text file that records the build steps once; then you just type make. Better still, make is incremental: it checks file timestamps and recompiles only the files that changed since last time, which turns a minute-long full rebuild into a one-second one. It's the standard build tool across the C/C++ world, and a tiny subset gets you everything you need day to day.

The whole idea is one repeated shape: a rule that says "to make X, which needs Y, run this command."

2 The anatomy of a rule

Target, prerequisites, recipe 📋

Every Makefile is built from rules, and every rule has the same three parts:

prog: main.o util.o          # target : prerequisites
→gcc main.o util.o -o prog   # recipe (MUST start with a TAB)
  • Target — the file to build (here prog), before the colon.
  • Prerequisites — what it depends on (main.o util.o), after the colon.
  • Recipe — the shell command(s) to build it, on the next line(s).
Read it as a sentence: "to build prog, which depends on main.o and util.o, run this gcc command." make uses the dependency list to decide whether the recipe needs to run, and to figure out what to build first. The one piece of syntax people trip on: the recipe line must begin with a real Tab character, not spaces (more on that in the trap).
3 A whole small Makefile

The ten lines you'll use 📄

Here's a complete, realistic Makefile for a small project — chained rules plus a couple of conveniences:

CC = gcc
CFLAGS = -Wall -g            # warnings on, debug info

prog: main.o util.o          # final program
→$(CC) main.o util.o -o prog

main.o: main.c util.h        # recompiles if main.c OR util.h changes
→$(CC) $(CFLAGS) -c main.c

util.o: util.c util.h
→$(CC) $(CFLAGS) -c util.c

clean:                       # a "phony" task, not a file
→rm -f prog *.o
A few everyday features are in here. Variables like CC and CFLAGS are defined once and used as $(CC) — change a flag in one place. The rules chain: building prog needs main.o, which has its own rule, so make builds the .o files first, then links. The util.h in each .o's prerequisites means editing the header rebuilds the files that include it. And clean is a handy task target you run with make clean.
4 Incremental builds

Only rebuild what changed ⏱️

The magic is timestamps. make rebuilds a target only if it's older than one of its prerequisites — so editing one file recompiles just that file (and whatever depends on it):

$ make            # first time: compiles everything, links prog
$ make            # nothing changed → "make: 'prog' is up to date."

# edit util.c, then:
$ make            # recompiles ONLY util.o, then relinks prog (skips main.o)
For each target, make compares modification times: if every prerequisite is older than the target, the target is up to date and its recipe is skipped. Touch util.c and only util.o is now out of date, so just that compile runs — then prog, now older than the fresh util.o, is relinked. On a big project this is the difference between waiting and not. Running make with nothing changed does nothing at all, which is exactly what you want.
5 Running it & targets

make, make clean, and friends 🎯

By default make builds the first target in the file; you can also name a target explicitly:

CommandWhat it does
makebuild the first target (here, prog)
make util.obuild just that target
make cleanrun the clean recipe (delete build files)

🚨 Recipes must start with a real Tab, and "phony" targets need declaring

The number-one Makefile error: indenting a recipe with spaces instead of a Tab. Make is strict — recipe lines must begin with an actual Tab character, or you get the cryptic *** missing separator. Stop. Configure your editor to keep the Tab in Makefiles. Second gotcha: a target like clean isn't a real file. If a file named clean ever exists in the folder, make sees the target as "up to date" and skips it. Declare such task-targets phony so make always runs them: add a line .PHONY: clean. (Common phony targets: all, clean, test, install.)

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

↹
Spaces instead of a Tab Recipe lines must start with a real Tab character. Spaces give *** missing separator. Stop. Set your editor to preserve tabs in Makefiles.
👻
Forgetting .PHONY Task targets like clean aren't files. If a file named clean exists, make thinks it's up to date and skips it. Declare .PHONY: clean all test.
🔗
Missing prerequisites If you don't list util.h as a prerequisite of the .o files that include it, editing the header won't trigger a rebuild — and you'll debug stale code. List every real dependency.

Practice exercises 💪

Easy One-rule Makefile. For a single-file program, write a Makefile with one rule: prog: main.c and a recipe that compiles it. Run make, then run it again and see "up to date."
Medium Multi-file + clean. Take your 10.1 three-file project and write the full Makefile from section 3 (with CC/CFLAGS variables and a clean target). Edit one .c and confirm only that file recompiles.
Stretch Prove the header dependency. With util.h listed as a prerequisite, edit only util.h and run make — confirm every .o that includes it rebuilds. Then remove it from the prerequisites and show that editing the header no longer triggers a rebuild (the bug).

🎓 What to remember

  • A Makefile records your build so you type make instead of long gcc commands.
  • Every rule is target : prerequisites + a recipe — "to build X (which needs Y), run this." Recipes start with a Tab.
  • make is incremental — it compares timestamps and rebuilds only what's out of date; unchanged builds do nothing.
  • Variables ($(CC), $(CFLAGS)) and chained rules keep it DRY; make builds the first target by default.
  • Declare task targets .PHONY (clean, all) so they always run regardless of same-named files.

🎯 PART VI · POINTERS

Visualizing Pointers

The single best way to understand pointers is to draw them: a variable is a labelled box holding a value, and a pointer is a box holding an arrow to another box. Once you see the boxes and arrows, every pointer operation becomes "what does the arrow do?"

1 Why this matters

See it and it makes sense 🧠

The syntax of pointers (&, *, int *p) is small, but it gets confusing fast when you only stare at the symbols. The cure is a picture. Every C programmer carries the same mental image — boxes and arrows — and reasons about pointers by drawing them. This lesson makes that picture explicit so you can run it in your head.

Nothing new syntactically here; we're cementing 6.2 by turning it visual. Grab a pen if you like — by the end, you should be able to sketch any small pointer program.

2 The picture

Boxes for values, arrows for pointers 📦

Draw a variable as a box with a label and a value. Draw a pointer as a box too — but instead of an ordinary value, it holds an arrow aimed at another box:

int x = 30;
int *p = &x;

//   p ──────▶ [ x | 30 ]
//   the arrow IS the address; *p means "follow the arrow"
The arrow is the address. Saying "p holds &x" and drawing "an arrow from p to x" are the same statement. To read *p, you put your finger on p, follow the arrow to x, and read what's in that box: 30.
3 Reassigning moves the arrow

p = &y swings it over ➡️

This is the idea that trips people up: assigning to p doesn't copy any values — it just re-aims the arrow at a different box:

int x = 30, y = 99;
int *p = &x;   //   p ──▶ [x|30]
p = &y;        //   p ──▶ [y|99]   (arrow now points at y; x untouched)
Before, the arrow pointed at x; after, it points at y. Neither x nor y changed — only which box p points at changed. Compare this with *p = &y's cousin from 6.2: p = ... moves the arrow, while *p = ... reaches through the arrow to change a value.
4 Many arrows, one box

Aliasing 🔗

Nothing stops two (or more) pointers from aiming at the same box. They're called aliases — different names for one variable. Change the value through one arrow, and every arrow sees it, because there's only one box:

int x = 5;
int *p = &x;
int *q = &x;     //   p ──▶ [x|5] ◀── q
*p = 8;          //   change through p…
printf("%d", *q);  //   8  — q sees it too
In the picture, p and q are two arrows landing on the same box. There's no copy involved — *p and *q are two routes to the one x. This is exactly why passing a pointer to a function lets it change your variable: the function's pointer is just another arrow to your box.
5 An arrow to nowhere

NULL — points at nothing 🚫

Sometimes a pointer shouldn't point anywhere yet. For that there's NULL — a special "points at nothing" value. In the picture, it's an arrow with no box on the end (often drawn going to ground):

int *p = NULL;   //   p ──▶ ∅   (nothing on the end)

if (p != NULL)   // always check before following the arrow
    printf("%d", *p);

🚨 Dereferencing a NULL (or unset) pointer

Following an arrow that points at nothing — *p when p is NULL — crashes your program (a segmentation fault). NULL itself is safe and useful as a "not pointing anywhere yet" marker; the danger is dereferencing it. Always make sure a pointer aims at a real box before you write *p. Set unaimed pointers to NULL so you can test them.

7 Quick check

One question 🎯

In the box-and-arrow picture, what does p = &y; do?
Hint: assigning to p moves the arrow; it copies no values.
8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📋
Thinking p = &y copies a value It only moves the arrow — changes which box p points at. To copy values you'd write *p = *q; or x = y;. Moving the arrow touches no data.
💥
Following an arrow to nowhere Dereferencing a NULL or never-aimed pointer crashes. In the picture there's no box on the end of the arrow — there's nothing to read. Check p != NULL first.
🎯
Mixing up p and *p in the picture Changing p moves the arrow; changing *p edits the value in the box at the arrow's tip. One re-aims, the other writes through.

Practice exercises 💪

Easy Draw it. On paper, sketch the boxes and arrow for int x = 5; int *p = &x;. Then do *p = 9; and update the value in the box. Run the same code and confirm x is 9.
Medium Move the arrow. With int x=1, y=2; int *p=&x;, print *p; then p = &y; and print *p again. The same expression *p gives 1 then 2 — because the arrow moved.
Stretch Safe NULL. Set int *p = NULL; and print it with %p (you'll see (nil) or 0). Guard with if (p != NULL) before dereferencing, then aim it at a real variable and deref safely.

🎓 What to remember

  • Draw a variable as a box with a value, and a pointer as a box holding an arrow to another box.
  • p = &x draws the arrow from p to x; *p follows it to the value.
  • Reassigning (p = &y) just swings the arrow to a new box — it copies no values.
  • Many arrows can land on one box (aliasing); changing through one is seen by all.
  • NULL is an arrow to nothing — safe to hold, never safe to dereference.

Binary I/O: fread & fwrite

Everything so far stored data as text — human-readable characters. fwrite and fread do the opposite: they copy a value's raw bytes straight to and from disk. It's fast, compact, and exact — but not human-readable and not portable. This is a brief tour of when (and when not) to reach for it, closing out Part IX.

1 Why this matters

Bytes, not characters 💾

With fprintf, the number 1000000 is stored as the seven characters "1000000" — readable, but the program has to convert the int into text on the way out and parse it back on the way in. Binary I/O skips all that: fwrite copies the int's raw 4 bytes to disk exactly as they sit in memory, and fread copies them back. That's faster and lossless (no rounding of doubles into decimal text), and it makes saving whole structs and arrays in one call trivial. The cost is that the file is unreadable in a text editor and not portable across different machines — so this is a brief, use-with-care overview.

Think of it as a memory snapshot on disk, rather than a typed-out description.

2 Text vs binary

Same value, two encodings 🔢

The data is the same; only the encoding differs. And to do binary I/O you open the file in binary mode — the b in the mode string:

FILE *f = fopen("data.bin", "wb");   // "wb" / "rb" — note the b

// text:    fprintf writes "1000000"  → 7 bytes of characters
// binary:  fwrite(&n, sizeof n, 1, f) → 4 bytes (the raw int)
The b tells the library "don't translate anything" — it matters on Windows, where text mode rewrites newlines (\n ⇄ \r\n); for binary data that translation would corrupt your bytes. On Linux/macOS "wb" and "w" behave the same, but always write the b for binary so the code is correct everywhere. Everything else about opening, checking for NULL, and fclose is exactly as in 9.1.
3 fwrite

Dump bytes to disk ⬇️

fwrite(ptr, size, count, f) writes count items, each size bytes, taken from ptr. It returns how many items it actually wrote:

int n = 1000000;
fwrite(&n, sizeof n, 1, f);          // one int (4 bytes)

int scores[5] = {10, 20, 30, 40, 50};
fwrite(scores, sizeof scores[0], 5, f); // a whole array in ONE call
The pattern is "address of the data, size of one element, how many elements." For a single value you pass &n and a count of 1; for an array you pass the array (already a pointer) and the element count. Saving an entire array or struct is a single call — no loop, no formatting. sizeof does the byte-counting for you, so you never hard-code 4 or 8.
4 fread

Read them back ⬆️

fread is the mirror image — same arguments, reading instead of writing. It returns the number of items actually read, which you should check (it's how you detect a short or empty file):

int scores[5];
size_t got = fread(scores, sizeof scores[0], 5, f);
if (got != 5) {                          // fewer than asked = short file / EOF
    fprintf(stderr, "expected 5, got %zu\n", got);
}
You read into a variable of the same type and size you wrote — the bytes only mean the right thing if they're interpreted as the original type. The return value is items, not bytes: ask for 5 and get 5 means success; getting fewer means the file ran out. A clean round-trip (fwrite then later fread with matching types) gives you back the exact value, bit for bit — which is the whole appeal.
5 The cautions

Exact, but not portable ⚠️

Binary's exactness comes with strings attached. The raw bytes depend on the machine that wrote them:

IssueWhy it bites
Endiannessbyte order of an int differs across CPUs
Type sizesint/long width can vary by platform
Struct paddingcompilers insert invisible gaps between fields

🚨 A binary file is a memory snapshot — don't trust it across machines (or pointers)

Because fwrite copies bytes exactly as they sit in memory, a file written on one machine may not read back correctly on another with different endianness, type sizes, or struct padding. So binary is great for a program saving and loading its own data on the same kind of system, but risky as an interchange format — for sharing data, prefer text (or a defined format like JSON). And never fwrite a pointer: you'd save a memory address that's meaningless the next time the program runs. Save the data it points to, not the pointer.

7 Quick check

One question 🎯

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📏
Reading into a different type than you wrote The bytes only mean the original value if interpreted as the same type and size. Write an int, read an int — don't mix.
🌍
Treating binary as an interchange format Endianness, type sizes, and padding differ across systems. For sharing data, use text or a defined format; keep binary for same-program save/load.
🔗
Writing a pointer with fwrite Saving a pointer stores an address that's meaningless on the next run. Save the data the pointer points to, never the pointer itself.

Practice exercises 💪

Easy Round-trip one value. Open "v.bin" with "wb", fwrite a single double, close it. Reopen with "rb", fread it back into another double, and confirm it's bit-for-bit identical.
Medium Save an array. Fill an int array, write it in one fwrite, then read it back in one fread into a fresh array and compare element by element. Check the return count both times.
Stretch Text vs binary size. Write the same array of 1000 ints two ways — once with fprintf (text) and once with fwrite (binary) — and compare the two file sizes on disk. Note which is smaller and think about why.

🎓 What to remember

  • Binary I/O writes raw bytes — fwrite/fread copy a value's memory directly, no text conversion.
  • Open with "wb" / "rb" (the b) so newlines aren't translated on Windows.
  • fwrite(ptr, size, count, f) and fread(...) move count items; both return the item count — check fread's.
  • Read into the same type you wrote; a matching round-trip is exact and lossless.
  • Not portable — endianness, type sizes, padding differ; never fwrite a pointer. Use it for same-program save/load.

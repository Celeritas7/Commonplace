break and continue

Sometimes you need to bail out of a loop early, or skip just one round and keep going. break exits the loop entirely; continue jumps to the next iteration. Two small keywords that give your loops an emergency exit and a skip button.

1 Why this matters

Loops with an exit and a skip 🧠

A normal loop only stops when its condition turns false. But real code often needs to bail out sooner — "stop searching the moment you find the match" — or skip a single round — "process every file except the hidden ones." break and continue give you that control without contorting your loop condition.

Think of them as a loop's two escape hatches. break is the emergency exit: it abandons the loop entirely and jumps to the code after it. continue is the skip button: it abandons just the current iteration and jumps ahead to the next one. Both make loops more expressive — and both have one subtlety each that trips people up. Let's take them one at a time.

2 break

The emergency exit 🚪

break immediately ends the loop it's in. Execution jumps straight to the first line after the loop — no more iterations, no final condition check.

// Stop as soon as we find the target
for (i = 0; i < n; i++) {
    if (data[i] == target) {
        printf("Found at %d\n", i);
        break;          // done — leave the loop now
    }
}
The classic use is searching: once you've found what you're looking for, there's no point checking the rest. break lets you stop right there. Without it, you'd either keep looping pointlessly or twist your loop condition into something harder to read. You already met break in lesson 3.2 — it's the same keyword that ends a switch case.
3 continue

The skip button ⏭️

continue abandons the rest of the current iteration and jumps ahead to the next one. The loop keeps running — you just skip whatever code came after the continue for this round.

// Print only the odd numbers
for (i = 1; i <= 10; i++) {
    if (i % 2 == 0) {
        continue;       // even? skip the rest, go to next i
    }
    printf("%d ", i);     // only reached for odd i
}                          // 1 3 5 7 9
continue is handy when you want to "filter" — handle most items, but skip the ones that don't qualify. The pattern is "if this one's no good, continue" near the top of the loop, so the rest of the body only runs for the items you care about. It keeps the main logic un-indented instead of buried inside a big if.
4 Where continue jumps

The infinite-loop trap 💥

Here's the subtlety that bites people. continue doesn't restart the loop from the top — it jumps to the loop's update/condition step. In a for loop, that means the update (i++) still runs. But in a while loop, the update is part of the body — and continue can skip right over it.

🚨 continue in a while loop can loop forever

If the variable that advances the loop is updated after the continue, then continue skips it — and the condition never changes:

int i = 0;
while (i < 5) {
    if (i == 2) continue;  // skips i++ → i stuck at 2 forever!
    printf("%d", i);
    i++;
}

The fix: make sure the loop variable advances before any continue, or use a for loop, where the update lives in the header and always runs.

In a…continue jumps to…Update runs?
for loopthe update, then the condition✅ Yes (it's in the header)
while loopthe condition directly⚠️ Only if it ran before the continue
5 Scope, nesting, and good taste

What they do and don't reach 🎯

  • They affect only the innermost loop. A break inside two nested loops exits just the inner one — the outer loop keeps going. (More on escaping nested loops next lesson.)
  • break vs return. break leaves the loop but stays in the function; return leaves the whole function. If finding your answer means you're completely done, return can be cleaner than break plus more code.
  • Don't overuse them. A loop riddled with breaks and continues in many places gets hard to follow. One well-placed early exit is clarity; five scattered ones are spaghetti.
A good rule of thumb: use break for "I'm done here" (searching, hitting a limit) and continue for "skip this one" (filtering). If you find yourself needing to break out of nested loops at once, that's a sign to either pull the loops into their own function (and return) or rethink the structure — plain break only escapes one level.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

♾️
continue in a while that skips the update If i++ sits after the continue, it never runs and the loop hangs. Advance the loop variable before any continue, or use a for loop where the update is in the header.
🪆
Expecting break to exit all nested loops break leaves only the innermost loop. The outer loop keeps going. To leave several at once, restructure into a function and return, or use a flag.
🍝
Scattering jumps everywhere A loop with breaks and continues sprinkled throughout becomes a maze. Prefer one clear early exit. If the logic needs many, the loop probably wants splitting up.

Practice exercises 💪

Easy First multiple. Loop from 1 to 100 and print the first number divisible by both 7 and 9, then stop with break. (It's 63.)
Medium Skip the negatives. Read 10 numbers and sum only the non-negative ones — use continue to skip any negative value. Print the running total at the end.
Stretch Input until valid. Use an infinite while (1) loop that reads a number and breaks out once the user enters one in the range 1–10, rejecting anything else with a message. Then explain why while (1) { ... break; } is a common, readable pattern.

🎓 What to remember

  • break exits the loop entirely and jumps to the code after it — great for searching.
  • continue skips the rest of the current iteration and moves to the next — great for filtering.
  • The while trap: continue can skip the update, causing an infinite loop. In a for, the header update always runs.
  • They affect only the innermost loop; break won't escape nested loops at once.
  • Use sparingly: one clear early exit aids readability; a swarm of jumps hurts it.

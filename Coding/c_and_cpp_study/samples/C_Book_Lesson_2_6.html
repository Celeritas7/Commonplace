Operator Precedence — The Rules You'll Actually Hit

When you write a + b * c, C doesn't read left to right — it knows * binds tighter than +. This lesson is the small set of precedence rules that bite beginners, plus a Parenthesizer that shows you exactly how C groups any expression.

1 Why this matters

One expression, one meaning 🧠

Every time you write more than one operator in a line — total = price * qty + tax — C has to decide what runs first. It never guesses and it never reads strictly left-to-right. It follows a fixed precedence table that's the same on every compiler, every machine.

The good news: you've actually been relying on this since grade-school math. 2 + 3 * 4 is 14, not 20, because multiplication binds tighter than addition. C uses the exact same idea — it just has more operators, and a few of them surprise people. Get precedence wrong and your code compiles cleanly but computes the wrong thing, which is the worst kind of bug. This lesson gives you the handful of rules that actually matter.

2 The mental model

Precedence = binding strength 🔗

Think of each operator as having a grip strength. A tighter grip means it grabs its operands first. In a + b * c, both + and * want b. Because * has the stronger grip, b goes to * — giving a + (b * c).

A useful trick: imagine C putting invisible parentheses around the tightest operators first, then the next tightest, and so on. The Parenthesizer below literally draws those parentheses for you. Once an expression is fully parenthesized, there's no ambiguity left — you just evaluate from the inside out.

Two separate questions are hiding here, and it helps to keep them apart:

  • Precedence — which different operator binds first (* vs +).
  • Associativity — when two operators have the same precedence, which side groups first (the - in 10 - 4 - 2). We'll get to it in a moment.
3 The table that bites

The rows worth knowing 🥇

C has around 15 precedence levels. You do not need to memorize all of them. Here are the ones that actually come up, strongest grip at the top:

The four rules that cover almost everything you'll write:

  • * / % before + -. a + b * c is a + (b * c).
  • Arithmetic before comparison. a + 1 > b is (a + 1) > b.
  • Comparison before && / ||. x > 0 && y > 0 needs no extra parens.
  • Assignment is loosest. x = a > b assigns the result of a > b to x.
4 Associativity

When two operators tie ⚖️

If two operators share the same rank, associativity breaks the tie. Almost everything in C is left-associative — it groups left to right, the way you'd read it:

ExpressionGroups asWhy it matters
10 - 4 - 2(10 - 4) - 2 = 4Not 10 - (4 - 2) = 8
100 / 5 / 2(100 / 5) / 2 = 10Order changes the answer
a = b = 0a = (b = 0)= is right-associative
a ? b : c ? d : ea ? b : (c ? d : e)Ternary groups right

Most of the time left-associativity is exactly what you expect, so you won't think about it. The two right-associative cases — assignment and the ternary — are the ones to remember. Right-associativity is what makes the chained assignment a = b = c = 0 work: it runs c = 0 first, then b = (that), then a = (that).

5 The famous C traps

Where precedence ambushes you 💥

A few C combinations are notorious because the precedence is the opposite of what reads naturally. These are worth burning into memory:

🚨 == binds tighter than the bitwise operators

You write if (flags & MASK == 0) meaning "mask the bits, then compare." But == outranks &, so C reads it as flags & (MASK == 0) — it compares first, then ANDs with a 0 or 1. Almost never what you meant. Always parenthesize: (flags & MASK) == 0.

🚨 Shifts are looser than + and -

1 << 2 + 3 looks like "shift 1 left by 2, then add 3" → you'd guess 7. But + binds tighter than <<, so it's 1 << (2 + 3) = 1 << 5 = 32. Parenthesize the shift amount when in doubt.

🚨 && binds tighter than ||

a || b && c means a || (b && c), not (a || b) && c. AND has the stronger grip. This one surprises nearly everyone — just add the parentheses and spell out your intent.

Notice the theme: when bitwise operators, shifts, or mixed &&/|| show up, your instinct is often wrong. The fix is never to memorize harder — it's to add parentheses. They cost nothing and they make the code say what it does.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🎭
Assuming bitwise binds like you'd read it x & MASK == 0 is x & (MASK == 0), not (x & MASK) == 0. Any time you mix & ^ | with == !=, wrap the bitwise part in parentheses.
📏
Forgetting that shifts are looser than + x << n + 1 shifts by n + 1, not "shift then add one." Write (x << n) + 1 if that's what you meant.
⚖️
Trusting && / || precedence without parens a || b && c means a || (b && c). Even when you know the rule, the reader (including future-you) shouldn't have to. Parenthesize mixed logic.

Practice exercises 💪

Easy Add the parens. Rewrite each with explicit parentheses showing how C groups it, then check in the Parenthesizer: a + b * c - d, a < b == c, a + b << c.
Medium Fix the bug. A programmer wrote if (count & 1 == 1) hoping to test whether count is odd. It always behaves the same regardless of count. Explain why, then fix it with one pair of parentheses.
Stretch Predict, then verify. Without running it, work out the value of 2 + 3 * 4 > 10 && 1 << 2 == 4. Fully parenthesize it on paper, evaluate inside-out, then confirm with the Parenthesizer. (Watch the == vs << trap.)

🎓 What to remember

  • Precedence = grip strength. Tighter operators grab their operands first; C inserts invisible parentheses accordingly.
  • The everyday order: * / % > + - > shifts > comparisons > && > || > assignment.
  • Associativity breaks ties. Nearly everything is left-to-right; = and ?: are right-to-left.
  • The trap trio: == beats bitwise & ^ |; + - beat shifts; && beats ||.
  • The fix is always the same: when in doubt, add parentheses. They cost nothing and remove all ambiguity.

Multiple Inheritance

C++ lets a class inherit from more than one base at once — useful when a type genuinely is two things (say, both drawable and serializable). It's powerful, but it brings two classic puzzles: name ambiguity when bases share a member, and the diamond problem when two bases share a common ancestor.

1 Why this matters

When a thing is two things 🧠

So far every class had a single base. But sometimes a type really does belong to two families at once: a FileLogger that is both a Logger and a Writable; a game entity that is both Drawable and Updatable. C++ supports multiple inheritance — listing several bases — so the derived class gets the members of all of them. It's a sharp tool: the cleanest use is combining small interface classes (abstract bases), and the things to watch for are name clashes and the famous diamond. Most other languages dropped multiple inheritance precisely because of those — C++ keeps it, with rules to manage it.

Know how it works, where it shines, and the two traps to steer around.

2 Declaring it

List several bases ➕

Separate the base classes with commas. The derived class gains the members of each:

struct Drawable  { void draw()   { /* ... */ } };
struct Updatable { void update() { /* ... */ } };

struct Sprite : public Drawable, public Updatable {
    // Sprite now has BOTH draw() and update()
};

Sprite s;
s.draw();      // from Drawable
s.update();    // from Updatable
A Sprite here is a Drawable and is an Updatable, so it can be used as either — and code that handles Drawable* or Updatable* works with it. When the two bases are independent like this, multiple inheritance is clean and handy: it's the natural way to say "this type supports both of these capabilities."
3 Name ambiguity

Two bases, same member 🔀

If both bases declare a member with the same name, an unqualified call is ambiguous — you must say which one you mean:

struct Base1 { void id() { /* ... */ } };
struct Base2 { void id() { /* ... */ } };
struct C : Base1, Base2 { };

C c;
c.id();          // ✗ error: ambiguous — Base1::id or Base2::id?
c.Base1::id();   // ✅ qualify to pick one
c.Base2::id();   // ✅ the other
The compiler won't guess. When two inherited names collide, you disambiguate with the scope operator: c.Base1::id(). It's not an error in the design necessarily — just C++ insisting you be explicit about which inherited member you mean. This is the first thing to watch for when you combine bases that might overlap.
4 The diamond problem

One ancestor, two paths 💎

The trickier case: two bases that themselves share a common ancestor. The derived class then inherits that ancestor twice:

struct Base { int v = 5; };
struct A : Base { };      // A has a Base
struct B : Base { };      // B has a Base too
struct C : A, B { };      // C has A's Base AND B's Base — two copies!

C c;
c.v;             // ✗ ambiguous: c.A::v or c.B::v? Two separate Bases.
Picture a diamond: Base at the top, A and B in the middle, C at the bottom inheriting both. Because A and B each carry their own Base, C ends up with two Base subobjects — two separate vs. Asking for c.v is ambiguous, and worse, you usually wanted a single shared Base, not two. That's the diamond problem.
5 Virtual inheritance

Share the ancestor ✨

The fix is virtual inheritance: mark the shared base virtual in the middle classes, and C gets just one shared Base:

struct Base { int v = 5; };
struct A : virtual Base { };   // "share" the Base
struct B : virtual Base { };
struct C : A, B { };

C c;
c.v;             // ✅ one shared Base → unambiguous, c.v == 5

🚨 Reach for interfaces before deep diamonds

Virtual inheritance solves the diamond, but multiple inheritance with shared state gets complicated fast (construction order, who initializes the shared base, etc.). In practice the safe, common use of multiple inheritance is combining interface classes — abstract bases with only pure virtual functions and no data (recall 7.4). With no data to duplicate, there's nothing to clash, and you get clean "implements both contracts" composition. Save real diamonds for when you genuinely need shared state.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔀
Leaving a name clash unqualified If two bases both declare id(), c.id() is ambiguous and won't compile. Qualify it: c.Base1::id().
💎
Forgetting virtual in a diamond Without virtual inheritance, C gets two copies of the shared base and c.v is ambiguous. Mark the shared base virtual in the middle classes to get a single one.
🧩
Using MI with state when an interface would do Multiple inheritance of data-carrying classes gets complex. Prefer combining abstract interface classes (pure virtual, no data) — clean and clash-free.

Practice exercises 💪

Easy Combine two. Make Drawable with draw() and Clickable with click(), then a Button : Drawable, Clickable. Create one and call both methods.
Medium Hit the ambiguity. Give two bases a same-named method and inherit both. Try the unqualified call (watch it fail), then fix it with obj.Base1::method().
Stretch Build & break a diamond. Create Base{int v}, A:Base, B:Base, C:A,B; show c.v is ambiguous. Add virtual to A's and B's inheritance and confirm c.v now works with a single shared Base.

🎓 What to remember

  • A class may inherit from multiple bases — struct C : A, B — gaining the members of all.
  • If two bases share a member name, qualify it: c.A::foo().
  • The diamond problem: two bases sharing a common ancestor gives the derived class two copies of it.
  • Virtual inheritance (: virtual Base) gives a single shared base, resolving the diamond.
  • Cleanest use: combine abstract interface classes (no data) — powerful, no clashes.

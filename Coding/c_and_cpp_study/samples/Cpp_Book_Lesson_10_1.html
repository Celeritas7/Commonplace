🧯 PART X · ERRORS AND EXCEPTIONS

Why Exceptions Exist

Things go wrong: a file won't open, input is malformed, memory runs out. The old answer — return an error code and check it after every call — clutters your logic and is easy to forget. Exceptions separate the error path from the happy path: you throw where the problem is, catch where you can handle it, and the language carries the error up the stack for you. This opens Part X.

1 Why this matters

Errors are part of programming 🧯

No matter how careful your code is, the world isn't: files go missing, networks drop, users type nonsense, memory fills up. A robust program has to detect these failures and respond. The question is how the error travels from where it's discovered (deep in some helper) to where it can actually be handled (often far up the call chain). C's answer was error codes — return a special value and have every caller check it. It works, but it tangles error-handling into every line and is dangerously easy to skip. C++'s answer is exceptions: a structured way to throw an error and have it automatically propagate to a matching handler, leaving your normal logic clean. This opens Part X — Errors and Exceptions.

This lesson is the why; the next lessons cover the how (throw/try/catch) in detail.

2 The error-code way

Check after every call 😮‍💨

With return codes, a function signals failure by what it returns, and every caller must check and pass the failure up by hand:

int load_config(Config *out) {
    int err = open_file(...);
    if (err) return err;         // check + bubble up
    err = read_header(...);
    if (err) return err;         // check + bubble up
    err = parse_body(...);
    if (err) return err;         // check + bubble up
    return 0;                    // success
}
Three problems jump out. First, the real logic is buried under if (err) return err; noise. Second, it's opt-in: forget one check and the error silently vanishes, and the program limps on with bad data. Third, some operations can't return a code at all — a constructor has no return value, and operators like + are supposed to return a result, not an error. Error codes leave those failure paths with nowhere to go.
3 The exception way

Throw here, catch there 🎯

With exceptions, the failing code throws, and a try/catch somewhere up the stack handles it. Everything in between stays clean — no checks:

void read_header(...) {
    if (bad) throw std::runtime_error("bad header");   // raise the error
}

try {
    load_config(&cfg);   // no per-call error checks inside
    use(cfg);
} catch (const std::exception& e) {       // handle it here
    std::cerr << "failed: " << e.what() << "\n";
}
The error is created at the point of failure (throw) and "flies" past every function in between straight to the nearest enclosing catch that matches its type. The intermediate functions don't need a single if (err) — they simply don't run their remaining code. Your happy path (load_config then use) reads as if nothing can fail, and the failure handling sits in one clearly-marked place. (Next lesson breaks down throw, try, and catch properly — this is the shape.)
4 Stack unwinding

The magic: RAII still runs 🧹

When an exception is thrown, C++ unwinds the stack — it exits each function on the way to the catch, and as it does, it runs the destructors of every local object. This is where Part VIII pays off:

void work() {
    std::ifstream file("data.txt");   // RAII: closes itself
    std::vector<int> buffer(1000);     // RAII: frees itself
    risky();   // throws → file & buffer destructors STILL run as the stack unwinds
    // ...code after risky() is skipped...
}
This is the deep reason exceptions are safe in modern C++: as the exception propagates out of work(), the local file and buffer are destroyed automatically — the file is closed, the memory freed — even though you wrote no cleanup. That's RAII (8.1) and exceptions working together: resources owned by stack objects are always released on the way out, by any exit including an exception. With raw new/manual cleanup, a thrown exception would leak; with RAII types, it just works. (It's also why a destructor should never itself throw.)
5 When to use them

Exceptional, not routine ⚖️

Exceptions are for unexpected failures, not ordinary control flow. The rule of thumb:

Use an exception for…Use a normal return for…
a file that won't open"user picked quit" (expected)
invalid input that breaks an invariant"item not found" in a search
out of memory, lost connectiona routine yes/no result

🚨 Don't use exceptions as ordinary control flow

Exceptions shine for genuinely exceptional conditions — the rare, hard-to-handle-locally failures. Don't use them for expected, everyday outcomes (a search that finds nothing, a loop that's done): that's slower and harder to read than a normal return value or an std::optional. The other side of the coin: an exception that's never caught calls std::terminate and aborts the program, so something up the stack should be ready to catch what you throw. Used well, exceptions keep the common path clean and put error handling where it belongs; overused, they hide control flow. (Some codebases even disable them for performance or determinism — but the standard library uses them throughout.)

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔁
Exceptions for ordinary control flow Don't throw for expected outcomes like "not found" or "user quit" — use a return value or std::optional. Reserve exceptions for genuinely exceptional failures.
🕳️
Throwing with nothing to catch An uncaught exception calls std::terminate and aborts. Make sure something up the stack (often main) is ready to catch what you throw.
💧
Manual cleanup that an exception skips Raw new/fopen with manual cleanup leaks when an exception jumps over it. Use RAII types (vector, string, unique_ptr, ifstream) so unwinding frees them automatically.

Practice exercises 💪

Easy Throw & catch. Write a function that throws std::runtime_error("boom"), call it inside a try, and catch (const std::exception& e) printing e.what(). Confirm code after the throw doesn't run.
Medium See the unwind. Make a class whose destructor prints "cleanup". Create one as a local in a function that then throws. Catch in main and confirm "cleanup" prints during the unwind, before the catch runs.
Stretch Codes vs. exceptions, side by side. Write a 3-level call chain (main → a → b) where b fails. Implement it once with return codes (checking at each level) and once with exceptions, and compare how much error-handling noise each version has.

🎓 What to remember

  • Errors must travel from where they're found to where they're handled — exceptions do this without per-call checks.
  • Error codes clutter logic, are easy to ignore, and can't report from constructors/operators.
  • Exceptions: throw at the failure, catch where you can handle it; intermediate code stays clean.
  • Stack unwinding runs destructors on the way out — with RAII types, resources are freed automatically even on an exception.
  • Use them for exceptional failures, not routine control flow; an uncaught exception terminates the program.

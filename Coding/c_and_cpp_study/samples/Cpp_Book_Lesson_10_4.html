Your Own Exception Class

Sometimes std::runtime_error isn't specific enough — you want callers to catch your error distinctly, and to carry extra data (an HTTP code, a filename, the bad value). The trick is simple: derive from a standard exception type so you join the hierarchy and inherit a working what().

1 Why this matters

When a standard type isn't enough 🧬

The standard hierarchy (10.3) covers generic failures, but real programs have domain-specific errors: a ConfigError, a NetworkError, a ParseError. Defining your own exception type does two things a bare std::runtime_error("...") can't. First, it gives callers a distinct type to catch — they can handle your error specifically while letting others propagate. Second, it lets the exception carry structured data — an HTTP status code, the offending filename, a line number — instead of cramming everything into a string. The golden rule: always derive from std::exception (usually via runtime_error or logic_error), so your type slots into the family and a catch (const std::exception&) still catches it.

This is just inheritance (Part VII) again — you're adding a new leaf to the tree you mapped in 10.3.

2 The easy way

Extend std::runtime_error ✅

The simplest correct custom exception inherits from std::runtime_error (or logic_error) and forwards the message to its constructor. You get a working what() for free — the base already stores and returns the string:

#include <stdexcept>

class ConfigError : public std::runtime_error {
public:
    using std::runtime_error::runtime_error;   // inherit the (string) constructor
};

// usage
throw ConfigError("missing 'port' key");   // what() == "missing 'port' key"
That using std::runtime_error::runtime_error; line is an inheriting constructor: it pulls in the base's constructors so ConfigError("...") works without you writing one. Because ConfigError is-a runtime_error is-a std::exception, all three of these catch it: catch (const ConfigError&), catch (const std::runtime_error&), and catch (const std::exception&). You wrote almost nothing and got a fully-integrated exception type — this is the form to reach for 90% of the time.
3 The manual way

Derive from std::exception & override what() 🔧

If you'd rather inherit straight from std::exception, you must provide what() yourself. Store the message as a member and return its c_str():

#include <exception>
#include <string>

class ParseError : public std::exception {
    std::string msg_;                  // the string OUTLIVES what()'s pointer
public:
    ParseError(std::string m) : msg_(std::move(m)) {}
    const char* what() const noexcept override {   // signature must match exactly
        return msg_.c_str();
    }
};
The signature matters: what() is declared const char* what() const noexcept in the base, so your override must match it exactly — write const noexcept override and let the compiler confirm you really are overriding (10.3, Part VII). The pointer from c_str() is only valid while msg_ lives — which is fine here because msg_ is a member that lives as long as the exception object. Storing the string as a member is the whole trick; returning c_str() of a temporary is the classic bug (see the gotcha below).
4 Carrying extra data

More than a message 📦

The real payoff of a custom type: attach structured data the handler can read. Add members and accessors alongside the message:

class HttpError : public std::runtime_error {
    int code_;
public:
    HttpError(int code, std::string msg)
        : std::runtime_error(std::move(msg)), code_(code) {}
    int code() const noexcept { return code_; }   // extra data the catcher can use
};

try {
    throw HttpError(404, "not found");
}
catch (const HttpError& e) {
    std::cout << e.code() << ": " << e.what();   // 404: not found
}
Here HttpError forwards the message to runtime_error (so what() still works) and adds its own code_ member with a code() accessor. A handler that catches HttpError& specifically can branch on e.code(); a generic std::exception& handler still catches it and can at least print what(). This is why custom exceptions beat error codes: the type tells you what went wrong, and the members carry the details, all the way up the stack to whoever can handle it.
5 Throwing & catching it

It plays by the same rules 🪜

Your type behaves exactly like a standard one: catch it specifically, or let a base handler take it. Order most-derived first (10.2) — a base listed first will swallow your derived type before its own clause is reached:

try {
    load_config();
}
catch (const ConfigError& e) { // most specific: handle config problems }
catch (const std::exception& e) { // backstop for everything else }

🚨 Don't return what() from a temporary, and always derive from std::exception

Two failure modes. First, the dangling what(): const char* what() const noexcept override { return ("error: " + name).c_str(); } builds a temporary std::string, takes a pointer into it, then destroys the temporary when the function returns — the caller reads freed memory (undefined behaviour). Build the full message once in the constructor and store it in a member; return that member's c_str(). Second, don't inherit from nothing: a custom exception that doesn't derive from std::exception can't be caught by catch (const std::exception&) — it'll slip past every generic handler and likely call std::terminate. Always root your type in the standard tree.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🧬
Not deriving from std::exception A custom exception that inherits from nothing can't be caught by catch (const std::exception&) and has no what(). Always derive from std::exception — usually via runtime_error or logic_error.
💥
Returning what() from a temporary return ("x" + s).c_str(); points into a string that's destroyed as what() returns — undefined behaviour. Build the message in the constructor, store it in a member, return that member's c_str().
✍️
Mismatching the what() signature The base declares const char* what() const noexcept. Drop the const or noexcept and you've hidden it, not overridden it. Write const noexcept override so the compiler checks you.

Practice exercises 💪

Easy The one-liner. Define class FileError : public std::runtime_error that inherits the base constructor with using. Throw FileError("cannot open data.txt"), catch it by const std::exception& e, and print e.what().
Medium Carry a code. Write HttpError deriving from std::runtime_error with an int code() accessor. Throw HttpError(404, "not found"), catch const HttpError& e, and print both e.code() and e.what().
Stretch Spot the dangling bug. Write a ParseError deriving from std::exception whose what() returns ("line " + std::to_string(n)).c_str() from a temporary. Observe the garbage/UB, then fix it by building the message in the constructor and storing it in a member.

🎓 What to remember

  • Always derive from std::exception — usually via std::runtime_error or std::logic_error — so your type joins the family and any base handler catches it.
  • The easy way: inherit from runtime_error and pull in its constructor with using — you get a working what() for free.
  • The manual way: from std::exception, store the message in a member and override const char* what() const noexcept override returning its c_str().
  • Carry data with extra members + accessors (code(), filename()); the handler that catches your type can read them.
  • Two traps: never return what() from a temporary string (dangling pointer), and match the what() signature exactly (const noexcept override).

Friend Functions

Private members are normally off-limits to outside code. A friend function is the deliberate exception: a non-member the class invites in to access its private parts. It's handy — chiefly for printing with << — but used sparingly, because it bends encapsulation.

1 Why this matters

An invited guest 🧠

In 6.3 you learned that private members are reachable only from inside the class — outside code is locked out. That's usually exactly what you want. But occasionally a function that isn't a member has a legitimate need to peek at the private data — the most common case being a function that prints an object with std::cout << obj. A friend function is how a class grants that one specific function access, without making everything public.

This is a short, "good to recognise" topic — you'll see friend in real code (especially around operator<<), so it's worth understanding, even though you'll write it rarely.

2 What a friend is

Access without membership 🤝

You declare a function a friend inside the class. That function can then touch the class's private members — even though it is not a member (no this, called as an ordinary free function):

class Box {
    int w = 5;                  // private
    friend int getWidth(Box b);  // grant getWidth access to privates
};

int getWidth(Box b) {            // a free function, not Box::getWidth
    return b.w;                 // ✅ allowed — it's a friend
}
Note the two halves: the friend declaration inside Box grants permission, and the function itself is defined outside like any normal function. You call it getWidth(b), not b.getWidth() — it has no object of its own, it just received a key to Box's private rooms.
3 The classic use

Printing with operator<< 🖨️

By far the most common reason to use friend is to let std::cout << obj work. That output operator can't be a member (its left operand is the stream, not your object), so it's a free function — and to read your private fields, it's made a friend:

class Point {
    int x = 1, y = 2;
    friend std::ostream& operator<<(std::ostream& os, Point p);
};

std::ostream& operator<<(std::ostream& os, Point p) {
    return os << "(" << p.x << ", " << p.y << ")";   // reads privates
}

std::cout << myPoint;   // (1, 2)
Don't worry about the operator<< mechanics yet — that's operator overloading, coming in 6.10. The point here is why it needs friend: a non-member function that genuinely must see the private state. This is the one pattern where friend is widely accepted as the right tool.
4 Granted, not taken

The class chooses its friends ⚖️

An important property: friendship is given by the class, not claimed by the function. A function can't declare itself a friend of some class — only the class can name who its friends are. And friendship is neither mutual nor inherited:

  • Not automatic: only the friend declaration inside the class grants access.
  • Not mutual: if A befriends B, that doesn't give A access to B.
  • Not inherited: a derived class doesn't inherit its base's friends.

🚨 Use friend sparingly

Every friend pokes a hole in encapsulation — it lets outside code reach private data, which is exactly what private was protecting. So reach for it only when there's a real need (mainly operator<< and a few tightly-coupled helpers). If a plain public method would do the job, prefer that. A class with a long list of friends is usually a design smell. (A whole class can also be a friend — friend class Other; — but that's even broader, so use it rarely.)

5 Friend vs member

At a glance 🔍

member functionfriend function
part of the class?yesno (free function)
has this?yesno

So a friend is the "outsider with a key": it lives outside the class and is called like any free function, but it was explicitly granted access to the private internals. Whenever you see friend in code, read it as "the class is trusting this function with its private data" — and ask whether that trust is really necessary.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🧩
Thinking a friend is a member A friend function is a non-member — no this, called as f(obj) not obj.f(). The friend declaration just grants it access; it doesn't make it part of the class.
🕳️
Overusing friend Every friend pokes a hole in encapsulation. Use it sparingly — mainly operator<< and tightly-coupled helpers. If a public method would do, prefer that.
🔁
Expecting friendship to be mutual or inherited It's neither. A befriending B gives B access to A, not the reverse — and derived classes don't inherit the base's friends.

Practice exercises 💪

Easy One friend. Give a class Account a private balance and a friend double peek(Account a) that returns it. Print the balance from main via peek.
Medium Print with <<. Add a friend std::ostream& operator<<(std::ostream&, Account) so std::cout << acc; prints the balance. (A taste of 6.10 — the friend is what lets it read the private field.)
Stretch Feel the difference. Write a non-friend function that tries return a.balance; and read the compiler error. Then add the friend declaration inside the class and watch it compile — exactly the access friend grants.

🎓 What to remember

  • A friend function can access a class's private members even though it isn't a member of the class.
  • Declare it with friend inside the class — the access is granted by the class, not taken.
  • A friend is a non-member: no this, called as a free function f(obj).
  • The classic use is operator<< for printing (full operator overloading in 6.10).
  • Use friend sparingly — it bends encapsulation; friendship isn't mutual or inherited.

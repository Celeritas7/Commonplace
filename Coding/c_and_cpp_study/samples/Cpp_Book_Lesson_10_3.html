🧯 PART X · ERRORS AND EXCEPTIONS

The Standard Exception Hierarchy

You've been throwing std::runtime_error — but it's one of a whole family. The standard library defines a tree of exception types, all rooted at std::exception, split into "your bug" (logic_error) and "runtime trouble" (runtime_error) branches. Knowing the tree lets you throw the right type and catch at the right level.

1 Why this matters

A ready-made family of errors 🌳

You don't have to invent an exception type for every failure — the standard library ships a coherent hierarchy of them, and uses them itself. They all inherit from one base, std::exception, so a single catch (const std::exception&) can handle any of them. Below the root they split into two big families: logic_error (mistakes a correct program should have prevented — bad arguments, out-of-range indices) and runtime_error (failures detected only while running — out of range values, overflow). Knowing this tree means you can throw the type that best describes a failure, and catch precisely the ones you can handle while letting a base handler mop up the rest.

It's just inheritance (Part VII) applied to errors — and you already know how catch matches a base to its derived types (10.2).

2 The root

std::exception & what() 🌱

At the top sits std::exception (from <exception>). Its key member is a virtual what() that returns a C-string describing the error:

try {
    risky();
}
catch (const std::exception& e) {   // catches ANY standard exception
    std::cout << e.what();          // the message (virtual → real type's text)
}
Because every standard exception derives from std::exception, catching const std::exception& is the universal safety net — it'll catch an out_of_range, a bad_alloc, a runtime_error, anything in the tree. And because what() is virtual (Part VII), calling it through the base reference returns the actual type's message. That's why catching by reference matters (10.2): it preserves the real object so what() tells you what truly went wrong.
3 The two branches

logic_error vs runtime_error ⚖️

Most of the everyday types (in <stdexcept>) hang off two intermediate classes that capture whose fault the error is:

BranchMeaning & common subclasses
logic_errora preventable bug: invalid_argument, out_of_range, length_error, domain_error
runtime_errora runtime failure: range_error, overflow_error, underflow_error
(other roots)bad_alloc (from new), bad_cast (from dynamic_cast)
The distinction is about cause: a logic_error signals a programming mistake that could, in principle, have been caught before running — you passed a negative size, indexed past the end, gave an out-of-domain argument. A runtime_error signals something that genuinely depends on runtime conditions and couldn't be checked in advance. Both inherit from std::exception, so a base handler catches either; the split mainly helps you choose the type to throw and lets callers catch one family without the other. A few important exceptions (like bad_alloc) derive straight from std::exception, not from these two.
4 Throwing the right one

Pick a type, add a message 🏷️

Construct a standard exception with a descriptive string; what() returns it. Reach for the most specific type that fits:

#include <stdexcept>

void set_age(int a) {
    if (a < 0)
        throw std::invalid_argument("age cannot be negative");   // a logic_error
    // ...
}

// the library throws these for you, too:
v.at(99);          // std::out_of_range  if 99 is past the end
std::stoi("abc");   // std::invalid_argument
new int[HUGE];     // std::bad_alloc  if allocation fails
Choosing a precise type (invalid_argument rather than a bare runtime_error) documents the failure and lets callers catch exactly that case. You'll also receive these from the standard library: vector::at throws out_of_range, std::stoi throws invalid_argument or out_of_range, and a failed new throws bad_alloc. Knowing which function throws which type is half of handling errors well.
5 Catching at the right level

Specific when you can, base as backstop 🪜

Use the hierarchy to catch as narrowly as you can handle, and keep a base handler as a safety net (remember: most-derived first, from 10.2):

try {
    process(input);
}
catch (const std::out_of_range& e) { // handle this one specifically }
catch (const std::exception& e)   { // everything else in the tree }

🚨 Always include <stdexcept>, and catch the base, not "everything but"

Two practical notes. First headers: std::exception lives in <exception>, but the everyday types (runtime_error, logic_error, out_of_range, invalid_argument, …) are in <stdexcept> — include it or you'll get "not declared" errors. Second, design: catching const std::exception& is the right backstop because it covers the whole standard tree (and any well-behaved custom exception, which should derive from it — next lesson). Don't try to enumerate every leaf type; catch the specific ones you can act on, then a single std::exception handler for the rest. And note catch (...) would also catch non-standard throws (a raw int, say), but unlike std::exception& it can't give you a what() message — prefer the typed base when you can.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📦
Forgetting <stdexcept> runtime_error, logic_error, out_of_range, etc. live in <stdexcept> (the base std::exception is in <exception>). Include it or they're "not declared".
🏷️
Throwing a vague type Throwing a bare std::exception or generic runtime_error for a bad argument hides intent. Pick the specific type — invalid_argument, out_of_range — that names the failure.
🔢
Throwing non-exception values throw 42; or throw "oops"; can't be caught by std::exception& and carry no what(). Throw a type derived from std::exception.

Practice exercises 💪

Easy Catch what the library throws. Call std::vector::at() with a bad index inside a try, catch const std::exception& e, and print e.what(). Note it's actually a std::out_of_range — caught via the base.
Medium Throw the right type. Write set_percentage(int p) that throws std::invalid_argument if p is outside 0–100, with a clear message. Catch it specifically, then add a base handler and confirm both can catch it.
Stretch Map the branches. In one try, throw (in separate runs) an out_of_range, a runtime_error, and a bad_alloc. With handlers for logic_error, runtime_error, and exception (in that order), predict then verify which clause catches each.

🎓 What to remember

  • std::exception is the root of the standard hierarchy; its virtual what() returns the message — catch const std::exception& as a universal backstop.
  • Two branches: logic_error (preventable bugs — invalid_argument, out_of_range) and runtime_error (runtime failures — overflow_error, …).
  • Throw the most specific type that fits, with a clear message; the library throws these too (at→out_of_range, new→bad_alloc).
  • Catch specific then base (most-derived first); a base handler catches every derived type.
  • Include <stdexcept> for the everyday types, and throw types derived from std::exception — not bare ints or strings.

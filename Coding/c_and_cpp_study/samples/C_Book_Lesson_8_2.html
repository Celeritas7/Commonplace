typedef — Naming Your Types

Writing struct Point on every line gets old fast — and unsigned long long is a mouthful. typedef gives any existing type a short, readable alias: define it once, then use the nicer name everywhere. It doesn't make a new type — just a new name for one you already have.

1 Why this matters

Shorter, clearer names 🧠

Last lesson you had to write struct Point every single time you wanted a point. That's noisy, and it gets worse with longer types like unsigned long long or struct LinkedListNode. typedef fixes it: you give a type a nickname once, then use that nickname everywhere. It's a small feature with a big payoff for readability — and it's how nearly every real C codebase makes its structs pleasant to use. It also gives you one place to change a type later, and lets a name communicate intent (typedef int Celsius;).

This is a quality-of-life tool for the custom types you started building in 8.1.

2 What typedef does

An alias for a type ✍️

The form is typedef <existing type> <new name>; — the new name comes last. After it, the new name means exactly the existing type:

typedef unsigned int uint;   // "uint" now means "unsigned int"

uint count = 5;          // identical to: unsigned int count = 5;
uint i, j, k;            // all three are unsigned int
Read a typedef just like a variable declaration, but with typedef in front: where you'd normally name a variable, you're naming a type instead. So typedef unsigned int uint; "declares uint" — except uint becomes a type alias, not a variable. From then on the two names are completely interchangeable.
3 typedef with structs

Drop the struct keyword 🧱

This is the most common use. You can alias an existing struct, or define the struct and alias it in one go:

// Option A: define the struct, then alias it
struct Point { int x; int y; };
typedef struct Point Point;   // now "Point" == "struct Point"

// Option B: define + alias together (very common)
typedef struct { int x; int y; } Point;

Point p = {3, 4};        // no more "struct" needed 🎉
With the alias in place, Point p; just works — no struct keyword. Option B uses an anonymous struct (no tag) and names it only through the typedef; it's the idiom you'll see most often. Either way, your code reads like Point is a built-in type, which is exactly the feel you want for your own types.
4 Why it's useful

Readable, portable, intentional 📖

A good alias makes code self-documenting: Celsius t = 20; reads better than int t = 20;. And if you alias real to float and later need more precision, you change the one typedef to double and the whole program follows. This is exactly why the standard library is full of typedefs like size_t and uint32_t — names that hide a concrete type behind a meaningful one.
5 An alias, not a new type

Same type underneath ⚠️

Important: a typedef creates a name, not a new distinct type. The alias and the original are fully interchangeable — there's no extra type checking between them:

typedef int Celsius;
typedef int Fahrenheit;

Celsius c = 20;
Fahrenheit f = c;        // compiles fine — both are just "int" 😬

🚨 typedef won't catch mixed-up units

Because Celsius and Fahrenheit are both aliases for int, the compiler treats them as identical and happily lets you assign one to the other — even though that's a bug in meaning. A typedef improves readability, not type safety. If you need the compiler to actually keep two int-based types apart, you'd wrap them in distinct structs instead. (C++ adds stronger tools for this; in C, typedef is purely a naming convenience.)

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔀
Putting the new name first The form is typedef <existing type> <new name>; — the alias comes last. typedef uint unsigned int; is backwards and won't compile.
⚠️
Expecting type safety A typedef is just a name. Celsius and Fahrenheit aliased to int are the same type — the compiler won't stop you mixing them. Use it for clarity, not safety.
➡️
Hiding a pointer inside an alias typedef char* String; works, but const String s means "const pointer," not "pointer to const" — surprising. Be cautious when a typedef buries a *.

Practice exercises 💪

Easy Alias a primitive. Write typedef unsigned char byte;, declare a byte b = 200;, and print it with %d. Confirm byte behaves exactly like unsigned char.
Medium Clean up your struct. Take the struct Point from 8.1 and add typedef struct { int x; int y; } Point;. Rewrite a small program to use Point everywhere — no struct keyword in sight.
Stretch A self-referential node. Define typedef struct Node { int value; struct Node *next; } Node;. Notice the tag Node is still needed inside the struct for the next pointer, even though the typedef names it for outside use. (You'll use exactly this for a linked list later in Part VIII.)

🎓 What to remember

  • typedef <type> <name>; creates an alias — the new name comes last.
  • The classic use is structs: typedef struct { … } Point; lets you drop the struct keyword.
  • It aids readability and intent (Celsius, byte) and gives one place to change a type.
  • It's an alias, not a new type — no extra type checking between the alias and the original.
  • Watch typedef'd pointers — a hidden * changes how const and multiple declarations behave.

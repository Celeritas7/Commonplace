Capstone: A Contacts CLI

Time to put it all together. We'll build a complete little program — a contacts manager you run in the terminal: add people, list them, search, delete, and save everything to a file so it's there next time. Structs, arrays, a menu loop, string handling, and file I/O — the whole C book in one project you can actually compile and use.

1 What we're building

The whole book in one program 📇

A contacts CLI: a text-menu program that stores a list of people (name + phone), lets you add, list, find, and delete them, and saves to a file so your contacts survive between runs. It's small — a couple hundred lines — but it exercises nearly everything: a struct for each contact (Part VIII), an array to hold them, a while menu loop with switch (Part III), string compare/copy (Part V), and reading & writing a file (Part IX). This lesson walks the design piece by piece; the interactive runs a working version so you can see it behave before you build your own.

The aim isn't new syntax — it's seeing how the parts you know snap together into something real.

2 The data model

A struct and an array 🗂️

One struct describes a contact; a fixed-size array (plus a count) holds them all:

// contacts.h
#define MAX_CONTACTS 100
#define NAME_LEN 50

typedef struct {
    char name[NAME_LEN];
    char phone[NAME_LEN];
} Contact;

typedef struct {
    Contact items[MAX_CONTACTS];
    int count;            // how many are actually used
} Book;
A Contact bundles a name and phone (fixed-length char arrays — simple and safe to fwrite later). The Book wraps the whole collection: a fixed array items and a count of how many slots are in use. Keeping the count with the array is the classic C idiom — every operation works on the first count entries and ignores the rest. Using a fixed cap (MAX_CONTACTS) keeps this capstone simple; a real one might grow with malloc/realloc (Part VII), which is a great stretch goal.
3 The menu loop

Read a choice, dispatch, repeat 🔁

The heart of any CLI tool: loop forever, show options, read the user's pick, and act on it:

int main(void) {
    Book book = { .count = 0 };
    load(&book, "contacts.dat");   // restore saved contacts

    int choice;
    do {
        printf("\n1 Add  2 List  3 Find  4 Delete  5 Save  0 Quit\n> ");
        if (scanf("%d", &choice) != 1) break;   // bad input → stop
        switch (choice) {
            case 1: add_contact(&book);    break;
            case 2: list_contacts(&book);  break;
            case 3: find_contact(&book);   break;
            case 4: delete_contact(&book); break;
            case 5: save(&book, "contacts.dat"); break;
        }
    } while (choice != 0);
}
A do…while loop is natural here — show the menu at least once, keep going until the user picks 0. Each menu number maps through a switch to a function that does one job, passing &book so the functions can read and modify the shared collection (pointers to a struct, Part VIII). Checking scanf's return (!= 1) guards against non-numeric input — a small but important robustness habit.
4 The operations

Add, find, delete 🛠️

Each operation is a short function over book->items and book->count. Here are the three that carry the logic:

void add_contact(Book *b) {
    if (b->count >= MAX_CONTACTS) { puts("Full!"); return; }  // bounds!
    Contact *c = &b->items[b->count];
    printf("Name: ");  scanf("%49s", c->name);    // %49s = no overflow
    printf("Phone: "); scanf("%49s", c->phone);
    b->count++;
}

int find_index(Book *b, const char *name) {
    for (int i = 0; i < b->count; i++)
        if (strcmp(b->items[i].name, name) == 0) return i;
    return -1;                       // not found
}

void delete_at(Book *b, int i) {
    b->items[i] = b->items[b->count - 1];   // move last into the gap
    b->count--;                            // shrink
}
Three small, focused functions. add checks the array isn't full before writing (avoiding the out-of-bounds UB from 10.5) and uses %49s so scanf can't overflow the 50-char buffer. find_index walks the used entries with strcmp and returns the index or -1 — the universal "not found" convention. delete_at uses a neat trick: copy the last contact into the deleted slot and drop the count by one (order doesn't matter for a contact list, and it avoids shifting everything down). Each does one thing well — easy to test, easy to read.
5 Persistence & building

Save, load, and compile 💾

To survive between runs, write the whole Book to a file and read it back — binary I/O (9.6) makes this two calls. And a tiny Makefile (10.3) builds it:

void save(Book *b, const char *path) {
    FILE *f = fopen(path, "wb");          // binary write
    if (!f) { perror("save"); return; }
    fwrite(&b->count, sizeof b->count, 1, f);          // how many
    fwrite(b->items, sizeof b->items[0], b->count, f);  // the contacts
    fclose(f);
}
void load(Book *b, const char *path) {
    FILE *f = fopen(path, "rb");
    if (!f) return;                          // no file yet = empty book, fine
    fread(&b->count, sizeof b->count, 1, f);
    fread(b->items, sizeof b->items[0], b->count, f);
    fclose(f);
}

🚨 Save the count first, and read back the same way you wrote

The save/load pair must agree exactly. Write the count first, then that many Contacts; on load, read the count first so you know how many to read back. Mismatch the order or the types and you'll read garbage (the binary-I/O caution from 9.6). A missing file on first run isn't an error — load just returns and you start with an empty book. Build it all with a Makefile: contacts: main.o contacts.o linking the two objects, exactly as in 10.3. (Binary save is simplest here; a real app might use a text format so the file is portable and human-readable — a good stretch.)

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📏
Adding without checking count < MAX Writing items[count] when the array is full is out-of-bounds UB (10.5). Always bounds-check before adding, and use %49s so scanf can't overflow a 50-char name.
💾
Save/load that don't match Read back exactly what you wrote, in the same order and types — count first, then that many contacts. Mismatch and you load garbage. And don't treat a missing file on first run as an error.
🔁
Forgetting to handle "not found" find_index returns -1 when there's no match. Check for it before using the index — otherwise find/delete on a missing name reads a bogus slot.

Build it yourself 💪

Easy Type it in and run it. Put the code above into contacts.h, contacts.c, and main.c, write the Makefile, and make it. Add a few contacts, list them, save, quit, re-run — confirm they're still there.
Medium Add an "edit" option. Add menu item 6 Edit that finds a contact by name and lets you change the phone number. Reuse find_index, and handle the not-found case cleanly.
Stretch Grow it & harden it. Switch from a fixed array to a malloc/realloc-grown one (Part VII) so there's no MAX_CONTACTS; switch the file format to text (one name,phone per line) so it's portable; and run the whole thing under AddressSanitizer to confirm it's clean.

🎓 What to remember

  • A real program is your toolkit combined — struct + array + count, a menu loop, string ops, and file I/O.
  • Model the data first: a Contact struct and a Book that pairs an array with its count.
  • One function per operation (add/list/find/delete), each taking &book — small, focused, testable.
  • Persist with save/load — write the count then the contacts, read them back the same way; a missing file just means "empty."
  • Be defensive: bounds-check before adding, limit scanf widths, and handle "not found" (-1).

Arrays as Function Arguments

When you pass an array to a function, it doesn't get copied — it decays to a pointer to its first element. That has two big consequences: the function can change your array, and it can no longer tell how long the array is. So you always pass the length too.

1 Why this matters

Functions work on whole collections 🧠

A function that prints a list, sums some numbers, or sorts your data needs the whole array, not one value. So you pass the array as an argument. But arrays behave differently from ints and doubles when handed to a function — they don't get copied. Understanding why is the key to almost every C function that touches arrays.

You met array decay back in 5.2: an array's name turns into a pointer to its first element in almost every expression. Passing it to a function is exactly such an expression. This lesson is decay in action — what the function actually receives, what it can do with it, and the one extra thing you must always hand over: the length.

2 The mental model

The function gets the address, not a copy ✏️

For a normal variable, the function gets a private copy — changing it inside doesn't affect the caller. An array is different: the function receives a pointer to the original. There's no copy. So when the function writes to a[i], it's writing into your array.

void bump(int a[], int n) {
    a[0] = 99;            // writes into the CALLER's array
}

int main(void) {
    int v[] = {1, 2, 3};
    bump(v, 3);
    printf("%d\n", v[0]); // 99  — the change stuck!
}
This is a feature, not a bug. It's how a function can fill, sort, or modify an array in place. It's also why arrays are cheap to pass — only an address (a few bytes) is handed over, no matter how huge the array is. But it means you give the function the power to change your data, so be deliberate about it.
3 The size is lost — pass it yourself

Always hand over the length 📏

Because the function only gets a pointer, it has no way to know how many elements the array holds. Inside the function, sizeof measures the pointer, not the array — so the trick from 5.1 (sizeof(arr)/sizeof(arr[0])) breaks. The fix is simple: pass the length as a second argument.

int sum(int a[], int n) {   // n = how many elements
    int total = 0;
    for (int i = 0; i < n; i++)
        total += a[i];
    return total;
}

sum(v, 3);                  // caller knows the size — tell the function

🚨 Using sizeof(a) inside the function

Inside sum, sizeof(a) is the size of a pointer (typically 8 bytes), not the array. So sizeof(a)/sizeof(a[0]) gives something like 8/4 = 2 regardless of the real length — a silent, wrong answer. The element count only exists where the array was declared; once it decays, that information is gone. Always pass n.

4 Three spellings, one meaning

int a[] is just int *a 🎭

You'll see array parameters written several ways. They all mean the exact same thing — a pointer. The compiler quietly rewrites them:

How you write itWhat it really isNote
int a[]int *amost common style
int a[10]int *athe 10 is ignored!
int *aint *athe honest version
The number in int a[10] is decoration — the compiler throws it away, and passing a 3-element array still compiles fine. That's exactly why it can't protect you and why you pass n separately. Pick the int a[] form for readability (it signals "this is an array"), but remember it's a pointer underneath.
5 Protect read-only arrays

Use const when you only read 🛡️

If a function only reads the array (like sum or print), mark the parameter const. Now the compiler will stop you from accidentally writing to it — a small promise that prevents a whole class of bugs.

int sum(const int a[], int n) {
    // a[i] = 0;   // ❌ compiler error — good, we promised not to write
    int t = 0;
    for (int i = 0; i < n; i++) t += a[i];
    return t;
}
Reach for const on every array parameter the function doesn't change. It documents intent ("I won't touch your data"), lets the compiler catch mistakes, and is the professional default. Leave const off only when the function is meant to modify the array — like a sort or a fill.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📏
Using sizeof inside the function to find the count Inside the function the parameter is a pointer, so sizeof(a) is the pointer's size (usually 8), not the array's. Always pass the length as a separate argument.
✏️
Forgetting the function edits your real array Since it gets a pointer, not a copy, anything it writes to a[i] changes the caller's data. If a function should only read, add const so the compiler enforces it.
↩️
Returning a local array You can't safely return an array declared inside a function — it disappears when the function ends, leaving a dangling pointer. Instead, have the caller pass in an array for the function to fill.

Practice exercises 💪

Easy Print an array. Write void printArray(const int a[], int n) that prints all n elements separated by spaces. Call it from main with a small array. Notice you had to pass n — the function couldn't have known it.
Medium Double in place. Write void doubleAll(int a[], int n) that multiplies every element by 2. Print the array in main before and after the call to prove the original was changed — that's the pointer at work.
Stretch sum, max, and const. Write sum and max functions that each take (const int a[], int n). Try adding a[0] = 0; inside one and watch the compiler reject it — proof that const is protecting the caller's data.

🎓 What to remember

  • Arrays decay to a pointer when passed — the function gets the address of the first element, not a copy.
  • The function edits the original: writing a[i] changes the caller's array, so changes persist.
  • The size is lost — sizeof inside the function measures the pointer. Always pass the length n.
  • int a[], int a[10], and int *a are identical parameters — all just a pointer (the number is ignored).
  • Mark read-only parameters const so the compiler stops accidental writes.

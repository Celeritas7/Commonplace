Move Semantics

Copying an owning object duplicates its resource — and that's wasteful when the source is a temporary about to be destroyed anyway. C++11 added a smarter option: move. Instead of deep-copying, you transfer ownership — steal the resource and leave the source empty. It's the difference between photocopying a document and just handing it over.

1 Why this matters

Don't copy what you can hand over 🧠

Last lesson, copying an owning object meant a deep copy: allocate a new resource and duplicate the contents. That's correct, but sometimes wasteful. Consider returning a big std::vector from a function, or doing v.push_back(makeHugeThing()) — the source is a temporary that's about to be destroyed. Deep-copying it just to throw the original away is pure waste. Move semantics (added in C++11) lets the new object steal the temporary's resource — take its pointer, leave it empty — which is a couple of pointer assignments instead of a full allocation-and-copy. Same result, far cheaper.

This is why modern C++ can return huge containers by value without fear: the compiler moves, it doesn't copy.

2 lvalues, rvalues, and T&&

What can be safely stolen from ⏳

To know when stealing is safe, C++ classifies expressions. An lvalue has a name and lives on — you can't gut it. An rvalue is a temporary or expiring value with no lasting name — stealing from it is harmless. The new type T&& (an rvalue reference) binds only to rvalues:

int x = 5;
int&  lref = x;      // lvalue reference — binds to a named thing
int&& rref = 5;      // rvalue reference — binds to a temporary (5)
int&& bad  = x;      // ERROR: x is an lvalue, not a temporary
The single & means "reference to something with a name" (an lvalue); the double && means "reference to a temporary" (an rvalue). The whole point of && is that it lets a function know "this argument is a temporary — its contents are yours to take." That's the hook move operations use: by overloading on const T& (copy) versus T&& (move), a class can do the cheap thing when given a temporary and the safe thing when given a named object.
3 The move constructor

Steal the pointer, null the source 🫳

A move constructor takes a T&&. It copies the source's pointer (not the buffer!) and then sets the source's pointer to nullptr so the source's destructor won't free what you just took:

struct Box {
    int* data;
    Box(int v) { data = new int(v); }
    ~Box() { delete data; }
    Box(const Box& o) { data = new int(*o.data); }   // COPY: deep, allocates
    Box(Box&& o) noexcept {                            // MOVE: steal, no allocation
        data = o.data;        // take the pointer
        o.data = nullptr;    // leave source empty so its dtor is a no-op
    }
};
Compare the two: the copy constructor calls new and duplicates the value; the move constructor just reassigns a pointer and nulls the original. After the move, the source o still exists and will be destroyed normally — but its data is nullptr, so delete nullptr (which is safe and does nothing) runs instead of double-freeing the buffer you stole. Mark move operations noexcept: it tells the standard library they won't throw, which lets containers like vector move your objects during reallocation instead of copying.
4 Move assignment

Release mine, take yours 🔁

Move assignment overwrites an object that already owns a resource, so it must free its own first, then steal the source's:

Box& operator=(Box&& o) noexcept {
    if (this == &o) return *this;   // guard self-move
    delete data;             // 1. release what we already own
    data = o.data;           // 2. steal the source's pointer
    o.data = nullptr;       // 3. leave the source empty
    return *this;            // 4. return *this for chaining
}
It mirrors copy assignment from 8.2, but step 2 is a pointer steal instead of a deep copy, and step 3 nulls the source. The structure is the same — release the old, take the new, return *this — which is why people group all five special members together. Once a class has correct move operations, assigning a temporary to it (box = makeBox();) transfers ownership for almost free instead of copying.
5 std::move & the Rule of Five

Opting in to a move 🎭

Moves happen automatically when the source is already a temporary (a return value, an unnamed object). But if you have a named object you're finished with, you can opt in with std::move, which simply casts it to an rvalue so the move overload is chosen:

Box a(5);
Box b = a;              // a is an lvalue → COPY
Box c = std::move(a);   // std::move(a) is an rvalue → MOVE (a is now empty)

🚨 std::move doesn't move — and the source isn't gone

Two surprises. First, std::move(a) does nothing at runtime — it's just a cast to T&& that permits a move to be selected; the actual stealing happens inside the move constructor/assignment. Second, after you move from a, it isn't destroyed — it's left in a valid but unspecified (usually empty) state. You may safely destroy it or assign it a new value, but don't rely on its old contents. The full set of special members is the Rule of Five: destructor, copy constructor, copy assignment, move constructor, move assignment — and the easiest way to get all five right is still the Rule of Zero: hold std::vector/std::string/std::unique_ptr members and let them move for you.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🫙
Using a moved-from object's value After std::move(a), a is valid but unspecified (usually empty). You may reassign or destroy it, but don't read its old contents.
🧨
Forgetting to null the source A move constructor that steals the pointer but doesn't set the source's to nullptr leads to a double free — both destructors free the same buffer. Always empty the source.
🎭
Thinking std::move moves something It's just a cast to an rvalue. The move only happens if a move constructor/assignment exists and gets selected — otherwise it silently falls back to a copy.

Practice exercises 💪

Easy Copy vs. move. Give a class a copy constructor that prints "copy" and a move constructor that prints "move". Then write T b = a; and T c = std::move(a); and confirm which message each prints.
Medium Steal and null. Write a Box owning an int* with a move constructor that takes the pointer and nulls the source. Move-construct a new box, then print whether the source's pointer is nullptr and confirm nothing double-frees.
Stretch Rule of Five → Zero. Implement all five special members for a raw-pointer class with print statements, then rewrite it to hold a std::vector<int> and delete all five. Show that returning it from a function still moves (no deep copy) for free.

🎓 What to remember

  • Move = transfer ownership, not duplicate: steal the resource, leave the source empty. Cheap where copy is expensive.
  • T&& is an rvalue reference — it binds to temporaries (rvalues), the things safe to steal from.
  • Move constructor / assignment copy the pointer and null the source (assignment also frees its own first); mark them noexcept.
  • std::move is just a cast to an rvalue — it selects the move; it doesn't move anything by itself.
  • A moved-from object is valid but unspecified; the full set is the Rule of Five — or sidestep it with the Rule of Zero.

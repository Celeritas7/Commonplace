noexcept

A function marked noexcept promises never to let an exception escape. It's part of the type, it documents intent, and — crucially — it tells the standard library it can move instead of copy. Keep the promise and you get speed; break it and the program calls std::terminate.

1 Why this matters

A promise the compiler can use 🤝

noexcept declares that a function will not let an exception escape. That's more than a comment — it's part of the function's type, checked and exploited by the compiler. It does two valuable things. It documents a guarantee callers can rely on (this operation is safe to use where throwing would be disastrous). And it unlocks optimizations — the single most important being that standard containers will move your objects instead of copying them during reallocation, but only if the move is noexcept. The flip side: it's a real promise. If a noexcept function throws anyway, there's no catching it — the program calls std::terminate on the spot.

This builds on everything in Part X: you know how exceptions propagate (10.1–10.2) and the types involved (10.3–10.4) — now you decide which functions promise not to throw.

2 The basics

The keyword, and the cost of lying 🚫

Append noexcept to a function declaration to promise it won't throw. A normal (unmarked) function is potentially-throwing:

void safe() noexcept { /* guaranteed not to throw */ }
void risky()          { /* may throw — the default */ }

// noexcept(bool) lets you state it explicitly:
void a() noexcept(true);    // same as noexcept
void b() noexcept(false);   // same as the default (may throw)
If an exception does try to escape a noexcept function, the runtime doesn't search for a handler — it calls std::terminate immediately, ending the program. There's no stack unwinding to a catch; a surrounding try/catch cannot save you. So noexcept is a guarantee you must actually keep: mark a function noexcept only when you're sure no exception (including ones from things it calls) can propagate out of it.
3 The big payoff

Why noexcept moves are faster 🚀

Here's the reason noexcept earns its keep. When a std::vector grows, it allocates a bigger buffer and relocates the existing elements. It would prefer to move them (cheap), but it must preserve the strong guarantee: if relocation fails partway, the original must be intact. A throwing move could leave things half-moved and unrecoverable — so the vector plays it safe and copies instead. Unless the move is noexcept:

Move constructor is…What vector does on reallocation
noexceptmoves elements — fast 🚀
potentially-throwingcopies elements — safe but slow 🐢
So a noexcept move constructor isn't just tidy — it can make your type dramatically faster inside every standard container. The library checks this with the trait std::is_nothrow_move_constructible<T>, and the compiler-generated move operations are noexcept automatically when every member's move is. The practical rule: mark your move constructor and move assignment noexcept whenever they genuinely can't throw (which, for a well-behaved type, is almost always).
4 The noexcept operator

noexcept(expr) asks a question 🔎

Confusingly, noexcept is also an operator. Written with parentheses around an expression, noexcept(expr) is a compile-time bool: true if expr is guaranteed not to throw, false otherwise. The expression is not evaluated — only its throwing-ness is inspected:

void f() noexcept;
void g();

static_assert(  noexcept(f()) );   // true  — f is noexcept
static_assert( !noexcept(g()) );   // true  — g may throw
The two uses look alike but do different jobs: noexcept as a specifier (no parens) makes a promise; noexcept(expr) as an operator (with parens) queries a promise. The operator is how the library writes conditional specifications — e.g. a wrapper that is noexcept exactly when the thing it wraps is: void wrap() noexcept(noexcept(inner()));. You'll mostly read this in template code rather than write it, but recognizing the noexcept(noexcept(...)) pattern stops it from looking like a typo.
5 What to mark

Where noexcept belongs 🏷️

Reach for noexcept on operations that genuinely can't fail and that the library cares about:

  • Move constructor & move assignment — the big win (faster containers).
  • swap — should never throw; many algorithms rely on it.
  • Destructors — already noexcept by default; don't fight it.
  • Trivial accessors — simple getters that just return a member.

🚨 Don't over-promise — and never throw from a destructor

Two cautions. First, don't slap noexcept on functions that legitimately throw (anything doing I/O, allocation you don't control, parsing). If such a function throws, std::terminate fires and no handler can intervene — you've turned a recoverable error into a crash. Mark noexcept only when the guarantee is real. Second, destructors are implicitly noexcept: if a destructor throws, the program terminates. Never let an exception escape a destructor — catch and swallow (or log) inside it instead. When in doubt, leave a function unmarked; potentially-throwing is the safe default.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

💥
Marking a throwing function noexcept If a noexcept function throws, std::terminate fires — a surrounding try/catch can't help. Only promise noexcept when it's genuinely true.
🐢
Forgetting noexcept on a move A move constructor that could be noexcept but isn't marked makes std::vector fall back to copying on reallocation — silently slower. Mark moves noexcept when they can't throw.
🔀
Confusing the specifier and the operator void f() noexcept (no parens) makes a promise; noexcept(expr) (parens) queries one and yields a bool. The noexcept(noexcept(...)) pattern is the operator inside the specifier.

Practice exercises 💪

Easy Query the promise. Declare void f() noexcept; and void g();. Print noexcept(f()) and noexcept(g()) and confirm you get 1 and 0. (Remember the expressions aren't actually called.)
Medium Watch move vs copy. Make a struct with a move constructor that prints "move" and a copy constructor that prints "copy". Put a few in a std::vector, force a reallocation, and watch which prints. Then remove noexcept from the move and observe it flip to copying.
Stretch Feel the terminate. Write void boom() noexcept { throw std::runtime_error("x"); } and call it inside a try { boom(); } catch (...) {}. Confirm the catch never runs and the program aborts — proof that noexcept is enforced, not advisory.

🎓 What to remember

  • noexcept promises a function won't let an exception escape — it's part of the type, not just a comment.
  • Breaking the promise calls std::terminate immediately; no try/catch can intercept it.
  • The payoff: std::vector (and friends) move instead of copy on reallocation only when the move is noexcept — faster, with the strong guarantee preserved.
  • The operator noexcept(expr) is a compile-time bool (unevaluated) — used for conditional noexcept(noexcept(...)).
  • Mark moves, swap, and trivial accessors noexcept; destructors already are. Don't over-promise — leave throwing functions unmarked.

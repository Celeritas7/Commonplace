override & final

Virtual functions are powerful but easy to get subtly wrong — a one-letter typo can silently create a brand-new function instead of an override. Two C++11 keywords fix that: override makes the compiler verify your intent, and final lets you seal a method or a whole class against further overriding.

1 Why this matters

Intent the compiler can check 🧠

You've been writing override since lesson 7.3 — now let's see exactly what it (and its partner final) buys you. Overriding a virtual function depends on the derived method matching the base's signature precisely. Get one detail wrong — a typo, a missing const, a different parameter type — and C++ doesn't override; it quietly makes a new, unrelated function, and the base version keeps running. These two keywords turn fuzzy intent into something the compiler enforces: override says "this must override," and final says "no one may override this further."

Small keywords, large payoff in safety and clarity.

2 override

"Make sure this really overrides" ✅

Put override after a derived method's signature. It compiles only if the method genuinely overrides a base virtual — otherwise you get an immediate error:

struct Animal { virtual std::string speak() { return "..."; } };

struct Dog : Animal {
    std::string speak() override { return "Woof"; }   // ✅ verified override
    // std::string speek() override {...}   // ❌ won't compile — no base "speek"
};
Without override, that speek typo would compile happily as a brand-new method, and Animal::speak() would keep running through a base pointer — a bug that can cost hours. With override, the compiler checks the base for a matching virtual and rejects the code if it doesn't find one. It costs nothing and catches a whole family of silent mistakes, so the guidance is simple: put override on every override.
3 final on a method

"No one may override this further" 🔒

final marks a virtual method as the last override — any class deriving further can't override it again:

struct Animal { virtual std::string speak() { return "..."; } };
struct Dog : Animal {
    std::string speak() override final { return "Woof"; }  // sealed here
};
struct Puppy : Dog {
    std::string speak() override { return "Yip"; }   // ❌ error: Dog::speak is final
};
Here Dog overrides speak() and then locks it: Puppy may inherit it but cannot change it. You can write override final together (verify and seal) or just final. Use it when a behaviour must not be altered downstream — it documents that intent and lets the compiler enforce it (and can even help it optimize the call).
4 final on a class

"You can't inherit from me" 🚫

Put final after a class name and nothing can derive from it at all:

struct Widget final {   // no class may inherit from Widget
    void draw() {}
};

struct Button : Widget { };   // ❌ error: cannot derive from 'final' class Widget
Sealing a whole class says "this type is complete — it isn't designed to be a base." That's useful for leaf types you don't want extended, and it gives the compiler a guarantee it can use to devirtualize calls (a small speed win). Reach for class-level final deliberately, though: it forecloses future extension, so use it when you're sure the type shouldn't be a base.
5 When to use which

Verify vs seal 📜

KeywordWhereMeaning
overrideafter a derived methodcompiler verifies it overrides a base virtual
finalafter a virtual methodno further class may override it
finalafter a class nameno class may inherit from it
Think of it as verify vs seal. Use override liberally — on essentially every override, for safety. Use final sparingly and on purpose, where you genuinely want to stop further overriding or inheritance. Both are contextual keywords: they only have special meaning in these positions, so existing variables named final still work elsewhere.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

✍️
Skipping override and getting a silent non-override Without it, a signature mismatch creates a new function and the base version still runs — no error. Add override so the compiler catches it instantly.
🔒
Thinking override seals the method override only verifies; it doesn't stop further overriding. To seal, use final (you can combine them: override final).
🚫
Sealing too eagerly with class final Marking a class final forecloses all future inheritance. It's great for true leaf types, but don't reach for it by default — only when you're sure the type shouldn't be a base.

Practice exercises 💪

Easy Catch a typo. Give Animal a virtual void speak() and write Dog with void speek() override {}. Watch it fail to compile, then fix the spelling and confirm it builds.
Medium Seal a method. Override speak() in Dog with override final, then try to override it again in a Puppy : Dog. Read the error that final produces.
Stretch Seal a class. Mark a small class final and attempt to derive from it. Then remove final and confirm the derivation now compiles — proving what the keyword was doing.

🎓 What to remember

  • override tells the compiler "this overrides a base virtual" — and it errors if it doesn't (catches typos / signature mismatches).
  • Without override, a mismatch silently creates a new function and the base version keeps running.
  • final on a method stops any further class from overriding it.
  • final on a class stops any class from inheriting from it.
  • Verify vs seal: use override everywhere; use final deliberately. Both are C++11.

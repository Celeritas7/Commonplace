dynamic_cast & RTTI

Polymorphism lets you hold any animal as an Animal*. But sometimes you need to ask: is this actually a Dog? — to reach a Dog-only member. dynamic_cast is the safe, checked way to find out: it gives you the derived pointer if the object really is that type, or nullptr if it isn't.

1 Why this matters

"Is this really a Dog?" 🧠

Most of the time you call virtual functions through a base pointer and let dynamic dispatch (7.3) do the right thing — that's the clean way. But occasionally you genuinely need to recover the concrete type: you have an Animal* and want to call something only Dog has. Guessing is dangerous — if the object isn't actually a Dog, treating it as one corrupts memory. dynamic_cast is the safe answer: it performs a checked downcast, asking the object at runtime "are you really a Dog?" and handing back either a valid Dog* or nullptr. The machinery behind that question is RTTI — Run-Time Type Information.

The safe way to go from a base pointer back down to a derived type.

2 dynamic_cast on pointers

Derived pointer, or nullptr 🎯

Give dynamic_cast the target pointer type and the base pointer. If the object really is that type, you get a usable pointer; if not, nullptr:

struct Animal { virtual ~Animal() = default; };   // must be polymorphic
struct Dog : Animal { void fetch() {} };

Animal* a = new Dog();
Dog* d = dynamic_cast<Dog*>(a);   // a really is a Dog → valid Dog*
if (d) d->fetch();              // safe to use Dog-only members
The form is dynamic_cast<Target*>(pointer). Here a points at a real Dog, so the cast succeeds and d is a valid Dog* you can use. The base class must be polymorphic — it needs at least one virtual function (a virtual destructor counts, and you want one anyway from 7.5) — because that's what lets the runtime store and check the real type.
3 Why nullptr matters

Always check the result ✅

The whole point is the failure case. If the object isn't the type you asked for, you get nullptr instead of a wrong pointer — so you check before using it:

Animal* a = new Cat();          // actually a Cat
Dog* d = dynamic_cast<Dog*>(a);

if (d) {
    d->fetch();                  // not reached
} else {
    // d is nullptr — a wasn't a Dog. Handle it safely. ✅
}
Because a Cat is not a Dog, the cast returns nullptr — no crash, no corruption, just a clear "no." This is what makes dynamic_cast safe: it can fail, and it tells you so. Compare static_cast<Dog*>(a), which does no runtime check — it just trusts you, and if you're wrong, using the result is undefined behaviour. Reach for dynamic_cast whenever a downcast might not hold, and always test the result for nullptr.
4 References & typeid

The rest of RTTI 🪪

On references there's no null to return, so a failed dynamic_cast throws std::bad_cast. And typeid reports an object's runtime type:

try {
    Dog& d = dynamic_cast<Dog&>(someAnimal);   // throws std::bad_cast if not a Dog
} catch (const std::bad_cast&) { /* handle */ }

// typeid: ask for the runtime type
if (typeid(*a) == typeid(Dog)) { /* a points at exactly a Dog */ }
These three — dynamic_cast, typeid, and the std::type_info it returns — make up RTTI. With pointers you check for nullptr; with references you catch std::bad_cast (a reference can't be "null"). typeid(*a) gives the most-derived runtime type, again only meaningful when the base is polymorphic. You'll mostly use the pointer form; the others are good to recognize.
5 Use it sparingly

Often a virtual function is better 📜

CastRuntime check?On failure
dynamic_cast<D*>yes (needs polymorphic base)nullptr

🚨 A forest of dynamic_casts is a design smell

If you find yourself writing if (dynamic_cast<Dog*>(a)) … else if (dynamic_cast<Cat*>(a)) …, that's usually a sign the behaviour belongs in a virtual function instead: add a virtual method to the base and let each derived type override it (7.3). The polymorphic call replaces the whole type-checking ladder, and it stays correct when you add new types. Keep dynamic_cast for the genuine cases — interfacing with code you can't change, or recovering a specific type at an API boundary.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

❓
Not checking for nullptr A pointer dynamic_cast can return nullptr. Using it without checking dereferences null → crash. Always test the result first.
🧱
Casting from a non-polymorphic base dynamic_cast needs the base to have at least one virtual function. No virtuals → it won't compile. (You'll usually have a virtual destructor anyway.)
🪜
Building a ladder of dynamic_casts Chaining type checks to pick behaviour is a smell — a virtual function in the base usually expresses it better and keeps working as new types are added.

Practice exercises 💪

Easy Success and failure. Make a polymorphic Animal with Dog and Cat derived. Point an Animal* at a Dog; dynamic_cast to Dog* (succeeds) and to Cat* (nullptr). Print which.
Medium Guard a Dog-only call. Give Dog a fetch() the base lacks. Write a function taking Animal* that calls fetch() only when the dynamic_cast<Dog*> succeeds.
Stretch Replace it with a virtual. Take a two-branch dynamic_cast if/else and refactor it into a single virtual function overridden by each derived class. Notice the call site gets shorter and needs no changes when you add a third type.

🎓 What to remember

  • dynamic_cast<D*>(base) is a checked downcast: a valid D* if the object really is a D, else nullptr.
  • Always check the pointer result for nullptr before using it.
  • On references, a failed cast throws std::bad_cast (no null to return).
  • Needs a polymorphic base (a virtual function); static_cast does no check and is unsafe if wrong.
  • Prefer a virtual function over chains of dynamic_cast — it's cleaner and extensible.

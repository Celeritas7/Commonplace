The range-for Loop

When you just want to visit every element of a container, the range-for (C++11) does it with no index, no size(), and no off-by-one. The one thing to master: whether your loop variable is a copy or a reference.

1 Why this matters

Visit everything, skip the bookkeeping 🧠

Last lesson you walked a container with an index: for (int i = 0; i < v.size(); i++), then used v[i] inside. It works, but it's fiddly — you manage a counter, get the < vs <= right, and write v[i] everywhere. The range-for (C++11) says what you actually mean: "for each element in this container, do this."

When all you need is to touch every element in order — printing, summing, transforming — the range-for is shorter, clearer, and impossible to get an off-by-one wrong, because there's no index to misjudge. It's the loop you'll write most often in modern C++. There's just one concept that makes or breaks it, and it's the whole reason this lesson exists: copy versus reference.

2 The syntax

for each element 🔁

std::vector<int> scores = {90, 75, 60};

for (int s : scores) {
    std::cout << s << ' ';
}                          // 90 75 60

Read for (int s : scores) as "for each int s in scores." Each time around, s takes the value of the next element. No counter, no [i], no condition to write — the loop knows when to stop because it knows where the container ends.

In practice you'll usually write auto instead of spelling out the element type, so the compiler figures it out: for (auto s : scores). That's the idiomatic form. It reads cleanly and adapts if the container's element type ever changes. The range-for works on std::vector, std::string, plain arrays, std::map, initializer lists like {1, 2, 3} — anything that knows its own beginning and end.
3 Copy vs reference

The one thing that matters ⭐

By default, the loop variable is a copy of each element. That means changing it inside the loop does not change the container:

for (auto s : scores) {
    s += 10;          // changes the COPY, not the vector
}
// scores is unchanged: still {90, 75, 60}

To actually modify the elements, ask for a reference with & — now the loop variable is the element, not a copy of it:

for (auto& s : scores) {
    s += 10;          // changes the REAL element
}
// scores is now {100, 85, 70}
FormWhat s isModifies container?
for (auto s : v)A copy of each elementNo
for (auto& s : v)A reference to the elementYes
for (const auto& s : v)A read-only referenceNo (and no copy)
The third form, const auto&, is the one seasoned C++ programmers reach for when they only need to read. It avoids copying each element (which matters a lot for big types like std::string or large structs) and guarantees you won't accidentally modify them. 4 Which form to use

A simple rule of thumb 🎯

You want to…Use
Modify each element in placefor (auto& x : v)
Read each element, no copying (especially big types)for (const auto& x : v)
Read small, cheap values (int, char) — copy is finefor (auto x : v)
A safe default that's hard to get wrong: if you're only reading, use const auto&. It's never wrong (no copy, no accidental change), and for small types the compiler optimizes away any overhead. Reach for plain auto& when you specifically need to write to the elements, and plain auto only when you actually want a throwaway copy to mess with.
5 When NOT to use range-for

The limits 🚧

The range-for is perfect for "visit every element," but it deliberately hides the index — so it's the wrong tool when you actually need that machinery:

  • You need the position. "Print element 3" or "compare v[i] with v[i+1]" needs an index — use a classic for.
  • You're not visiting all of them. Every other element, the first half, backwards — the indexed for gives you that control.
  • You change the container's size while looping. Adding or removing elements during a range-for can break it. Don't resize mid-loop.

🚨 The silent copy that does nothing

The most common range-for bug: you write for (auto x : v) x = something; meaning to change the container, but x is a copy, so the container is untouched and your loop does nothing visible. If you meant to modify, you needed auto&. The compiler won't warn you — the code is perfectly legal, just pointless.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📋
Expecting auto x to modify the container for (auto x : v) x = ...; changes only the copy — the container is untouched. To modify in place, use auto&.
🐘
Copying big elements for no reason for (auto s : bigStrings) copies every string each iteration. If you're only reading, use const auto& s — no copies, and it can't be accidentally modified.
🚧
Resizing the container mid-loop Adding or removing elements during a range-for can invalidate it and crash. If you need to change the size while iterating, use a classic indexed loop and be careful.

Practice exercises 💪

Easy Sum a vector. Given std::vector<int> nums = {4, 8, 15, 16, 23, 42}, use a range-for with const auto& to add them all up and print the total.
Medium Double in place. Take a std::vector<int> and double every element in the vector itself using a range-for. Print it before and after to prove it changed — then switch your loop to auto (no &) and confirm the change vanishes.
Stretch Count vowels in a string. Loop over a std::string with a range-for (a string is a container of char). For each character, check if it's a vowel and count it. Use const auto& since you're only reading.

🎓 What to remember

  • for (auto x : container) visits every element in order — no index, no off-by-one (C++11).
  • Default is a copy: changing x doesn't change the container.
  • Use auto& to modify elements in place; const auto& to read without copying.
  • Safe default for read-only loops: const auto& — never wrong, never copies.
  • Use a classic for when you need the index, skip elements, or change the container's size.

🧬 PART VII · INHERITANCE & POLYMORPHISM

Virtual Functions

Inheritance lets a Dog be an Animal. Virtual functions let you call the right behaviour through a base pointer: Animal* a = &dog; a->speak(); should say "Woof". This runtime choice — dynamic dispatch — is the heart of polymorphism in C++.

1 Why this matters

One pointer, many behaviours 🧠

Here's the dream: write code that works with an Animal, and have it automatically do the right thing whether that animal is really a Dog, a Cat, or a Cow. You loop over a list of Animal* and call speak() on each — and each makes its own sound. This is polymorphism: many types, treated uniformly through a common base, each behaving as itself. virtual functions are the mechanism that makes it happen, and they're what most of object-oriented C++ is built on.

This is the payoff of Part VII. Inheritance set it up; virtual delivers the magic.

2 The problem without virtual

A base pointer calls the base 🤔

Suppose Dog overrides Animal's speak(), but speak() is an ordinary (non-virtual) method. Call it through an Animal pointer and you get the base version — not the dog's:

struct Animal { std::string speak() { return "..."; } };
struct Dog : Animal { std::string speak() { return "Woof"; } };

Dog d;
Animal* a = &d;
a->speak();   // "..."  — the BASE version, not "Woof"! 🤔
The compiler picks the function based on the pointer's declared type (Animal*), not what it actually points at. That's static dispatch — decided at compile time. It's almost never what you want when you're working through a base pointer: you wanted the dog to bark, and you got a generic "..." instead.
3 virtual — let the object decide

Dynamic dispatch ✨

Add the keyword virtual to the base method, and everything changes. Now the call looks at the actual object at runtime and runs its version:

struct Animal { virtual std::string speak() { return "..."; } };
struct Dog : Animal { std::string speak() override { return "Woof"; } };

Dog d;
Animal* a = &d;
a->speak();   // "Woof"  — the DOG's version! ✨
With virtual, the choice of which speak() to run is made at runtime, based on the real type of the object — that's dynamic dispatch. A single line, a->speak(), now does different things depending on what a points at. That's polymorphism in action, and it all hinges on that one keyword in the base class.
4 override — let the compiler check

Catch silent mistakes ✅

You may have noticed override on the derived method. It's optional but strongly recommended: it tells the compiler "this is meant to override a base virtual," so it can verify that it actually does:

struct Dog : Animal {
    std::string speak() override { return "Woof"; }   // ✅ checked
    // std::string speek() override {...}   // ❌ won't compile — typo, no such base virtual
};
Without override, a tiny mismatch — a typo, a wrong parameter type, a missing const — quietly creates a new function instead of an override, and the base version keeps running. The bug is maddening to find. Adding override turns it into an immediate compile error. Use it on every override; it's free insurance (C++11).
5 Why it's powerful

Treat many types as one 🔁

The real power shows up when you collect different derived objects and handle them through a base pointer. One loop, and each object behaves as itself:

std::vector<Animal*> zoo = { &dog, &cat, &cow };
for (Animal* a : zoo)
    std::cout << a->speak();   // "Woof" "Meow" "Moo" — each its own!

🚨 A polymorphic base needs a virtual destructor

If you ever delete a derived object through a base pointer (common with heap allocation), the base class must have a virtual destructor — otherwise only the base part is destroyed and the derived part leaks (undefined behaviour). The rule of thumb: any class with virtual functions should have a virtual destructor (virtual ~Animal() = default;). We'll cover the details in the next lesson; for now, just know it goes hand-in-hand with virtual methods.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🚫
Forgetting virtual Without it, a call through a base pointer/reference runs the base version, not the override (static dispatch). Mark the base method virtual to get polymorphism.
✍️
A silent non-override (signature mismatch) If your "override" has a slightly different signature, it's a brand-new function and the base version still runs. Add override so the compiler catches the mismatch.
🧹
No virtual destructor on a polymorphic base Deleting a derived object through a base pointer without a virtual destructor destroys only the base part (undefined behaviour). A class with virtual functions should have virtual ~Base() = default;.

Practice exercises 💪

Easy Make it bark. Give Animal a virtual std::string speak() returning "..." and Dog an override returning "Woof". Call speak() through an Animal& that refers to a Dog — confirm you get "Woof".
Medium Feel the difference. Remove virtual and run again — now the same call returns "..." (the base version). Add it back and watch "Woof" return. That one keyword is the whole effect.
Stretch A talking zoo. Make Cat and Cow too, put several animals as Animal* in a std::vector, and loop calling speak() — each prints its own sound. Runtime polymorphism over a collection.

🎓 What to remember

  • A virtual function in the base can be overridden by a derived class.
  • Called through a base pointer or reference, the derived version runs — this is dynamic dispatch (runtime polymorphism).
  • Without virtual, the call uses the pointer's static type and runs the base version.
  • Mark overrides with override (C++11) so the compiler verifies they really override.
  • A polymorphic base needs a virtual destructor — coming up next.

Why Functions Exist

A function is a named block of code you can run whenever you want, as many times as you want. Before you learn to write one, it's worth knowing why they exist at all — they fight repetition, give your ideas names, and let you test one small piece at a time.

1 Why this matters

Copy-paste is a quiet disaster 🧠

You already know main() — every program you've written lives inside it. A function is just another named block like main, except you name it and you decide when it runs. You've also been calling functions all along: printf and scanf are functions someone else wrote. This lesson is about why you'd write your own.

Imagine you need to print a tidy, boxed heading three times in a program. Without functions, you copy the same five lines three times. It works — until you want to change the box style. Now you're hunting for all three copies and editing each one, hoping you didn't miss any. Multiply that across a real program and small changes become an afternoon of error-prone find-and-replace. Functions are the cure: write the work once, give it a name, and call that name everywhere.

2 The mental model

A named little machine 📦

Picture a kitchen appliance. A blender has a slot for ingredients (the input), it does one well-defined job, and it hands you back a smoothie (the output). You don't rebuild the motor each time — you drop things in and press the button. A function is exactly that: a reusable machine with a name.

// the machine, defined once
int square(int n) {   // input: a number n
    return n * n;       // output: n times itself
}

// press the button as often as you like
printf("%d\n", square(3));   // 9
printf("%d\n", square(10));  // 100
Three words name the whole idea: define the machine once (give it a name and a body), then call it by that name whenever you need the job done. The details of how to write the inputs and the return are the whole of lesson 4.2 — here, just hold the picture: a function is a named box of work you can press a button on.
3 The three reasons

DRY, names, and testing 🎯

Functions earn their keep three ways. Keep these three words in your head — they're why experienced programmers reach for a function almost by reflex:

ReasonWhat it meansThe payoff
DRYDon't Repeat Yourself — write the logic onceFix or improve it in one place
NamingA good name says what a block doesCode reads like sentences
TestingEach function is a small, isolated pieceCheck one part without the rest
DRY is the famous one: every copy of a piece of logic is a place a bug can hide and a place you'll forget to update. Naming is underrated — total = price_with_tax(p); tells a reader the intent instantly, where ten raw lines of arithmetic don't. Testing falls out for free: a function with clear inputs and one output is something you can poke at on its own, long before the whole program is finished.
4 A worked example

Three copies become one ♻️

Here's the repetition trap in the wild. Say we convert Celsius to Fahrenheit in three spots. Copy-pasted, it looks like this:

// the SAME formula, written three times
printf("%.1f\n", body * 9.0 / 5.0 + 32);
printf("%.1f\n", oven * 9.0 / 5.0 + 32);
printf("%.1f\n", fridge * 9.0 / 5.0 + 32);

If you later learn the formula was typed wrong, you must fix it in three places. Pull it into a function and the duplication vanishes:

double to_fahrenheit(double c) {
    return c * 9.0 / 5.0 + 32;   // the rule lives here, once
}

printf("%.1f\n", to_fahrenheit(body));
printf("%.1f\n", to_fahrenheit(oven));
printf("%.1f\n", to_fahrenheit(fridge));
Look what changed. The formula now appears exactly once. Each call site reads as plain English — "to Fahrenheit of body." And if the rule ever changes, you edit a single line and all three callers get it automatically. That's all three reasons at once: DRY, a good name, and a piece you could test by itself with to_fahrenheit(100) and checking you get 212.
5 Naming well

A name is a tiny promise 🏷️

Since naming is half the value of a function, it's worth a moment. A function does something, so its name usually starts with a verb, and it should describe the result, not the mechanics:

  • Actions read as verbs: print_report, save_score, draw_box. The reader knows what happens without opening the body.
  • Yes/no answers start with is_ or has_: is_even, has_won. The name tells you it returns true or false.
  • Avoid vague mush: doStuff, process, thing2 promise nothing. If you can't name it clearly, the function might be doing too many jobs at once.
A function name is a promise to whoever reads the call later — often future-you. if (is_valid(card)) keeps its promise; if (check(card)) makes you go read the body to find out what "check" decided. Good names are the difference between code you can skim and code you have to decode.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🧱
The do-everything function If a function reads input, calculates, prints, and saves a file all at once, it's really four functions in a trenchcoat. Each function should do one clear job — that's what makes it nameable, reusable, and testable.
🔤
Vague names Names like doIt, process, or temp tell a reader nothing. A function name should describe its result so the call site reads clearly — price_with_tax(p), not calc(p).
🙅
Refusing to make functions ("it's just a few lines") The third time you paste the same logic, stop and extract it. Duplication doesn't feel expensive today — it costs you the day a bug lives in three copies and you fix only two.

Practice exercises 💪

Easy Spot the repetition. Find a program you've written (or one of the earlier lessons' examples) where the same few lines appear more than once. Write down a good verb-based name for the function you'd extract from it.
Medium Name five. For each job, write the function name you'd choose: (1) prints a row of dashes, (2) returns true if a number is positive, (3) converts kilometres to miles, (4) finds the largest of three numbers, (5) clears the screen. Compare your verbs and is_ prefixes against the naming rules above.
Stretch Plan a refactor. Take the temperature example and sketch (on paper) a tiny program that prints body, oven, and fridge temperatures in Fahrenheit using a single to_fahrenheit function. Don't worry about perfect syntax yet — lesson 4.2 fills that in — just lay out the one definition and the three calls.

🎓 What to remember

  • A function is a named, reusable block of work — like main, but you name it and you decide when it runs.
  • You define it once and call it by name as many times as you need.
  • Three reasons: DRY (write logic once), naming (code reads clearly), testing (check one small piece in isolation).
  • Name it for what it does — verbs for actions, is_/has_ for yes/no answers, never vague mush.
  • One function, one job. If it's hard to name, it's probably doing too much.

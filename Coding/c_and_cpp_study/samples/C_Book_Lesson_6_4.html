🎯 PART VI · POINTERS

Pointer Arithmetic

Add 1 to an int pointer and it doesn't move one byte — it moves to the next int. Pointer arithmetic counts in elements, automatically scaled by the type's size. This is the secret machinery behind arrays, and the reason p[i] and *(p + i) are the same thing.

1 Why this matters

Walking through memory 🧠

In 6.3 you saw a pointer as an arrow to one box. But memory is a row of boxes, and pointers can move along it. Add to a pointer and the arrow slides to a neighbouring box; subtract and it slides back. This is how you scan an array, process a buffer, or walk a string — and it's why arrays and pointers are so deeply linked in C.

The one rule that makes it all sensible: arithmetic on a pointer counts in elements, not bytes. The compiler knows the type, so it scales every step by the right size for you. Let's see exactly what that means.

2 p + 1 means "next element"

Scaled by the type 📏

If p is an int * pointing at address 100, then p + 1 is not 101 — it's 104, because an int is 4 bytes. The compiler multiplies your + 1 by sizeof(int) so the pointer lands on the next int:

int *p;   // suppose p == 100
p + 1     // → 104  (one int forward, +4 bytes)
p + 2     // → 108  (two ints forward, +8 bytes)

char *c;  // suppose c == 100
c + 1     // → 101  (char is 1 byte, so +1)
The same + 1 moves a char * by 1 byte but an int * by 4 and a double * by 8. You never compute the byte offset yourself — you think in elements ("two forward"), and the type's sizeof handles the rest. That's why the pointer's type matters so much.
3 Arrays are pointers in disguise

arr + i points at element i 🔗

Remember array decay from 5.2: an array's name becomes a pointer to its first element. So arr is like &arr[0], and adding moves along the array:

int arr[] = {10, 20, 30, 40};

arr        // → &arr[0]  (points at 10)
arr + 2    // → &arr[2]  (points at 30)
*(arr + 2) // → 30       (follow it to the value)
Because the boxes of an array sit side by side in memory, stepping a pointer through them visits each element in turn. arr + i is the address of element i, and *(arr + i) is its value. This is the bridge between the two ideas: an array is a block of elements, and a pointer can walk it.
4 The big reveal

p[i] is just *(p + i) ✨

Here's the punchline: array indexing is defined as pointer arithmetic. The square brackets are pure shorthand — p[i] means exactly *(p + i):

int arr[] = {10, 20, 30};
int *p = arr;

p[1];        // 20
*(p + 1);    // 20  — identical
++p;         // p now points at arr[1]; *p == 20
Once you see p[i] == *(p + i), indexing and pointer arithmetic stop being two topics — they're one. ++p walks the pointer one element forward (handy in loops), and you can use either p[i] or *(p + i), whichever reads better. (Fun fact: since addition commutes, arr[i] even equals i[arr] — true, but please never write that.)
5 Differences & walking

From begin to end 🚶

Subtracting two pointers into the same array gives the number of elements between them (again, not bytes). That lets you walk an array with a pointer, stopping at one-past-the-end:

int arr[] = {10, 20, 30, 40, 50};

for (int *p = arr; p < arr + 5; ++p)
    printf("%d ", *p);   // 10 20 30 40 50

(arr + 5) - arr            // → 5  (element count)

🚨 Walking off the end

You may form the address one past the last element (arr + 5 above) and compare against it — that's the standard loop bound. But you must never dereference it: *(arr + 5) reads memory the array doesn't own (undefined behaviour). The same goes for stepping before arr. Keep the pointer inside [arr, arr + n], and only read while it's strictly below arr + n.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📏
Thinking p + 1 adds one byte It adds sizeof(*p). For an int * that's typically 4 bytes; for a char * it's 1. You count in elements — the compiler scales for you.
🧱
Walking off the end Forming arr + n (one past the last) is fine for a loop bound, but dereferencing it — or going before arr — is undefined behaviour. Read only while p < arr + n.
🎯
Confusing p + i with *(p + i) p + i is an address (a pointer to element i); *(p + i) — i.e. p[i] — is the value stored there. One locates, the other reads.

Practice exercises 💪

Easy Three ways in. With int arr[5]; int *p = arr;, print *p, *(p + 1), and p[2]. Confirm they equal arr[0], arr[1], arr[2].
Medium Walk it with a pointer. Loop for (int *p = arr; p < arr + 5; ++p) printf("%d ", *p);. Notice the stop condition p < arr + 5 and that you never dereference arr + 5.
Stretch Prove the scaling. Print (void *)arr, (void *)(arr + 1), and (void *)(arr + 2). The addresses rise by sizeof(int) each step. Then confirm &arr[i] equals arr + i for each i.

🎓 What to remember

  • Pointer arithmetic counts elements, not bytes: p + 1 advances by sizeof(*p).
  • An array name decays to a pointer to its first element; arr + i points at element i.
  • p[i] is exactly *(p + i) — indexing is pointer arithmetic.
  • ++p walks forward one element; q - p gives the number of elements between them.
  • You may form one-past-the-end (arr + n) for loop bounds, but never dereference it.

Concepts

Templates accept any type — but if the type can't do what the body needs, you get a wall of cryptic errors deep inside the library. Concepts (C++20) fix that: you state a template's requirements up front, so an unsuitable type is rejected with one clear message right at the call, and your template documents what it expects. A brief tour to close Part IX.

1 Why this matters

The cryptic-error problem 🧠

Back in 9.1 you saw that a template is "duck typed": the body just has to be valid for whatever T you use, and if it isn't, you get a compile error. The trouble is where and how that error appears — deep inside the template (or worse, inside the standard library), as dozens of lines of unreadable diagnostics, far from your actual mistake. Concepts (added in C++20) let you attach a named requirement to a template parameter — "T must be sortable," "T must be an integer" — so the compiler checks it at the call site and reports a single, plain-English failure. The result: clearer errors, and templates that say what they expect. This is a brief look to round out the templates Part.

Same templates you already know — now with a "must satisfy" label on the type.

2 Constraining a template

State what T must be 🛂

The standard library ships ready-made concepts (in <concepts>). Use one to constrain a parameter — here, "T must be an integer type":

#include <concepts>

template<std::integral T>       // T must satisfy the 'integral' concept
T half(T x) { return x / 2; }

half(10);     // ✅ int is integral → 5
half(2.5);    // ❌ double is NOT integral → clear error at this line
template<std::integral T> reads "for any type T that is an integral type." Call half with an int or long and it compiles; call it with a double and the compiler stops here with a message like "constraints not satisfied: double does not satisfy integral" — instead of erroring three layers down inside half's body. Common standard concepts include std::integral, std::floating_point, std::same_as<U>, and std::sortable. (An equivalent longer form is template<typename T> requires std::integral<T>.)
3 Why it's better

One clear error, not fifty 🎯

The payoff is mostly about diagnostics and intent — the constrained and unconstrained versions behave the same for valid types:

Without conceptsWith concepts
error appears deep in the template bodyerror at the call: "T isn't integral"
pages of template-instantiation noiseone short, readable message
requirements hidden in the coderequirements stated in the signature
A constrained template is also self-documenting: template<std::integral T> tells a reader exactly what types are allowed, without digging into the body. And concepts participate in overload resolution — you can have one version for integers and another for floating-point, and the compiler picks the one whose constraints fit. For everyday code the headline win is simply: when you misuse a template, the error tells you what you did wrong in plain language.
4 Writing your own

Define a named requirement 📐

You can define your own concept with the concept keyword — a compile-time boolean about a type. A requires block lists operations the type must support:

template<typename T>
concept Addable = requires(T a, T b) {
    a + b;                  // T must support a + b
};

template<Addable T>          // use it like any concept
T sum(T a, T b) { return a + b; }
Addable is true for any type where a + b compiles. Now template<Addable T> only accepts such types, and rejects the rest with a clear "doesn't satisfy Addable." The requires(...) block is just a list of expressions that must be valid — you're describing the interface a type needs, by example. This is how the standard concepts are built, and how you'd express domain requirements ("Drawable," "Serializable") in your own code.
5 Scope & cautions

A polish, not a revolution 🎓

Concepts make templates friendlier, but keep their place in perspective — especially as a newcomer:

🚨 Concepts check requirements, not correctness — and they're C++20

A concept verifies that a type supports certain operations (it has +, it's an integer), not that your logic is right — a type can satisfy Addable and still give a result you didn't intend. They're also a C++20 feature: you need a recent compiler and -std=c++20, and you'll see plenty of existing code that constrains templates the old way (SFINAE / static_assert) instead. So treat concepts as a quality-of-life improvement — clearer errors and self-documenting signatures — rather than something you must use everywhere. As a beginner, the key skills are recognising a constrained template (template<std::integral T>) and reaching for a standard concept when you want a friendlier error; writing your own concepts comes later.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🎯
Thinking a concept checks correctness A concept verifies a type supports the needed operations, not that your logic is right. A type can satisfy the constraint and still produce a wrong answer.
🗓️
Forgetting it's C++20 Concepts need a recent compiler and -std=c++20. Older code constrains templates with SFINAE or static_assert instead — you'll meet both.
🧱
Over-engineering custom concepts early As a beginner, lean on the standard concepts (std::integral, etc.) for friendlier errors. Writing elaborate custom concepts is an advanced skill for later.

Practice exercises 💪

Easy Constrain it. Write template<std::integral T> T triple(T x) { return x * 3; } (with -std=c++20). Call it with an int (works) and a double (read the clear "not integral" error).
Medium Two constrained overloads. Write a describe with one template<std::integral T> version returning "whole number" and one template<std::floating_point T> returning "decimal". Call with an int and a double and confirm each picks the right one.
Stretch Your own concept. Define template<typename T> concept Printable = requires(T x) { std::cout << x; }; and a template<Printable T> void show(T x). Try it with a type that has operator<< and one that doesn't, and compare the error messages with and without the concept.

🎓 What to remember

  • Concepts (C++20) attach a named requirement to a template parameter — template<std::integral T> — so unsuitable types are rejected up front.
  • The win is diagnostics & intent: one clear error at the call instead of cryptic noise deep in the template, and a self-documenting signature.
  • Use standard concepts (std::integral, std::floating_point, std::same_as, …) or write your own with concept name = requires(...) { ... };.
  • They check requirements, not correctness, and need -std=c++20 — a quality-of-life polish, not a must-use.
  • As a beginner: recognise a constrained template and reach for a standard concept for friendlier errors.

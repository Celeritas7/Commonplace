const and Immutability

One little keyword that makes your code safer: const promises a value won't change, and the compiler holds you to it. Master const variables, const references for fast safe parameters, and the famous pointer-const puzzle.

1 Why this matters

A promise the compiler enforces 💭

const marks a value as read-only. Once you write const double PI = 3.14159;, any line that tries to change PI is a compile error — not a runtime surprise, an error the compiler catches before your program ever runs. It's a promise you make ("I won't modify this") that the compiler turns into a guarantee.

Why is that worth a whole lesson? Because immutability is one of the cheapest ways to prevent bugs. A value that can't change can't be changed by accident. const also documents intent — a reader sees const and instantly knows "this stays put." And it unlocks performance: passing big objects by const reference avoids copying them while guaranteeing the function won't mess them up. C++ programmers call the habit of using const wherever possible const correctness, and it's a hallmark of solid code.

2 const variables

The basics 🔒

A const variable must be given a value when you declare it, and can never be reassigned afterward:

CodeResult
const int maxLives = 3;✅ Fine — initialized once
maxLives = 5;❌ Error — can't assign to const
const int x;❌ Error — must initialize
int y = maxLives + 1;✅ Fine — reading is always allowed
Reading a const is always fine — the restriction is only on changing it. You can copy it, add to it, pass it around, print it. You just can't assign a new value back into the const variable itself.

You'll also meet constexpr (C++11), a stronger cousin that means "const and computable at compile time." constexpr int size = 10; can be used where the compiler needs a constant, like an array length. For now, reach for const by default; use constexpr when you specifically need a compile-time constant.

3 const references

Fast and safe parameters 🚀

Here's where const earns its keep day to day. When a function takes a big object — a std::string, a std::vector — passing it by value copies the whole thing. Passing by plain reference avoids the copy but lets the function modify your original. const reference gives you both wins: no copy, no modification.

ParameterCopy?Can modify caller's object?
std::string sYes (slow)No (it's a copy)
std::string& sNoYes — risky if unintended
const std::string& sNoNo — compiler enforced ✅
The rule of thumb: for any parameter you only need to read, and that isn't a tiny type like int or double, take it by const&. void greet(const std::string& name) is the idiomatic way to pass a string you won't change. Inside, name behaves like the original but is read-only — try to modify it and you get a compile error.
4 Pointers and const

The two-const puzzle 🧩

Pointers have two things that could be constant: the value they point at, and the pointer itself. const can lock either, or both. The trick to reading them is right to left:

DeclarationRead right-to-leftWhat's locked
int* ppointer to intNothing
const int* ppointer to const intThe value (*p read-only)
int* const pconst pointer to intThe pointer (can't repoint)
const int* const pconst pointer to const intBoth

The position of const relative to the * is what matters. const before the * protects the pointed-to value; const after the * protects the pointer. The Pointer Decoder below lets you toggle each one and watch what becomes legal.

A memory hook: read the declaration backwards, starting from the variable name. int* const p → "p is a const pointer to int" → the pointer is frozen. const int* p → "p is a pointer to int that's const" → the int is frozen. The thing nearest the word const is the thing that can't change.
5 Where const trips people up

The gotchas ⚖️

🚨 const objects can only call const methods

If you have a const std::string& s, you can call s.size() or s.substr(...) (they don't modify) but not anything that would change it. A class method that promises not to modify the object is marked const after its parameter list — int size() const; — and only those can be called on a const object. You'll write your own in the classes part of the book.

🚨 const-ness is "shallow" through a pointer

int* const p freezes the pointer, but *p = 99; still works — the value isn't const. People assume one const locks everything. Decide which you actually want to protect: the pointer, the pointee, or both, and place const accordingly.

The throughline: const is precise about what it freezes. A const variable freezes that variable; a const& freezes the function's view of the argument; in a pointer, the side the const sits on decides whether the pointer or the value is locked. Be deliberate, and let the compiler enforce your intent.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📋
Passing big objects by value out of habit void print(std::string s) copies the whole string every call. Use const std::string& s — no copy, and the compiler guarantees you won't modify the caller's string.
🔁
Misreading pointer-const const int* p does not mean the pointer is const — it means the value is. You can still do p = &other;. Read right-to-left, and remember: const before * locks the value.
🧊
Adding const too late Sprinkling const onto a finished codebase fights you, because one const forces another. Reach for const as you write — "const by default," loosen only when you truly need to mutate.

🎓 What to remember

  • const = read-only, enforced at compile time. Must initialize; can never reassign; reading is always fine.
  • const& parameters are the idiom for "pass a big object I'll only read" — no copy, no accidental modification.
  • Pointer-const, read right-to-left: const before * locks the value; const after * locks the pointer.
  • const objects call only const methods — methods that promise not to modify.
  • Be const-correct: reach for const by default; loosen only when you genuinely need to mutate.

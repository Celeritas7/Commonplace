RAII — The Central Idea

Forgetting to free memory, close a file, or release a lock is the source of countless bugs. C++'s answer is its defining idiom: RAII — tie a resource's lifetime to an object's lifetime. Acquire it in the constructor, release it in the destructor, and cleanup happens automatically when the object goes out of scope. This opens Part VIII.

1 Why this matters

Cleanup you can't forget 🧠

Every resource you acquire — heap memory (new), an open file, a network socket, a mutex lock — must eventually be released, exactly once. Doing that by hand is shockingly error-prone: you forget the delete, an early return skips the cleanup, or an exception jumps over it entirely. C++ solves this with RAII (Resource Acquisition Is Initialization): wrap the resource in an object that acquires it in its constructor and releases it in its destructor. Because C++ guarantees the destructor runs when the object leaves scope — on every path out — the cleanup can't be skipped. This is the idea the whole of modern C++ is built on, and it opens Part VIII — Modern Resource Management.

It's why you've been told to give classes a destructor that tidies up.

2 The idea

Acquire in ctor, release in dtor 🏗️

Tie the resource to the object. The constructor takes ownership; the destructor gives it back. The object is the resource's keeper:

class File {
    std::FILE* f;
public:
    File(const char* name) { f = std::fopen(name, "r"); }   // acquire
    ~File() { if (f) std::fclose(f); }                       // release
    // ... methods that use f ...
};

void read() {
    File file("data.txt");   // opened here
    // ... use file ...
}                            // ← file's destructor runs → fclose, automatically
When read() ends, the local file goes out of scope and its destructor fires, closing the handle — you never write fclose at the call site. The resource's lifetime is now welded to the object's lifetime. Acquire when the object is born; release when it dies. That's the entire pattern, and its name ("acquisition is initialization") describes the first half.
3 Why it's bulletproof

Destructors run on every exit 🛡️

Compare the fragile manual version with the RAII one. The manual one leaks if anything jumps over the cleanup:

// Manual — leaks on early return OR exception
void f() {
    std::FILE* fp = std::fopen("d.txt", "r");
    if (bad()) return;            // ← fclose never runs! leak
    risky();                       // ← throws? fclose never runs! leak
    std::fclose(fp);               // only reached on the happy path
}

// RAII — cleans up no matter how f() exits
void f() {
    File file("d.txt");          // dtor will run on EVERY path out
    if (bad()) return;            // ✅ file closed by ~File()
    risky();                       // ✅ throws? stack unwinds → ~File() still runs
}                                  // ✅ normal exit → ~File() runs
This is the magic: when control leaves a scope — by reaching the end, by return, or by an exception unwinding the stack — C++ runs the destructors of all local objects automatically. So an RAII object releases its resource on every exit path, with no extra code. The manual version, by contrast, only cleans up if execution happens to reach the fclose line. RAII turns "remember to clean up everywhere" into "clean up once, in the destructor."
4 Order & ownership

Last in, first out 🔁

When a scope ends, its local objects are destroyed in reverse order of construction — the same rule you saw with base/derived destructors (7.5):

{
    Lock  lock(mtx);     // 1st constructed
    File  file("d");     // 2nd constructed
    // ... work ...
}   // ~File() runs first, then ~Lock() — reverse order, automatically
Each object owns exactly one resource and cleans up exactly that one, and the reverse order means dependencies unwind sensibly (you release the file before the lock that guarded it). The principle underneath is single ownership with a clear lifetime: every resource has one owning object responsible for it. Get the ownership right and the cleanup takes care of itself.

🚨 You've been using RAII all along

Every one of those standard types is an RAII wrapper: it grabs a resource in its constructor and frees it in its destructor, so you never call delete or unlock yourself. That's why a std::vector "just works" with no manual cleanup, and why modern C++ rarely writes raw new/delete. The rest of Part VIII builds directly on this idea: how copying and moving an owning object should behave (copy/move semantics, the Rule of 0/3/5) and the standard smart pointers (unique_ptr, shared_ptr) that put RAII around heap memory for you.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🧹
Manual cleanup on every path Calling delete/fclose by hand leaks whenever an early return or exception skips it. Wrap the resource in an RAII object and let the destructor handle it once.
🪤
Releasing in the wrong place Don't free a resource in a regular method or rely on reaching some line. The destructor is the one place guaranteed to run — release there.
📋
Reinventing what the library gives you Before writing your own RAII wrapper, reach for std::vector, std::string, std::unique_ptr, or std::lock_guard — they already do RAII correctly.

Practice exercises 💪

Easy Trace a destructor. Write a class whose constructor prints "open" and destructor prints "close". Create one inside a block { } and confirm "close" prints when the block ends.
Medium Beat the early return. Put your tracer object in a function that returns early under some condition. Confirm the destructor still runs (cleanup happens) on both the early and normal paths.
Stretch A real wrapper. Write a tiny FileRAII that fopens in its constructor and fcloses in its destructor (guarding against null). Use it to read a file, then throw partway through and confirm the file still closes.

🎓 What to remember

  • RAII = tie a resource's lifetime to an object's: acquire in the constructor, release in the destructor.
  • The destructor runs on every exit — normal end, return, or exception — so cleanup can't be skipped.
  • Objects are destroyed in reverse order of construction; each owns and releases one resource.
  • The standard library is RAII: vector, string, unique_ptr, lock_guard all clean up in their destructors.
  • It's the foundation of Part VIII — copy/move semantics and smart pointers all build on it.

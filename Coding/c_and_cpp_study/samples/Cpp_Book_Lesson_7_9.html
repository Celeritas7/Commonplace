🧬 PART VII · INHERITANCE & POLYMORPHISM

Object Slicing

Polymorphism works through pointers and references. Copy a derived object into a base value and something quietly goes wrong: the derived part is sliced off — its extra data and its virtual behaviour are lost, leaving only the base. It compiles fine and gives no warning, which is exactly what makes it a classic trap.

1 Why this matters

The derived part vanishes 🧠

Every polymorphism lesson so far insisted on pointers or references — Animal*, Animal& — and here's the reason laid bare. A derived object like Dog is bigger than its base Animal: it carries extra members and a link to its virtual functions. When you copy a Dog into a plain Animal value, there's only room for the Animal part, so the extra Dog data is dropped on the floor and the object becomes a genuine Animal — virtual calls now run the base version. This silent loss is called object slicing, and recognizing where it happens is the whole lesson.

It explains the "why pointers/references" rule you've been following.

2 What slicing is

Copy into a base value 🪒

Assign or initialize a base value from a derived object, and only the base subobject is copied — the rest is sliced away:

struct Animal { virtual std::string speak() { return "..."; } };
struct Dog : Animal { std::string speak() override { return "Woof"; } };

Dog d;
Animal a = d;        // SLICE: only the Animal part of d is copied into a
a.speak();           // "..."  — a is a real Animal now, not a Dog 🪒
a is a variable of type Animal — it has exactly the size and identity of an Animal, so it cannot hold a Dog. The copy takes the Animal-shaped portion of d and ignores the rest. Crucially, a.speak() now returns "...": because a truly is an Animal, even a virtual call resolves to Animal::speak(). The dog is gone.
3 Where it bites

Pass-by-value and containers 📦

Slicing usually sneaks in through a by-value function parameter or a container of base values — both copy:

void announce(Animal a) {        // takes Animal BY VALUE → slices any derived arg
    std::cout << a.speak();
}
announce(Dog{});                  // prints "..." — the Dog was sliced into an Animal

std::vector<Animal> zoo;          // a vector of Animal VALUES
zoo.push_back(Dog{});             // the Dog is sliced as it's copied in → just an Animal
Both look perfectly reasonable and compile without complaint — that's the danger. Any time a derived object is copied into a base-typed value (a parameter, a vector element, a return value), it's sliced. A std::vector<Animal> can only ever hold Animals, so every Dog you put in becomes an Animal the moment it's stored.
4 The fix

Reference or pointer, not value ✅

Keep the object as itself by referring to it indirectly — a reference, a pointer, or a smart pointer. No copy into a base value, no slice:

void announce(const Animal& a) {       // by reference → no slice
    std::cout << a.speak();
}
announce(Dog{});                  // "Woof" ✅ — virtual dispatch on the real Dog

std::vector<Animal*> zoo;          // pointers — each element still its real type
// better: std::vector<std::unique_ptr<Animal>> for owned, polymorphic objects (Part VIII)
A reference or pointer doesn't copy the object — it refers to the original, which stays a Dog, so virtual dispatch (7.3) still runs Dog::speak(). The rule that ties together all of Part VII: handle polymorphic objects through Base& or Base*, never by base value. For containers of owned polymorphic objects you'll reach for std::vector<std::unique_ptr<Base>> — coming up in Part VIII.
5 At a glance

Value slices, indirection keeps 📜

How you hold itSliced?Virtual call runs
Animal a = dog;yes — copied into a valueAnimal::speak()
Animal& a = dog;no — refers to the DogDog::speak()
Animal* a = &dog;no — points at the DogDog::speak()

🚨 Slicing is silent — no error, no warning

The compiler is happy to copy a Dog into an Animal; it's a legal copy of the base part. So slicing never announces itself — you only notice when the "wrong" (base) behaviour shows up at runtime, or derived data mysteriously disappears. The defence is a habit, not a diagnostic: whenever you're dealing with a class hierarchy and polymorphism, pass and store through & or *. If you see a base type used by value with derived objects, suspect slicing.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📥
Passing a polymorphic object by value void f(Base b) slices any derived argument. Take it by reference instead: void f(const Base& b) (or Base*).
📦
Storing derived objects in a vector<Base> A container of base values slices everything it stores. Use vector<Base*> or vector<unique_ptr<Base>> to keep each element its real type.
🤫
Expecting a warning Slicing compiles silently — it's a valid base-part copy. There's no error to catch it, so it's a habit to avoid, not a diagnostic to rely on.

Practice exercises 💪

Easy See the slice. Make Animal/Dog with a virtual speak(). Do Animal a = Dog{}; std::cout << a.speak(); and confirm it prints the base "...".
Medium Fix the function. Write void announce(Animal a), call it with a Dog (sliced → base). Change the parameter to const Animal& and confirm it now prints "Woof".
Stretch Container comparison. Put a few Dogs and Cats in a std::vector<Animal> and loop calling speak() — all base. Switch to std::vector<Animal*> and see each speak as itself.

🎓 What to remember

  • Object slicing = copying a derived object into a base value, which keeps only the base part.
  • A sliced object is a real base — virtual calls run the base version; derived data is lost.
  • It sneaks in via base-value parameters, vector<Base>, and base-value returns — each copies.
  • Fix: use Base&, Base*, or unique_ptr<Base> — refer to the object, don't copy it into a base value.
  • It's silent — no error or warning; avoid it by habit. That wraps Part VII! 🎉

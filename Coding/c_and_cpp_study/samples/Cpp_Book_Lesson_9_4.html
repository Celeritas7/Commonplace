Specialisation & Non-Type Parameters

Two smaller template features, briefly. Specialisation lets you write a custom version of a template for one specific type, when the generic one isn't right. Non-type parameters let a template take a compile-time value (like an array size), not just a type. You'll meet both in the standard library — here's enough to recognise and use them.

1 Why this matters

Two refinements on templates 🔧

The plain templates from 9.1–9.2 cover most needs, but two extensions come up often enough to know. First, sometimes the generic template almost works — but one particular type needs different handling (printing a bool as "true"/"false" instead of 1/0, say). Template specialisation lets you provide a tailored version for that one type. Second, a template parameter doesn't have to be a type — it can be a compile-time value, called a non-type parameter, which is how a fixed-size array like std::array<int, 5> bakes its size into the type. This is a brief tour: enough to recognise both when you see them and reach for them when they fit.

Neither is something you'll write daily — but both appear throughout the standard library.

2 Template specialisation

A custom version for one type 🎯

Write the general template, then add a specialisation — a separate definition for a specific type, marked template<>:

template<typename T>            // general version
std::string describe(T) { return "some value"; }

template<>                       // specialisation for bool
std::string describe<bool>(bool b) { return b ? "true" : "false"; }

describe(42);     // general → "some value"
describe(true);   // specialisation → "true"
The empty template<> says "this is a specialisation, not a new template." When you call describe, the compiler picks the specialised version if the type matches exactly (bool here), and falls back to the general template for everything else. So one name, type-driven dispatch, all resolved at compile time. You're overriding the recipe for a single chosen type while keeping the generic one for the rest.
3 When to specialise

Different behaviour, same name 🔀

Specialise when a particular type needs to behave differently — for correctness or efficiency — than the generic logic provides:

  • Correctness: a generic serialize<T> might handle most types, but std::string or const char* needs quoting.
  • Efficiency: a generic algorithm might have a much faster version for one type (e.g. memcpy-able trivial types).
  • Famous example: std::vector<bool> is a specialisation that packs bits — handy, but notoriously surprising.
The pattern keeps a single, consistent interface (describe, serialize) while letting one type opt out of the default. Reach for it sparingly: most of the time the generic template is enough, and overusing specialisation makes code harder to follow. (For functions, a plain overload is often simpler than a specialisation; class templates are where specialisation earns its keep.) Brief by design — recognise it, use it when a type genuinely needs its own version.
4 Non-type parameters

A value, not a type 🔢

A template parameter can be a compile-time value — most often an int. It becomes part of the type and can be used like any constant:

template<typename T, int N>        // N is a non-type parameter (a value)
struct FixedArray {
    T data[N];                       // N sizes the array — known at compile time
    int size() const { return N; }
};

FixedArray<int, 5> a;   // an array of exactly 5 ints; size() == 5
FixedArray<int, 3> b;   // a DIFFERENT type — 3 is part of it
N isn't a type — it's a number fixed at compile time, so data[N] is a valid array size and N can be returned or computed with directly. This is exactly how std::array<T, N> works: the size lives in the type, so it costs nothing at runtime and the compiler knows it everywhere. Note FixedArray<int, 5> and FixedArray<int, 3> are different types — the value is as much part of the type as T is.
5 The rules & cautions

Must be a constant 🎓

Non-type arguments must be known at compile time — they're true compile-time constants (this is where constexpr from 9.3 connects):

🚨 A non-type argument must be a compile-time constant

You can write FixedArray<int, 5> or FixedArray<int, N> where N is a constexpr — but not FixedArray<int, n> with a runtime n (e.g. read from input): that's a compile error, because the size must be baked into the type while compiling. If you need a runtime size, you want std::vector, not a non-type-parameter array. Two more notes: non-type parameters are traditionally limited to integral/enum/pointer kinds (C++20 broadens this), and each distinct value makes a separate instantiation — FixedArray<int, 5> and <int, 6> are unrelated types, so a function accepting "any size" must itself be templated on N. Both features here are occasional tools — know them, don't overuse them.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🏃
A runtime value as a non-type argument FixedArray<int, n> with a runtime n won't compile — the value must be a compile-time constant. Need a runtime size? Use std::vector.
🔀
Expecting <int,5> and <int,6> to interchange Different non-type values make different, unrelated types. A function that works for "any size" must itself be a template on N.
🧩
Over-specialising Reach for specialisation only when a type genuinely needs different behaviour. For functions a plain overload is often clearer; keep the generic template as the default.

Practice exercises 💪

Easy Specialise for bool. Write a generic template<typename T> std::string kind(T) returning "value", plus a template<> specialisation for bool returning "flag". Call it with an int and a bool and confirm each picks the right version.
Medium Fixed-size array. Write template<typename T, int N> struct FixedArray { T data[N]; int size() const { return N; } };. Make a FixedArray<int, 4>, fill it, and print size(). Try FixedArray<int, n> with a runtime n and watch it fail.
Stretch Compare with std::array. Use std::array<int, 5> and note it's the same idea as your FixedArray — a non-type size parameter. Write a function template template<int N> int sum(const std::array<int, N>&) that works for any size, and call it with arrays of different N.

🎓 What to remember

  • Template specialisation (template<>) gives one specific type its own version; the compiler picks it over the general template when the type matches.
  • Specialise sparingly — for correctness or speed on a particular type; the generic template stays the default for everything else.
  • Non-type parameters (template<int N>) make a template take a compile-time value, baking it into the type (e.g. an array size).
  • Non-type arguments must be compile-time constants — that's the link to constexpr; runtime values need std::vector.
  • Both are how the STL works — std::array<T, N> and specialisations like std::vector<bool> — occasional tools worth recognising.

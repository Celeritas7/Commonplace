📁 PART IX · FILES AND THE OUTSIDE WORLD

Standard Streams & Pipes

You've been using files you opened yourself. But every C program starts with three streams already open: stdin for input, stdout for normal output, stderr for errors. They're plain FILE *s — and because the shell can redirect and pipe them, getting these right is what lets your program plug into the whole Unix toolbox.

1 Why this matters

Three streams, already open 🔌

Before your main runs, the system hands every program three open streams: stdin (standard input — where keyboard or piped input arrives), stdout (standard output — your normal results), and stderr (standard error — messages and diagnostics). You've used them already without knowing: printf writes to stdout and scanf reads from stdin. They're all FILE *, so every function from this Part works on them too. The payoff is that the shell can rewire these streams to files or to other programs — and a program that uses them correctly becomes a reusable building block.

This is the idea behind the Unix philosophy: small programs that read stdin and write stdout snap together like Lego.

2 They're just FILE*

printf is fprintf(stdout, …) 🖨️

The three streams are predefined FILE * values (from <stdio.h>). Anything you can do to a file you opened, you can do to them:

printf("hello\n");                 // exactly fprintf(stdout, "hello\n")
fprintf(stdout, "answer: %d\n", 42);  // normal output
fprintf(stderr, "error: %s\n", msg); // diagnostics go here, NOT stdout

char line[128];
fgets(line, sizeof line, stdin);    // read a line of input (like 9.2)
StreamPurposeShortcut
stdininputscanf, getchar, fgets(…, stdin)
stdoutnormal outputprintf, putchar
stderrerrors / logsfprintf(stderr, …)
There's no magic: printf(...) is literally shorthand for fprintf(stdout, ...), and scanf(...) is fscanf(stdin, ...). The only one without a one-word shortcut is stderr — you write to it explicitly with fprintf(stderr, …), which is exactly how you signal "this is a message, not a result."
3 Why stderr is separate

Keep errors out of the data ⚠️

Why two output streams? Because output and commentary about output are different things. Results go to stdout; everything else — errors, warnings, progress — goes to stderr:

if (!file) {
    fprintf(stderr, "could not open %s\n", name);  // to stderr
    return 1;
}
printf("%d\n", total);   // the actual result → stdout
Keeping them apart means someone can capture your real output (redirect stdout to a file, or pipe it into another program) while still seeing the error messages on their screen. If errors went to stdout, they'd be mixed into the saved data or fed as bogus input to the next program. As a bonus, stderr is typically unbuffered (or line-buffered), so error messages appear immediately rather than waiting in a buffer — handy when a program crashes.
4 Redirection

The shell rewires the streams 🔀

Here's the powerful part: your program doesn't change. The shell can point the streams somewhere else before your program even starts:

CommandWhat the shell does
prog > out.txtstdout → the file out.txt (stderr still to screen)
prog 2> err.txtstderr → the file err.txt (stdout still to screen)
prog < in.txtstdin ← the file in.txt (instead of the keyboard)
prog > out 2>&1stdout → out, and stderr → wherever stdout goes
When you run prog > out.txt, the program still just calls printf — but those bytes now land in out.txt instead of the terminal, because the OS connected stdout to the file. Same code, different destination. This is why a program should write its real output with printf/stdout: it makes the output capturable. The 2 in 2> is the number of the stderr stream (0 = stdin, 1 = stdout, 2 = stderr).
5 Pipes

One program's output, another's input 🪈

A pipe (|) connects the stdout of one program directly to the stdin of the next — no temp file needed:

cat log.txt | grep ERROR | sort     # each | feeds stdout → next stdin

🚨 Printing errors to stdout breaks pipes

If your program prints an error with printf (i.e. to stdout) instead of fprintf(stderr, …), that error text gets piped into the next program as if it were data — corrupting the pipeline — and it vanishes if someone redirects stdout to a file. Errors belong on stderr precisely so they stay out of the data stream and remain visible. One more gotcha: stdout is fully buffered when it's not a terminal (e.g. into a pipe or file), so if you interleave stdout and stderr writes they can appear out of order — call fflush(stdout) before writing to stderr if order matters.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

⚠️
Printing errors to stdout Use fprintf(stderr, …) for messages. On stdout, errors get redirected into files or piped into the next program as fake data.
🧱
Hard-coding a filename instead of reading stdin A tool that reads stdin and writes stdout composes with pipes and redirection. Tying it to a fixed file makes it un-pluggable.
🔀
Assuming output order across streams stdout is buffered when piped/redirected, so stdout and stderr lines can interleave oddly. fflush(stdout) before a stderr write if order matters.

Practice exercises 💪

Easy cat, lite. Read stdin line by line with fgets and echo each to stdout. Run it three ways: type input, prog < file.txt, and echo hi | prog — same code, three sources.
Medium Results vs. messages. Write a program that prints its computed answer to stdout and a "done in N steps" note to stderr. Run prog > out.txt and confirm out.txt has only the answer while the note still shows on screen.
Stretch A pipe stage. Build a filter that reads stdin, uppercases each line, and writes stdout — with any errors to stderr. Drop it into a pipeline: cat file | yourprog | sort and confirm it behaves like a real Unix tool.

🎓 What to remember

  • Three streams open automatically: stdin (input), stdout (output), stderr (errors) — all FILE *.
  • printf = fprintf(stdout, …), scanf = fscanf(stdin, …); write errors with fprintf(stderr, …).
  • Keep errors on stderr so real output stays clean and capturable, and messages stay visible.
  • The shell redirects with > (stdout), 2> (stderr), < (stdin) — your code is unchanged.
  • A pipe | wires one program's stdout into the next's stdin — the heart of composing Unix tools.

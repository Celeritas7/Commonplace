Recursion

A function is allowed to call itself. That sounds like a recipe for an infinite loop — and it is, unless you give it a way to stop. Done right, recursion solves naturally self-similar problems in a few elegant lines.

1 Why this matters

Some problems contain themselves 🧠

A few problems are self-similar: a factorial is a number times a smaller factorial; a folder contains files and smaller folders; a row of dominoes falls one, then the rest. For those, a loop can feel clumsy, but a function that calls a smaller version of itself fits like a glove. That's recursion.

You already know functions call other functions. Recursion is just a function calling itself — with a slightly smaller job each time — until the job is small enough to answer outright. The magic and the danger are the same thing: if it never reaches that "small enough" case, it calls forever. So every recursion has two parts you must get right.

2 The two parts

Base case + recursive case 🧩

Every recursive function needs a base case (a condition that stops, returning an answer directly) and a recursive case (which does a little work and calls itself on a smaller input). Here's the classic, factorial:

int fact(int n) {
    if (n <= 1) return 1;          // base case: stop here
    return n * fact(n - 1);        // recursive case: smaller n
}
Read it as a definition: "the factorial of n is n times the factorial of n − 1, and the factorial of 1 is just 1." The recursive case must move toward the base case — here n - 1 shrinks n on every call, so it's guaranteed to hit n <= 1 eventually. Miss that, and it never stops.
3 The call stack

Calls wait, then unwind 📚

How can fact(4) finish before fact(3) does? It can't — so it waits. Each call is paused mid-expression, stacked on top of the previous one, until the base case returns. Then the answers flow back down, each waiting call finishing its multiplication:

fact(4) = 4 * fact(3)
        = 4 * (3 * fact(2))
        = 4 * (3 * (2 * fact(1)))
        = 4 * (3 * (2 * 1))      // base case reached
        = 4 * (3 * 2)
        = 4 * 6
        = 24
That pile of paused calls is the call stack — the same mechanism behind every ordinary function call, just stacked deeper. Each call keeps its own copy of n (remember scope: locals are per-call), which is why fact(4) still knows its n is 4 when control finally returns to it. 4 Another shape

Do work on the way down — or up ⬇️⬆️

Where you put the work relative to the recursive call changes the order things happen. Compare a countdown and a count-up:

void down(int n) {
    if (n == 0) return;
    std::cout << n << " ";   // print BEFORE recursing
    down(n - 1);
}   // down(3) → 3 2 1

void up(int n) {
    if (n == 0) return;
    up(n - 1);
    std::cout << n << " ";   // print AFTER recursing
}   // up(3) → 1 2 3
Same structure, opposite output. down prints on the way down the stack, so the biggest number comes first. up recurses all the way to the base case first, then prints as the stack unwinds — so the smallest prints first. Tracing where the cout sits relative to the recursive call is the key to predicting recursive output.
5 Watch out

No base case, no mercy 💥

If the recursion never reaches its base case, each call stacks another frame until the call stack runs out of memory — a stack overflow, and your program crashes.

🚨 Missing or unreachable base case

int bad(int n) { return n + bad(n - 1); } has no if to stop it, so it calls forever (well — until the stack overflows and it crashes). Always include a base case, and make sure every recursive call moves toward it. And remember: anything recursion does, a loop can do too — recursion is for when the self-similar shape makes the code clearer, not a tool you must use.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

💥
No base case (or you can't reach it) Without a stopping condition — or if the recursive call doesn't shrink toward it — the function calls forever and overflows the stack. Always check: does every path move toward the base case?
🔙
Forgetting to return the recursive result In a value-returning recursion, write return n * fact(n - 1);. Calling fact(n - 1) without return throws the answer away and the result is wrong.
🔀
Mixing up "before" vs "after" the call Work done before the recursive call happens on the way down; work done after happens as the stack unwinds. That placement decides the output order (the down vs up example).

Practice exercises 💪

Easy Countdown. Write void countdown(int n) that prints n, n-1, …, 1 then "Liftoff!", using recursion with a base case at n == 0.
Medium Sum to n. Write int sumTo(int n) that returns 1 + 2 + … + n recursively (sumTo(n) = n + sumTo(n-1), base case sumTo(0) = 0). Test it against the formula n*(n+1)/2.
Stretch Fibonacci, then reflect. Write int fib(int n) returning the nth Fibonacci number (base cases fib(0)=0, fib(1)=1; otherwise fib(n-1)+fib(n-2)). Then think about why fib(40) is slow — it re-computes the same calls many times. (A loop, or remembering results, fixes it.)

🎓 What to remember

  • Recursion is a function calling itself on a smaller input.
  • Two parts: a base case that stops, and a recursive case that moves toward it.
  • Calls stack up and wait, then unwind once the base case returns — each call keeps its own variables.
  • Work before vs after the recursive call decides whether things happen on the way down or back up.
  • No reachable base case → stack overflow. Anything recursion does, a loop can too — use it when the shape fits.

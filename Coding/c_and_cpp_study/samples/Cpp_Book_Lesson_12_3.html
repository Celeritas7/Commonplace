Capstone: An Expression Evaluator

A calculator that reads text like 2 + 3 * 4 and gets 14, not 20 — because * binds tighter than +. The beautiful part: we don't write a precedence table. We layer the grammar into expression → term → factor, and recursion does the rest.

1 What we're building

Order of operations, from scratch ✖️➕

We'll build an arithmetic evaluator: give it a string like 2 + 3 * 4 or (10 - 4) / 2 and it returns the numeric answer, respecting operator precedence (* and / before + and -) and parentheses. It's a close cousin of the JSON parser (12.2) — same recursive-descent technique — but it adds the one thing JSON didn't have: precedence. The elegant trick is that we never compare operator priorities by hand. Instead we split the grammar into three levels — an expression is terms added and subtracted; a term is factors multiplied and divided; a factor is a number or a parenthesised expression. Because multiplication is resolved one level below addition, it naturally binds tighter.

Try it below, then read the five pieces. By the end you'll see why "precedence" is really just "which rule calls which."

2 The grammar

Three levels = precedence 🪜

The whole design is captured in three rules. Read each as "a ___ is a ___ followed by zero or more operators and more ___s":

RuleDefinition
expressionterm (( + | - ) term)*
termfactor (( * | / ) factor)*
factornumber | ( expression ) | - factor
Precedence is baked into this layering. expression only knows about +/-, and for each operand it asks term; term only knows about */-… /, and asks factor for its operands. So in 2 + 3 * 4, the expression level sees 2, then +, then asks term for the next operand — and term greedily consumes 3 * 4 = 12 before handing it back. The * got evaluated "deeper," so it bound tighter. factor is the bottom: a literal number, or a whole (expression) that loops back to the top (giving parentheses their override power), or a leading - for negatives.
3 The skeleton

One cursor, three functions 🚶

As in the JSON parser, a small class scans the string with an index. Each grammar rule becomes a function returning a double:

class Evaluator {
    const std::string& s_;
    size_t i_ = 0;
public:
    Evaluator(const std::string& s) : s_(s) {}
    double eval() { double v = expression(); skipWs();
                   if (i_ != s_.size()) throw std::runtime_error("trailing input");
                   return v; }
private:
    void skipWs() { while (i_ < s_.size() && std::isspace(s_[i_])) ++i_; }
    char peek() { skipWs(); return i_ < s_.size() ? s_[i_] : '\0'; }

    double expression();
    double term();
    double factor();
    double number();
};
Nothing here is new after 12.2 — it's the same recursive-descent skeleton, just returning double instead of a tree. skipWs and peek are the same helpers. eval() is the public entry: parse a full expression, skip trailing spaces, and complain if anything is left over (so "2 + 3 x" is rejected rather than silently giving 5). The three private functions expression/term/factor mirror the three grammar rules exactly.
4 The two operator levels

expression and term 🔁

Each level grabs one operand, then loops while it sees one of its operators — accumulating left to right (so subtraction and division associate correctly):

double Evaluator::expression() {
    double v = term();                       // first operand
    while (true) {
        char c = peek();
        if      (c == '+') { ++i_; v += term(); }   // + and - live here
        else if (c == '-') { ++i_; v -= term(); }
        else break;
    }
    return v;
}

double Evaluator::term() {
    double v = factor();                     // first operand
    while (true) {
        char c = peek();
        if      (c == '*') { ++i_; v *= factor(); } // * and / live one level down
        else if (c == '/') {
            ++i_; double d = factor();
            if (d == 0) throw std::runtime_error("division by zero");
            v /= d;
        } else break;
    }
    return v;
}
These two functions are nearly identical — they differ only in which operators they handle and that term calls factor while expression calls term. That difference is the entire precedence mechanism. The while loop makes each operator left-associative: 10 - 4 - 3 becomes (10 - 4) - 3 = 3, not 10 - (4 - 3). And division checks its divisor first, throwing the kind of runtime error you'd catch with the machinery from Part X.
5 The bottom & the entry

factor, numbers, and going public 🌳

factor is where recursion bottoms out — or loops back to the top for parentheses:

double Evaluator::factor() {
    char c = peek();
    if (c == '(') {
        ++i_;
        double v = expression();              // back to the top!
        if (peek() != ')') throw std::runtime_error("expected ')'");
        ++i_;
        return v;
    }
    if (c == '-') { ++i_; return -factor(); }    // unary minus
    return number();                          // a plain number
}

double evaluate(const std::string& s) {       // the public entry point
    return Evaluator(s).eval();
}

🚨 Parentheses loop back to the top — that's the recursion

When factor meets a (, it calls expression() again — the highest-level rule — to evaluate whatever is inside, then expects the matching ). That single call is what makes parentheses override precedence: (2 + 3) * 4 works because the inner 2 + 3 is a complete sub-expression evaluated before the * ever sees it. It's the same mutual recursion as the JSON parser, just with arithmetic. Two things to get right: always consume the closing ) (and error if it's missing), and remember factor handles unary minus (-5, or -(3 + 1)) by negating whatever factor follows. Miss the trailing-input check in eval() and a typo like 2 ** 3 would quietly evaluate as 2.

8 Mistakes, extensions, recap

Wrap up 📚

Three mistakes to avoid 🚨

🪜
Handling all operators in one function If expression handled + - * / together, you'd lose precedence (left-to-right only). The layering into expression/term/factor is the precedence — keep the levels separate.
➗
Not guarding division by zero Dividing by zero is undefined for integers and gives inf/nan for doubles. Check the divisor and throw a clear error (Part X) before dividing.
🔚
Not rejecting leftover input After parsing one expression, make sure nothing's left (eval()'s trailing-input check). Otherwise "2 + 3 )" or "4 5" would parse the first bit and ignore the junk.

Extend it 💪

Easy Add modulo. Support the % operator at the same level as * and / (in term). Decide how it should behave for doubles, or restrict the evaluator to integers.
Medium Exponentiation. Add a ^ operator that binds tighter than * and is right-associative (2 ^ 3 ^ 2 = 512). That needs a new level between term and factor — and a right-associative shape (recursion, not a loop).
Stretch Variables & functions. Let factor also recognise names: a variable (x) looked up in a std::map<std::string,double>, and calls like sqrt(2) or max(3, 7) that parse an argument list (each argument a full expression).

🎓 What you built

  • An arithmetic evaluator that respects precedence and parentheses — text in, number out.
  • Precedence without a table: the grammar layers into expression (+/-) → term (*//) → factor, so deeper rules bind tighter.
  • Left-associativity from the while loop at each level; parentheses from factor calling expression again.
  • Recursion is the engine — the same recursive-descent shape as the JSON parser, now producing a value.
  • Robustness: guard divide-by-zero and reject leftover input, reporting errors clearly.

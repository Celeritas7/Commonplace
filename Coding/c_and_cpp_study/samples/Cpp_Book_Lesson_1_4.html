How a C++ Program Is Built

In lesson 1.3 you ran g++ hello.cpp -o hello. It looked atomic. It isn't — that command is four separate tools running in sequence. Today we open the lid.

1 Why this matters

The black box is actually a pipeline 🔧

When you typed g++ -std=c++20 hello.cpp -o hello, g++ ran four separate tools in sequence: preprocessor, compiler, assembler, linker. Knowing this matters more in C++ than C, because the linker is where C++'s most famous error class lives — "undefined reference," "multiple definition," and friends — and you can't debug them without knowing which stage is talking.

This lesson is lighter on theory and bigger on the pipeline picture. The interactive shows you the four stages and what your file looks like after each one. By the end you'll understand the journey from hello.cpp (five lines of text) to hello (~20 KB of machine code).

2 The four stages, in one breath

Preprocessor → Compiler → Assembler → Linker 🏗️

Here's the whole story in four bullets. We'll click into each one in the next section.
  1. Preprocessor — handles every line starting with #. Mostly pastes in the contents of each #include'd header. In C++, headers can be huge — <iostream> alone pulls in thousands of lines.
  2. Compiler — takes the now-expanded C++ and translates it into assembly language, specific to your CPU. This is the slow step in C++; templates and inlining happen here.
  3. Assembler — converts assembly into machine code — actual binary bytes. The output is an object file (.o) with placeholders for things from other files / libraries.
  4. Linker — stitches your object file together with the C++ standard library (for std::cout etc.) and any other libraries, producing the final executable. This is where most C++ ecosystem mystery happens.

g++ hides all of this behind one command. But you can ask it to stop after any stage and inspect the intermediate file — exactly what we'll do in the interactive below.

4 All four, in one command

Why g++ "just works" ✨

The convenience of typing one g++ command hides a remarkable amount of engineering. The pipeline below corresponds to four separate executables that g++ runs on your behalf: 
$ g++ -std=c++20 hello.cpp -o hello
# = cpp → cc1plus → as → ld, all in one shot

Pro habit for later: when you build with cmake --build . or run the F5 button in VS Code, those tools ultimately invoke g++ (or clang++ or MSVC), which in turn runs this same four-stage pipeline. Build systems orchestrate; the pipeline does the actual work.

And the error-message rosetta stone you'll use forever:

  • "No such file or directory" (on a #include) → preprocessor failed.
  • "expected ';' before '...'", "no matching function for call to..." → compiler failed.
  • "undefined reference to...", "multiple definition of..." → linker failed.

6 Common mistakes

Three traps that signal you skipped this lesson 🚧

🙅
"The compiler is the only thing that runs." In C++ especially, the linker is the part where most ecosystem pain lives — undefined references, multiple definitions, ABI mismatches. When you say "the compiler" loosely, fine; when you're debugging, name the exact tool.
🙅
"Object files (.o) are runnable on their own." They're not. An object file contains machine code, but with placeholders for everything it references but doesn't define — including all the C++ standard library. The linker fills those placeholders in. Until then, you've got a half-assembled program.
🙅
"Templates are like normal functions." They're not, and the compile pipeline is why. Templates aren't compiled until used; the compiler generates fresh code for each unique combination of types you instantiate. This is part of why C++ compile times are slow — and it's also why linker errors with templates look so scary (the error message includes the full template instantiation chain). We'll meet templates properly in Part IX.
7 Your turn

Watch the pipeline yourself 💪

Easy Stop after each stage. Open a terminal in your hello.cpp folder. Run these four commands in order: 
$ g++ -std=c++20 -E hello.cpp -o hello.ii
$ g++ -std=c++20 -S hello.ii -o hello.s
$ g++ -std=c++20 -c hello.s -o hello.o
$ g++ hello.o -o hello
Open each file. Notice how each is bigger and stranger than the last. hello.ii will be massive (~30,000 lines) because <iostream> includes hundreds of other headers transitively.
Medium Count the lines. After step 1 above (g++ -E), check wc -l hello.ii (Linux/Mac) or open the file in your editor (Windows). Your original 5-line program will have expanded to tens of thousands of lines. That's why C++ compile times are slow — the compiler chews through all of this every time.
Stretch godbolt is the best lab. Open godbolt.org. Paste your hello.cpp on the left, set the compiler to x86-64 gcc 13.2, set Compiler options to -std=c++20 -O2. Look at the assembly on the right. Now toggle -O0 instead of -O2 and look again. The unoptimized version is much longer. That's the compiler's optimization stage doing real work between your code and the CPU.

🎓 Quick recap

  • g++ hello.cpp -o hello runs four tools in sequence: preprocessor (cpp), compiler (cc1plus), assembler (as), linker (ld).
  • Preprocessor stops at -E → .ii file. In C++, the output is huge because headers transitively include thousands of lines.
  • Compiler stops at -S → .s file. Turns C++ into assembly. The slow step in C++ because of templates and optimization.
  • Assembler stops at -c → .o (object file). Binary, with placeholders for unresolved references.
  • Linker stitches everything together with libraries (libstdc++ provides std::cout, etc.) into the final executable.
  • Error-message lookup table: "no such file" = preprocessor, "expected ';'" / "no matching function" = compiler, "undefined reference" / "multiple definition" = linker.

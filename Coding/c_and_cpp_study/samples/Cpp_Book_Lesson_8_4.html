♻️ PART VIII · MODERN RESOURCE MANAGEMENT

Rule of 0 / 3 / 5

You've now met all the pieces — destructor, copy operations, move operations. They aren't independent: they form a coordinated set, and getting one without the others is how bugs sneak in. These three rules of thumb tell you exactly how many you should write yourself — and the surprising best answer is usually zero.

1 Why this matters

They come as a set 🧠

Over the last three lessons you saw the destructor (8.1), copy constructor and copy assignment (8.2), and move constructor and move assignment (8.3). The trap is treating them as five unrelated functions. They're not — they all answer the same question: "what happens to the resource this object owns when it's destroyed, copied, or moved?" Get one right and the others wrong, and your class is subtly broken (a double free, a silent slow copy, a leak). The "Rule of 0/3/5" is the community's compact wisdom for keeping them consistent: it tells you, based on what your class holds, exactly how many of the five to write.

Spoiler: the rule you should reach for almost every time is the Rule of Zero.

2 The five special members

The cast of characters ⚙️

These are the five functions the compiler will generate for you if you don't write them — the ones that govern an object's lifetime:

struct T {
    ~T();                       // 1. destructor       — clean up
    T(const T&);              // 2. copy constructor — build from an lvalue
    T& operator=(const T&);   // 3. copy assignment  — overwrite from an lvalue
    T(T&&);                   // 4. move constructor — build from an rvalue (C++11)
    T& operator=(T&&);        // 5. move assignment  — overwrite from an rvalue (C++11)
};
By default the compiler synthesises all five, and for a class made of value-like members (numbers, std::string, std::vector) those defaults are exactly right — each member is copied or moved correctly. The rules below only matter when your class owns a raw resource (a new'd pointer, a file handle, a socket) where the compiler's member-wise default does the wrong thing.
3 Rule of Three

One means three 📐

The classic (pre-C++11) rule: if you need to write a custom destructor, copy constructor, or copy assignment, you almost certainly need all three. Why? Because needing any one of them is a sign your class manages a resource by hand:

// Has a destructor that frees data → MUST also define both copies
~Box() { delete data; }            // custom dtor → resource ownership
Box(const Box& o);              // ...so define copy ctor (deep copy)
Box& operator=(const Box& o);   // ...and copy assignment (deep + free old)
If you write the destructor but let the copies default, the defaults do a shallow copy (8.2) — two objects share one pointer and both free it: a double free. The three are a package because they all depend on the same fact: "this object owns something that needs deliberate cleanup and duplication." Write one, write them all.
4 Rule of Five

Add the two moves ✋

Since C++11 there are five special members, so the rule grows: if your class manages a resource, define all five — the destructor, both copy operations, and both move operations. The moves let your class transfer ownership cheaply (8.3) instead of always deep-copying:

If you write……you should write
any of dtor / copy ctor / copy =all three (Rule of Three)
and you want cheap transfers+ move ctor & move = (Rule of Five)

🚨 Declaring a destructor or copy quietly kills your moves

Here's the subtle part that bites people: if you declare a destructor (or a copy operation), the compiler stops generating the move operations for you. Your class still works — it just silently falls back to copying everywhere a move could have happened, so a class you thought was fast quietly does deep copies. This is exactly why the rule jumps from three to five: once you're writing any of them by hand, you've turned off the automatic moves and need to provide them yourself (or explicitly = default them) to keep moves working.

5 Rule of Zero

The one to actually aim for 🎯

The modern best practice flips the whole problem around: don't manage raw resources yourself at all, so you write none of the five and the compiler's defaults are automatically correct. Hold members that already follow the Rule of Five internally:

struct Box {
    std::vector<int> data;   // vector owns the memory and handles copy + move
    std::string  name;   // string does too
    // no destructor, no copy/move — the defaults are correct!
};
Because std::vector and std::string (and std::unique_ptr, coming next) each clean up, copy, and move themselves correctly, a class built only from them gets correct compiler-generated versions of all five — for free. No double frees, no leaks, no forgotten move. This is the Rule of Zero, and it's the goal: reach for vector/string/smart pointers as members, and you almost never write a special member again. Hand-writing the Rule of Five is for the rare case where you're building one of those resource wrappers yourself.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

💥
Writing only the destructor A custom destructor with default (shallow) copies = double free. If you write one of the three, write all three — or better, avoid the raw resource entirely.
🐌
Forgetting that a dtor suppresses moves Declaring a destructor (or copy) stops the compiler generating move operations, so your class silently copies instead of moving. Add the moves (or = default) to keep them.
🔧
Hand-writing the Rule of Five when you don't need to If your members are already RAII types, writing special members by hand just adds bugs. Prefer the Rule of Zero — let vector/string/unique_ptr do the work.

Practice exercises 💪

Easy Count the generated. Write a struct of two ints with no special members. Copy it and move it, and confirm both work — the compiler gave you all five for free (Rule of Zero).
Medium Watch a move vanish. Add a destructor that just prints "~" to a class with a copy ctor (prints "C") and a move ctor (prints "M"). Then add a second class with the same copy/move but no user destructor. Move both and see which one actually moves.
Stretch Five → Zero. Implement a raw-pointer Buffer with a correct Rule of Five (all five members, with prints). Then rewrite it to hold a std::vector<int> and delete all five. Prove the rewrite still copies and moves correctly with no hand-written members.

🎓 What to remember

  • Five special members govern lifetime: destructor, copy ctor, copy =, move ctor, move = — all auto-generated by default.
  • Rule of Three: need a custom destructor, copy ctor, or copy assignment? Write all three.
  • Rule of Five: managing a resource by hand? Define all five, including the two moves.
  • Declaring a dtor or copy suppresses the implicit moves — your class silently copies unless you add them back.
  • Rule of Zero (the goal): hold RAII members so you write none of the five and the defaults are correct.

🔀 PART III · MAKING DECISIONS

if, else, else if

Up to now your programs ran straight through, top to bottom. The if statement is the first fork in the road: it lets your code take different paths depending on what's true. This is where programs start to feel alive.

1 Why this matters

Programs that choose 🧠

Every useful program reacts to its situation. "If the password is correct, let them in." "If the cart is empty, show a message." "If the temperature is below zero, warn about ice." That word — if — is a real keyword in C, and it's the foundation of every decision your code will ever make.

So far your programs have been a straight line: do step 1, then step 2, then step 3. The if statement adds a fork. The program checks a condition, and only runs certain code when that condition is true. Add an else and you get a second path for when it's false. Chain a few together and your program can sort any situation into the right response. This whole lesson is that one idea, plus the handful of ways it trips people up.

2 The if statement

One condition, one block 🚦

The shape of an if is a condition in parentheses, followed by the code to run when it's true:

if (temperature < 0) {
    printf("Watch out for ice!\n");
}

C evaluates the condition temperature < 0. If it's true (any nonzero value), the code in the braces runs. If it's false (zero), C skips the whole block and carries on below it.

Remember from the operators lessons: a C condition doesn't have to be a comparison. Any expression that's nonzero counts as true, and zero counts as false. So if (count) means "if count isn't zero," and if (ptr) means "if ptr isn't null." Comparisons like a >= 18 just produce 1 or 0, which is why they work as conditions.
3 else and else if

Adding the other paths 🛤️

else gives you a fallback that runs when the if condition is false. And else if lets you test a series of conditions in order, picking the first one that's true:

if (score >= 90) {
    grade = 'A';
} else if (score >= 80) {
    grade = 'B';
} else if (score >= 70) {
    grade = 'C';
} else {
    grade = 'F';
}
The key rule of an if / else if chain: C checks the conditions top to bottom and stops at the first true one. Once a branch runs, every later branch is skipped — even if its condition would also be true. That's why you order them carefully: score >= 90 comes before score >= 80, because a 95 satisfies both, and you want the A. Exactly one branch runs (the else catches everything left over). 4 Braces and blocks

Why you should always use { } 🧱

If the body is a single statement, C lets you drop the braces: if (x > 0) printf("positive\n"); is legal. But this convenience is the source of two nasty bugs, so the strong advice is: always use braces, even for one line.

🚨 The "I forgot braces" bug

Without braces, only the next single statement belongs to the if. So in:

if (loggedIn)
    showDashboard();
    logAccess();  // runs ALWAYS — not part of the if!

logAccess() looks indented under the if, but indentation means nothing to C. It runs every time. Braces would have caught it.

🚨 The stray semicolon

if (x > 0); — that semicolon right after the ) is an empty statement that becomes the entire body. Whatever you wrote next runs unconditionally. It compiles cleanly and silently does the wrong thing.

5 Conditions and the equals trap

The bug that bites every beginner 🪤

Inside an if, you compare with == (two equals). The single = is assignment — and using it by accident is the most famous bug in all of C:

// BUG: assigns 0 to x, then tests 0 (false)
if (x = 0) { ... }

// CORRECT: compares x against 0
if (x == 0) { ... }
if (x = 0) compiles fine. It assigns 0 to x, then the value of that assignment — 0 — becomes the condition, which is false. With if (x = 5) it's worse: it sets x to 5 and the condition is always true. A defensive trick some programmers use is to put the constant first — if (0 == x) — because if (0 = x) won't compile, catching the typo. Mostly, just turn on compiler warnings (-Wall), which flag if (x = 0) automatically.

7 Predict the output

Quick check 🎯

What does this print? 
int x = 5;
if (x = 0)
    printf("zero\n");
else
    printf("nonzero\n");
Hint: that's a single =, not ==. What value does x = 0 produce, and is it true or false?
8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🟰
Writing = instead of == if (x = 5) assigns 5 and is always true; if (x == 5) compares. Turn on -Wall and the compiler warns you. When in doubt, put the constant first: if (5 == x).
🧱
Dropping braces and adding a second line Without { }, only the next single statement belongs to the if. The line after it runs unconditionally, no matter how it's indented. Always use braces.
❗
A semicolon right after if (...) if (cond); makes the body an empty statement — the real code below runs every time. Watch for that stray ;.

Practice exercises 💪

Easy Pass or fail. Read an int score and print "Pass" if it's 50 or above, otherwise "Fail". Use a single if / else with braces.
Medium Grade ladder. Recreate the A/B/C/D/F grader from this lesson as a real program. Test it with 95, 85, 72, 65, and 40 — match the Branch Tracer's answers. Make sure a 90 gets an A, not a B (boundary check!).
Stretch Categorize a number. Read an int and print whether it's "negative", "zero", or "positive" using an if / else if / else chain. Then add: if it's positive, also say whether it's even or odd, using a nested if inside that branch.

🎓 What to remember

  • if (condition) { ... } runs its block only when the condition is true (nonzero).
  • else is the fallback; else if chains test top to bottom and stop at the first true one.
  • Always use braces { } — bare bodies invite the missing-brace and stray-semicolon bugs.
  • Compare with ==, not =. A single = assigns and is a classic silent bug.
  • Any nonzero value is true; zero is false. Comparisons just produce 1 or 0.

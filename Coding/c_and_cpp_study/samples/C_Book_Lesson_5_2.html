Arrays Don't Know Their Size

Here's a surprise about C arrays: a function can't ask one how long it is. The array "decays" to a bare pointer the moment you pass it, losing its length. The fix is simple and you'll use it constantly — pass the length alongside the array.

1 Why this matters

A list with no length tag 🧠

In the last lesson you looped over an array by writing the size yourself: for (i = 0; i < 5; i++). You might wonder — why not just ask the array its length, like v.size() in C++? The blunt answer: a C array doesn't store its length anywhere. It's just a run of values in memory with no count attached, and certain operations lose even the little the compiler knew.

This isn't a small detail — it's the reason almost every C function that takes an array also takes an int length right beside it (you've seen main(int argc, char *argv[]) do exactly this). Understanding why turns a confusing rule into an obvious one, and it sets up pointers — the next big chapter — perfectly.

2 The sizeof trick

Where the size is known 📐

In the same function where you declared the array, the compiler does know its total size, and you can recover the element count with a classic idiom: total bytes divided by one element's bytes.

int a[5];                          // assume int = 4 bytes

sizeof(a);                          // 20  — the whole array (5 × 4)
sizeof(a[0]);                       // 4   — one element
sizeof(a) / sizeof(a[0]);           // 5   — the element COUNT ✓
sizeof(a) / sizeof(a[0]) reads as "how many elements fit in the whole array" and gives you the count without hard-coding 5. It's handy and worth knowing. But — and this is the whole lesson — it only works where a is a real array. Cross a function boundary and it quietly breaks.
3 Array decay

Pass it, and it becomes a pointer 👻

When you pass an array to a function, C doesn't copy the whole array — it passes the address of the first element. Inside the function, the parameter is a pointer, not an array. This is called array decay, and it means sizeof inside the function measures the pointer, not the array:

void f(int a[]) {        // 'a' here is really int* — a pointer
    sizeof(a);            // 8 (pointer size), NOT 20!
    sizeof(a) / sizeof(a[0]); // 8 / 4 = 2 — WRONG, the array had 5
}

int main(void) {
    int a[5];
    sizeof(a);            // 20 here — still a real array
    f(a);                 // inside f, the length is gone
}
int a[] in a parameter list is a polite fiction — the compiler treats it exactly like int *a. So inside f, the sizeof trick gives a meaningless answer (often 2 on a 64-bit machine: 8-byte pointer ÷ 4-byte int). The array's length simply did not travel with it. This is the single most surprising thing about C arrays, and now you know it's by design.
4 The fix

Pass the length yourself 🤝

Since the array can't carry its length, you carry it — as a second parameter. This is the universal C convention for array functions:

int sum(int a[], int n) {   // take the array AND its length
    int total = 0;
    for (int i = 0; i < n; i++)
        total += a[i];
    return total;
}

int main(void) {
    int a[5] = {10, 20, 30, 40, 50};
    int n = sizeof(a) / sizeof(a[0]);  // compute the count HERE (= 5)
    printf("%d\n", sum(a, n));        // pass both → 150
}
The pattern is always the same: compute the length in main (where the array is real, using the sizeof trick or a #define), then hand both the array and the length to the function. The function trusts the n you gave it. This is exactly why you'll see (int arr[], int n) everywhere in C.
5 Other ways to mark the end

Lengths, sentinels, and constants 🧩

  • Pass the length (most common): f(arr, n). Simple and explicit.
  • A named constant: #define N 5 (or const int N = 5;) used everywhere, so there's one number to change.
  • A sentinel value: mark the end with a special value. C strings do exactly this — a '\0' byte marks where the text stops, which is how strlen finds the length without being told. (That's lesson 5.4.)

🚨 Trusting sizeof inside a function

Writing for (i = 0; i < sizeof(a)/sizeof(a[0]); i++) inside a function that received a as a parameter is a real bug: a is a pointer there, so the loop runs the wrong number of times (often just 2). The sizeof-count trick is only valid in the scope where the array was actually declared. Pass the length instead.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

👻
Using the sizeof trick inside a function Once an array is a parameter, it's a pointer — sizeof(a)/sizeof(a[0]) there is wrong. Compute the count where the array is declared, and pass it in.
📏
Forgetting to pass the length A function that takes int a[] has no idea how long it is. Always give it the count too: f(int a[], int n). Without n it can only guess — and guessing reads out of bounds.
🔢
Passing the wrong length The n you pass must match the real array. Pass a bigger n than the array has and the function walks off the end — the same undefined behaviour as any out-of-bounds access.

Practice exercises 💪

Easy Print an array. Write void print_array(int a[], int n) that prints all n elements separated by spaces. Call it from main, computing n with the sizeof trick.
Medium Prove decay. In main, print sizeof(a) for an int a[5]. Then pass a to a function that prints sizeof(a) too. Watch the numbers differ — and explain why in a comment.
Stretch Max with length. Write int max_of(int a[], int n) that returns the largest of the first n elements. Test it, then try passing a wrong (too-large) n and notice how the function has no way to catch it — the discipline is entirely yours.

🎓 What to remember

  • A C array doesn't store its length — there's no .size() to ask.
  • The sizeof(a)/sizeof(a[0]) trick gives the element count, but only where a is a real array.
  • Arrays decay to pointers when passed to a function, so sizeof there measures the pointer, not the array.
  • The fix: pass the length as a second parameter — f(int a[], int n).
  • Other markers: a named constant, or a sentinel value (like the '\0' that ends C strings).

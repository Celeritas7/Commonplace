Debugging with gdb

When a program misbehaves, sprinkling printfs everywhere is slow guesswork. gdb (the GNU debugger) lets you pause your running program, step through it one line at a time, and inspect any variable's real value. It has hundreds of commands — but five of them cover almost everything you'll do. This is your practical starter kit.

1 Why this matters

Look inside, don't guess 🐛

Everyone debugs with printf at first — add a print, recompile, run, read, repeat. It works, but it's slow and you have to guess where to look before you even start. A debugger flips that around: it runs your program under supervision so you can stop it at any line, look at every variable's actual value, and step forward instruction by instruction watching things change. When a program crashes, it can even tell you the exact line and the chain of function calls that led there. gdb is the standard debugger for C on Linux (with lldb as the close cousin on macOS), and learning a handful of commands repays itself the first time you'd otherwise have spent an hour adding prints.

The one prerequisite: compile with debug info so gdb can map machine code back to your source.

2 Getting in

Compile with -g, then run gdb 🚪

gdb needs your program built with -g, which embeds the source line and variable info. Then you launch gdb on the program:

gcc -g -o prog main.c        # -g = include debug info
gdb ./prog                     # start gdb on your program
# now you're at the (gdb) prompt, program not yet running
Without -g, gdb still runs but can only show raw addresses and assembly — useless for everyday debugging. With it, you get your real variable names and line numbers. Starting gdb drops you at a (gdb) prompt where you type commands; your program isn't running yet — you typically set a breakpoint first, then start it. (The -g flag is exactly the one the CFLAGS in your Makefile from 10.3 already included.)
3 The five commands

Break, run, print, next, backtrace 🎯

This tiny set handles the vast majority of debugging sessions:

Command (short)What it does
break main / break 12 (b)pause when execution reaches a function or line
run (r)start the program (stops at the first breakpoint)
print x (p)show the current value of a variable or expression
next / step (n / s)run the next line — step goes into calls, next over them
backtrace (bt)show the call stack — who called whom to get here
A typical session reads like a sentence: break where you suspect trouble, run to get there, print the variables to see what's actually true, then next a few times watching them change until the wrong value appears. Add continue (c) to resume to the next breakpoint, and quit (q) to leave — but break/run/print/next/backtrace are the core. Most commands have one-letter shortcuts, and pressing Enter repeats the last command (great for stepping with n).
4 A real session

Cornering a bug 🔎

Here's the rhythm — break before the suspect spot, run, then print and step to watch the wrong value emerge:

(gdb) break 8          # pause at line 8
(gdb) run              # start; stops at line 8
(gdb) print sum        # $1 = 0   ... so far so good
(gdb) next             # run one line
(gdb) print sum        # $2 = 5   ... watch it change
(gdb) print i          # $3 = 3   ... aha, i is wrong!
You're not changing the program — you're observing it. Each print shows the variable's actual value at that exact point, which is how you catch the moment reality diverges from what you expected. If the program crashes, just run it under gdb and, when it stops, type backtrace — gdb points at the precise line of the crash and the call chain that led there, which alone solves a huge fraction of segfaults.
5 Crashes & backtrace

Where did it die? 💥

The single highest-value move: when a program segfaults, run it in gdb and read the backtrace. gdb stops at the crash and tells you exactly where:

(gdb) run
Program received signal SIGSEGV, Segmentation fault.
0x... in process (p=0x0) at main.c:14
14:    return p->value;        # crashed dereferencing p
(gdb) backtrace
#0  process (p=0x0) at main.c:14   # p is NULL here!
#1  main () at main.c:22           # called from line 22

🚨 Debugging an optimized build with no -g is misery

Two setup mistakes waste hours. First, forgetting -g: gdb can't show your variables or lines, only hex addresses — always build debug builds with -g. Second, debugging an optimized build (-O2): the compiler reorders and merges code, so the line gdb stops on and the variables it shows may not match your source ("value optimized out", lines jumping around). For debugging, compile with -g and without -O (or -Og, which optimizes lightly but keeps debugging sane). And note gdb shows you where and what, but not why — you still reason about the bug; gdb just hands you the facts instead of guesses.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🏷️
Forgetting -g Without debug info gdb shows hex addresses, not your variables or lines. Always build debug builds with -g (it's in your Makefile's CFLAGS).
⚙️
Debugging an optimized build -O2 reorders code so lines and variables don't match your source ("optimized out"). Debug with -g and no -O (or -Og).
🪜
Confusing step and next step dives into a called function; next runs it and stops on the line after. Use next for code you trust, step to investigate a call.

Practice exercises 💪

Easy First session. Compile a small program with -g, run gdb ./prog, set break main, run, then next a few times and print a variable at each step. Just get comfortable at the (gdb) prompt.
Medium Catch a wrong value. Write a loop that computes a sum with an off-by-one or wrong index. Break at the loop, step through with next, and print the accumulator and index each pass until you see where it goes wrong.
Stretch Backtrace a segfault. Deliberately dereference a NULL pointer inside a helper function called from main. Run it under gdb, and from the crash use backtrace to identify both the crashing line and the caller. Then print the pointer to confirm it's 0x0.

🎓 What to remember

  • A debugger beats printf — pause the program, inspect any variable, step line by line.
  • Compile with -g (and without -O) so gdb sees your source lines and variable names.
  • The five commands: break, run, print, next/step, backtrace — cover almost everything.
  • On a crash, backtrace shows the exact line and the call chain that led there — the fastest segfault finder.
  • next steps over calls, step goes into them; Enter repeats the last command, and most commands have one-letter shortcuts.

Reading a File Line by Line

Last lesson you met fgets. Now we build the single most common file task in all of C: walk through a file one line at a time, from the first line to the last. The pattern is tiny and you'll use it forever — but it hides three sharp edges: the leftover newline, lines longer than your buffer, and the infamous feof trap.

1 Why this matters

One line at a time 📄

Almost every program that reads text does it line by line: counting words in a file, parsing a config, scanning a log, loading a list of names. C has no built-in "read all lines" — you write a small loop yourself, and it's the same loop every time. Master it once and you can read any text file. The tool is fgets from 9.1, but using it correctly in a loop has a few wrinkles that trip up nearly every beginner. This lesson nails the canonical loop and the three things that bite.

It's short — but getting the loop condition right is what separates working code from a subtle bug.

2 The canonical loop

while (fgets(...)) 🔁

Here it is — the loop you'll write for the rest of your C life. Open, loop on fgets until it returns NULL, close:

FILE *f = fopen("notes.txt", "r");
if (!f) { perror("open"); return 1; }

char line[256];
while (fgets(line, sizeof line, f) != NULL) {   // reads one line each pass
    printf("%s", line);   // line still has its '\n'
}
fclose(f);
Each pass through the loop, fgets reads one line into line and returns the buffer (a non-NULL pointer). When there are no more lines, it returns NULL and the loop stops cleanly. Test the return value directly in the loop condition — while (fgets(...) != NULL) — because the return value is exactly the "did I get a line?" signal. Notice sizeof line as the size argument: it always matches the array, so you can never tell fgets the wrong size.
3 The leftover newline

fgets keeps the \n — trim it ✂️

fgets stores the line including the trailing \n. That surprises people: "Ada" in the file comes back as "Ada\n". When you compare or print it, that newline is still there. The clean one-liner to chop it off:

// after fgets(line, sizeof line, f):
line[strcspn(line, "\n")] = '\0';   // replace the first '\n' with end-of-string
strcspn(line, "\n") (from <string.h>) returns the index of the first \n — or the length of the string if there's none. Writing '\0' at that index turns the newline into a string terminator, neatly trimming it. If the line had no newline (the last line of a file sometimes doesn't), strcspn returns the length and you harmlessly overwrite the existing '\0'. One safe line, every case handled — much tidier than poking at line[strlen(line)-1], which misbehaves on empty input.
4 When the buffer is too small

A line longer than the buffer 📏

fgets(buf, size, f) reads at most size - 1 characters. If a line is longer than that, it does not overflow (that's the whole point of fgets) — it reads as much as fits and leaves the rest for the next call:

// file line is: "abcdefgh\n"   (8 chars + newline)
char buf[5];                 // holds 4 chars + '\0'
fgets(buf, sizeof buf, f);   // buf = "abcd"   (no '\n' yet!)
fgets(buf, sizeof buf, f);   // buf = "efgh"
fgets(buf, sizeof buf, f);   // buf = "\n"     (the rest of the line)
The tell for "this was a partial read" is that the buffer does not end in \n (and you haven't hit end of file). A correct line is one that ends in \n — or the very last line, which may end at EOF instead. This is why a comfortable buffer matters: pick something larger than any line you reasonably expect (256, 1024). If you must handle arbitrarily long lines, you loop reading chunks until you see the \n, growing storage as you go — but for everyday files, a generous fixed buffer is fine.
5 EOF vs. error

Don't loop on feof 🐛

It's tempting to write while (!feof(f)) — and it's wrong. feof only becomes true after a read has already failed at end of file, so the loop runs one extra time on stale data:

// BUG: prints the last line twice
while (!feof(f)) {
    fgets(line, sizeof line, f);   // fails at EOF, leaves line unchanged
    printf("%s", line);            // prints the OLD line again
}

🚨 while (!feof(f)) reads one time too many

End-of-file isn't known until a read tries and comes up empty. So feof is still false when you're sitting on the last line; you read it, print it, loop again, the next fgets fails (but you don't check!), and line still holds the previous line — printed a second time. The fix is the loop from section 2: test what fgets returns. Reserve feof / ferror for after the loop, to ask why it stopped: if (ferror(f)) perror("read"); tells you a real I/O error from a normal end of file.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🐛
Looping on while (!feof(f)) It runs one extra time and reprocesses the last line. Loop on what fgets returns instead, and check feof/ferror only after the loop.
✂️
Forgetting the trailing \n fgets keeps the newline, so "Ada\n" won't match "Ada". Trim it with line[strcspn(line, "\n")] = '\0'; before comparing.
📏
Assuming one fgets = one whole line If a line is longer than the buffer, you only get part of it; the rest comes next call. Use a generous buffer, or detect a missing \n to know the line was cut.

Practice exercises 💪

Easy Count the lines. Open a file with "r" and loop with fgets, incrementing a counter each pass. Print the total at the end — a one-screen wc -l.
Medium Number and clean. Read each line, trim the \n with strcspn, and print it back as 1: line, 2: line, … so the numbering sits flush against the text with no stray blank lines.
Stretch grep, lite. Take a search word from argv[1], read a file line by line, and print only the lines that contain it (use strstr). After the loop, call ferror and report if the read stopped on an error rather than EOF.

🎓 What to remember

  • The loop: while (fgets(line, sizeof line, f) != NULL) { ... } — stop when it returns NULL.
  • fgets keeps the \n — trim with line[strcspn(line, "\n")] = '\0'; when you don't want it.
  • It reads at most size - 1 chars — a too-long line comes back in pieces (no \n = partial).
  • Never loop on while (!feof(f)) — it reprocesses the last line. Test fgets's return value.
  • After the loop, use ferror(f) to tell a real I/O error from a normal end of file.

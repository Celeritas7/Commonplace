Members, Methods & this

A class bundles data members (state) with member functions — methods — that act on that state. Inside a method you use the members directly, and there's a hidden helper, this, that quietly points at the object the method was called on.

1 Why this matters

Behaviour lives with the data 🧠

In 6.1 you bundled data into a class and set it from outside with the dot. But the real power of a class is putting the behaviour inside too. A method (member function) lives in the class and works on that object's own data — so a BankAccount can deposit() into itself, a Counter can increment() itself. The data and the code that manages it travel together.

This lesson answers two questions: how does a method reach the object's data, and what is that mysterious this keyword you'll see in C++ code? Both turn out to be simple once you see them in action.

2 Members, up close

Data members and methods 📦

A class has two kinds of members. Data members hold the object's state; member functions (methods) are functions defined inside the class that act on it. Crucially, inside a method you refer to the data members by their bare names — no dot needed:

struct Counter {
    int count = 0;            // data member (state)

    void increment() {        // method (behaviour)
        count++;              // just 'count' — this object's count
    }
    int get() { return count; }
};
Inside increment(), writing count means "this object's count." You don't (and can't) write c.count there, because the method doesn't know it's called c — it just works on whichever object it was called on. The object is implicit.
3 Every call runs on an object

c.increment() works on c 🎯

You call a method on an object with the dot, and it operates on that object's data. Call it on a different object and it touches that one instead:

Counter a, b;
a.increment();  a.increment();   // a.count → 2
b.increment();                   // b.count → 1

std::cout << a.get() << " " << b.get();  // 2 1
Same method, different objects, independent results. The code of increment() exists once, but each call is bound to one object — a.increment() bumps a's count, b.increment() bumps b's. That binding is the heart of object-oriented code: the method always knows which object it's working on.
4 The hidden helper

this — a pointer to the object 👉

How does increment() know which object it's on? C++ secretly passes a pointer called this into every method — it points at the object the method was called on. So count is really shorthand for this->count:

void increment() {
    count++;          // what you write
    // this->count++;  // what it really means — 'this' points at the object
}
this is a pointer (remember pointers — it holds the object's address), so you reach members through it with the arrow ->. You rarely write this by hand, because the bare member name already does the job. But knowing it exists explains why a method magically operates on the right object.
5 When you actually write this

Untangling a name clash ✨

The common reason to spell out this is when a parameter has the same name as a data member. Then the bare name refers to the parameter, and this-> is how you reach the member:

struct Box {
    int width;
    void setWidth(int width) {     // parameter shadows the member
        this->width = width;       // member = parameter
    }
};

🚨 A parameter that shadows a member

Inside setWidth(int width), the bare name width means the parameter, not the member. Writing width = width; just assigns the parameter to itself and the member never changes — a silent bug. Use this->width = width; so the left side clearly means the member. (Renaming the parameter, e.g. w, avoids the clash too.)

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🪞
A parameter shadowing a member If a parameter shares a member's name, the bare name means the parameter. width = width; does nothing useful — use this->width = width; (or rename the parameter).
🏷️
Using c.count inside a method Inside the method you don't know the object's outside name. Use the member directly — count, not c.count. The object is implicit (it's this).
👯
Thinking one call changes every object A method runs only on the object you called it on. a.increment() changes a alone; b is untouched. Same code, separate data.

Practice exercises 💪

Easy A greeter. Give struct Person { std::string name; } a method void greet() { std::cout << "Hi, I'm " << name; }. Make a person, set the name, and call greet() — note it uses name directly.
Medium Two counters. Build the Counter with increment() and get(). Make two counters, increment them a different number of times, and print both totals to prove each keeps its own count.
Stretch The name clash. Write void setName(std::string name) { this->name = name; } on a struct with a name member. Confirm it sets the member. Then try it without this-> and watch the member stay unchanged — feel why this matters.

🎓 What to remember

  • A class has data members (state) and member functions/methods (behaviour) that act on it.
  • Inside a method, use member names directly — the object is implicit, no dot needed.
  • A method runs on one object: c.increment() changes c's data, nobody else's.
  • this is a pointer to that object; count is short for this->count.
  • Spell out this-> mainly to disambiguate a parameter that shares a member's name.

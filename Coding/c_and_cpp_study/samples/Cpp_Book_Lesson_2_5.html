Operators and Expressions

You know the operators. Now: how C++ decides what to compute first in a + b * c && d. Precedence, associativity, short-circuit, and the Precedence Explorer that walks any expression for you.

1 Why this matters

Expressions are how C++ thinks 💭

An expression is any piece of code that produces a value. 3 + 4 is an expression. x > 5 is an expression. a = b * c + d is an expression. In C++ you spend most of your time writing them, and the compiler reads them with very specific rules about what runs first.

Get the rules wrong and your code does something subtly different from what you meant — and it usually compiles fine. The classic example: a + b * c isn't (a + b) * c; it's a + (b * c). * binds tighter than +. This is operator precedence, and it's the topic of the lesson.

Important C++ difference: in C, the comparison and logical operators return int (1 or 0). In C++ they return a true bool (true or false). The conversions are usually invisible — but if you std::cout a comparison you'll see 1 by default. Add std::boolalpha to print true / false instead.
3 Precedence — who runs first

The pecking order 🥇

When several operators sit in one expression, C++ has a fixed table that decides which one binds tighter. You don't need to memorize all 17 levels — you need the handful that bite. Highest binds first:

RankOperatorsMnemonic
1()   []   .Calls, indexing, member access
2!   ++   --   (unary -)Unary ops — bind tightly to one operand
3*   /   %Multiplicative
4+   -Additive
5<   >   <=   >=Relational
6==   !=Equality (looser than relational!)
7&&Logical AND
8||Logical OR (looser than AND)
9?:Ternary
10=   +=   -= ...Assignment (lowest — runs last)

The three rules that catch beginners:

  • Arithmetic before comparison. a + 1 > b means (a + 1) > b, not a + (1 > b).
  • Comparison before logical. x > 0 && y > 0 means (x > 0) && (y > 0) — no extra parens needed.
  • AND binds tighter than OR. a || b && c means a || (b && c). This one surprises everyone — always parenthesize.
4 Associativity — left or right

What happens when ranks tie ⚖️

If two operators of the same rank meet, associativity decides who runs first. Most operators are left-associative — they group left to right:

ExpressionGroupingWhy
10 - 4 - 2(10 - 4) - 2 = 4- is left-associative
100 / 5 / 2(100 / 5) / 2 = 10/ is left-associative
a = b = 5a = (b = 5)= is right-associative (rare)
a ? b : c ? d : ea ? b : (c ? d : e)Ternary is right-associative

🚨 Equality has lower precedence than relational

You'd think a < b < c would mean "b is between a and c." It doesn't. < is left-associative, so it becomes (a < b) < c — first a < b gives a bool (true/false which is 1/0), then 1 < c or 0 < c. Almost certainly not what you wanted. Write a < b && b < c instead.

5 Short-circuit and side effects

C++ is lazy with logic ⚡

&& and || have a special property: they evaluate left to right and stop the moment the answer is known.

ExpressionWhat happens
false && expensive()expensive() is never called — left side already decides the result.
true || expensive()Same — expensive() is skipped.
ptr != nullptr && ptr->value > 0The dereference is safe — only happens if ptr is non-null.
n != 0 && total / n > 100The division is safe — only happens if n isn't zero.

That's why putting your safety check first matters. Reversing the order — ptr->value > 0 && ptr != nullptr — will crash, because the nullptr check happens after you've already tried to read through a null pointer.

Operators that DON'T short-circuit: the bitwise versions & and |. They always evaluate both sides. If you write ptr != nullptr & ptr->value > 0 (one ampersand!) — boom, segfault. Same trap for | vs ||.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔗
Chained comparisons that don't mean what they look like if (0 < x < 10) doesn't check that x is between 0 and 10. It evaluates as (0 < x) < 10, which is always true (a bool is 0 or 1, both less than 10). Write if (x > 0 && x < 10).
⚖️
Trusting && and || precedence without parens a || b && c means a || (b && c), not (a || b) && c. The reader (including future-you) will pause to figure it out. Add the parens — they cost nothing and signal intent.
💥
Single & or | instead of && / || The bitwise versions still compile, still produce a value the if accepts — but they don't short-circuit. If you wrote p != nullptr & p->ok(), the ->ok() call happens even when p is null. Crash.

Practice exercises 💪

Easy Add the parens. For each expression below, write it with explicit parentheses showing how C++ groups it: a + b * c - d, x > 0 || y > 0 && z > 0, a == b == c. Check your answers in the explorer.
Medium In-range check. Write a single bool expression that's true when x is between 10 and 20 inclusive. Then write a version that's true when x is outside that range. No if — just the expression.
Stretch Order matters. Suppose you have a int *p that may be nullptr. Write a single if that prints "ok" when p is non-null AND *p is between 1 and 100. Reverse the order of the two checks and explain why the new version crashes.

🎓 What to remember

  • Expression = anything that produces a value. Most C++ code is expressions, and precedence dictates the order.
  • Precedence shortcut: * / before + -; arithmetic before comparison; comparison before &&; && before ||; assignment last.
  • Associativity: most operators group left-to-right. = and ?: are right-associative.
  • Short-circuit: && and || stop early. Put safety checks first. Don't use the bitwise & or | by mistake.
  • When in doubt, parenthesize. The compiler doesn't care; the reader does.

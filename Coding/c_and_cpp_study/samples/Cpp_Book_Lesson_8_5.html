std::unique_ptr

Everything in this Part has been building to this: a ready-made RAII wrapper around a heap object. std::unique_ptr owns exactly one thing, hands you pointer-like access, and deletes it automatically when it goes out of scope — no leaks, no double frees, even when exceptions fly. It's the smart pointer you'll reach for by default.

1 Why this matters

RAII in a box 🧠

Raw new and delete are a constant source of bugs: forget the delete and you leak; delete twice and you crash; throw an exception between them and the cleanup is skipped. RAII (8.1) said "wrap the resource in an object whose destructor frees it." std::unique_ptr is that wrapper, already written for you and applied to heap memory. It holds a pointer, behaves like one, and runs delete in its destructor — so the object it owns is freed exactly once, automatically, on every exit path. This is the practical payoff of the whole Part.

The rule of thumb in modern C++: don't write delete yourself — let a smart pointer do it.

2 Creating & using

make_unique, then use it like a pointer ✨

Create one with std::make_unique<T>(args) (from <memory>). Then use -> and * exactly as you would a raw pointer:

#include <memory>

auto p = std::make_unique<Widget>("gear", 5);   // new Widget("gear", 5), owned by p
p->spin();              // use it like a pointer
int n = (*p).size();    // or dereference
// ...no delete needed: when p goes out of scope, the Widget is destroyed
make_unique<Widget>(...) allocates a Widget on the heap, forwards your arguments to its constructor, and returns a unique_ptr<Widget> that owns it. From then on p acts just like a Widget* — p->member, *p — but you never call delete. Prefer make_unique over unique_ptr<Widget>(new Widget(...)): it's shorter, and it's exception-safe (no raw new left dangling if something throws mid-expression).
3 Unique ownership

Can't copy — only move 🚚

The "unique" is the whole point: exactly one unique_ptr owns the object at a time. So you can't copy it (that would make two owners) — you can only move it, transferring ownership:

auto p = std::make_unique<Widget>();
auto q = p;              // ❌ COMPILE ERROR — copy is deleted
auto q = std::move(p);   // ✅ ownership moves p → q; p is now nullptr
This is move semantics (8.3) made concrete: unique_ptr is the canonical move-only type. Its copy constructor is = deleted, so an accidental copy is a compile error — the compiler protects you from creating two owners that would both delete the object. To hand ownership elsewhere, you std::move it, which leaves the source holding nullptr. Returning a unique_ptr from a function moves it out automatically (no std::move needed there).
4 The operations

get, reset, release 🔧

A few member functions cover the rest of what you need:

CallWhat it does
p.get()returns the raw pointer (non-owning — don't delete it)
p.reset()deletes the owned object now; p becomes nullptr
p.reset(q)deletes the old object, then owns raw pointer q
p.release()gives up ownership, returns the raw pointer (YOU must delete it)
if (p)true if p currently owns something
Use get() when a function just needs to look at the object (pass the raw pointer; ownership stays with the unique_ptr). reset() frees early or swaps in a new object. release() is the rare escape hatch — it hands you the raw pointer and steps back, making you responsible for the delete, so reach for it sparingly. To pass ownership into a function, take a unique_ptr by value and std::move into it; to just use the object, take a Widget* or Widget&.
5 Why prefer it

No delete, and Rule of Zero 🎯

A unique_ptr is (almost) free — the same size as a raw pointer, with no runtime overhead for the common case — but it buys you safety and clarity. And as a class member, it makes the Rule of Zero (8.4) just work:

class Engine {
    std::unique_ptr<Turbo> turbo;   // owns a Turbo; auto-deleted with the Engine
public:
    Engine() : turbo(std::make_unique<Turbo>()) {}
    // no destructor, no copy/move written — and it's correct & leak-free
};

🚨 Don't let a raw pointer outlive (or double-own) the object

Two pitfalls. First, never build two owning smart pointers from the same raw pointer — Widget* w = new Widget; unique_ptr<Widget> a(w); unique_ptr<Widget> b(w); gives two owners that both delete it → double free. (Using make_unique makes this almost impossible.) Second, a raw pointer from get() is non-owning and borrowed: if the unique_ptr is destroyed or reset, that raw pointer dangles — using it afterward is undefined behaviour. Treat get() results as short-lived observers, never as something to store or delete.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

👯
Two owners from one raw pointer Building two smart pointers from the same new'd object means both delete it — a double free. Use make_unique so there's never a loose raw pointer to double-own.
🪢
Trying to copy a unique_ptr Copy is deleted (that's the point). To transfer ownership use std::move; to just look at the object pass get() or a reference.
💀
Keeping a get() pointer too long The raw pointer from get() is borrowed and non-owning. After the unique_ptr resets or dies, it dangles — never store it or delete it.

Practice exercises 💪

Easy Auto-cleanup. Give a class a constructor that prints "made" and a destructor that prints "freed". Create one with make_unique inside a block { } and confirm "freed" prints when the block ends — with no delete.
Medium Hand it over. Make a unique_ptr, then move it into a second variable. Print whether the original is now nullptr (it should be) and use the new owner. Try writing auto q = p; and watch it fail to compile.
Stretch Owning member. Write a class that holds a std::unique_ptr<T> member and write none of the five special members (Rule of Zero). Show it's automatically non-copyable but movable, and that the T is freed when the owner dies — no destructor needed.

🎓 What to remember

  • std::unique_ptr<T> owns a heap object and deletes it automatically when it goes out of scope — RAII for memory.
  • Make one with std::make_unique<T>(args); use it like a pointer with -> and *. No manual delete.
  • Unique ownership: can't be copied (compile error), only std::moved — it's the canonical move-only type.
  • get() peeks, reset() frees now, release() hands off ownership (then it's your job to delete).
  • As a member it gives you the Rule of Zero — no leaks, exception-safe, and basically zero overhead.

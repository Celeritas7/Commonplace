🧰 PART IV · FUNCTIONS

Defining and Calling Functions

You've been calling functions since your first std::cout. Now you'll write your own: give a block of code a return type, a name, some parameters, and a body — then call it by name and watch control flow jump in and back out.

1 Why this matters

From calling to defining 🧠

Every program you've written lives inside one function already: int main(). And every std::cout, std::sort, or std::getline is a call to a function someone else defined. Functions are how C++ programs are built — small named pieces that call each other. This lesson is where you start writing your own pieces.

A function lets you take a chunk of work, give it a name and a clear set of inputs and outputs, and reuse it anywhere. The payoff is the same trio every language shares — less repetition, clearer code, easier testing — but C++ gives you precise control over exactly what goes in and what comes back. We'll start with the simplest, most common shape: a function that takes some values and returns one.

2 The anatomy

Four parts, one shape 🔧

Every function definition has the same four parts. Here's one, labelled:

int add(int a, int b) // return type · name · parameters
{                            // body starts
    return a + b;          // hand a value back to the caller
}                            // body ends
  • Return type (int) — what kind of value the function hands back. Use void if it hands back nothing.
  • Name (add) — what you'll call it by. Pick a clear, verb-like name.
  • Parameters (int a, int b) — named inputs, each with a type. The function works with these as ordinary variables.
  • Body (the { … }) — the statements that run when it's called. A return sends a value back and ends the function.
Read the first line as a sentence: "add takes two ints and gives back an int." That first line is called the function's signature — it's the contract. The { … } below is how the contract is fulfilled.
3 Calling it

Control jumps in, then comes back ↩️

You call a function by writing its name and passing values for its parameters. The values you pass are called arguments. When the call happens, your current code pauses, the function runs, and its returned value takes the place of the call:

int main() {
    int sum = add(2, 3);   // arguments 2 and 3 → a=2, b=3
    std::cout << sum;       // prints 5
}
Here's the key mental picture: at add(2, 3), main pauses. Control jumps into add, which runs with a = 2 and b = 3, hits return 5, and hands 5 back. Control then returns to exactly where it paused — the call add(2, 3) becomes the value 5, which gets stored in sum. The Call Flow stepper later in this lesson animates this jump-and-return.
4 When nothing comes back

void and the bare return 📭

Not every function produces a value. Some just do something — print a banner, save a file. Those return void, meaning "no value back":

void greet(std::string name) {
    std::cout << "Hello, " << name << "!\n";
}   // no return value needed

greet("Ada");   // called for its effect, not a value
A void function can still use return; on its own — with no value — to exit early. And a function that returns a value must hand one back on every path, or the compiler complains. One exception you already use: main is allowed to skip its return 0; — C++ adds it for you.
5 Where to put it

Define before you call 📍

C++ reads your file top to bottom. When it reaches a call, it must already know what that function is. The simplest rule for now: put your function above main.

int square(int n) { return n * n; }   // defined first…

int main() {
    std::cout << square(5);              // …so main can see it. ✅
}

🚨 Calling a function the compiler hasn't met yet

If square is defined below main, the compiler hits the call before it knows square exists and errors with something like 'square' was not declared in this scope. Two fixes: move the definition above main, or write a one-line declaration (a prototype) up top. Prototypes are exactly what lesson 4.3 is about — for now, just define above main.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

📍
Calling a function defined below main The compiler reads top-down, so define the function above main (or add a prototype). Otherwise: 'name' was not declared in this scope.
🚪
Forgetting to return a value A non-void function must return a value on every path. Leaving one path with no return is undefined behaviour — the compiler usually warns, and you should always fix it.
🔢
Mismatching arguments and parameters Pass arguments in the same order and types the parameters expect. add(2, 3) sets a = 2, b = 3 — swapping them silently swaps the meaning, and wrong types may convert in surprising ways.

Practice exercises 💪

Easy Write cube. Define int cube(int n) that returns n * n * n, then call it from main to print the cube of 3. Remember to define it above main.
Medium A void banner. Write void banner(std::string text) that prints the text surrounded by a line of = above and below. Call it twice with different strings — that's DRY in action.
Stretch Compose calls. Write int max2(int a, int b) returning the larger of two numbers. Then find the largest of three values by calling it twice: max2(max2(x, y), z). Notice how a returned value flows straight into the next call.

🎓 What to remember

  • A function definition has four parts: return type, name, parameters, body.
  • Call it by name, passing arguments; the call becomes the returned value.
  • Control jumps in and returns: the caller pauses, the function runs, then the caller resumes where it left off.
  • void means no value back; a bare return; exits early. Non-void must return on every path.
  • Define above main (or declare a prototype — that's lesson 4.3) so the compiler knows it first.

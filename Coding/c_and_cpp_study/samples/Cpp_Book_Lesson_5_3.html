std::map

A vector finds things by position. A map finds them by a key — a name, a word, an id — paired with a value. "What's Alice's score?" "How many times does this word appear?" That's a map: a dictionary of key → value.

1 Why this matters

Look it up by name 🧠

A vector answers "what's at position 3?" But often you want "what's the value for this key?" — a phone book (name → number), a tally (word → count), a settings table (option → value). A std::map stores key → value pairs and lets you look up a value instantly by its key. It's the C++ equivalent of a dictionary.

This is one of the most useful containers there is. Counting things, grouping things, remembering an answer for each input — all become a couple of lines with a map. We'll cover creating one, the look-up syntax (with its one famous surprise), iterating, and the choice between std::map and its faster cousin std::unordered_map.

2 Creating & inserting

Key in the brackets 🗝️

Include <map>. The template takes two types — the key type and the value type. You set a value by assigning to m[key]:

#include <map>

std::map<std::string, int> score;   // key: string, value: int
score["alice"] = 90;                  // add/update a pair
score["bob"]   = 85;

std::map<std::string, int> m = {       // or initialise up front
    {"x", 1}, {"y", 2}
};
The key goes in the square brackets instead of a position number. Assigning to m[key] either updates that key (if it exists) or inserts it (if it doesn't) — so each key appears at most once. The key can be almost any type (strings and ints are the common ones), and the value can be anything, including another container.
3 Looking up

Reading a value — and the surprise 👀

Read a value with the same m[key] syntax. But here's the famous catch: if the key isn't there, m[key] doesn't error — it silently creates the key with a default value (0 for an int) and returns that:

std::cout << score["alice"];   // 90 — it exists
std::cout << score["zoe"];     // 0  — and "zoe" is now IN the map!

To check whether a key exists without inserting it, use count (or find, or C++20's contains):

if (score.count("zoe"))            // 1 if present, 0 if not — no insert
    std::cout << score["zoe"];
That auto-insert behaviour is a feature for counting (m[word]++ just works on a brand-new word, starting it at 0 then bumping to 1) but a trap when you only meant to peek. Rule: to look up safely, check with count/find first; reading with m[key] may add the key.
4 Iterating

Walk the pairs (in sorted order) 🔁

A range-for over a map gives you each key → value pair. With C++17 structured bindings you can name the two halves cleanly:

for (const auto& [name, pts] : score) {   // C++17: name + value
    std::cout << name << ": " << pts << "\n";
}
// alice: 90
// bob: 85   — note: keys come out SORTED (alice before bob)
A std::map always keeps its keys in sorted order, so iterating walks them alphabetically (or numerically). That's handy when you want sorted output for free — and it's the key difference from unordered_map, which doesn't promise any order. The Word Counter below shows keys re-sorting as you add them.
5 map vs unordered_map

Sorted, or fast? ⚖️

OrderLookup speedUse when
std::mapsorted by keyO(log n)you need keys in order
std::unordered_mapno orderO(1) averageyou just need fast lookups
They have the same interface — m[key], count, range-for — so swapping one for the other is usually a one-word change (#include <unordered_map>). Default to unordered_map for speed, and reach for map when you need the keys sorted. The examples here use map so the output is tidy, but the counting pattern is identical either way.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

👻
Peeking with m[key] and accidentally inserting Reading a missing key with m[key] creates it (at the default value). To check existence without inserting, use m.count(key) or m.find(key) != m.end().
🔁
Expecting insertion order std::map iterates in sorted key order, not the order you inserted. If you need insertion order, a map isn't the tool (a vector of pairs is). unordered_map has no order at all.
🔑
Assuming one key can hold many values Each key maps to exactly one value; assigning m[k] again overwrites it. If you need multiple values per key, store a std::vector as the value (or use std::multimap).

Practice exercises 💪

Easy Phone book. Make a std::map<std::string, std::string> of name → phone number, add three people, and look one up by name. Then iterate and print all entries (they'll come out sorted).
Medium Letter frequency. Given a word, count how many times each letter appears using a std::map<char,int> and the m[c]++ trick. Print each letter with its count.
Stretch Safe lookup. Write a function that looks up a key and returns its value, or a default (say −1) if the key is absent — without inserting the key. Use find (or count) so a missing key never gets added.

🎓 What to remember

  • std::map<K, V> stores key → value pairs; look up by key with m[key] (needs #include <map>).
  • Assigning m[key] inserts or updates; each key appears once.
  • Reading a missing key with m[key] inserts it at a default value — check with count/find to peek safely.
  • Iterating gives key→value pairs in sorted key order (use structured bindings: for (auto& [k, v] : m)).
  • unordered_map is the faster, unordered cousin — same interface; prefer it unless you need sorted keys.

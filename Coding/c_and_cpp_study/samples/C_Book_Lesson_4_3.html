Function Prototypes

So far you've defined functions above main so the compiler meets them first. A prototype lifts that restriction: it's a one-line promise — "this function exists, here's its shape" — that lets you put definitions anywhere, even in other files.

1 Why this matters

The "define it above" rule, lifted 🧠

In lessons 4.1 and 4.2 there was one rule: define a function above main so the compiler has met it before the call. That works, but it forces an awkward order — all your helpers first, main last. A prototype is the fix: declare the function's shape up top, and the compiler is happy to see the call even though the real definition comes later.

The compiler reads your file strictly top to bottom. When it hits area(3, 4), it has to already know what area is — how many arguments, what types, what it returns. A prototype gives it exactly that information without the full body, so you can write main first and your helpers below, or even split functions across multiple files.

2 What a prototype is

The header line, then a semicolon 📣

A prototype is just the function's first line — return type, name, and parameter types — followed by a semicolon instead of a body:

int area(int w, int h);   // prototype — note the semicolon, no { }

Compare it to the full definition, which is the same first line plus the body:

int area(int w, int h) {   // definition
    return w * h;
}
The prototype is a declaration (it announces the function exists); the version with the body is the definition (it says what the function does). The parameter names in a prototype are optional — int area(int, int); is perfectly valid — because the compiler only needs the types to check your calls. Many people keep the names anyway as documentation.
3 Where things go

Prototype up top, definition below 🗂️

The usual layout: prototypes near the top (just after your #includes), main next, and the full definitions below it:

#include <stdio.h>

int area(int w, int h);    // 1. prototype — the promise

int main(void) {
    printf("%d\n", area(3, 4)); // 2. call — compiler already knows area ✓
    return 0;
}

int area(int w, int h) {    // 3. definition — the promise kept
    return w * h;
}
This is the pattern you'll see in almost every real C program, and it's why header files (.h) exist: a header is basically a collection of prototypes that lets one file call functions defined in another. You met this idea already — #include <stdio.h> pulls in the prototype for printf so your code can call it.
4 Why bother

Three real payoffs 🎯

  • Read top-down. main can sit at the top where a reader looks first, with the helper details below — instead of wading through every helper before reaching the point.
  • Multiple files. Big programs split functions across files. A prototype (usually in a shared .h) lets one file call a function whose body lives in another.
  • Functions that call each other. If a calls b and b calls a, no single ordering puts both definitions first. A prototype for one of them breaks the deadlock.
The prototype is also a contract the compiler enforces. Once it knows area takes two ints, calling area(3) or area("hi", 4) is caught as an error right at the call — instead of becoming a mysterious crash at run time.
5 Keep them in sync

One shape, two places 🔗

The prototype and the definition describe the same function, so their signatures must match — same return type, same parameter types, in the same order. If they disagree, the compiler complains about conflicting types.

🚨 Calling with no prototype (implicit declaration)

If you call a function the compiler hasn't seen — no prototype, and the definition is further down — older C assumes it returns int and accepts whatever you pass. That "implicit declaration" silently hides bugs (modern compilers warn or reject it). The fix is exactly this lesson: add a prototype up top, or move the definition above the call. Don't ignore an "implicit declaration of function" warning.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔚
Forgetting the semicolon (or adding a body) A prototype ends with ; and has no { }. Write int area(int, int); — not int area(int, int) (missing semicolon) and not a second full body.
🔀
Prototype and definition that don't match If the prototype says double f(int) but the definition is int f(int), you get a conflicting-types error. Keep the return type and parameter types identical in both places.
🤫
Ignoring "implicit declaration" warnings That warning means you called a function with no prototype in scope. It's not harmless — add the prototype (or move the definition up) rather than letting the compiler guess.

Practice exercises 💪

Easy Reorder with a prototype. Take your square program from 4.2 and rearrange it so main comes first and square's definition comes last. Add the prototype int square(int); at the top to make it compile.
Medium Three helpers. Write prototypes for int add(int, int), int sub(int, int), and int mul(int, int) at the top, call all three from main, and put their definitions below. Notice how readable the top of the file becomes.
Stretch Mutual calls. Write int is_even(int n) and int is_odd(int n) where each calls the other (is_even(n) returns is_odd(n-1), with base cases at 0). You'll need a prototype for one of them — there's no ordering that defines both first.

🎓 What to remember

  • A prototype is a declaration: the header line (return type, name, parameter types) + a semicolon, no body.
  • It lets you call a function before its definition — so main can go first and helpers below.
  • Parameter names are optional in a prototype; only the types matter to the compiler.
  • Prototype and definition must match in return type and parameter types.
  • Headers (.h) are collections of prototypes — that's how files share functions, and why #include exists.

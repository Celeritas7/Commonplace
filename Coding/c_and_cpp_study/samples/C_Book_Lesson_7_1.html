🧠 PART VII · MEMORY MANAGEMENT

Stack vs Heap

Your program's memory has two regions you'll actually think about: the stack, where local variables live automatically, and the heap, a big pool you request memory from and manage yourself. Knowing which is which explains lifetimes, array sizes, and why you'll soon reach for malloc.

1 Why this matters

Where do variables live? 🧠

Every variable your program uses sits somewhere in memory — and that "somewhere" falls into one of two regions you'll care about: the stack and the heap. Up to now everything has quietly lived on the stack. But the stack has limits: things on it vanish when their scope ends, and its size is fixed. When you need memory that outlives a function, or whose size you only learn at runtime, you'll turn to the heap. Understanding the two regions is the foundation for everything in this part.

This lesson is the map; the next ones (malloc, free, leaks) are the tools. Get the mental model of stack vs heap and the rest of memory management clicks into place.

2 The stack

Automatic and scope-bound 📚

The stack is where your local variables and function calls live. Every time a function runs, it gets a "frame" of stack space for its locals; when the function returns, that frame is automatically reclaimed. You don't manage it — the compiler does:

void f(void) {
    int x = 5;       // x lives on the stack
    int arr[10];     // so does this array
}                    // at this brace, x and arr are gone — automatically
The stack is fast (allocating is just bumping a pointer) and effortless (no cleanup to remember). But it has two catches: things on it die at the end of their scope, and the stack has a limited size. Every local variable you've ever written has lived here — it's the default home for data.
3 The heap

A pool you request and return 🗄️

The heap is a large pool of memory you can request chunks from while the program runs, using malloc (next lesson). What you get back lives until you hand it back with free — it is not tied to any scope:

int *p = malloc(100);   // ask the heap for 100 bytes (you'll meet malloc in 7.2)
// ... use the memory; it stays alive across function calls ...
free(p);                // YOU must give it back — the heap won't do it for you
The heap's superpower is control: memory you allocate there outlives the function that created it, and its size can be decided at runtime (you can ask for exactly as much as you need). The price is responsibility — every block you take, you must return. Forget, and you "leak" memory (a topic of its own in 7.4).

The headline contrast is automatic vs manual. The stack manages itself and is fast but rigid; the heap is flexible and long-lived but puts the cleanup on you. Most of memory management is really about using the heap responsibly — taking what you need and giving it back.
5 When to use which

Default to the stack 🎯

The rule of thumb is easy. Use the stack (ordinary local variables) by default — it's simpler and faster. Reach for the heap only when you have a real reason:

  • the memory must outlive the function that created it,
  • its size isn't known until runtime (e.g. it depends on user input), or
  • it's too big to sit comfortably on the limited stack.
If none of those apply, a plain local on the stack is the right choice — no malloc, no free, no chance to leak. When one of them does apply, the heap is exactly the tool, and the next lessons show you how to use it safely. Keep this checklist in mind and you'll rarely reach for the heap when you don't need to.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🧹
Thinking heap memory cleans itself up It doesn't. Every block you take from the heap with malloc, you must give back with free, or you leak it. Only the stack is automatic.
📦
Putting huge arrays on the stack The stack is limited. A giant local array (like int big[10000000];) can overflow it and crash. For large or runtime-sized data, use the heap.
👻
Returning a pointer to a stack variable The stack frame disappears when the function returns, leaving a dangling pointer (6.8). If the data must outlive the function, put it on the heap instead.

Practice exercises 💪

Easy Name the region. For each, say stack or heap: a local int, a function's parameter, a block from malloc, a fixed-size local array. Check yourself (stack, stack, heap, stack).
Medium Spot the address. Print the address of a normal local variable with %p. It's a stack address — note how it differs run-to-run, and that it lives only while the function runs.
Stretch Choose the tool. You're writing a function that builds a list whose length depends on user input and returns it to the caller. Which region must that list use, and why? (Heap — it must outlive the function and its size is runtime-decided. Next lesson: how.)

🎓 What to remember

  • Two regions matter: the stack and the heap.
  • The stack holds locals and function calls — automatic, fast, scope-bound, limited in size.
  • The heap is a pool you request from at runtime (malloc) and must return (free) — flexible, lives until you free it.
  • Default to the stack; use the heap when memory must outlive a function, has a runtime-decided size, or is too big for the stack.
  • Heap memory is manual — forget to free and you leak. That's where the next lessons go.

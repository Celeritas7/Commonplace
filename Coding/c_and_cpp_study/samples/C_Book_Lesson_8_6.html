union & Bit Fields

Two specialised layout tools to round out Part VIII. A union lets several members share the same memory — only one is valid at a time. Bit fields let a struct's members be measured in bits, packing flags and small values tightly. You'll meet both in systems code; here's just enough to recognize and use them.

1 Why this matters

Layout control, two ways 🧠

A struct gives every member its own storage, side by side. Sometimes you want the opposite — members that overlap because only one is ever in use at a time (a value that's either an int or a float). That's a union. Other times you want to pack many tiny values into as few bytes as possible (eight on/off flags in a single byte). That's bit fields. Both are specialised, low-level tools — you'll see them in device drivers, network protocols, and memory-tight code — and most everyday programs don't need them. This brief lesson closes Part VIII so you'll recognize them when they appear.

Specialised, but worth knowing. Then Part VIII is complete. 🎉

2 union

Members that share memory 🔲

A union looks just like a struct, but all its members occupy the same storage. Its size is that of its largest member, and only one member holds a valid value at any moment:

union Value {
    int    i;     // these three
    float  f;     // all live at the
    char   c;     // same address
};

union Value v;
v.i = 42;          // now i is valid
v.f = 3.14f;       // writing f OVERWRITES the shared bytes — i is now stale
// sizeof(union Value) == 4 (the size of the largest member)
Think of a union as a single chunk of memory with several labels for reading it as different types. Writing one member reuses the same bytes, so it clobbers the others. The payoff is space: when a value can only be one of several types at a time, a union stores it in the size of the biggest option instead of all of them added together.
3 The tagged union

Remember which member is active 🏷️

Since a union doesn't track which member is valid, the safe pattern is to pair it with an enum tag (from 8.3) inside a struct:

enum Kind { INT, FLOAT };

struct Number {
    enum Kind kind;        // which member is active?
    union { int i; float f; } u;
};

struct Number n = { INT, { .i = 42 } };
if (n.kind == INT) printf("%d\n", n.u.i);   // read the RIGHT member
The kind field records which union member currently holds the real value, so you always read the one you wrote. This tagged union (also called a variant) is how you safely store "one of several types" — a building block for interpreters, message formats, and JSON-like values. It combines three Part VIII tools: struct, enum, and union.
4 Bit fields

Members measured in bits 🔬

A struct member can be given an explicit width in bits with : N. Several pack into a single integer, saving space for flags and small ranges:

struct Flags {
    unsigned ready : 1;    // 1 bit  → 0 or 1
    unsigned mode  : 2;    // 2 bits → 0..3
    unsigned level : 3;    // 3 bits → 0..7
};
// all six bits pack into one unsigned, instead of 3 separate ints

struct Flags f = {1, 2, 5};
printf("%d %d %d\n", f.ready, f.mode, f.level);   // 1 2 5
Bit fields let you describe hardware registers and packed protocol headers directly: each field is exactly as wide as it needs to be. You read and write them like normal members (f.mode = 3), and the compiler handles the shifting and masking. They're a precise, readable alternative to doing bit operations by hand — but, like unions, they're a specialised tool.
5 Cautions

Know the sharp edges ⚠️

ToolKey factWatch out
unionsize = largest memberonly one member valid at a time
tagged unionenum tag + unionkeep the tag in sync
bit fieldwidth in bitslayout is implementation-defined

🚨 Don't read a union member you didn't write

Writing v.f then reading v.i reinterprets the same bytes as a different type — type punning. The result depends on the machine's representation and byte order (endianness), so it's not portable; always read the member you last wrote (that's what the tag is for). Likewise, the exact bit order and packing of bit fields is implementation-defined — handy for compact in-memory flags, but don't rely on their layout for a portable on-disk or on-the-wire binary format.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🔲
Treating a union like a struct A union's members overlap — only the last one you wrote is valid. Writing one clobbers the others; don't expect to store an int and a float at the same time.
🧭
Reading the wrong member (type punning) Reading a member you didn't write reinterprets the bytes — result depends on representation and endianness, so it's non-portable. Track the active member with a tag.
📐
Relying on bit-field layout The bit order and packing of bit fields are implementation-defined. Great for compact in-memory flags; don't use them for portable on-disk/on-wire binary formats.

Practice exercises 💪

Easy Measure a union. Define union U { char c; int i; double d; }; and print sizeof(union U). Confirm it equals sizeof(double), the largest member.
Medium Tagged number. Build the struct Number with an enum Kind tag and a union { int i; float f; }. Write a function that prints it using the right member based on the tag.
Stretch Pack some flags. Make a struct of bit fields for a tiny "status byte" (a few 1- and 2-bit fields). Set and print them, and compare sizeof with the same fields written as separate ints.

🎓 What to remember

  • A union overlaps its members in one chunk; its size is the largest member, and only one member is valid at a time.
  • A tagged union (enum tag + union in a struct) safely records which member is active.
  • Bit fields (unsigned x : N;) give members an exact bit width — compact flags and registers.
  • Type punning (reading a member you didn't write) is non-portable; bit-field layout is implementation-defined.
  • Specialised tools — handy in systems code, rarely needed in everyday programs. Part VIII complete! 🎉

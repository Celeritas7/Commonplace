if, else, and if-init

The if statement is your program's first fork in the road — run this code when something's true, that code otherwise. C++ adds a modern twist: the if with initializer (C++17), which lets you create a variable and test it in one tidy line.

1 Why this matters

Programs that choose 🧠

Every useful program reacts to its situation. "If the password matches, log them in." "If the file opened, read it." "If the score beats the record, celebrate." That word — if — is a real C++ keyword, and it's the foundation of every decision your code makes.

So far your programs ran straight down the page: statement, statement, statement. The if statement adds a branch — code that runs only when a condition holds. Add else for the alternative, chain else if for a series of choices, and your program can respond intelligently to anything. C++ keeps all of this familiar from C, then adds one genuinely useful upgrade — the if-initializer — that keeps short-lived variables from cluttering your scope. We'll build up to it.

2 The if statement

One condition, one block 🚦

The shape is a condition in parentheses, then a block to run when it's true:

if (temperature < 0) {
    std::cout << "Watch out for ice!\n";
}

C++ evaluates temperature < 0. A comparison produces a bool — true or false. If it's true, the block runs; if false, C++ skips it and continues below.

Conditions don't have to be comparisons. Any value that converts to bool works: if (count) means "if count isn't zero," if (ptr) means "if ptr isn't null," and if (vec.empty()) tests a real bool directly. Unlike C, C++ leans on the genuine bool type — but the everyday feel is the same: nonzero / non-null / true takes the branch.
3 else and else if

Adding the other paths 🛤️

else runs when the if was false; else if tests more conditions in order, taking the first true one:

if (score >= 90) {
    grade = 'A';
} else if (score >= 80) {
    grade = 'B';
} else if (score >= 70) {
    grade = 'C';
} else {
    grade = 'F';
}
The rule of an if / else if chain: C++ checks conditions top to bottom and stops at the first true one. Once a branch runs, the rest are skipped — even if they'd also be true. That's why order matters: score >= 90 must come before score >= 80, since a 95 satisfies both and you want the A. Exactly one branch runs; the else mops up everything left. 4 The if with initializer

A C++17 tidiness upgrade C++17

Often you create a variable just to test it in an if. The classic example is checking the result of a function before using it. In older C++ you'd write:

// the variable leaks into the surrounding scope
auto it = myMap.find(key);
if (it != myMap.end()) {
    use(it->second);
}
// 'it' is still here, cluttering the scope...

C++17's if with initializer lets you declare that variable inside the if, scoped to the if and its else only:

if (auto it = myMap.find(key); it != myMap.end()) {
    use(it->second);       // 'it' is in scope here
} else {
    std::cout << "not found\n";  // and here too
}
// 'it' is gone — out of scope, can't be misused
The shape is if (init; condition) — an initializer, a semicolon, then the condition, just like a for loop's header. The variable lives for the whole if / else and then disappears. This keeps short-lived helpers from leaking into the rest of your function, which means fewer name clashes and less chance of using a stale value later. 5 Braces and the equals trap

Two habits that save you 🛡️

Two pieces of advice carry over from C and are just as important in C++. First, always use braces, even for a one-line body — bare bodies hide bugs where a second line silently runs unconditionally, or a stray ; becomes an empty body.

🚨 = is assignment, == is comparison

if (x = 5) assigns 5 to x and is always true; you meant if (x == 5). Good news for C++: if your variable is a bool, many compilers warn loudly, and writing conditions against real bools (like if (found)) sidesteps the trap entirely. Still, compile with -Wall and stay alert.

The second habit is to prefer clear, bool-valued conditions. if (isReady) reads better and is safer than if (status != 0). C++'s real bool type and named predicates (empty(), has_value(), contains()) let your conditions say what they mean.

8 Mistakes, exercises, recap

Wrap up 📚

Three mistakes to avoid 🚨

🟰
Writing = instead of == if (x = 5) assigns and is always true. Use == to compare, prefer bool conditions like if (isReady), and compile with -Wall.
🧱
Dropping braces Without { }, only the next single statement is controlled by the if. A second indented line runs unconditionally. Always brace your bodies.
🔭
Using an if-init variable after the if The whole point of if (init; cond) is that the variable is scoped to the if/else. Reaching for it afterward is a compile error — if you need it later, declare it before the if instead.

Practice exercises 💪

Easy Pass or fail. Read an int score and print "Pass" if it's 50 or above, else "Fail". Use one if / else with braces and a clear condition.
Medium Grade ladder. Build the A/B/C/D/F grader as a real program and test it with 95, 85, 72, 65, 40 — match the Branch Tracer. Verify a 90 earns an A, not a B.
Stretch Use if-init. Given a std::string s, use if (auto pos = s.find('@'); pos != std::string::npos) to print where an @ appears, or "no @" otherwise. Then confirm that pos is not usable after the if — try it and read the compiler error.

🎓 What to remember

  • if (condition) { ... } runs its block only when the condition is true.
  • else / else if chains test top to bottom and stop at the first true branch.
  • if-init (C++17): if (init; cond) scopes a variable to the if/else and nowhere else — tidy and safe.
  • Always brace bodies, and compare with ==, never =.
  • Prefer clear bool conditions — if (isReady) over if (status != 0).
